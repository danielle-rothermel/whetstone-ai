GEPA Run Design

whetstone-ai · 2026-07-20
Vocab & Defs Concrete Changes Eval COPRO MIPROv2 GEPA Codex CLI agent Quick Test

GEPA is a multi-step Optimization Run. Each outer request executes exactly one durable, structured, tool-using GEPA step and returns one immutable Step Result; the next Step Request consumes prior result and state references. A step may select a parent, obtain fresh bounded Rollout Trace projections, reflect and propose a child, compare it on the same minibatch, fully evaluate an accepted child, and update optimizer state. Those are internal subphases, never separate outer request kinds. Only the completed run returns ordered Optimization Proposals for official evaluation.

Purple marks what differs from the Eval run — the optimization machinery; everything unmarked matches the eval design.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • GEPA step — one restartable optimizer invocation within the umbrella Optimization Run. It owns the complete select → diagnose → reflect → compare → accept → full-evaluate → update transition, even when an early terminal condition skips later subphases.
  • Step Request — the one immutable outer input shape for every GEPA step: run/step/config identities, deterministic seed or prior RNG state, immutable prior Step Result and state snapshot references, initial candidates when no prior state exists, Tool Configs, and remaining budgets.
  • Step Result — one immutable durable result for a GEPA step, sufficient to audit the transition and restart the next step without replaying successful effects.
  • state snapshot — immutable post-step references to the proposal catalog, per-instance/objective frontiers, optional active archive, RNG state, and budget counters. The next request consumes it by reference; no mutable in-memory object is the restart authority.
  • Whetstone multi-objective GEPA variant — the named whetstone_multi_objective/v1 experiment extension: same-minibatch strict Pareto improvement across correctness ↑ and compression ↓, followed by full 24-task evaluation, plus an optional capacity-32 active index. It is not claimed to be exact upstream GEPA.
  • proposal catalog — optimizer-owned, append-only index of every canonical, valid, unique proposal evaluated in the run, including proposals later rejected, dominated or evicted. Needed: final return fallback must not depend on active archive membership.
  • archive — optional capacity-32 active index in the Whetstone multi-objective GEPA variant. It caches proposal catalog, frontier, Tool Result, and Rollout references for bounded active-state operations; eviction removes only cache membership and never changes the durable catalog or frontiers.
  • archive policy — variant-only identity-bearing configuration fixing archive_capacity = 32 and deterministic eviction. It is an experiment extension, not reference GEPA acceptance or frontier semantics.
  • Whetstone experiment nondominated set — reporting view over all accepted, fully evaluated catalog entries under correctness ↑ and compression ↓. The legacy fragment ID is retained for stable links; the view is not limited by archive eviction.
  • parent / child — the candidate a mutation starts from and the candidate it produces. Needed: describes lineage during evolution.
  • reflection LM — the optimizer-owned LM that reads bounded Rollout Trace projections and writes a mutated encoder template. Needed: distinguishes it from the encoder and decoder LMs inside rollouts.
  • bounded Rollout Trace projection — a versioned, size-limited selection of preprocessing and code-test facts resolved from Rollout references; it records truncation and never replaces the source results. Rollout Trace is page-local shorthand for this projection, not a normative Trace or Result. Needed: it is GEPA's reflective evidence.
  • per-instance/objective frontier — durable mapping from each full-evaluation task and objective to its best accepted candidates; parents are sampled from its union. This is reference-shaped diversity state and stays distinct from the official plot's Pareto Front.
  • comparison minibatch — the exact sampled task IDs used both for the parent's fresh diagnostic evaluation and the child's acceptance evaluation in one GEPA step.
Reference semantics and the named extension. Reference GEPA accepts only a child that strictly improves on its parent on the same sampled minibatch, then fully evaluates the accepted child and updates per-instance/objective frontiers. This page preserves that ordering. The Whetstone multi-objective GEPA variant defines “strictly improves” as strict Pareto improvement on the same minibatch—no objective worsens and at least one improves—and uses the fixed 24-task, two-objective evaluation only after acceptance. Its capacity-32 archive is a bounded active index, not the semantic frontier or acceptance authority.

Example Setup

Mutation Surface & Initial Candidates

  • Mutation surface: {user_prompt_template} on the Encoder LLM Call Node — nothing else may change
  • Initial candidates: encoder Templates A and B (full text in the eval doc)
  • One GEPA run per Model Route — the same 4 routes as the eval experiment; each route resolves to a complete output-affecting Provider Call Config identity that contributes to Rollout Variant identity

This walkthrough follows (openrouter, deepseek/deepseek-v4-flash).

Toolkit

  • evaluate_minibatch — evaluates one candidate on caller-supplied stable task IDs of size minibatch_size = 3 × optimization_repeat_count = 1 at optimization_budget_ratio = 0.5; returns one Tool Result referenced by a terminal Tool Call Store Entry, with Rollout references, objective values, and a bounded Rollout Trace projection. Parent and child calls in one step MUST name the same task IDs.
  • evaluate_subset — fully evaluates an accepted child on objective_task_count = 24 tasks × optimization_repeat_count = 1 at optimization_budget_ratio = 0.5; returns one Tool Result referenced by a terminal Tool Call Store Entry, with Rollout references and per-instance/objective values used to update frontiers.

Each Tool Config references a versioned Tool Definition and supplies endpoint, an ordinary Eval Config to bind under the internal role, Reward Policy, capacity/timeout/policy refs, Tool Call Store namespace, idempotency and refusal rules. Every Step Request serializes the same pinned configs and policy identities; Runtime Tool Handles are reconstructed and never serialize. The Whetstone Tool Call Store, keyed by (tool_config_hash, call_id), owns acceptance/refusal/completion/capacity across retry and restart. A tool invocation is usable only through its terminal Tool Call Store Entry, Tool Result, and referenced Rollout Results.

Optimization Budget & Return

  • max_reflection_lm_calls = 8 across reflective mutation attempts
  • Hard budget: max_optimization_rollouts = 400
  • Run return: proposal_return_count = 4 ordered Optimization Proposals, or a failed Optimization Result with no official materialization; every nonterminal request still returns one durable Step Result

Guidance: "evolve the encoder user prompt template using fresh same-minibatch evidence." Guidance does not override acceptance_policy = same_minibatch_strict_pareto/v1.

Official Evaluation

  • Unchanged from the eval design: HumanEval+ official_task_count = 164 @ pinned revision · official_repeat_count = 3 · zstd-19 ÷ reference length
  • Variance Source: budget_ratio_values = (0.25, 0.5, 0.75, 1.0)
  • Selection: official_selection_count = 3 returned candidate groups

official_rollout_count = D × official_task_count × official_repeat_count, where D ≤ max_admitted_graph_count = 12; 5,904 only in the explicit nonconvergent example.

Repeated Durable GEPA Steps, Then Official Evaluation

Repeated durable GEPA steps · nested reflection and step loops with explicit exits
Step Result restart minimum. Every Step Result contains all of: run/step/config identities; deterministic seed and RNG state; parent and lineage; sampled tasks and evaluation/tool hashes; parent and child result references; bounded trace projection and truncation; reflection evidence, attempts, and cost; canonical child and diff validation; acceptance inputs, decision, and policy; accepted-child full-evaluation references; catalog/frontier/archive delta; remaining budgets; and continue|complete|failed. Absence of any applicable field is explicit and typed, never inferred from a missing event.

Detailed Walkthrough: Repeated Step Requests and Durable Step Results

Designer

Freezes gepa_variant = whetstone_multi_objective/v1, starting candidates A/B, mutation surface, deterministic run seed, acceptance_policy = same_minibatch_strict_pareto/v1, diagnostic/comparison and full-evaluation Tool Configs, variant archive policy, and budgets. Tool Configs reference versioned Tool Definitions and own Eval Config, Reward Policy, endpoint, Tool Call Store namespace, capacity, timeout, idempotency, and refusal configuration.

Evaluation harness

Issues step 1's Step Request under one Optimization Run identity. It contains no prior Step Result/state refs, so it carries A/B; later requests carry immutable prior refs instead. The harness reconstructs non-serializable Runtime Tool Handles and binds the pinned Tool Call Store namespaces.

One request → one GEPA step → one Step Result
Optimizer

Validates run/step/config identities, prior references, budgets, and deterministic continuation state. On first use only, it calls evaluate_subset for A and B, seeds the proposal catalog and per-instance/objective frontiers, and records two terminal Tool Results plus their Tool Call Store Entries with initial_candidate_count × objective_task_count = 2 × 24 = 48 Rollout refs. It then samples a parent from the frontier union using the persisted RNG state.

durable evidenceseed Tool Result/Rollout refs when applicable; pre-step state refs; sampled parent and lineage; pre/post selection RNG state
Optimizer via tool

Samples stable task IDs with minibatch_size = 3 and invokes evaluate_minibatch(parent, task_ids). The call obtains fresh parent behavior for this step; it does not substitute a cached full-evaluation row for same-minibatch comparison.

tool outputterminal Tool Call Store Entry and Tool Result ref/hash; exactly 3 parent Rollout Result refs; objective inputs; bounded Rollout Trace projection ref/hash, projection version, and truncation
Optimizer

The reflection LM reads the parent template and fresh bounded Rollout Trace projection, then proposes a child. Canonicalization, schema validation, uniqueness, and Diff Check enforce the template-only mutation surface. Invalid or duplicate output records provider evidence and retries only this internal reflection subphase with a deterministic (run_id, step_index, attempt_index) nonce, bounded by max_reflection_attempts_per_step = 3 and max_reflection_lm_calls = 8.

durable evidenceevery reflection request/response or failure reference, attempt nonce, validation outcome, normalized usage/cost, canonical child assignment and identity, and diff evidence
Optimizer via tool

For a valid unique child, invokes evaluate_minibatch(child, task_ids) with the exact parent diagnostic task IDs and matching comparison Eval Config. The Whetstone multi-objective GEPA variant accepts only strict Pareto improvement: neither minibatch objective worsens and at least one improves. Equality, a trade-off, or regression rejects the child for this step. This is the named multi-objective extension of reference GEPA's same-minibatch strict-improvement gate.

tool outputseparate terminal Tool Call Store Entry and Tool Result ref/hash; exactly 3 child Rollout Result refs; parent/child acceptance inputs, acceptance-policy hash, and deterministic decision
Optimizer via tool

If accepted, calls evaluate_subset(child) under the pinned Eval Config bound through a context carrying the internal role: objective_task_count = 24 × optimization_repeat_count = 1. Only this accepted-child full evaluation updates the per-instance/objective frontiers. Rejected children skip the call; their proposal, comparison evidence, and rejection remain durable.

tool outputfor acceptance, terminal Tool Result ref/hash + 24 full-evaluation Rollout Result refs and per-instance/objective values; otherwise an explicit not-applicable reason, never a fabricated empty result
Optimizer

Appends the canonical valid child and evidence to the proposal catalog, updates frontiers only when accepted and fully evaluated, and applies the variant active archive delta. At archive_capacity = 32, deterministic eviction removes cache membership only; it cannot erase catalog/frontier history or reverse acceptance. The step decrements Rollout and reflection-call budgets and freezes the next state snapshot.

state deltacatalog/frontier/archive before/after refs and typed delta; budget consumed/remaining; post-step RNG state
Optimizer

Returns exactly one immutable Step Result with the complete restart minimum and status continue, complete, or failed. Invalid/refused/rejected algorithm outcomes are recorded transitions; failed is reserved for a causal inability to produce the durable step result or a terminal run condition that cannot satisfy its contract.

outputone Step Result Object Reference/content hash and one post-step state-snapshot ref; no internal subphase response escapes as another outer result
Evaluation harness

On continue, verifies the committed references and issues the next request with the same run/config identities, incremented step identity, and immutable prior Step Result/state refs. This is the outer loop back-edge shown above. It never converts internal diagnostic, reflection, comparison, full-evaluation, or update subphases into request kinds.

Optimizer

In the all-valid example, optimization_step_count = max_reflection_lm_calls = 8. It derives the upper plan planned_optimization_rollout_count = initial_candidate_count × objective_task_count + optimization_step_count × (minibatch_size + minibatch_size + objective_task_count) = 2 × 24 + 8 × (3 + 3 + 24) = 288, below max_optimization_rollouts = 400. Rejected children omit the 24-task full evaluation; invalid/duplicate reflection retries add no Rollouts and may consume the global call cap in fewer than eight steps.

Optimizer

The terminal complete Step Result returns proposal_return_count = 4 unique accepted, fully evaluated proposals ordered from the durable proposal catalog and frontier state under the frozen variant return policy. The harness persists the umbrella Optimization Result with every Step Result reference. If the contract cannot be met, the run terminates failed and official materialization does not begin.

Evaluation harness

Validates all four proposals and expands materialization_count = proposal_return_count × len(budget_ratio_values) = 4 × 4 = 16 Graph Config/Materialization Record pairs. The Selection Policy selects official_selection_count = 3 groups as 12 ordered slots, then deduplicates by full Graph Hash to D ≤ max_admitted_graph_count = 12. Official records preserve every selected record → Graph Hash → result/aggregate mapping.

Evaluator

Submits exactly official_rollout_count = D × official_task_count × official_repeat_count Rollout Work Items with no duplicate Rollout Execution Keys. At D = 12, official_rollout_count = 12 × 164 × 3 = 5,904. Each immutable Rollout Result retains its nested Graph Run Result and provider/operational evidence; atomic binding remains authoritative.

outputauthoritative bound Rollout Result refs or explicit FAILED stages; no internal Tool Result is promoted to official evidence
Evaluation harness

Aggregates complete ordinary Rollout Results referenced and certified by the Official Evaluation Record into Average Binary Test Pass Rate and Mean Compression Ratio. In the explicit nonconvergent case, the three selected evolved proposals produce 12 Rollout Aggregates across four ratios; eligible A/B baselines may be reused only with matching Rollout Keys, official attestation, certification, and environment compatibility.

outputexactly D selected distinct-hash aggregates plus eligible baselines; official analysis never reads internal acceptance as truth

What lands on the plot: five curves per Model Route, and only official results

Per Model Route, eligible A/B baselines and three evolved templates retain separately attributable ratio slots, while converged slots share the planned/result-key set and Rollout Aggregate for their Graph Hash. The resulting distinct-hash aggregates are each averaged over 164 tasks × 3 repeats; in the explicit nonconvergent example they form five candidate curves with four aggregates each. The natural readings are whether evolution beat the starting templates and how the official Pareto Front moves. Internal Tool Results never appear here; their Evaluation Contexts lack official authority.

Additional Notes

Frontiers, cache, and official front

  • The durable per-instance/objective frontiers update only after same-minibatch acceptance and accepted-child full evaluation; their union steers parent sampling
  • The capacity-32 archive is a Whetstone variant cache. Eviction cannot delete frontier/catalog truth or change acceptance
  • The Whetstone experiment nondominated set is an internal reporting/return view; the plot's Pareto Front is recomputed from official Rollout Aggregates

Internal Evaluation vs Official Evaluation

  • Tool and official measurement policies have independent eval_config_hash identities. Their identical Evaluation Procedure Config is included in graph_hash; only Sampling/Aggregation remain outside it
  • The authorized Evaluation Context carries the official role, authority, and attestation; only the Official Evaluation Record certifies referenced ordinary Results as official, and reuse also requires a compatible environment
  • The 3-task comparison minibatch and 24-task accepted-child full evaluation are separate Eval Config sampling scopes bound under the internal role; whether the 24 tasks overlap the official 164 is left to experimental designers

Durable optimizer state crosses step boundaries by reference

  • Every Step Result references an immutable post-step state snapshot; the next request never relies on surviving process memory
  • The append-only proposal catalog and frontiers retain durable proposal/evaluation history; the variant archive only bounds active cache membership
  • All three resolve Tool Results through terminal Tool Call Store Entries; Tool Results and referenced Rollout Results remain immutable in dr-store
  • The Result Store separately owns only the authoritative Rollout Execution Key → Rollout Result Object Reference binding

Candidate = one plotted line

  • A child inherits everything outside the mutation surface from its parent
  • Each proposal emits 4 Graph Config/Materialization Record pairs — the fixed ratio axis unrolled into its curve
  • Ordered multi-return: 4 candidates back, the frozen selection policy decides how many go on

Versus reference GEPA

  • Reference-shaped: reflective mutation from fresh behavior; parent and child comparison on the same minibatch; strict improvement before accepted-child full evaluation; per-instance/objective frontier updates and diverse parent sampling
  • Named extension: whetstone_multi_objective/v1 interprets strict improvement as correctness/compression strict Pareto improvement and adds a non-authoritative capacity-32 active cache
  • Whetstone integration: repeated durable GEPA steps, immutable Step Results/state refs, bounded Rollout Trace projections, and concrete Tool Configs/Runtime Tool Handles/Tool Call Store Entries
  • dspy's GEPA is reference-only, never a dependency
