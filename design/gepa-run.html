GEPA uses a reflection LM to evolve encoder templates from rollout traces under internal tool evaluation; official evaluation remains authoritative.

Purple marks what differs from the Eval run — the optimization machinery; everything unmarked matches the eval design.

Temporary vocabulary — terms this doc needs beyond the vocab page; flagged here for review, not added there.
  • archive — the optimizer's in-memory table of per-task scores for every candidate it has evaluated through its tools. Needed: GEPA's parent selection and return ranking read it.
  • parent / child — the candidate a mutation starts from and the candidate it produces. Needed: describes lineage during evolution.
  • reflection LM — the optimizer-owned LM that reads rollout traces and writes a mutated encoder template. Needed: distinguishes it from the encoder and decoder LMs inside rollouts.
  • rollout trace — the node outputs and failure detail a tool attaches to a rollout result: encoded_code_rep, code_generation, failing-test output, preprocess failure cause. Needed: it is GEPA's reflective evidence.
  • per-task frontier — the candidates whose archive row is best on at least one task; parents are sampled from it. Needed: must stay distinct from the plot's pareto front.
  • minibatch — the 3-task sample a tool evaluates one candidate on per iteration. Needed: names the reflective evidence's scope.

Example Setup

Mutation Surface & Initial Candidates

  • Mutation surface: {user_prompt_template} on the Encoder LLM Call Node — nothing else may change
  • Initial candidates: encoder Templates A and B (full text in the eval doc)
  • One GEPA run per (provider, model) pair — the same 4 pairs as the eval experiment

This walkthrough follows (openrouter, deepseek/deepseek-v4-flash).

Toolkit

  • evaluate_minibatch — 3 tasks × 1 seed at budget ratio 0.5; returns rollout results with rollout traces
  • evaluate_subset — 24 tasks × 1 seed at budget ratio 0.5; scores only

Each tool swaps in its own cheaper Internal Eval Config: same preprocess and metrics extraction as the Official Eval Config, smaller task set, fewer seeds. Minibatches are sampled per iteration from the 24-task subset.

Optimization Budget & Return

  • 8 iterations of reflective mutation
  • Caps: 400 optimization rollouts · 8 reflection LM calls
  • Return: ordered list, at most 4 candidates

Guidance: "evolve the encoder user prompt template toward the correctness–compression front."

Official Evaluation

  • Unchanged from the eval design: HumanEval+ (164) @ pinned revision · 3 seeds · zstd-19 ÷ reference length
  • Variance source: budget ratios 0.25 / 0.5 / 0.75 / 1.0
  • Selection: officially evaluate the top 3 returned candidates

3 × 4 ratios × 164 × 3 = 5,904 rollouts per run.

Full run walkthrough: seed the archive, evolve eight times, return the front

Designer

Freezes the experiment: initial candidates A and B expanded across the 4 budget ratios into 8 initial rollout configs, each carrying an Internal Eval Config whose hash matches the Official Eval Config; mutation surface = {user_prompt_template}; the two tool definitions, each carrying its cheaper Internal Eval Config; the budget caps; the selection policy (officially evaluate top 3); and the Official Eval Config — identical to the eval run's. Everything serialized, everything hashed.

Evaluation harness

Issues the OptimizationRequest to the GEPA optimizer: 8 initial rollout configs, the mutation surface, handles for evaluate_minibatch and evaluate_subset, the guidance string, and the budget caps.

Optimizer

Seeds the archive: evaluate_subset on Templates A and B at ratio 0.5 — 48 rollouts. The archive now holds two rows of 24 per-task scores.

output48 internal rollout results under the subset tool's Internal Eval Config — optimization data from tool execution
Evolution loop · iterations 1–8
Optimizer

Computes the per-task frontier over the archive — a candidate survives if its row is best on at least one task — and stochastically samples a parent P from it. Diversity pressure, not greedy best. Zero rollouts spent.

Evaluation harness

Serves evaluate_minibatch(P): 3 tasks sampled from the subset, 1 seed, ratio 0.5.

output3 rollout results with rollout traces: encoded_code_rep, code_generation, failing-test output, preprocess failure causes
Optimizer

The reflection LM reads P's template plus the traces and writes a new template. Iteration 2, concretely: the traces show descriptions dropping edge-case behavior the tests exercise, so the new template adds "state the boundary conditions the tests exercise." The child candidate's rollout configs are assembled: the parent's, with only {user_prompt_template} changed.

outputone reflection LM call counted against the reflection cap, separate from rollout spend
Evaluation harness

Serves evaluate_minibatch(child) on the same 3 task ids. Acceptance is strict: the child's summed Eval Node score must exceed the parent's. Iteration 3 fails here — the child compressed harder but regressed correctness on 2 of 3 tasks, and is dropped; its rollout results remain optimization data.

output3 rollout results; acceptance verdict either way
Optimizer

On acceptance (iterations 1, 2, 4, 6, 8): evaluate_subset(child) — 24 rollouts — adds the child's row to the archive; the per-task frontier recomputes on the next iteration.

outputarchive grows to 7 rows: A, B, and five evolved candidates
Optimizer

Aggregates each archive row and computes the archive's correctness×compression front — 4 of the 7 candidates. Returns the OptimizationResult: an ordered list of 4 candidates as 16 rollout configs (each candidate = its 4 budget-ratio configs), plus advisory annotations — each candidate's parent and the reflection excerpt that produced it. Advisory provenance, never a score. Spend: ≈216 of 400 rollouts, 8 of 8 reflection calls.

Evaluation harness

Diff check: all 16 returned rollout configs differ from the initial ones only in {user_prompt_template} — consistent with the mutation surface; 16/16 pass. Selection applies the frozen policy and admits the top 3 candidates — 12 rollout configs — to official evaluation.

Evaluator

Submits 5,904 jobs — one per (rollout config, task, seed) — as a dr-platform campaign with labels {template, ratio}. Whetstone's failure taxonomy retries transient errors; permanent failures land as rollout results with a cause, not gaps.

output5,904 official rollout results keyed (rollout config hash, official eval config hash, task, seed)
Evaluation harness

Aggregates from official rollout results only: per rollout config, the Average Binary Test Pass Rate and mean measured compression — 12 rollout aggregates. Templates A and B's 8 rollout aggregates carry over from the eval run unchanged, because their rollout config and official eval config hashes are identical.

output20 rollout aggregates for this (provider, model) pair — five candidate curves of 4

What lands on the plot: five curves per pair, and only official results

Per (provider, model) pair: five candidate curves — Templates A and B (reused from the eval run) and the three evolved templates — each a distribution of 4 rollout aggregates across the budget ratios, each aggregate averaged over 164 tasks × 3 seeds. The natural readings: whether evolution beat the initial templates at matched ratio, and how the pareto front moves once the evolved candidates join. Nothing measured during optimization appears here because those results came from internal tool execution; their cheaper Internal Eval Configs also give them distinct identities.

Additional Notes

Three fronts, only one is a result

  • The per-task frontier steers parent sampling — diversity machinery
  • The archive's correctness×compression front ranks the return — selection machinery
  • The plot's pareto front is computed from official rollout aggregates — the only front that is a result

Internal Evaluation vs Official Evaluation

  • A tool's cheaper Internal Eval Config sits inside rollout_config_hash, keeping measurement identities disjoint from returned configs whose Internal Eval Config hash matches the Official Eval Config
  • Officialness comes from execution in the official harness context; the tool uses the same preprocess and metrics extraction, but a smaller task set, fewer seeds, and one ratio
  • Whether the 24-task subset may overlap the official 164 is left to experimental designers

The archive is optimizer state

  • In-memory, assembled entirely from tool returns; the harness owns none of it
  • Persistence enters only at the final results & analysis stage, and is deliberately not detailed here

Candidate = one plotted line

  • A child inherits everything outside the mutation surface from its parent
  • Each candidate spans 4 rollout configs — the variance source unrolled across the budget ratios
  • Ordered multi-return: 4 candidates back, the frozen selection policy decides how many go on

Versus reference GEPA

  • Kept, because it works: reflective mutation from real failures, strict-improvement acceptance on the same minibatch, per-task frontier parent sampling
  • Ours: rollout traces come from rollout results, not framework trace capture; internal evaluation lives in tool definitions as cheaper Internal Eval Configs — guidance, not enforcement
  • dspy's GEPA is reference-only, never a dependency
