GEPA evolves against the ledger: the archive lives environment-side, reflection is the mutator

whetstone-ai · system design walkthrough 4 of 5 · 2026-07-19
Vocab & Defs Eval COPRO MIPROv2 GEPA Codex CLI agent
GEPA is the richest control plane in the family: an evolutionary loop whose mutation operator is a reflection LM reading real failure evidence. The key structural move here is that GEPA's candidate archive — the per-task score matrix its pareto frontier is computed from — is not internal state. It is a ledger view, read back through the environment each iteration. That one move buys crash-resume for free, lets concurrent optimizer runs share memoized evaluations, and makes the frontier auditable after the run. And because the archive's frontier is a candidate set spanning trade-offs, GEPA is where multi-return earns its keep: the run returns a front, not a point.

This configuration: one seed prompt, four evolution iterations, a frontier returned

One seed treatment (baseline encoder prompt); mutation surface = the encoder prompt parameter only. The toolkit holds two tools: E_mini (3-task minibatch whose records carry full feedback material) and E_val (25-task validation sample). The loop runs four iterations of reflective mutation with strict-improvement acceptance, then returns the two-objective frontier from the archive as an ordered list of at most 4 candidates. The experiment certifies the top 3.

Two adaptations of reference GEPA, both flagged as recommendations: the reflective dataset comes from ledger records (sandbox test failures, preprocessing failure causes, code-leakage flags) rather than framework trace capture; and selection is a hybrid — per-task frontier for parent diversity, correctness×compression front for what gets returned.

Full run walkthrough: seed, evolve four times, return the front

Designer

Freezes the experiment config: seed spec T0, mutation surface = encoder prompt, toolkit {Emini, Eval}, requested return "the frontier, max 4", selection = certify top 3, Ccert. Everything serialized and digested.

ledgerexperiment config registered under its digest; T0, Emini, Eval, Ccert digests recorded
Environment

Issues the OptimizationRequest: seed, mutation surface, tool handles, guidance ("evolve toward the correctness–compression front").

Optimizer

Seeds the archive: calls Eval(T0). The archive now has one row of 25 per-task scores — held in the ledger, not in GEPA.

ledger25 records (T0, Eval): pass/fail, compressed bytes, costs
Evolution loop · iterations 1–4
Optimizer

read_archive query → per-task score matrix for all treatments so far. Computes the per-task pareto frontier (a candidate survives if it is best on at least one task) and stochastically samples a parent P from it — diversity pressure, not greedy best.

ledgerarchive query logged; zero eval cost
Environment

Emini(P) on 3 sampled tasks. Records return with full feedback material: which tests failed and how, preprocessing Absent causes, whether the description leaked literal code.

ledger3 records (P, Emini) incl. failure text and leakage flags
Optimizer

The reflection LM reads P's current encoder prompt plus the feedback-annotated examples and writes a new encoder prompt (e.g., iteration 2: "descriptions dropped edge-case behavior the tests exercise — state boundary conditions explicitly"). The child spec C is assembled: identical to P except inside the mutation surface.

ledgerreflection-LM tokens tracked as control-plane cost, separate from task-LM cost
Environment

Emini(C) on the same 3 task ids. Acceptance is strict: sum of child scores must exceed the parent's. Iteration 3 fails here — the child compressed harder but regressed correctness on 2 of 3 tasks, and is dropped. Its records stay in the ledger: a rejected child is still a memoized, auditable data point, and a later run that proposes the same prompt gets the answer for free.

ledger3 records (C, Emini); acceptance verdict recorded either way
Environment

On acceptance (iterations 1, 2, 4): Eval(C) promotes the child to the archive — 25 per-task records — and the frontier recomputes on the next read_archive.

ledger25 records (C, Eval); archive grows to 2, 3, then 4 rows
Optimizer

Final read_archive: aggregates Eval scores per treatment and computes the correctness×compression front — this, not the per-task frontier, ranks the return. Returns OptimizationResult: ordered list of 4 specs spanning the trade-off (best-correctness first) plus advisory annotations — each candidate's parent and the feedback excerpt that drove its mutation.

ledgerresult recorded; annotations land in the provenance store, never as scores
Environment

Diff validation: all 4 specs differ from the seed lineage only in the encoder prompt parameter — consistent with the declared mutation surface. Selection policy certifies the top 3.

Certifier

3 treatments × 40 tasks under Ccert. Compression computed on every description; correctness from the sandbox; failures classified and retried per the whetstone taxonomy.

ledger120 records tagged (Ti, Ccert) — the only records the plot may use
Environment

Aggregates certified records; the plot gains 3 points and the front updates. The run report shows total spend split by kind: task-LM (tool + certification evals) vs reflection-LM (control plane).

ledger3 certified aggregates; run cost totals: ~206 task-LM executions, 4 reflection calls

What lands on the plot: a certified slice of the evolved front

GEPA's returned set is not one prompt but a front — in this run, one candidate holding the correctness ceiling, one trading a few points of pass rate for markedly shorter descriptions, one in between. All three certified points carry Ccert's digest; every Emini/Eval measurement made during evolution coexists in the same ledger and is excluded from the plot by construction. The rejected iteration-3 child is visible in the ledger too — the run's full search history is reconstructible from records alone, with no GEPA-internal state to recover.

Versus dspy's GEPA: no two-package engine/adapter split — the environment is the engine's world; no framework trace capture — ledger records carry the feedback; the archive is externalized to the ledger instead of living in optimizer memory; and the two-objective front becomes the native selection structure for what the run returns, rather than a per-task proxy. What stayed, because it works: reflective mutation from real failures, strict-improvement acceptance on the same minibatch, and frontier-based parent sampling for diversity.
