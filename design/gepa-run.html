GEPA uses a reflection LM to evolve encoder templates from bounded Rollout Trace projections of referenced Rollout Results; on success it returns four ordered Optimization Proposals, while official evaluation remains authoritative.

Purple marks what differs from the Eval run — the optimization machinery; everything unmarked matches the eval design.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • proposal catalog — optimizer-owned, append-only index of every canonical, valid, unique proposal evaluated in the run, including proposals later rejected, dominated or evicted. Needed: final return fallback must not depend on active archive membership.
  • archive — the capacity-32 active index over proposal catalog entries and their persisted Tool Result/Rollout references, with derived per-task Objective vectors. Needed: GEPA's parent selection and archive nondominated set return ranking read it; eviction removes only active membership.
  • archive policy — identity-bearing optimizer configuration fixing archive archive_capacity = 32, dominance/equality/incomparability handling, nondominated preservation, deterministic eviction order, and return ordering; the Optimization Request carries its immutable reference and hash.
  • archive nondominated set — active archive entries not dominated on the fixed correctness-maximizing and compression-minimizing objective population; it supplies the archive nondominated set return ranking.
  • parent / child — the candidate a mutation starts from and the candidate it produces. Needed: describes lineage during evolution.
  • reflection LM — the optimizer-owned LM that reads bounded Rollout Trace projections and writes a mutated encoder template. Needed: distinguishes it from the encoder and decoder LMs inside rollouts.
  • bounded Rollout Trace projection — a versioned, size-limited selection of preprocessing and code-test facts resolved from Rollout references; it records truncation and never replaces the source results. Rollout Trace is page-local shorthand for this projection, not a normative Trace or Result. Needed: it is GEPA's reflective evidence.
  • per-task frontier — the candidates whose archive row is best on at least one task; parents are sampled from it. Needed: must stay distinct from the plot's Pareto Front.
  • minibatch — the 3-task sample a tool evaluates one candidate on per iteration. Needed: names the reflective evidence's scope.

Example Setup

Mutation Surface & Initial Candidates

  • Mutation surface: {user_prompt_template} on the Encoder LLM Call Node — nothing else may change
  • Initial candidates: encoder Templates A and B (full text in the eval doc)
  • One GEPA run per Model Route — the same 4 routes as the eval experiment; each route resolves to a complete output-affecting Provider Call Config identity that contributes to Rollout Variant identity

This walkthrough follows (openrouter, deepseek/deepseek-v4-flash).

Toolkit

  • evaluate_minibatch — minibatch_size = 3 tasks × optimization_repeat_count = 1 at optimization_budget_ratio = 0.5; returns Rollout references plus a bounded Rollout Trace projection
  • evaluate_subset — objective_task_count = 24 tasks × optimization_repeat_count = 1 at optimization_budget_ratio = 0.5; returns Rollout references and Objective vectors

Each Tool Spec references a versioned Tool Definition and supplies endpoint, internal Eval Config, Reward Policy, capacity/timeout/policy refs, ledger namespace, idempotency and refusal rules. The Optimization Request serializes those Tool Specs plus an identity-bearing archive_policy_ref/hash whose capacity is 32; Runtime Tool Handles do not serialize. Whetstone Tool Call Ledgers keyed by tool_spec_hash + call_id own acceptance/refusal/completion/capacity across retry and restart.

Optimization Budget & Return

  • max_reflection_lm_calls = 8 across reflective mutation attempts
  • Hard budget: max_optimization_rollouts = 400
  • Return: proposal_return_count = 4 ordered Optimization Proposals, or a failed Optimization Result with no official materialization

Guidance: "evolve the encoder user prompt template toward the archive nondominated set."

Official Evaluation

  • Unchanged from the eval design: HumanEval+ official_task_count = 164 @ pinned revision · official_repeat_count = 3 · zstd-19 ÷ reference length
  • Variance Source: budget_ratio_values = (0.25, 0.5, 0.75, 1.0)
  • Selection: official_selection_count = 3 returned candidate groups

official_rollout_count = D × official_task_count × official_repeat_count, where D ≤ max_admitted_graph_count = 12; 5,904 only in the explicit nonconvergent example.

Full run walkthrough: seed the catalog/archive, spend eight reflection LM calls, return the archive nondominated set

Designer

Freezes candidates, axes, mutation surface, concrete objective/diagnostic Tool Specs, and the identity-bearing archive policy ref/hash with capacity 32. Tool Specs reference versioned Tool Definitions and own Eval Config, Reward Policy, endpoint, ledger and capacity/refusal configuration.

Evaluation harness

Issues the Optimization Request with Tool Specs; the harness constructs non-serializable Runtime Tool Handles and binds Whetstone Tool Call Ledgers.

Optimizer

Seeds the Proposal Catalog and active archive: evaluate_subset on Templates A and B at ratio 0.5 — 48 objective rollouts. Both contain the two canonical, valid, unique proposals and their rows of 24 per-task scores.

output48 internal Rollout references under the subset tool's Eval Config and Evaluation Context
Evolution loop · max_reflection_lm_calls = 8; max_reflection_attempts_per_iteration = 3
Optimizer

Computes the per-task frontier over the archive — a candidate survives if its row is best on at least one task — and stochastically samples a parent P from it. Diversity pressure, not greedy best. Zero rollouts spent.

Evaluation harness

Loads P's archived Objective Vector over the fixed 24-task Internal Eval Config and serves a 3-task diagnostic projection for reflection. The diagnostic subset never supplies acceptance objectives.

output3 Rollout refs plus bounded Rollout Trace projections: encoder/Submission refs, preprocessing causes, selected candidate test facts; truncation and projection version recorded
Optimizer

The reflection LM reads P's template plus bounded Rollout Trace projections and writes a template. The harness canonicalizes and validates it against the template-only mutation surface and the Proposal Catalog. Invalid or duplicate output is recorded with full request/response evidence and retried using a deterministic (iteration, attempt) nonce, at most 3 attempts in that iteration and never beyond the global cap of 8 calls. Every call consumes the reflection cap; invalid/duplicate attempts consume no Rollouts. A valid unique child inherits every output-affecting value except {user_prompt_template}.

outputone persisted reflection LM call evidence; valid unique template or explicit invalid/duplicate reason
Evaluation harness

Only for a valid unique child, evaluates the same fixed 24-task objective population and the same 3 diagnostic task IDs, then appends the child to the Proposal Catalog. Child dominates → replace parent in the active archive; parent dominates or equals → reject child from active membership; incomparable → retain both. Rejected/dominated children remain cataloged.

output24 objective Rollout refs + 3 diagnostic refs; catalog entry and deterministic active-archive verdict
Optimizer

Applies the active-archive verdict, then enforces capacity 32: preserve nondominated entries as space permits and evict deterministic worst by (pass ascending, compression descending, proposal ID descending). Eviction never deletes a Proposal Catalog entry.

outputcapacity-32 active archive plus append-only catalog, both indexing immutable Tool Results
Optimizer

After max_reflection_lm_calls = 8, returns proposal_return_count = 4 unique proposals: first the archive nondominated set ordered by pass descending, compression ascending, proposal ID ascending; then, if needed, one pass over not-yet-selected Proposal Catalog entries in that same total order. The all-valid example derives planned_optimization_rollout_count = initial_candidate_count × objective_task_count + max_reflection_lm_calls × (minibatch_size + objective_task_count + minibatch_size) = 2 × 24 + 8 × (3 + 24 + 3) = 288, below the hard max_optimization_rollouts = 400. Invalid/duplicate reflection LM calls add no Rollouts. If fewer than proposal_return_count catalog entries exist, returns a failed Optimization Result and the harness performs no official materialization.

Evaluation harness

Validates all 4 proposals, expands every valid proposal over 4 ratios → 16 Graph Config outputs and 16 Materialization Records, and preserves optimizer order as selection input. The Selection Policy selects the top 3 proposal groups as 12 ordered record slots, then deduplicates the selected records by full Graph Hash to D ≤ 12 admitted graphs. The fourth proposal's 4 record slots persist unselected; they are not officially evaluated and need no selected official-lineage mapping. Official Evaluation Records and Official Plot Manifests MUST preserve the ordered mapping from every selected Materialization Record reference through graph_hash to the shared planned/result-key set and aggregate reference, including when multiple selected records share one graph and aggregate.

Evaluator

Submits exactly official_rollout_count = D × official_task_count × official_repeat_count Rollout Work Items with no duplicate Rollout Execution Keys; the explicit nonconvergent example has D = max_admitted_graph_count = 12 and official_rollout_count = 12 × 164 × 3 = 5,904. Every Provider Call Attempt runs in one @DBOS.step(retries_allowed=False) with native retries zero; DBOS checkpoints completed typed observations and replay reconstructs Node Outcomes and the Graph Run Result. dr-store persists one immutable Rollout Result containing that nested evidence, not an attempt ledger or a separate Graph Run Result, and atomic binding remains authoritative.

outputauthoritative bound refs or explicit FAILED Stages; dr-platform does not own uniqueness
Evaluation harness

Aggregates complete official Results: Average Binary Test Pass Rate and Mean Compression Ratio derived from integer Compressed Description Length. In the nonconvergent case, the 3 newly evaluated evolved proposals produce 12 Rollout Aggregates (3 × 4 ratios); 8 eligible A/B baseline aggregates (2 × 4 ratios) are reused only when Rollout Keys, official attestation and environment compatibility match.

outputexactly D new aggregates plus eligible distinct-hash baselines; 20 total in the explicit nonconvergent example

What lands on the plot: five curves per Model Route, and only official results

Per Model Route, eligible A/B baselines and three evolved templates retain separately attributable ratio slots, while converged slots share the planned/result-key set and Rollout Aggregate for their Graph Hash. The resulting distinct-hash aggregates are each averaged over 164 tasks × 3 repeats; in the explicit nonconvergent example they form five candidate curves with four aggregates each. The natural readings are whether evolution beat the starting templates and how the official Pareto Front moves. Internal Tool Results never appear here; their Evaluation Contexts lack official authority.

Additional Notes

Three fronts, only one is a result

  • The per-task frontier steers parent sampling — diversity machinery
  • The archive's archive nondominated set ranks the return — selection machinery
  • The plot's Pareto Front is computed from official rollout aggregates — the only front that is a result

Internal Evaluation vs Official Evaluation

  • Tool and official measurement policies have independent eval_config_hash identities. Their identical Evaluation Procedure Config is included in graph_hash; only Sampling/Aggregation remain outside it
  • Officialness comes only from the authorized Evaluation Context and write path; reuse also requires official attestation and a compatible environment
  • Whether the 24-task subset may overlap the official 164 is left to experimental designers

The catalog and archive are optimizer state

  • The append-only Proposal Catalog keeps every valid unique evaluated proposal; the capacity-32 active archive controls parent sampling and may evict catalog entries from active membership
  • Both in-memory indexes resolve Tool Results through terminal Tool Call Ledger entries; the ledger owns each Tool Result's completion and reference, and Tool Results plus referenced Rollout Results remain immutable in dr-store
  • The Result Store separately owns only the authoritative Rollout Execution Key → Rollout Result Object Reference binding

Candidate = one plotted line

  • A child inherits everything outside the mutation surface from its parent
  • Each proposal emits 4 Graph Config/Materialization Record pairs — the fixed ratio axis unrolled into its curve
  • Ordered multi-return: 4 candidates back, the frozen selection policy decides how many go on

Versus reference GEPA

  • Kept: reflective mutation from real failures and diverse parent sampling; archive update now uses explicit dominance/equality/incomparability on one fixed objective population
  • Ours: bounded Rollout Trace projections resolve from immutable Rollout references; concrete Tool Specs bind internal measurement and Reward Policy
  • dspy's GEPA is reference-only, never a dependency
