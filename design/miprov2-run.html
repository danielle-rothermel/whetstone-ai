MIPROv2 builds its candidate pool up front, then spends two tools searching it

whetstone-ai · system design walkthrough 3 of 5 · 2026-07-19
Eval COPRO MIPROv2 GEPA Codex CLI agent
MIPROv2 is a deterministic in-process policy with three internal stages: bootstrap demos from seed-treatment runs that passed their tests, propose grounded instructions with a proposal LM, then search the instruction×demo grid. The search sampler is internal state the contract never sees; what the contract sees is two designer-issued tools — a cheap noisy minibatch tool and an expensive promotion tool — whose records land in the ledger under separate config digests. The pool is frozen after proposal: nothing in the search loop can invent a new instruction in response to failures, and this doc leaves that limitation visible on purpose.

This configuration: two mutable regions, two tools, one grid search

One seed treatment; the mutation surface declares two named mutable regions of the encoder node — the instruction text and a demo block — which is the first configuration in the family where the mutation-surface declaration does real work: the diff validator checks each returned spec against exactly these two regions. The toolkit holds Emini (8-task minibatch per call, cheap, noisy) and Epromote (35-task sample, expensive, reserved for promoting the best-average combination). The request asks for 3 returned candidates; the selection policy certifies the top 2 under Ccert.

The demo mapping is a real adaptation decision, not a port: in our encoder-optimization setting, "demos" are few-shot (ground-truth code → description) pairs embedded in the encoder prompt, and bootstrapping keeps the pairs from seed runs where the full pipeline passed its tests — pass/fail replaces dspy's per-trace metric filter. Marked visualization-only throughout.

Full run walkthrough: bootstrap, propose once, then search

Designer

Freezes the experiment config: seed spec, two-region mutation surface, Emini + Epromote tool configs, requested return 3, selection = certify top 2, Ccert. Everything serialized and digested.

ledgerexperiment config, seed treatment digest, and both tool-config digests registered
Environment

Issues the OptimizationRequest: seed, mutation surface, toolkit handles for Emini and Epromote, guidance (return 3).

Optimizer

Stage 1 — bootstrap. Runs the unmodified seed through Emini ~10 times; harvests (code → description) pairs from tasks where the full pipeline passed; assembles 4 candidate demo sets.

ledger~10 Emini calls × 8 tasks, tagged (seed digest, Emini digest); costs accrue to this optimization step
Optimizer

Stage 2 — grounded proposal. One proposal-LM pass grounded in a dataset summary, a treatment-spec summary, and a prompting tip yields 4 instruction candidates; the seed instruction is kept as candidate 0. The pool is now frozen — 4 instructions × 4 demo sets = 16 combinations.

ledgerproposal-LM tokens tracked as control-plane cost; no eval records
Search loop · ×10 trials
Optimizer

Sampler picks a combination (instructioni, demosetj); assembles the candidate spec by rewriting the two mutable regions; spends one Emini call on it; feeds the noisy 8-task score back into its internal sampler state.

ledger1 Emini call × 8 per-task records, tagged (candidate digest, Emini digest)
Optimizer

Every 5 trials: the combination with the best average minibatch score so far is promoted to one Epromote call (35 tasks) — the trustworthy score that ranks finalists.

ledger1 Epromote call × 35 per-task records, tagged (candidate digest, Epromote digest)
Optimizer

Returns OptimizationResult: the top 3 combinations as full specs, ordered by promotion score (combinations never promoted rank below promoted ones), plus the 10-trial attempt history as advisory annotations.

ledgeroptimization step totals: ~22 tool calls (~150 pipeline runs), proposal-LM tokens, wall time
Environment

Diff validation: each returned spec differs from the seed only in the instruction text and demo block — the two declared regions. Selection policy admits the top 2 to certification. Annotations go to the provenance store as advisory context.

Certifier

Runs the 2 selected candidates × 40 Ccert tasks through the full pipeline; compression measured on every description, correctness from the sandbox; whetstone failure taxonomy classifies and retries.

ledger80 per-task records tagged (candidate digest, Ccert digest)
Environment

Aggregates from certified records only; emits the report and the two certified points against the seed's baseline.

ledger2 certified aggregates; pareto front derived as a view

What lands on the plot: two certified points from a 16-combination pool

Only the two certified candidates reach the correctness–compression plot, tagged with Ccert's digest. The ~150 pipeline runs the search consumed remain queryable in the ledger under the Emini and Epromote digests — visible as the run's true cost, excluded from the plot by construction. The third returned candidate stays uncertified but recorded: a later experiment can certify it without re-running the search.

Against dspy's MIPROv2: the hard Optuna dependency disappears from the contract (the sampler is an optimizer internal), and the hard-coded minibatch/full-eval constants become two explicit designer-issued tools with separate ledger identities. What survives unchanged is the fixed pool — stage 3 can only recombine what stages 1–2 produced, never react to failures with a new instruction. That limitation is exactly what GEPA's reflection loop addresses.
