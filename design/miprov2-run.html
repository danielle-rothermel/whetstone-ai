MIPROv2 bootstraps demo sets, proposes instructions once, and searches a frozen instruction × demo grid with cheap tools before official evaluation.

Purple marks what differs from the Eval run — the optimization machinery; everything unmarked matches the Eval flow.

Temporary vocabulary — terms this doc needs beyond the vocab page:
  • instruction — the directive text of an encoder user prompt template, apart from the demo block and the prompt variables. Needed: proposing instructions is one axis of MIPROv2's search.
  • demo block — a formatted run of few-shot (ground-truth code → encoded representation) pairs inside the encoder user prompt template. Needed: choosing its contents is the other axis.
  • demo set — one particular choice of pairs for the demo block (the empty set counts). Needed: the units the bootstrap stage produces.
  • combination — one (instruction, demo set) pair, assembled into a full encoder user prompt template. Needed: the unit the sampler scores.
  • proposal LM — the LM the optimizer calls to draft instructions; an optimizer internal, not a node of the architecture. Needed: to charge optimizer-side LM cost to the right place.
  • trial — one sampler step: pick a combination, spend one minibatch tool call on it. Needed: the unit of the search budget.
  • minibatch tool · promotion tool — this example's two designer-issued evaluation tools (8-task and 35-task cheaper Score Node Eval Configs). Needed: the walkthrough charges rollouts to one tool or the other.

1 (Provider, Model) Pair

The eval experiment's other three pairs get three more independent optimization runs; this doc walks one.

Initial Candidates & Mutation Surface

Candidate variable: the encoder user prompt template — nothing else.

  • Initial candidates: encoder templates A and B (full text in the eval doc)
  • Mutation surface: encoder {user_prompt_template} — the one variable the optimizer may change

Toolkit: Two Evaluation Tools

Each tool runs rollouts under a fixed cheaper Score Node Eval Config — same preprocess and metrics extraction as official, smaller task subset, 1 seed, budget_ratio 0.5 only.

  • minibatch tool — 8 tasks × 1 seed: cheap, noisy
  • promotion tool — 35 tasks × 1 seed: the trustworthy in-run score

Budgets and Official Evaluation

  • Optimization budget: ≤ 250 tool-run rollouts + proposal LM tokens (guidance)
  • Return: 3 rollout configs, ordered (guidance)
  • Selection: officially evaluate the top 2
  • Official eval config: the eval experiment's — HumanEval+ (164) @ pinned revision, 3 seeds, zstd-19

Full run walkthrough: bootstrap, propose once, then search

Designer

Freezes the experiment: the generator emits 2 initial rollout configs (templates A and B at budget_ratio 0.5, Score Node Eval Config matching the official eval config); the mutation surface names encoder {user_prompt_template}; the two tool definitions carry their cheaper Score Node Eval Configs; guidance asks for 3 returned rollout configs within ≤ 250 tool-run rollouts; selection = officially evaluate the top 2. Everything serialized and hashed.

Evaluation harness

Issues the OptimizationRequest: 2 initial rollout configs, mutation surface, toolkit (minibatch + promotion), guidance.

Optimizer

Stage 1 — bootstrap demo sets. Spends 6 minibatch calls (3 on each initial rollout config) = 48 rollouts; harvests (ground-truth code → encoded representation) pairs from rollouts with Binary Test Pass Score = 1; assembles 3 demo sets of 3 pairs, plus the empty demo set.

output48 rollout results under tool rollout config hashes (cheaper Score Node Eval Config inside each hash) — never official; cost charged to the optimization stage
Optimizer

Stage 2 — grounded instruction proposal. One proposal LM pass, grounded in a dataset summary, an architecture summary, and a prompting tip, drafts 4 new instructions; templates A's and B's instructions are kept, giving 6. The pool is now frozen: 6 instructions × 4 demo sets = 24 combinations.

outputa frozen pool of 24 combinations; proposal LM tokens charged to the optimization stage; no rollout results
Search loop · ×12 trials
Optimizer

The sampler picks a combination, assembles it into a full encoder user prompt template, sets it into a tool-run rollout config, and spends one minibatch call — 8 tasks × 1 seed. The noisy score feeds the sampler's internal state.

output8 rollout results under the minibatch tool's rollout config hash
Optimizer

Every 6 trials: the combination with the best average minibatch score so far gets one promotion call — 35 tasks × 1 seed — the trustworthy in-run score that ranks finalists.

output35 rollout results under the promotion tool's rollout config hash
Optimizer

One trial, concretely — trial 7: the sampler picks (instruction 4, "name the function's contract, signature, and edge cases", demo set 2). The minibatch call runs 8 rollouts at budget_ratio 0.5; 6 of 8 pass, mean measured compression ≈ 0.41. The sampler nudges toward instruction 4's region of the grid.

Optimizer

Returns the OptimizationResult: the top 3 combinations as full rollout configs — identical to an initial rollout config everywhere outside encoder {user_prompt_template}, so each carries the official-matching Score Node Eval Config — ordered by promotion score, then average minibatch score. The 12-trial attempt history rides along as advisory provenance, never as a score.

outputrun totals: 20 tool calls, 214 of the ≤ 250 budgeted rollouts, proposal LM tokens, wall time — all advisory
Evaluation harness

Diff check: each of the 3 returned rollout configs differs from an initial one only in encoder {user_prompt_template} — consistent with the mutation surface; 3/3 pass. Selection admits the top 2 to official evaluation. The attempt history is persisted as advisory provenance.

Evaluator

Expands each admitted candidate across the 4 budget ratios → 8 rollout configs, and submits 8 × 164 × 3 = 3,936 jobs — one per (rollout config, task, seed) — as a dr-platform campaign. Whetstone's failure taxonomy retries transient errors; permanent failures stay rollout results, not gaps.

output3,936 official rollout results, each keyed (rollout config hash, official eval config hash, task, seed)
Evaluation harness

Aggregates from official rollout results only: per rollout config, per-task mean Binary Test Pass Score over 3 seeds, then unweighted mean over the 164 tasks, plus the same aggregation for measured compression.

output8 rollout aggregates — each optimized candidate's distribution of 4 across the budget ratios

What lands on the plot: two optimized candidates against the A/B baselines

The two admitted candidates land as two new budget-response curves — 8 rollout aggregates, each one rollout config averaged over 164 tasks × 3 seeds — read directly against the template A and B baselines from the Eval run: those official rollout results are reusable as-is, since the baseline rollout configs and the official eval config hash to the same identities in both experiments. The ~214 rollouts the search consumed stay queryable under the tool rollout config hashes — the run's true cost, excluded from the plot by construction. The third returned rollout config stays un-evaluated but persisted: a later experiment can officially evaluate it without re-running the search.

Additional Notes

One variable, two regions inside it

  • The contract sees one mutation surface entry: encoder {user_prompt_template}
  • The instruction + demo block structure is optimizer-internal — MIPROv2 assembles a combination into plain template text before returning it
  • The diff check therefore stays simple: one variable may differ, nothing else

Tools carry the cheap evaluation

  • Each tool holds a fixed cheaper Score Node Eval Config: task subset, 1 seed, budget_ratio 0.5 only — same preprocess and metrics extraction as official
  • The cheap config sits inside the tool-run rollout config's hash, so tool results and official results can never mix
  • Only the harness's official evaluation produces official rollout results
  • Whether tool task subsets may overlap the official 164 is a sample-holdout question for experimental designers — not litigated here

The pool is frozen after proposal

  • Stage 3 can only recombine what stages 1–2 produced — no mid-run invention of a new instruction in response to failures
  • A preserved MIPROv2 limitation, kept visible on purpose; GEPA's reflective loop is the answer to it

Against dspy's MIPROv2

  • The Bayesian sampler (Optuna/TPE in dspy) becomes an optimizer internal — no library dependency crosses the contract
  • dspy's hard-coded minibatch and full-eval constants become two designer-issued tools with their own identities
  • dspy stays reference-only — this is a reimplementation, never a dependency

Contract fit

  • Guidance, not enforcement: return count and budget live in the request and tool definitions — never enforced on the optimizer
  • Ordered multi-return; the attempt history is advisory provenance, never a score
  • Both contract schemas are serializable first: in-process here, and the same schemas cross a process boundary for the codex CLI agent
  • Eval is the identity optimizer — empty mutation surface, empty toolkit; this run is the family's first with both nonempty axes of MIPROv2's grid in play
