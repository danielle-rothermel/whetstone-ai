MIPROv2 bootstraps demo sets, proposes instructions once, and searches a frozen instruction × demo grid through runtime Handles constructed from serialized Tool Specs before returning three ordered Optimization Proposals.

Purple marks what differs from the Eval run — the optimization machinery; everything unmarked matches the Eval flow.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • instruction — the directive text of an encoder user prompt template, apart from the demo block and the prompt variables. Needed: proposing instructions is one axis of MIPROv2's search.
  • demo block — a formatted run of few-shot (ground-truth code → encoded representation) pairs inside the encoder user prompt template. Needed: choosing its contents is the other axis.
  • demo set — one particular choice of pairs for the demo block (the empty set counts). Needed: the units the bootstrap stage produces.
  • combination — one (instruction, demo set) pair, assembled into a full encoder user prompt template. Needed: the unit the sampler scores.
  • proposal LM — the LM the optimizer calls to draft instructions; an optimizer internal, not a node of the Graph. Needed: to charge optimizer-side LM cost to the right place.
  • trial — one sampler step: pick a combination, spend one minibatch tool call on it. Needed: the unit of the search budget.
  • minibatch tool · promotion tool — two Tool Specs referencing versioned interface Definitions and configuring 8-task and 35-task internal measurement.
Algorithm-local identity. MIPROv2 owns three versioned identity domains. For each one, the owning code selects the stated Identity Payload, wraps it in the exact Identity Document {schema, schema_version, payload}, and computes the full 64-character lowercase SHA-256 Identity Hash over canonical identity JSON.
  • Demo Set Identity Document — schema=miprov2.demo-set, schema_version=1; payload selects the ordered demo pairs, each with exact ground_truth_code and encoded_representation. The empty set has an empty pair list.
  • Instruction Identity Document — schema=miprov2.instruction, schema_version=1; payload selects only the exact validated instruction_text. Templates A/B and generated instructions use this same identity domain; attempt nonce, evidence and cost are excluded.
  • Trial Combination Identity Document — schema=miprov2.trial-combination, schema_version=1; payload selects the full instruction_identity_hash and demo_set_identity_hash. Trial ID, score, Reward and attempt evidence are excluded.
Identity Hash equality in the matching schema and version establishes algorithm-local equality. A Content Hash instead covers a complete canonical persisted object for storage integrity and never substitutes for any of these identities.

Initial Candidates & Mutation Surface

Candidate variable: the encoder user prompt template — nothing else.

  • Initial candidates: encoder templates A and B (full text in the eval doc)
  • Mutation surface: encoder {user_prompt_template} — the one variable the optimizer may change

Toolkit: Two Evaluation Tools

Each Tool Spec is a concrete Config: Definition ref/version, endpoint, internal Eval Config, Reward Policy, capacity/timeout/policy refs, ledger namespace, idempotency and refusal rules.

  • minibatch tool — 8 tasks × 1 repeat: cheap, noisy
  • promotion tool — 35 tasks × 1 repeat: the trustworthy in-run Reward
  • Optimization Request serializes Tool Specs; non-serializable runtime Handles bind them to services
  • Whetstone Tool Call Ledgers keyed by full Tool Spec Identity Hash plus call_id, spelled tool_spec_hash + call_id, own acceptance/refusal/completion and capacity accounting across retry/restart

Budgets and Official Evaluation

  • Optimization budget: ≤ 250 tool-run rollouts + proposal LM tokens (guidance)
  • Result shape: on success, exactly 3 ordered Optimization Proposals; cardinality failure returns an explicit failed Optimization Result and permits no official materialization
  • Selection: officially evaluate the top 2
  • Official Eval Config: the eval experiment's — HumanEval+ (164) @ pinned revision, 3 repeats, zstd-19

Full run walkthrough: bootstrap, propose once, then search

Designer

Freezes starting candidates, axes, mutation surface and two concrete Tool Specs. Each references a versioned Tool Definition and owns its internal Eval Config, Reward Policy and ledger/capacity/refusal configuration.

Evaluation harness

Issues the OptimizationRequest with the two Tool Specs; the harness constructs runtime Handles and binds durable Tool Call Ledgers.

Optimizer

Stage 1 — bootstrap demo sets. Spends 48 Rollouts and deterministically assembles 3 nonempty demo sets plus empty. Build the version-1 Demo Set Identity Document for each and require four distinct full Identity Hashes; a duplicate Demo Set identity, detected Demo Set Identity Hash collision or insufficient valid pairs ends with a failed Optimization Result before official materialization.

output48 internal Rollout references; treatment and Eval Config identities remain separate; cost charged to optimization
Optimizer

Stage 2 — distinct instruction identity construction. Build Instruction Identity Documents for A/B, then generate with the deterministic nonce/attempt sequence until four valid instructions have full Identity Hashes distinct from A/B and each other. Invalid or repeated Documents consume proposal-LM evidence/cost; a different Document with an already-used full Identity Hash is an explicit collision failure. Pair the resulting instruction and demo-set Identity Hashes in version-1 Trial Combination Identity Documents and require exactly 24 distinct identities; any detected Trial Combination Identity Hash collision, terminal proposal-LM failure or exhaustion of the request's fixed, identity-bearing instruction-attempt cap yields a failed Optimization Result and no official materialization.

outputsuccessful path: 6 distinct Instruction Identity Hashes × 4 distinct Demo Set Identity Hashes = 24 distinct Trial Combination Identity Hashes
Search loop · ×12 trials
Optimizer

Seeded TPE scores remaining choices; deterministic sampling chooses one not-yet-sampled Trial Combination Identity Hash, assigns a trial ID and spends one 8-task minibatch call. Twelve iterations therefore produce 12 distinct trial proposals.

output8 Rollout refs under minibatch Rollout Execution Keys plus Reward, cost and provenance
Optimizer

After all 12 cheap trials: group proposals by their full Trial Combination Identity Hash, order the 12 distinct identities by cheap Reward with stable proposal-ID tie-break, and promote exactly the top 2 to the 35-task Tool Spec.

output2 × 35 = 70 Rollout refs under promotion Rollout Execution Keys plus Reward, cost and provenance
Optimizer

One trial — trial 7 evaluates instruction 4 + demo set 2 on 8 tasks: 6 pass, Mean Compression Ratio ≈0.41 from integer Compressed Description Length facts. That cheap Reward updates TPE state.

Optimizer

Orders successful promoted proposals first, then cheap fallback, and returns exactly the first 3 proposals with distinct Trial Combination Identity Hashes. Because the successful path has 12 distinct trial identities, the fallback is constructive. Any violated minimum returns a failed Optimization Result; the harness performs no diff check, selection or official materialization.

outputsuccessful planned path: 20 tool calls, 214 Rollouts + proposal-LM attempts; failure path stops before official materialization
Evaluation harness

Diff-checks the 3 proposals, selects the top 2 while preserving optimizer order, then materializes each across four ratios → 8 Rollout Configs. The attempt history persists as advisory provenance.

Evaluator

Submits 3,936 Work Items, one per Rollout Execution Key. After Node retries exhaust, the executor immutably persists the terminal Result in dr-store and asks the Whetstone Result Store to atomically bind the key to that Object Reference. Unbound plus reference binds; the same reference replays as idempotent success. Only either path resolving to that authoritative bound reference makes the Stage SUCCEEDED, and dr-platform transports only that reference. Failure that leaves the key unbound makes the Stage FAILED and permits an operator-added Stage Attempt. A different-reference conflict keeps the existing authoritative binding, cannot authorize replacement and does not make the conflicting candidate a success.

output3,936 officially attested Rollout Results with immutable references
Evaluation harness

Aggregates complete official results only. Exhausted infrastructure failures keep the aggregate incomplete; deliberate re-evaluation uses a new context or Repeat, never replay/replacement of a terminal Result.

output8 rollout aggregates — each optimized candidate's distribution of 4 across the budget ratios

What lands on the plot: two optimized candidates against the A/B baselines

The two admitted proposals land as two new budget-response curves — 8 Rollout Aggregates, each one Rollout Config averaged over 164 tasks × 3 repeats — beside eligible A/B baselines. Baseline reuse requires matching Rollout Keys, official attestation and compatible environment. The ~214 internal Rollouts remain queryable as the search's measured cost but stay outside the official plot. The third proposal persists unselected and can later be materialized and officially evaluated without rerunning search.

Additional Notes

One variable, two regions inside it

  • The contract sees one mutation surface entry: encoder {user_prompt_template}
  • The instruction + demo block structure is optimizer-internal — MIPROv2 assembles a combination into plain template text before returning it
  • The diff check therefore stays simple: one variable may differ, nothing else

Tools carry the internal evaluation

  • Each Tool Spec binds a fixed cheaper Eval Config and Reward Policy; its Rollout Config independently fixes budget_ratio at 0.5
  • Different full Eval Config Identity Hashes distinguish measurement policy; the internal Evaluation Context and write path determine authority
  • Only the authorized official Evaluation Context/write path produces official Rollout Results
  • Whether tool task subsets may overlap the official 164 is a sample-holdout question for experimental designers — not litigated here

The pool is frozen after proposal

  • Stage 3 can only recombine what stages 1–2 produced — no mid-run invention of a new instruction in response to failures
  • A preserved MIPROv2 limitation, kept visible on purpose; GEPA's reflective loop is the answer to it

Against dspy's MIPROv2

  • The Bayesian sampler (Optuna/TPE in dspy) becomes an optimizer internal — no library dependency crosses the contract
  • dspy's hard-coded minibatch and full-eval constants become two designer-issued tools with their own identities
  • dspy stays reference-only — this is a reimplementation, never a dependency

Contract fit

  • Guidance, not enforcement: search budget and strategy are guidance; proposal schema, allowed diffs and Tool Handle capacity/refusal are hard contract behavior
  • Ordered multi-return; the attempt history is advisory provenance, never Reward or an official Objective
  • Both contract schemas are serializable first: in-process here, and the same schemas cross a process boundary for the codex CLI agent
  • Eval is the identity optimizer — empty mutation surface, empty toolkit; this run is the family's first with both nonempty axes of MIPROv2's grid in play
