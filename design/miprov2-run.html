MIPROv2 Run Design

A MIPROv2 Optimization Run is an ordered history of single Optimization Steps: initialize and bootstrap candidate pools, fully evaluate the default program, run seeded TPE minibatch trials, periodically and finally promote the best eligible combination to full evaluation, then complete with ordered Optimization Proposals. Each optimizer invocation only proposes candidates and an Evaluation Intent. Whetstone validates, materializes and evaluates outside that invocation, persists an immutable Optimization Step Result, and passes its references to the next invocation.

Purple marks the optimization machinery; everything unmarked follows the Eval run. The step boundary is deliberate: optimizer code never executes evaluation.

Algorithm-local vocabulary — these anchors describe this MIPROv2 protocol without adding shared Whetstone vocabulary:
  • Optimization Step — one restartable transition within an Optimization Run. It consumes one immutable Optimization Step Request and ends only after Whetstone persists its immutable Optimization Step Result.
  • Optimization Step Request — the serializable input to one optimizer invocation: run identity, step index and kind, immutable prior Step Result and history references, pool identities, sampler/study-state reference, fixed hyperparameters, budget state and output contract. It contains no mutable evaluator capability.
  • Optimization Step Result — Whetstone's immutable record of one completed step: accepted candidates, resolved Evaluation Intents, measurement and aggregate references, observed scores, updated history/study-state references, budget accounting and terminal status. The next Step Request references it; it is never edited in place.
  • Evaluation Intent — an immutable optimizer-produced request for Whetstone to measure a named candidate or combination. It selects the candidate/combination identity, exact target Eval Config reference and identity, required Evaluation Context role/policy, purpose (bootstrap, baseline_full, minibatch or promotion_full), and a correlation key back to the run and step. Task/repeat scope comes from the target Eval Config. An intent contains no score or result and does not itself perform evaluation.
  • instruction — directive text in the encoder user prompt template, apart from the demo block and prompt variables.
  • demo block — a formatted run of few-shot ground-truth-code → encoded-representation pairs in the encoder user prompt template.
  • demo set — one particular choice of pairs for the demo block; the empty set counts.
  • combination — one (instruction, demo set) pair assembled into a complete encoder user prompt template.
  • proposal LM — the LM called to draft instructions; it is optimizer-side compute, not a node of the Graph.
  • trial — one sampler step that may choose a previously sampled combination and requests one minibatch measurement.
  • cardinality failure — an explicit failed terminal Step Result when required pool or returned-proposal cardinality cannot be met. It prevents official materialization.
In this page, unevaluated always means “no completed Step Result records a measurement of this exact combination under the exact target Eval Config identity.” Full evaluation under one Eval Config does not make the combination evaluated under another.
Algorithm-local identity. MIPROv2 owns three versioned identity domains. Each owner selects the stated Identity Payload, wraps it in {schema, schema_version, payload}, and computes the full lowercase SHA-256 Identity Hash over canonical identity JSON.
  • Demo Set Identity Document — schema=miprov2.demo-set, schema_version=1; payload selects the ordered demo pairs with exact ground_truth_code and encoded_representation. Its Demo Set Identity Hash establishes demo-set equality.
  • Instruction Identity Document — schema=miprov2.instruction, schema_version=1; payload selects exact validated instruction_text. Attempt nonce, evidence and cost are excluded. Its Instruction Identity Hash establishes instruction equality.
  • Trial Combination Identity Document — schema=miprov2.trial-combination, schema_version=1; payload selects the full instruction and demo-set Identity Hashes. Trial ID and scores are excluded. Its Trial Combination Identity Hash groups repeated trials and establishes combination equality.
A Content Hash covers a complete persisted object for storage integrity and never substitutes for these identities.

Initial Candidates & Mutation Surface

  • Default program: encoder template A with the empty demo set
  • Initial candidates: encoder templates A and B
  • Mutation surface: encoder {user_prompt_template} only
  • Internal axes: instruction × demo set, assembled to plain template text before validation

Two Eval Configs bound under the internal Evaluation Role

  • Minibatch: minibatch_task_count=8 × 1 repeat
  • Full: full_eval_task_count=35 × 1 repeat
  • Both fix Sampling and Aggregation policy and use the same graph-assigned Evaluation Procedure Config
  • Evaluation authority stays with Whetstone between optimizer invocations

Result and official-evaluation contract

  • Success returns exactly returned_proposal_count=3 distinct, ordered, measured proposals
  • A cardinality failure returns a failed Step Result and permits no official materialization
  • Whetstone validates and officially evaluates every returned proposal across all four fixed ratios
  • Eval Config bound under the official Evaluation Role: HumanEval+ (164) at a pinned revision, three repeats, zstd-19

Boundary invariant. An optimizer invocation cannot validate a final graph, materialize it, select task/repeat work, execute a Rollout, or manufacture a score. Whetstone resolves each Evaluation Intent against the exact referenced Eval Config, performs those operations through the ordinary evaluation path, and records the evidence in the Step Result. A retry with the same immutable inputs must recover or reproduce the same transition identity; later state is carried by a new object reference, never mutation.

Full Run Walkthrough

Designer

Freezes Initial Candidates, the single-entry Mutation Surface, algorithm hyperparameters, two ordinary Eval Configs for internal-role contexts, one ordinary Eval Config for the official-role context, and the output contract. The Evaluation Procedure Config assigned to the graph is identical across internal and official contexts.

Step 0 · initialize

The first Step Request has no prior result. The optimizer records sampler=tpe and seed=9, registers the default combination, and emits bootstrap candidates and bootstrap Evaluation Intents as needed to construct demo evidence. Whetstone evaluates each intent externally; subsequent bootstrap Step Requests reference those Step Results until four valid demo-set identities exist.

resultnum_demo_set_candidates=4, each bounded by max_bootstrapped_demos=3 and max_labeled_demos=0; bootstrap measurement count is recorded, not assumed
Pool construction

The proposal LM attempts instructions at proposal_temperature=1.0. Templates A/B and accepted generated text use the same Instruction Identity Document. Duplicate text may consume an attempt but cannot add a pool member. Reaching instruction_attempt_cap=12 before six distinct valid identities produces a cardinality failure.

resultnum_instruction_candidates=6 × num_demo_set_candidates=4 = combination_pool_size=24; the pool is then frozen
Baseline full step

Before search begins, the optimizer emits a baseline_full Evaluation Intent for the default program under the exact full Eval Config bound through a context carrying the internal role. Whetstone validates and fully evaluates it. The resulting full score is inserted into the TPE study and the default combination becomes evaluated only for that Eval Config identity.

result35 Rollout references, aggregate/score, study-state reference and full-evaluation history for the default combination
TPE search · num_trials=12 minibatch trial steps
Trial step

The seeded sampler chooses any combination in the frozen pool, including one used by an earlier trial, and emits one minibatch Evaluation Intent. Whetstone measures it under the exact minibatch Eval Config and persists the Step Result. The study groups observations by Trial Combination Identity Hash and uses the arithmetic mean of all minibatch scores for that combination; repeated trials add observations rather than replacing them.

each resultminibatch_task_count=8 Rollout references + minibatch score + updated per-combination count, sum and mean + study-state reference
Promotion step

After every minibatch_full_eval_steps=5 trial results, choose the combination with the highest mean minibatch score that remains unevaluated under the exact target full Eval Config; use stable identity-hash tie-breaks. Emit one promotion_full Evaluation Intent. Whetstone performs the full evaluation, and the next Step Result feeds that full score into the TPE study.

cadenceafter trials 5 and 10 in this example; each successful promotion records 35 Rollout references and a full score
Final promotion step

After trial 12, because the run ended off the five-step cadence, perform the same highest-mean eligibility check and full evaluation once more. Promotion remains cadence-driven throughout the search. If every trial-observed combination is already fully evaluated under the target Eval Config, persist a deterministic no-op Step Result with that reason and no Evaluation Intent.

resultscheduled_promotion_count=3 opportunities at trials 5, 10 and 12; actual completed full measurements and no-ops remain explicit
Completion step

The completion Step Request references the entire immutable run history and final study state. It orders combinations with completed full scores first by full score, then fills remaining positions from distinct trial-observed combinations by mean minibatch score; both tiers use deterministic identity-hash tie-breaks. It returns exactly returned_proposal_count=3 distinct measured proposals. Completion does not add unscheduled internal evaluations. Fewer than three distinct measured combinations is a cardinality failure, never an unmeasured fallback.

resultterminal Optimization Result with three ordered proposals, their combination identities, source Step Result and measurement references, tier/rank evidence and complete optimization cost
Evaluation harness

Revalidates all three proposals against the Mutation Surface, expands each across fixed_ratio_count=4, and persists 12 Graph Configs and 12 Materialization Records. The Selection Policy admits every returned proposal for final official evaluation. Planning deduplicates only identical full Graph Hashes, producing D ≤ 12 distinct official graphs while preserving the ordered proposal/materialization-to-result lineage.

resultofficial_materialization_count=12; every returned proposal has official work or an explicit shared-graph mapping
Evaluator

Executes exactly D × official_task_count=164 × official_repeat_count=3 official Rollout Work Items with no duplicate Rollout Execution Keys. Ordinary Results retain their authorized Evaluation Context carrying the official role and immutable evidence; the Official Evaluation Record certifies their references.

resultD × 164 × 3 ordinary Rollout Results certified by the Official Evaluation Record; 5,904 when D=12
Evaluation harness

Aggregates complete ordinary results referenced and certified by the Official Evaluation Record and writes the plot manifest. Baseline reuse still requires matching Rollout Keys, official attestation and compatible environment.

What Lands on the Plot

All three returned Optimization Proposals receive final official evaluation. In the nonconvergent case, they produce three new four-point budget-response curves — 12 certified Rollout Aggregates, each over 164 tasks × 3 repeats — beside eligible template A/B baselines. The optimization history remains queryable as measured search cost and record-local provenance but stays outside the official plot: bootstrap measurements, the default-program full score, repeated minibatch observations, their per-combination means, and periodic/final promotion scores are not certified by being used in the study.

Additional Notes

One mutation, two internal axes

  • The contract exposes one Mutation Surface entry: encoder {user_prompt_template}.
  • MIPROv2 assembles instruction + demo block to plain template text before Whetstone validates the candidate.
  • The pool freezes after initialization; search recombines pool members and does not invent instructions mid-trial.

Evaluation stays external

  • Evaluation Intents identify desired measurements; Eval Configs determine task/repeat scope.
  • Only Whetstone materializes Graph Configs, creates work, executes Rollouts and records scores.
  • Internal and official contexts remain distinct even when they share a graph and Evaluation Procedure Config.

Faithful search semantics

  • Fully evaluate the default program before the first trial.
  • Allow repeated combinations and average their minibatch scores by combination.
  • At cadence and final boundaries, fully evaluate the highest mean-minibatch combination not yet evaluated under that exact full Eval Config, then feed the full score into the study.

Identity and retry behavior

  • Trial identity and combination identity are different: multiple trials may reference the same combination hash.
  • Step Requests and Results are immutable and reference-addressed, so restart does not depend on optimizer process memory.
  • Changing Eval Config identity changes evaluation eligibility and measurement history.

Contract fit

  • Hyperparameters, Eval Config values, output cardinality and derived counts are separately registered.
  • Proposal schema and allowed diffs are hard validation behavior; sampler strategy remains optimizer-owned.
  • The final official evaluation covers every returned proposal and is not a reuse of internal promotion evidence.
