This page turns the shared vocabulary and definitions into nine bounded implementation workstreams: dr-serialize identity; immutable storage; platform integration, orchestration, retry, and concurrency; dr-providers transport ownership; dr-code evaluation-kernel ownership; graph/Whetstone binding; the durable optimizer and Reward protocol; Objectives and official selection; and authority, certification, and publication. A cross-repository breaking release sequence makes those contracts usable through versioned PyPI dependencies without compatibility shims or Git revision pins.

Decisions at a Glance

Identity Hash

A domain identifier: full SHA-256 over the Canonical Identity JSON derived from an exact, self-describing, versioned Identity Document. The domain selects the identity-bearing payload inside the document; dr-serialize validates, canonicalizes, and hashes it.

Content Hash

A storage-integrity identifier: SHA-256 over the complete canonical persisted record. dr-store uses it to address and verify immutable content; no domain field selection occurs.

Retry Boundary

Whetstone owns bounded semantic retry using its failure taxonomy. Each Provider Call Attempt is one @DBOS.step(retries_allowed=False); DBOS checkpoints returned typed attempt observations, while Whetstone deterministically sleeps, backs off, and invokes later attempts. No custom attempt ledger or effect fence is added.

Provider Concurrency Boundary

Whetstone derives collision-free labels from every exact (provider, protocol, model) Provider Quota Identity, applies every relevant label to multi-route Rollout Work Items, and configures mandatory empty-selector plus exact-label Stage capacities. dr-platform supplies only generic label/selector admission. Capacity covers coarse ADMITTED-Stage occupancy, including backoff.

Graph and Node Contract

dr-graph supplies one versioned variable-bearing Graph Definition that materializes Graph Configs. Whetstone uses that same artifact in the Rollout Definition role. Each Graph Config contains concrete Node configs, unique Node IDs, identity-bearing Node Input Sources and Variable assignments, declared Node Output fields, and exactly one terminal/sink Node.

Evaluation Identity Partition

The Eval Node statically assigns an Evaluation Procedure Config reference, so Procedure changes alter both graph_hash and eval_config_hash. Sampling or Aggregation changes alter only eval_config_hash. Evaluation Context owns authority, officialness, and realized-environment attestation.

These hashes answer different questions. An Identity Hash asks whether two values have the same domain identity under a named schema version. A Content Hash asks whether two complete persisted records are byte-for-byte equivalent after canonicalization. Neither may substitute for the other.

Identity Document contract

{
  "schema": "example.config",
  "schema_version": 1,
  "payload": {"identity_field": "value"}
}

Illustrative values only. The three field names and their presence are exact; each domain owns its actual schema name, version, and payload.

Finite JSON contract

The accepted recursive value set is JSON null, booleans, strings, finite numbers, arrays of accepted values, and objects with string keys and accepted values. Validation does not invoke custom serializers or apply lossy diagnostic normalization.

Canonicalization begins only after validation succeeds, so unsupported runtime values cannot silently collapse onto an identity.

Coordinated Breaking Identity Reset

Adopt the new identity contract as one coordinated break across dr-serialize, dr-graph, dr-code, dr-platform, dr-providers where they expose identity-bearing values, and whetstone-ai. Update producers, consumers, fixtures, and stored test values together.

No compatibility aliases, legacy-hash acceptance, or dual-read paths. Old and new identities are intentionally distinct. Any persisted development data, caches, indexes, or references keyed by replaced identities must be regenerated or cleanly reseeded.

Object Reference

{
  "schema": "example.record",
  "content_hash": "<64-character lowercase SHA-256>"
}

The reference carries the declared schema and full Content Hash as one typed value. Consumers may pass it opaquely; dr-store validates both components when resolving it.

The Content Hash covers the complete canonical persisted record—including fields irrelevant to domain identity—so changing provenance or terminal evidence changes the Object Reference even when domain identity stays constant.

Explicit exclusions

  • No Rollout Execution Key or other Whetstone key type
  • No Rollout Result model or evaluation vocabulary
  • No official-evaluation authorization or attestation rules
  • No workflow lifecycle, retry, campaign, stage, or attempt state
  • No speculative backend, replication, or query features

Atomic Key-to-Reference Binding

dr-store provides one generic atomic operation for binding an application-supplied key to an Object Reference. The key is opaque to dr-store; the caller owns its meaning and validation.

Existing state for keyRequested referenceRequired result
UnboundAny valid stored Object ReferenceBind Atomically create the binding and return success.
Bound to AASame Return idempotent success without writing a replacement.
Bound to ABConflict Return an explicit different-reference conflict; keep A unchanged.
Bound to AOverwrite, clear, or rebind requestReject The public contract exposes no overwrite path.

Concurrent binds for one previously unbound key must produce exactly one durable winner. Every loser observes either idempotent same-reference success or an explicit different-reference conflict.

Workstream 3 · Platform Integration, Orchestration, Retry, and Concurrency

Register a concrete versioned Orchestration Pipeline in dr-platform, transport typed Work Request and Result references opaquely, and bind Whetstone's retry and provider-capacity policies to generic Stage mechanics. Platform Stage Attempt and DBOS Durability Replay evidence are sufficient; no additional stage-evidence type is introduced.

Three events, three meanings

  • Provider Call Attempt: one checkpointed logical attempt observation, not a guarantee of one physical wire call.
  • Durability Replay: deterministic recovery that reuses checkpointed Provider Call Attempt observations, reconstructs Node Outcomes and the Graph Run Result, and re-enters only uncheckpointed work.
  • Platform Stage Attempt: the platform-native execution attempt; a later one is created only by explicit operational recovery.

Terminality is result-based

An exhausted semantic failure is expected domain output. When Whetstone produces its causal evidence, persists the terminal Rollout Result, and binds the exact Object Reference, the Stage is operationally SUCCEEDED.

The Stage is FAILED only when execution cannot produce or persist a terminal Result, or cannot bind or confirm its exact reference. Semantic exhaustion alone is not operational failure.

Accepted external-effect ambiguity. A crash or ambiguous timeout after the provider effect but before the DBOS checkpoint may create duplicate physical calls or billing. Only the checkpointed physical outcome is retained. The design does not guarantee exactly-once provider effects or evidence for every billed call.
Raw evidence is primary. Retain the least-processed request and response or failure bodies on success and failure. Normalized usage, cost, and diagnostics may coexist with them but never replace them. Do not preemptively cap, truncate, allowlist, generally redact, or assign a retention policy to those bodies; if a backend limit is encountered, surface it rather than silently truncating.

Operator Retry Gate

A Whetstone operator command or gateway first verifies that the platform Stage is FAILED and the Rollout Execution Key is unbound, then invokes the rebuilt dr-platform retry_stage. The retried Whetstone executor repeats the binding check before any new semantic effect because a concurrent actor may have won after the gateway check.

The Result Store's atomic binding is authoritative. An unbound key may acquire one reference and the same reference may replay idempotently. A different-reference conflict preserves the existing binding; it cannot overwrite the winner, authorize replacement, or turn the conflicting candidate into the terminal Result. An orphan Object Reference that was persisted but never bound does not itself prohibit operator retry.

Provider Concurrency Control

Use the existing generic Stage Control selector-capacity mechanism as a deliberately coarse, conservative cross-worker cap. dr-providers supplies the best-effort Provider Quota Identity, exactly (provider, protocol, model). Whetstone derives collision-free labels, applies every relevant label to multi-route Rollout Work Items, and configures dr-platform's mandatory default empty-selector capacity plus every corresponding exact-label capacity. dr-platform enforces all matching controls without provider-specific code.

Provider Quota Identity and labels

  • The identity is exactly (provider, protocol, model).
  • It has no credential or account component and no arbitrary override.
  • Whetstone derives labels collision-free from the complete tuple.
  • Work that may route through more than one tuple carries every relevant label.

Accepted operating constraints

  • Capacity counts concurrently ADMITTED rollout-execution Stages across workers, not observed physical calls.
  • The default empty-selector control is mandatory Stage configuration; without it, work remains READY and unadmitted even if provider-specific controls exist.
  • Admission must satisfy the empty-selector control and every matching exact-label control; no matching control replaces or bypasses another.
  • A control is scoped to one pipeline key/version, stage key, selector, and shared database/schema.
  • Admission occupies capacity for the full Stage, including non-provider work and backoff between semantic attempts.
  • Stuck work retains capacity until it becomes terminal or an explicit sweep repairs its platform state.
  • The cap is intentionally conservative; it does not promise strict fairness or perfect quota utilization.
  • Independent accounts using the same tuple may be conservatively grouped.
  • A shared quota exposed under different tuples may be under-controlled.
Provider Concurrency Control non-goals. This workstream does not implement exact call-boundary concurrency, requests-per-minute or tokens-per-minute accounting, token reservation, global, cross-stage, or cross-version quotas, strict fairness, adaptive header-driven limiting, or credential-aware quota identity.
Provider Calls non-goals. No provider idempotency keys; exactly-once external effects; custom provider-attempt ledger; exact call-boundary limiter; RPM, TPM, or token reservation; global, cross-stage, or cross-version quotas; fairness or adaptive header limiting; credential-aware quota identity; generalized body-size, truncation, redaction, or retention policy; nested native or DBOS automatic retries; or provider-specific dr-platform logic.

Workstream 7 · Durable Optimizer and Reward Protocol

Implement one algorithm-neutral harness around an Optimization Run's ordered durable Steps. Each Step consumes one immutable Optimization Step Request and becomes terminal only when one immutable Optimization Step Result is persisted. The harness owns request validation, execution-mode dispatch, external evaluation and finalization, restart, idempotency, budgets, evidence, and terminal Optimization Result assembly; algorithm adapters own only algorithm-specific proposal or tool-use logic.

Schemas and reference behavior

  • Optimization Step Request: run/Step/config identities; kind/index; prior Step Result/state/history Object References plus Content Hashes; candidates/pools; hyperparameters and budget state; output contract; optional Tool Configs with typed Tool Definition references plus Identity Hashes.
  • Evaluation Intent: candidate identity; exact target Eval Config typed reference plus Identity Hash; Evaluation Context role/policy; purpose; run/Step correlation; no score or result.
  • Optimization Step Result: request Object Reference plus Content Hash; proposed/accepted candidates; resolved intent and evaluation or tool-evidence Object References plus Content Hashes; Tool Call Store Entries; state/history delta or snapshot Object Reference plus Content Hash; budget accounting; terminal Step status.
  • Optimization Result: ordered proposals, every ordered Step Result Object Reference plus Content Hash, record-local provenance, and cost; official evaluation follows.

Restart and idempotency

  • A replay resolves the immutable Step Request and any existing Step Result before invoking algorithm code or external effects.
  • The same request and same persisted Step Result replay idempotently; a different Result for the same Step identity conflicts and never replaces the winner.
  • A crash before finalization resumes from the immutable request, typed checkpointed proposal output, state/history, resolved-evaluation Object References plus Content Hashes, and Tool Call Store Entries. It does not rerun a completed proposal invocation or deserialize an opaque mutable optimizer pickle as truth.
  • A later Step or back-edge is forbidden until the preceding Step Result is persisted. Budget state is carried forward from that Result, not recomputed from process memory.

Acceptance Criteria

Identity reset is complete when

  • Recursive validation rejects every unsupported or non-finite value before canonicalization.
  • Only the complete, self-describing, versioned, three-field Identity Document is accepted.
  • Equivalent Identity Documents produce identical Canonical Identity JSON and a 64-character lowercase SHA-256 Identity Hash across repositories.
  • Every affected domain documents and tests its selected payload, schema, and version.
  • All dependent fixtures, golden hashes, keys, and references use the new identities.
  • No alias, legacy hash, truncated identity, or dual-read branch remains.
  • Diagnostic-only hashes are unchanged unless separately justified and reviewed.

Storage integration is complete when

  • An immutable put returns the schema-typed reference with content_hash for the complete canonical record and verifies its Content Hash on write.
  • Every get verifies schema and Content Hash before returning content; corruption and mismatch fail explicitly.
  • Repeated identical puts do not create mutable alternatives, and different content cannot replace an existing reference.
  • Atomic binding exhibits the exact unbound, same, different, and no-overwrite behavior under concurrency.
  • dr-store's public contract contains no Whetstone or workflow concepts.
  • Whetstone persists and uniquely binds terminal Rollout Results while dr-platform transports only the opaque reference.

Implementation Implications

Persisted identities reset

Identity-bearing test data and development state cannot be read as though old and new hashes were equivalent. Regeneration or a clean reseed is part of the coordinated release.

Records can share identity

Two complete records may describe the same domain identity while differing in evidence or provenance. They then share an Identity Hash but have different Content Hashes and Object References.

Policy stays above storage

dr-store guarantees immutable verified content and one-way bindings. Whetstone decides which key may be bound, what constitutes a terminal result, and which evaluation is official.
