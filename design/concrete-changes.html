This page turns the shared vocabulary and definitions into four bounded implementation workstreams: the coordinated dr-serialize identity reset, the new dr-store dependency, durable semantic retry, and coarse provider concurrency control. A cross-repository release sequence makes those contracts usable without unreleased Git revisions. The scope deliberately stops before broader Provider Calls planning: it adds no evaluation-algorithm policy, compatibility path, storage-backend choice, exact request-rate or token limiter, or provider-specific logic to dr-platform.

Decisions at a Glance

Identity Hash

A domain identifier: full SHA-256 over the Canonical Identity JSON derived from an exact, self-describing, versioned Identity Document. The domain selects the identity-bearing payload inside the document; dr-serialize validates, canonicalizes, and hashes it.

Content Hash

A storage-integrity identifier: SHA-256 over the complete canonical persisted record. dr-store uses it to address and verify immutable content; no domain field selection occurs.

Retry Boundary

Whetstone owns bounded automatic Node/provider semantic retries and their durable attempt evidence inside one platform Stage. DBOS replay remains within a Platform Stage Attempt, while explicit operator retry_stage may create a later Platform Stage Attempt only after operational failure and only while the Rollout Execution Key is unbound.

Provider Concurrency Boundary

Whetstone labels rollout work by non-secret provider throttle identity and configures both dr-platform's mandatory default empty-selector Stage capacity and every provider-specific exact-label selector capacity. All matching controls apply. Without the empty-selector control, the Stage is unconfigured and its work remains READY and unadmitted. dr-platform supplies generic coarse admission over those controls; dr-providers supplies or improves the identity. This controls full-Stage occupancy, not exact physical calls, requests per minute, tokens per minute, or token reservation.

These hashes answer different questions. An Identity Hash asks whether two values have the same domain identity under a named schema version. A Content Hash asks whether two complete persisted records are byte-for-byte equivalent after canonicalization. Neither may substitute for the other.

Identity Document contract

{
  "schema": "example.config",
  "schema_version": 1,
  "payload": {"identity_field": "value"}
}

Illustrative values only. The three field names and their presence are exact; each domain owns its actual schema name, version, and payload.

Finite JSON contract

The accepted recursive value set is JSON null, booleans, strings, finite numbers, arrays of accepted values, and objects with string keys and accepted values. Validation does not invoke custom serializers or apply lossy diagnostic normalization.

Canonicalization begins only after validation succeeds, so unsupported runtime values cannot silently collapse onto an identity.

Coordinated Breaking Identity Reset

Adopt the new identity contract as one coordinated break across dr-serialize, dr-graph, dr-code, dr-platform, dr-providers where they expose identity-bearing values, and whetstone-ai. Update producers, consumers, fixtures, and stored test values together.

No compatibility aliases, legacy-hash acceptance, or dual-read paths. Old and new identities are intentionally distinct. Any persisted development data, caches, indexes, or references keyed by replaced identities must be regenerated or cleanly reseeded.

Workstream 2 · New dr-store Dependency

Depends on dr-serializeUse the shared strict JSON validation and canonical JSON implementation. Do not create a second canonicalization dialect.
Typed ObjectReferenceBind a declared record schema to the Content Hash of the complete canonical persisted record.
Immutable putAddress content by its reference, verify the content hash during the write, and never replace different content at an existing reference.
Verified getResolve the exact referenced object and recompute its content hash on every read; reject missing, schema-mismatched, or corrupted content.
Append-only storageRepeated identical puts may be idempotent. Stored records are never edited or deleted through the contract.

Object Reference

{
  "schema": "example.record",
  "content_hash": "<64-character lowercase SHA-256>"
}

The reference carries the declared schema and full Content Hash as one typed value. Consumers may pass it opaquely; dr-store validates both components when resolving it.

The Content Hash covers the complete canonical persisted record—including fields irrelevant to domain identity—so changing provenance or terminal evidence changes the Object Reference even when domain identity stays constant.

Explicit exclusions

  • No Rollout Execution Key or other Whetstone key type
  • No Rollout Result model or evaluation vocabulary
  • No official-evaluation authorization or attestation rules
  • No workflow lifecycle, retry, campaign, stage, or attempt state
  • No speculative backend, retention, replication, or query features

Atomic Key-to-Reference Binding

dr-store provides one generic atomic operation for binding an application-supplied key to an ObjectReference. The key is opaque to dr-store; the caller owns its meaning and validation.

Existing state for keyRequested referenceRequired result
UnboundAny valid stored ObjectReferenceBind Atomically create the binding and return success.
Bound to AASame Return idempotent success without writing a replacement.
Bound to ABConflict Return an explicit different-reference conflict; keep A unchanged.
Bound to AOverwrite, clear, or rebind requestReject The public contract exposes no overwrite path.

Concurrent binds for one previously unbound key must produce exactly one durable winner. Every loser observes either idempotent same-reference success or an explicit different-reference conflict.

Three events, three meanings

  • Provider Attempt or Node Attempt: one Whetstone semantic try with immutable evidence and a durable fence.
  • DBOS replay: recovery of the application Stage body under the same Platform Stage Attempt; it resumes fenced state and creates no duplicate completed semantic attempt.
  • Platform Stage Attempt: the platform-native execution attempt; a later one is created only by explicit operational recovery.

Terminality is result-based

An exhausted semantic failure is expected domain output. When Whetstone produces its causal evidence, persists the terminal Rollout Result, and binds the exact ObjectReference, the Stage is operationally SUCCEEDED.

The Stage is FAILED only when execution cannot produce or persist a terminal Result, or cannot bind or confirm its exact reference. Semantic exhaustion alone is not operational failure.

Operator Retry Gate

A Whetstone operator command or gateway first verifies that the platform Stage is FAILED and the Rollout Execution Key is unbound, then invokes the rebuilt dr-platform retry_stage. The retried Whetstone executor repeats the binding check before any new semantic effect because a concurrent actor may have won after the gateway check.

The Result Store's atomic binding is authoritative. An unbound key may acquire one reference and the same reference may replay idempotently. A different-reference conflict preserves the existing binding; it cannot overwrite the winner, authorize replacement, or turn the conflicting candidate into the terminal Result. An orphan ObjectReference that was persisted but never bound does not itself prohibit operator retry.

Workstream 4 · Provider Concurrency Control

Use the existing generic Stage Control selector-capacity mechanism as a deliberately coarse, conservative cross-worker cap. Whetstone designs the Work Item labels and capacity policy and must configure dr-platform's mandatory default empty-selector capacity for the Stage plus a capacity for every provider-specific exact-label selector. dr-platform enforces every matching control together, and dr-providers exposes or improves the non-secret throttle identity from which those labels are derived.

Throttle identity and labels

  • The default throttle identity normally includes provider or service, endpoint or protocol, model or deployment, and a non-secret credential or account profile.
  • A provider integration may explicitly group configurations by the provider-native quota boundary when its actual quota model differs from that default.
  • Labels and persisted identity material never contain API keys, access tokens, credential values, or other secret material.
  • Work that may route to more than one provider quota carries every relevant label, preventing an alternate route from escaping its configured cap.

Accepted operating constraints

  • Capacity counts concurrently ADMITTED rollout-execution Stages across workers, not observed physical calls.
  • The default empty-selector control is mandatory Stage configuration; without it, work remains READY and unadmitted even if provider-specific controls exist.
  • Admission must satisfy the empty-selector control and every matching provider-specific exact-label control; no matching control replaces or bypasses another.
  • A control is scoped to one pipeline key/version, stage key, selector, and shared database/schema.
  • Admission occupies capacity for the full Stage, including non-provider work and backoff between semantic attempts.
  • Stuck work retains capacity until it becomes terminal or an explicit sweep repairs its platform state.
  • The cap is intentionally conservative; it does not promise strict fairness or perfect quota utilization.
Concurrency non-goals. This workstream does not implement exact physical-call concurrency, requests-per-minute or tokens-per-minute accounting, token reservation, global quotas, cross-stage or cross-version quotas, strict fairness, or adaptive header-driven limiting. Do not add a call-boundary limiter until a real, substantial need demonstrates that coarse Stage admission is insufficient.

Implementation Order

Inventory and freeze domain identity schemas

Classify existing hashes, enumerate identity-bearing dependents, and record the new payload fields, schema names, and versions in each affected repository before changing outputs.

Land and release the dr-serialize contract

Implement strict validation, exact Identity Document construction, canonicalization, and full SHA-256 output with contract tests and golden vectors.

Perform one coordinated identity reset

Update affected domain producers and consumers in dependency order; regenerate all golden values and fixtures; update cross-repository references and indexes together, with no legacy path.

Create dr-store on the stable serialization contract

Add the typed ObjectReference with its content_hash field, complete-record hashing, immutable put, verified get, append-only behavior, and generic atomic key binding. Keep the contract backend-neutral.

Add the Whetstone specialization

Implement the Result Store as the Rollout Execution Key → Rollout Result ObjectReference mapping and move complete result persistence behind it without moving Whetstone policy into dr-store.

Release the bounded dr-providers support

Add and test the non-secret throttle identity/default and the durable caller's ability to enforce native retry zero. Publish the package without adding semantic retry coordination or broader provider-call controls.

Rebuild and release the generic dr-platform boundaries

Verify selector capacity over Work Item labels and the explicit operator retry_stage operation, while keeping provider/node policy out of the platform. Publish the rebuilt version that replaces the superseded revision.

Implement Whetstone durable retry

Add the bounded semantic-attempt coordinator, retry-zero provider execution, durable fences/checkpoints, immutable attempt evidence, terminal exhausted-failure Results, and Result Store binding gate. Add the operator command/gateway and executor-side race recheck around retry_stage.

Configure coarse provider concurrency

Configure dr-platform's mandatory default empty-selector capacity for the Stage. Then derive non-secret throttle labels for every possible route and configure a conservative capacity for every provider-specific exact-label selector. Verify that admission satisfies all matching controls together and that omission of the empty-selector control leaves work READY and unadmitted. Exercise cross-worker full-Stage occupancy and stuck-work behavior in one shared database/schema.

Consume releases and verify end to end

Publish all remaining dependency packages, replace Git revisions with released versions, lock them, and record them in provenance. Then exercise persistence, replay fencing, retry races, binding conflicts, selector capacity, and corruption detection across repository boundaries before real runs.

Acceptance Criteria

Identity reset is complete when

  • Recursive validation rejects every unsupported or non-finite value before canonicalization.
  • Only the complete, self-describing, versioned, three-field Identity Document is accepted.
  • Equivalent Identity Documents produce identical Canonical Identity JSON and a 64-character lowercase SHA-256 Identity Hash across repositories.
  • Every affected domain documents and tests its selected payload, schema, and version.
  • All dependent fixtures, golden hashes, keys, and references use the new identities.
  • No alias, legacy hash, truncated identity, or dual-read branch remains.
  • Diagnostic-only hashes are unchanged unless separately justified and reviewed.

Storage integration is complete when

  • An immutable put returns the schema-typed reference with content_hash for the complete canonical record and verifies its Content Hash on write.
  • Every get verifies schema and Content Hash before returning content; corruption and mismatch fail explicitly.
  • Repeated identical puts do not create mutable alternatives, and different content cannot replace an existing reference.
  • Atomic binding exhibits the exact unbound, same, different, and no-overwrite behavior under concurrency.
  • dr-store's public contract contains no Whetstone or workflow concepts.
  • Whetstone persists and uniquely binds terminal Rollout Results while dr-platform transports only the opaque reference.

Retry implementation is complete when

  • Whetstone bounds and records Node/provider semantic attempts inside a Stage while durable dr-providers execution demonstrably uses native retries of zero.
  • A forced whole-body DBOS replay cannot repeat a semantic attempt whose durable fence/checkpoint and immutable completion evidence already exist.
  • Provider/Node Attempts, replayed Stage invocations, and Platform Stage Attempts retain distinct identities and evidence.
  • The operator gateway rejects retry unless the Stage is FAILED and the Rollout Execution Key is unbound; the retried executor rechecks that the key remains unbound before a new semantic effect.
  • A race that binds after the gateway check is stopped by the executor recheck or authoritative atomic conflict; no different reference overwrites or authorizes replacement.
  • A persisted and bound exhausted-failure Result makes the Stage operationally SUCCEEDED; only inability to produce, persist, bind, or confirm the exact Result leaves it FAILED.
  • Ownership remains explicit: dr-platform supplies generic Platform Stage Attempt mechanics and retry_stage; Whetstone supplies bounded semantic retry policy, attempt fencing, and operator eligibility/race checks; DBOS supplies replay.
  • dr-platform tests contain no provider/node semantic retry classification or policy.

Provider concurrency is complete when

  • Every provider route produces labels from a documented non-secret throttle identity, including provider/service, endpoint/protocol, model/deployment, and credential/account profile by default or an explicit provider-native quota group.
  • Multi-route work carries all relevant labels, and no label or persisted identity exposes secret material.
  • Whetstone configures the mandatory default empty-selector Stage capacity and a capacity for every provider-specific exact-label selector.
  • An admission test proves all matching controls apply together; a missing empty-selector control leaves the Stage unconfigured and its work READY and unadmitted.
  • A cross-worker test against one shared database/schema proves matching ADMITTED Stages never exceed the configured selector capacity within its pipeline key/version, stage key, and selector scope.
  • Tests prove capacity is held for the full Stage and by stuck work until terminal transition or sweep.
  • No call-boundary limiter, exact call/RPM/TPM or token-reservation path, global/cross-stage/cross-version quota, fairness scheduler, or adaptive header policy is introduced.
  • dr-platform treats labels and selectors generically; Whetstone alone defines their provider-quota meaning and capacity policy.

Release integration is complete when

  • dr-serialize, dr-store, dr-providers, the rebuilt dr-platform, and every other runtime dependency repository needed for real runs are public and their required packages are released on PyPI.
  • Whetstone no longer depends on the superseded dr-platform revision or any other Git revision dependency needed by this scope.
  • The locked environment resolves only the intended released package versions, and execution/environment provenance records those exact versions.
  • End-to-end checks prove the four workstreams together without importing broader Provider Calls policy into dr-providers or dr-platform.

Implementation Implications

Persisted identities reset

Identity-bearing test data and development state cannot be read as though old and new hashes were equivalent. Regeneration or a clean reseed is part of the coordinated release.

Records can share identity

Two complete records may describe the same domain identity while differing in evidence or provenance. They then share an Identity Hash but have different Content Hashes and ObjectReferences.

Policy stays above storage

dr-store guarantees immutable verified content and one-way bindings. Whetstone decides which key may be bound, what constitutes a terminal result, and which evaluation is official.
