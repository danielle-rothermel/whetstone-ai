COPRO Run Design

whetstone-ai · 2026-07-20

COPRO runs an in-process proposal loop that rewrites encoder prompts, scores each attempt through its issued tool, and returns ordered rollout configs for official evaluation.

Purple marks what differs from the Eval run; everything unmarked matches it.

Temporary vocabulary — terms this doc needs beyond the vocab page, flagged for review:
  • Proposal LM — the optimizer-owned model that writes new encoder user prompt templates. Needed because COPRO makes its own LM calls, distinct from the encoder and decoder inside a rollout.
  • Breadth / depth — COPRO's two knobs: templates proposed per round, and number of proposal rounds. Needed to name the loop's shape; they travel in guidance.
  • Attempt history — the score-ranked list of (template, tool score) pairs fed back to the proposal LM. Needed because it is COPRO's only state.
  • Tool score — the aggregate a tool call returns to the optimizer: Average Binary Test Pass Rate and mean measured compression under the tool's Score Node Eval Config. Needed to keep what COPRO ranks by distinct from official rollout aggregates.

Mutation Surface and Initial Candidates

  • Mutation surface: {user_prompt_template} of the Encoder LM Node — the only variable the optimizer may change
  • Initial candidates: encoder templates A and B from the eval experiment (full text in the eval doc)
  • A candidate here is one template value; provider and model are fixed within a run

Toolkit and Optimization Budget

  • One tool: evaluate_candidate(template) — 20 rollouts under a cheaper Score Node Eval Config: 20-task HumanEval+ subset · 1 seed · budget ratio 0.5 · same codec, preprocess, and metrics extraction
  • Capacity 12 calls — a constraint of the tool definition
  • Guidance: breadth 4 · depth 2 · return 3

Official Evaluation

  • Same official eval config as the eval experiment: HumanEval+ (164 tasks) @ pinned revision · 3 seeds · zstd-19
  • Selection: official-evaluate top 2 returned candidates, each across the 4 budget ratios

Per pair: 10 tool calls = 200 tool rollouts; official pass = 8 × 164 × 3 = 3,936 rollouts.

Full run walkthrough: one (provider, model) pair, ten tool calls, one ranked return

Designer

Freezes the experiment: the generator expands initial candidates A and B across the 4 budget ratios into 8 initial rollout configs, each carrying the official Score Node Eval Config. The mutation surface, toolkit, guidance, selection policy, and official eval config are serialized alongside. One such run per (provider, model) pair; this walkthrough follows (openrouter, deepseek/deepseek-v4-flash).

Evaluation harness

Issues the OptimizationRequest to the COPRO optimizer: 8 initial rollout configs, mutation surface {user_prompt_template}, toolkit {evaluate_candidate, capacity 12}, guidance (breadth 4, depth 2, return 3).

Optimizer

Baselines the initial candidates: evaluate_candidate(A) and evaluate_candidate(B), 20 rollouts each under the tool's Score Node Eval Config. The attempt history is seeded — A: 0.60 pass / 0.34 compression, B: 0.65 / 0.36.

output40 rollout results keyed by the tool rollout config hashes — proxies by key, never official
Optimizer

Round 1 of 2 — seed proposal: the proposal LM sees templates A and B with their tool scores and returns four new templates P1–P4. Four tool calls score them; the attempt history now ranks 6 attempts.

outputproposal-LM tokens metered as optimization cost; 80 more tool rollout results
Optimizer

One tool call, concretely — evaluate_candidate(P3): the tool sets {user_prompt_template} = P3 in its tool rollout config (20-task subset, 1 seed, ratio 0.5) — a hash distinct from every official rollout config — and runs 20 rollouts through the rollout generation stage. The aggregate comes back as the tool score: 0.70 pass, 0.33 compression.

output20 rollout results under the tool rollout config hash; one (template, tool score) pair appended to the attempt history
Optimizer

Round 2 of 2 — history proposal: all attempts formatted score-ranked ascending, best last, and the proposal LM returns P5–P8. Four more tool calls; the history holds 10 attempts. 10 of the tool's 12-call capacity spent.

outputproposal-LM tokens metered; 80 more tool rollout results — 200 in all
Optimizer

Returns the OptimizationResult: candidates ordered by tool score — P6, P2, B — each expanded across the 4 budget ratios by setting only {user_prompt_template} in the initial rollout configs: 12 rollout configs in all. The full attempt history rides along as advisory provenance, never a score.

outputOptimizationResult serialized under its hash; run cost totals: 10 tool calls, 200 tool rollouts, proposal-LM tokens
Evaluation harness

Diff check: each of the 12 returned rollout configs differs from an initial one only inside the mutation surface — B's four are byte-identical to initial rollout configs, which the surface permits. 12/12 pass. Selection (official-evaluate top 2) admits P6 and P2: 8 rollout configs.

outputattempt history stored as advisory provenance; 8 rollout configs admitted to official evaluation
Evaluator

Submits 3,936 jobs — one per (rollout config, task, seed) — as a dr-platform campaign. Per-model selector capacity keeps the provider throttled independently; whetstone's failure taxonomy retries transient errors and keeps permanent failures as rollout results with a cause.

output3,936 official rollout results — one per (rollout config hash, official eval config hash, task, seed)
Evaluation harness

Aggregates from official rollout results only: per rollout config, per-task mean Binary Test Pass Score over 3 seeds, then unweighted mean over 164 tasks, plus mean measured compression. A and B need no new work — their rollout configs are hash-identical to the eval experiment's, so their official rollout results are reused.

output8 new rollout aggregates — P6 and P2's curves across the ratios, joining the A and B baselines

What lands on the plot: two optimized curves per pair, priced end to end

Per (provider, model) pair, the two surviving candidates land as their own distributions of 4 rollout aggregates across the budget ratios, next to the A and B baseline curves reused from the eval experiment; the pareto front is recomputed from official rollout results only. The 200 tool rollout results stay out of the plot by key — their rollout config hashes carry the tool's Score Node Eval Config — but remain available for post-hoc analysis of how well tool scores predicted official results. Because proposal-LM tokens and every tool rollout were metered, the run has a total price: COPRO's curves can be compared against any other optimizer's on quality and on resources consumed, without COPRO's cooperation.

Additional Notes

Faithful to COPRO; priced by the toolkit

  • Kept, deliberately: greedy ascent, fixed breadth and depth, no convergence check, a score-only signal — COPRO is the baseline optimizer and its simplicity is the point
  • Changed: dspy's COPRO hard-codes full-trainset scoring; here evaluation cost is a property of the issued tool — a minibatch COPRO is the same optimizer handed a different toolkit
  • Candidates and rollout configs are identified by hash, not by object identity

Tool results and official results never mix

  • The tool's cheaper Score Node Eval Config sits inside the tool rollout config hash — disjoint by key from every official rollout config
  • Only the Evaluator's official evaluation produces official rollout results; plots read nothing else
  • Tool scores may inform selection policies and post-hoc analysis — never the plot

Guidance, not enforcement

  • Breadth, depth, and return count travel as guidance — requests the optimizer follows or doesn't
  • Hard constraints live in tool definitions: capacity 12, the 20-task subset, 1 seed
  • After return, the diff check compares each rollout config against the initial ones and the mutation surface

Candidates and the variance source

  • Within a run, a candidate is one {user_prompt_template} value; provider and model are fixed
  • Variance source = budget_ratio, spreading each returned candidate into 4 rollout aggregates — its curve
  • Ordered multi-return: an initial candidate (here B) may be returned; identical rollout configs pass the diff check trivially

Sample holdout is the designer's call

  • Whether the tool's 20-task subset may overlap the official 164 tasks is left to experimental designers
  • The serialized experiment carries the choice either way; this doc does not litigate it

Contract fit

  • OptimizationRequest and OptimizationResult are serializable schemas first: COPRO is called in-process, and the same schemas cross a process boundary for the codex CLI agent
  • Anything beyond the ordered rollout configs — here, the attempt history — is advisory provenance, never a score
