COPRO Optimization Run Design

COPRO is a multi-step Optimization Run: each proposal round is one proposal-only Optimization Step, Whetstone evaluates the returned batch outside the optimizer invocation, and only the completed run returns ordered Optimization Proposals for official evaluation.

Purple marks what differs from the Eval run; everything unmarked matches it.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • Proposal LM — the optimizer-owned model that writes new encoder user prompt templates. It is distinct from the encoder and decoder inside an evaluated Rollout.
  • Breadth — exact number of candidates evaluated at each depth. The seed depth contains breadth−1 generated templates followed by the original template; later depths contain breadth generated templates.
  • Depth — exact number of sequential proposal steps in the Optimization Run.
  • Seed Proposal — depth-0 proposal mode: the Proposal LM receives the one original prompt template and generates breadth−1 alternatives.
  • History Proposal — later-depth proposal mode: the Proposal LM receives up to breadth unique best attempts from prior depths, displayed in increasing score order, and generates breadth alternatives.
  • Attempt History — immutable ordered entries containing stable candidate identity, template, evaluation references, Reward, and record-local provenance. Whetstone creates each new history version after external evaluation; COPRO reads it but never mutates it in place.
DSPy COPRO semantics on Whetstone primitives. The run accepts exactly one original prompt. It preserves DSPy's prompt shape, breadth/depth control flow, defaults, stable score ordering, duplicate handling, and statistics. Whetstone intentionally omits DSPy Signature input/output descriptions and output-prefix optimization: the mutable value is the complete user_prompt_template, and placeholder safety is validated after generation. Model responses are not expected to match byte-for-byte because provider calls are nondeterministic.

Mutation Surface and Original Prompt

  • Mutation surface: {user_prompt_template} of the Encoder LLM Call Node — the only variable the optimizer may change
  • Original prompt: encoder template A from the eval experiment (full text in the eval doc)
  • A candidate here is one template value; the complete encoder and decoder Provider Call Config identities are fixed within a run

Proposal Steps

  • breadth=4 evaluated occurrences per depth: three generated plus original at depth 0, then four generated at depth 1
  • depth=2 steps: round 1 is Seed Proposal; round 2 is History Proposal
  • Each optimizer invocation receives an immutable Attempt History or reference and returns proposals plus an Evaluation Intent
  • COPRO receives no Tool Config, Runtime Tool Handle, or Tool Call Store and performs no evaluation

Official Evaluation

  • Same ordinary Eval Config as the eval experiment, bound through an Evaluation Context carrying the official role: HumanEval+ (164 tasks) @ pinned revision · 3 repeats · zstd-19
  • Selection: official-evaluate top 2 returned candidates, each across the 4 budget ratios

Per Model Route: evaluated_occurrence_count=8 → internal_rollout_count=160; after exact-template deduplication, the complete ranked set contains U≤8 candidates. Official selection still yields D≤8 graph slots and at most 3,936 official Rollouts.

Full Run Walkthrough: Two Depths, Eight Evaluated Occurrences

Designer

Freezes the registry values, starting candidates, mutation surface, ordinary Eval Configs for the step and official-role contexts, Reward Policy, materialization axis, and Selection Policy. No evaluation interface is granted to COPRO.

Whetstone

Issues S1's Optimization Step Request with proposal_mode=seed_proposal, the one original prompt A, empty measured history, the mutation surface, breadth=4, and proposal_lm_config_ref.

COPRO

The Proposal LM returns P1–P3. COPRO appends unchanged A last and emits an Evaluation Intent for each of the four occurrences. The invocation ends without evaluating any candidate.

outputthree generated mutations followed by A; four Evaluation Intents; Proposal LM request/response, usage, cost, and provenance
Whetstone

Validates and materializes the seed batch at step_eval_budget_ratio=0.5, executes step_rollout_count=breadth×step_task_count×step_repeat_count=80, derives Reward, freezes H1, and persists S1. Exact duplicate templates remain separate measured occurrences.

outputfour Materialization Records; 80 Rollout Result refs; four Rewards; four-entry occurrence history H1; immutable S1
Whetstone

Issues S2's Optimization Step Request with proposal_mode=history_proposal and H1. COPRO retains the best observation per exact template, selects up to four unique attempts, and presents them from lower to higher Reward.

COPRO

The Proposal LM reads the selected history and returns P4–P7 plus four Evaluation Intents. It cannot observe results that Whetstone has not yet produced.

outputfour proposed mutations and Proposal LM evidence; no evaluation output
Whetstone

Runs the second external 80-Rollout batch, then creates H2 and S2. All evaluated_occurrence_count=8 observations are now comparable under one step Eval Config and Reward Policy.

outputinternal_rollout_count=80+80=160; eight-entry occurrence history H2; immutable S2
Whetstone

Terminalizes from H2: exact-template duplicates are collapsed, a duplicate replaces its retained observation only when its new Reward is strictly higher, and stable ties preserve first-seen order. COPRO returns the complete unique set in descending Reward order, plus step/history references, total_calls=8, optional statistics, and record-local optimizer provenance.

Official Evaluator

Selects the top two candidates downstream, materializes selected_slot_count=8, deduplicates to D≤8, and executes official_rollout_count=D×official_task_count×official_repeat_count under an authorized Evaluation Context carrying the official role. The illustrative nonconvergent example derives 3,936 Rollouts; the Official Evaluation Record certifies the referenced ordinary Rollout Results.

What Lands on the Plot: Official Results Only

Per Model Route, the downstream Selection Policy chooses official_selected_proposal_count=2 survivors from COPRO's complete ranked unique set. Each retains four separately attributable materialization slots across variance_source_values, while converged slots share the planned/result-key set and Rollout Aggregate for their Graph Hash. The explicit nonconvergent example has D=8. The Pareto Front is recomputed from official results only. The derived internal_rollout_count=160 stays outside the plot because those Results have internal Evaluation Contexts; they remain available to study whether step Reward predicted official Objectives. Proposal LM and Rollout costs remain attributable through the two Step Results.

Additional Notes

Proposal-only means proposal-only

  • A COPRO invocation receives candidates and immutable history and emits proposed mutations plus Evaluation Intent
  • It receives no Tool Config, Runtime Tool Handle, Tool Call Store, Rollout Result, or mutable evaluation service
  • Whetstone alone validates, materializes, evaluates, derives Reward, and creates the immutable Step Result

State crosses explicit boundaries

  • H1 and H2 are immutable occurrence histories; there is no separately evaluated baseline or pre-seed history
  • Depth 0 is Seed Proposal; every later depth is History Proposal
  • Final selection occurs only from the history containing the last externally evaluated batch

DSPy COPRO semantics

  • Exactly one original prompt is appended last to the breadth−1 generated seed proposals
  • generated_candidate_count=breadth×depth−1; evaluated_occurrence_count=breadth×depth
  • Every occurrence is evaluated before exact-template deduplication; there is no early stopping or optimizer-side output truncation

Internal and official measurement stay separate

  • step_eval_config_ref and official_eval_config_ref have independent Eval Config hashes and Evaluation Contexts
  • Their shared Evaluation Procedure Config remains in Graph Config identity; Sampling and Aggregation distinguish measurement regimes
  • Only ordinary Results referenced and certified by an authority-issued Official Evaluation Record are plot-eligible

Config versus derived counts

  • breadth, depth, init_temperature, and track_stats are algorithm hyperparameters; the one original prompt is run input
  • Task sets and repeats belong to Eval Config; requested/selected cardinalities belong to output and Selection Policy
  • Attempts, Rollouts, materializations, selected slots, D, and plot curves are derived rather than separately configurable

Holdout remains explicit experiment design

  • Whether the step_task_count=20 subset overlaps the official_task_count=164 set is pinned by the referenced Sampling Configs
  • Identity keeps the regimes separate, but it does not decide whether overlap is scientifically appropriate
