COPRO rewrites encoder prompts, evaluates attempts through runtime Tool Handles constructed from serialized Tool Specs, and returns three ordered Optimization Proposals for harness-controlled materialization.

Purple marks what differs from the Eval run; everything unmarked matches it.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • Proposal LM — the optimizer-owned model that writes new encoder user prompt templates. Needed because COPRO makes its own LM calls, distinct from the encoder and decoder inside a rollout.
  • Breadth / depth — COPRO's two knobs: templates proposed per round, and number of proposal rounds. Needed to name the loop's shape; they travel in guidance.
  • Attempt history — the Reward-ranked list of (stable proposal ID, template, Reward) entries fed back to the proposal LM. Needed because it is COPRO's only state.

Mutation Surface and Initial Candidates

  • Mutation surface: {user_prompt_template} of the Encoder LLM Call Node — the only variable the optimizer may change
  • Initial candidates: encoder templates A and B from the eval experiment (full text in the eval doc)
  • A candidate here is one template value; the complete encoder and decoder Provider Call Config identities are fixed within a run

Toolkit and Optimization Budget

  • Tool Definition = versioned evaluate_candidate(template) interface; its Tool Spec Config supplies definition ref/version, endpoint identity, internal Eval Config, Reward Policy, capacity 12, timeout/policy refs, ledger namespace, idempotency and refusal rules
  • The Optimization Request serializes that Tool Spec. A non-serializable runtime Tool Handle is constructed from it
  • The durable Whetstone Tool Call Ledger is keyed by tool_spec_hash + call_id and owns acceptance, refusal, completion and capacity accounting across retry/restart
  • Guidance: breadth 4 · depth 2 · return 3

Official Evaluation

  • Same official Eval Config as the eval experiment: HumanEval+ (164 tasks) @ pinned revision · 3 repeats · zstd-19
  • Selection: official-evaluate top 2 returned candidates, each across the 4 budget ratios

Per Model Route: 10 tool calls = 200 tool rollouts; official pass = 8 × 164 × 3 = 3,936 rollouts.

Full run walkthrough: one Model Route, ten tool calls, one ranked return

Designer

Freezes the candidates, axes, mutation surface, concrete Tool Spec, guidance, selection and independent Eval Configs/contexts. The Tool Spec references the versioned Tool Definition and owns Reward Policy plus ledger/capacity/refusal configuration.

Evaluation harness

Issues the OptimizationRequest with the serialized Tool Spec. The harness constructs the non-serializable Tool Handle and binds it to the durable Tool Call Ledger.

Optimizer

Baselines the starting candidates: evaluate_candidate(A) and evaluate_candidate(B), 20 Rollouts each under the tool's internal Evaluation Context. The attempt history is seeded — A: 0.60 pass / 0.34 compression, B: 0.65 / 0.36.

output40 internal Rollout references and two Tool Results
Optimizer

Round 1 of 2 — seed proposal: the proposal LM sees templates A and B with their Rewards and returns four new templates P1–P4. Four tool calls evaluate them; the attempt history now ranks 6 attempts.

outputproposal-LM tokens metered as optimization cost; 80 more tool rollout results
Optimizer

One tool call — the ledger key (tool_spec_hash, copro-r1-p3) is accepted once, consumes capacity once, and records completion/refusal. The Handle runs 20 Rollouts and the Tool Spec's Reward Policy derives Reward 0.70 pass / 0.33 compression. Same-key replay returns the completed Tool Result.

output20 Rollout references, Reward Policy/version, lexicographic Reward, usage, cost and provenance; one history entry
Optimizer

Round 2 of 2 — history proposal: all attempts formatted Reward-ranked ascending, best last, and the proposal LM returns P5–P8. Four more tool calls; the history holds 10 attempts. 10 of the Tool Handle's 12-call capacity spent.

outputproposal-LM tokens metered; 80 more tool rollout results — 200 in all
Optimizer

Returns the OptimizationResult: three proposals ordered by the fixed comparator — maximum pass rate, then minimum compression, then stable proposal ID — P6, P2, B. The full attempt history rides along as advisory provenance, never an official Objective.

outputOptimizationResult serialized under its hash; run cost totals: 10 tool calls, 200 tool rollouts, proposal-LM tokens
Evaluation harness

Diff-checks all three proposals against the mutation surface, then materializes each over four ratios: 12 Rollout Configs. Selection preserves proposal order and admits P6 and P2: 8 Rollout Configs.

outputattempt history stored as advisory provenance; 8 rollout configs admitted to official evaluation
Evaluator

Creates one Platform Campaign and 3,936 Work Items, one per Rollout Execution Key. After Node retries, each terminal Result is immutably persisted to dr-store and atomically bound in the Whetstone Result Store: unbound binds, and same-ref replay is idempotent. Only then does dr-platform return the bound ref as output_reference and mark the Stage SUCCEEDED. Failure to produce or persist the Result, or a binding/storage failure leaving the key unbound, raises FAILED and alone permits operator retry; a different-ref conflict preserves the existing binding, fails the conflicting attempt, and permits no overwrite or retry replacement.

outputauthoritative bound success/exhausted-failure refs plus explicit FAILED Stages; dr-platform does not own uniqueness
Evaluation harness

Aggregates complete official results to Average Binary Test Pass Rate and Mean Compression Ratio derived from integer Compressed Description Length. Reuse requires matching Rollout Keys, official attestation and compatible environment.

output8 new rollout aggregates — P6 and P2's curves across the ratios, joining the A and B baselines

What lands on the plot: two optimized curves per Model Route, priced end to end

Per Model Route, the two surviving candidates land as their own distributions of 4 Rollout Aggregates across the budget ratios, next to eligible A/B baselines; the Pareto front is recomputed from official results only. The 200 tool Rollouts stay out of the plot because their Evaluation Context is internal; they remain available to study how well Reward predicted official Objectives. Proposal-LM tokens and every tool Rollout are metered, so COPRO's curves can be compared on quality and total resources consumed.

Additional Notes

Faithful to COPRO; priced by the toolkit

  • Kept, deliberately: greedy ascent, fixed breadth and depth, no convergence check, a score-only signal — COPRO is the baseline optimizer and its simplicity is the point
  • Changed: dspy's COPRO hard-codes full-trainset scoring; here evaluation cost is a property of the issued tool — a minibatch COPRO is the same optimizer handed a different toolkit
  • Candidates and rollout configs are identified by hash, not by object identity

Tool results and official results never mix

  • Tool and official measurement use independent eval_config_hash and Evaluation Context identities; neither sits inside rollout_config_hash
  • Only the authorized official context and write path confer officialness; plots read nothing else
  • Tool Reward may inform proposal order and selection; official plots use Objectives and Rollout Aggregates

Guidance, not enforcement

  • Breadth, depth, and return count travel as guidance — requests the optimizer follows or doesn't
  • The Tool Definition defines the interface; the Tool Spec configures capacity 12, tasks, policy and ledger rules; the runtime Handle binds services
  • After return, the harness diff-checks proposals and alone materializes concrete Rollout Configs

Candidates and the variance source

  • Within a run, a candidate is one {user_prompt_template} value; provider and model are fixed
  • Variance source = budget_ratio, spreading each returned candidate into 4 rollout aggregates — its curve
  • Ordered multi-return: a starting candidate (here B) may be proposed unchanged; its allowed-diff check passes trivially

Sample holdout is the designer's call

  • Whether the tool's 20-task subset may overlap the official 164 tasks is left to experimental designers
  • The serialized experiment carries the choice either way; this doc does not litigate it

Contract fit

  • OptimizationRequest and OptimizationResult are serializable schemas first: COPRO is called in-process, and the same schemas cross a process boundary for the codex CLI agent
  • Anything beyond the ordered Optimization Proposals — here, the attempt history — is advisory provenance, never an official Objective
