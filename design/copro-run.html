COPRO Optimization Run Design

whetstone-ai · 2026-07-20
Vocab & Defs Concrete Changes Eval COPRO MIPROv2 GEPA Codex CLI agent Quick Test

COPRO is a multi-step Optimization Run: each proposal round is one proposal-only Optimization Step, Whetstone evaluates the returned batch outside the optimizer invocation, and only the completed run returns ordered Optimization Proposals for official evaluation.

Purple marks what differs from the Eval run; everything unmarked matches it.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • Proposal LM — the optimizer-owned model that writes new encoder user prompt templates. It is distinct from the encoder and decoder inside an evaluated Rollout.
  • Breadth — exact number of new candidate templates returned by each proposal step in this Whetstone variant.
  • Depth — exact number of sequential proposal steps in the Optimization Run.
  • Seed Proposal — round-1 proposal mode: the Proposal LM receives the starting candidates and their externally measured entries.
  • History Proposal — later-round proposal mode: the Proposal LM receives the complete Reward-ranked history produced through the preceding step.
  • Attempt History — immutable ordered entries containing stable candidate identity, template, evaluation references, Reward, and provenance. Whetstone creates each new history version after external evaluation; COPRO reads it but never mutates it in place.
Named Whetstone variant. copro_variant=whetstone_multi_seed/v1 defines breadth as new proposals per step and supports multiple evaluated starting candidates. Reference DSPy COPRO counts its original prompt inside the first breadth-sized batch; this variant instead evaluates initial_candidate_count seeds before round 1 and returns exactly breadth new candidates in every round. All counts below follow the Whetstone definition.

Example Setup

One Run per Model Route

Same four routes as the eval experiment, displayed as (Provider Service, requested Model). Complete encoder and decoder Provider Call Config refs/hashes are output-affecting, remain fixed within the run, and contribute to each materialized Rollout Variant identity. The walkthrough follows one run.

  • openrouter, qwen/qwen3-coder-flash
  • openrouter, google/gemini-3.1-flash-lite

Mutation Surface and Initial Candidates

  • Mutation surface: {user_prompt_template} of the Encoder LLM Call Node — the only variable the optimizer may change
  • Initial candidates: encoder templates A and B from the eval experiment (full text in the eval doc)
  • A candidate here is one template value; the complete encoder and decoder Provider Call Config identities are fixed within a run

Proposal Steps

  • breadth=4 new candidates per proposal step
  • depth=2 steps: round 1 is Seed Proposal; round 2 is History Proposal
  • Each optimizer invocation receives an immutable Attempt History or reference and returns proposals plus an Evaluation Intent
  • COPRO receives no Tool Spec, Runtime Tool Handle, or Tool Call Ledger and performs no evaluation

Official Evaluation

  • Same official Eval Config as the eval experiment: HumanEval+ (164 tasks) @ pinned revision · 3 repeats · zstd-19
  • Selection: official-evaluate top 2 returned candidates, each across the 4 budget ratios

Per Model Route: attempt_count=10 externally measured candidates → internal_rollout_count=200; official_rollout_count=D×official_task_count×official_repeat_count, where D≤8. The illustrative nonconvergent case derives 3,936 official Rollouts.

What Lands on the Plot: Official Results Only

Per Model Route, official_selected_proposal_count=2 survivors retain four separately attributable materialization slots across variance_source_values, while converged slots share the planned/result-key set and Rollout Aggregate for their Graph Hash. The explicit nonconvergent example has D=8. The Pareto Front is recomputed from official results only. The derived internal_rollout_count=200 stays outside the plot because those Results have internal Evaluation Contexts; they remain available to study whether step Reward predicted official Objectives. Proposal LM and Rollout costs remain attributable through the two Step Results.

Additional Notes

Proposal-only means proposal-only

  • A COPRO invocation receives candidates and immutable history and emits proposed mutations plus Evaluation Intent
  • It receives no Tool Spec, Runtime Tool Handle, Tool Call Ledger, Rollout Result, or mutable evaluation service
  • Whetstone alone validates, materializes, evaluates, derives Reward, and creates the immutable Step Result

State crosses explicit boundaries

  • H0, H1, and H2 are immutable history versions; S1 and S2 name their exact before/after transition
  • Round 1 is Seed Proposal; every later round is History Proposal
  • Final selection occurs only from the history containing the last externally evaluated batch

Multi-seed COPRO semantics

  • breadth means new candidates per step, not a first batch that includes one original prompt
  • attempt_count=initial_candidate_count+breadth×depth
  • This named variant preserves fixed breadth/depth, Reward-ranked history, greedy final selection, and no convergence check

Internal and official measurement stay separate

  • step_eval_config_ref and official_eval_config_ref have independent Eval Config hashes and Evaluation Contexts
  • Their shared Evaluation Procedure Config remains in Graph Config identity; Sampling and Aggregation distinguish measurement regimes
  • Only official authority and write path produce plot-eligible Results

Config versus derived counts

  • breadth and depth are algorithm hyperparameters; initial_candidates is run input and initial_candidate_count is derived from it
  • Task sets and repeats belong to Eval Config; requested/selected cardinalities belong to output and Selection Policy
  • Attempts, Rollouts, materializations, selected slots, D, and plot curves are derived rather than separately configurable

Holdout remains explicit experiment design

  • Whether the step_task_count=20 subset overlaps the official_task_count=164 set is pinned by the referenced Sampling Configs
  • Identity keeps the regimes separate, but it does not decide whether overlap is scientifically appropriate
