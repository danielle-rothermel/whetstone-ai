COPRO is a deterministic policy: propose from ranked history, spend the tool, return a ranked list

whetstone-ai · system design walkthrough 2 of 5 · 2026-07-19
Eval COPRO MIPROv2 GEPA Codex CLI agent
COPRO is the simplest real optimizer on the contract: a deterministic in-process loop that owns two meta-prompts and nothing else. It proposes encoder-prompt rewrites with a proposal LM, scores every candidate by spending its one eval tool, feeds the score-ranked attempt history back to the proposal LM for a fixed number of rounds, and returns an ordered candidate list. All scores it ranks by are ledger records; all state it keeps is ephemeral loop state. Nothing about the environment changes from the Eval configuration — only the control plane grew a brain.

This configuration: one seed, breadth 4 × depth 2, one unlimited tool

The experiment optimizes one seed treatment's encoder prompt. The mutation surface is a single parameter: the encoder prompt of the encode node — the decoder prompt, models, and graph shape are untouchable. The toolkit holds one eval tool, Eeval (20 fixed HumanEval+ tasks, same encoder/decoder models as certification, unlimited call capacity — faithful to dspy COPRO's full-sample scoring). Guidance asks for breadth 4, depth 2, return 3; the selection policy certifies the top 2 as ranked. Breadth and depth are knobs of this optimizer, not the contract.

Unlike dspy's COPRO, which hard-codes a full-trainset evaluation for every candidate, eval cost here is entirely a property of the tool the designer issues. A minibatch COPRO is the same optimizer handed a different toolkit config — that substitution without code change is the whole point of toolkit-as-experiment-design.

What lands on the plot: two certified points, and a fully priced run

Only the two certified candidates reach the correctness–compression plot. The 240 Eeval per-task records stay in the ledger as proxy measurements — excluded from the plot by config digest, but available to selection policies and to post-hoc analysis of how well tool scores predicted certified scores. Because proposal-LM tokens and all 12 tool calls were metered, this run has a total price: COPRO's certified front can be compared against any other optimizer's on quality and on resources consumed, without COPRO having cooperated in any way.

What changed from dspy's COPRO: full-trainset evaluation is no longer hard-coded — the tool defines eval cost, so minibatch COPRO is a toolkit swap, not a rewrite; deepcopy-per-candidate and id()-keyed bookkeeping are replaced by treatment digests; and spend is visible in the ledger instead of invisible inside the loop. What stayed, deliberately: greedy coordinate ascent, a fixed depth with no convergence check, and a score-only signal — COPRO is the baseline optimizer, and its simplicity is the point.
