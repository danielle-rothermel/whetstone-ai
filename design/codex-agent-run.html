Temporary vocabulary — terms this doc needs beyond the vocab page:
  • bridge — the transport that exposes the toolkit to a process we don't own and carries tool calls and results across the process boundary (MCP in this example). Needed because this is the only doc where the contract leaves our process.
  • tool capacity — the number of calls a tool definition holds; the call after the last receives a structured refusal from the tool itself. Needed to state the optimization budget as tool design — guidance, not enforcement.
  • control cost — the optimizer's own resource use (agent tokens, wall clock), reported when the agent exits. Advisory metadata for comparing optimizers, never a score. Needed because an agentic optimizer has overhead its tool calls don't show.

Example Setup

Reused from the Eval run

  • Same encoder → decoder → score architecture
  • Same 4 (provider, model) pairs
  • Variance source: budget_ratio = 0.25, 0.5, 0.75, 1.0
  • HumanEval+ (164 tasks) @ pinned revision · 3 seeds · zstd-19
  • Same official eval config

Mutation surface & initial candidates

  • Mutation surface: the encoder {user_prompt_template} — nothing else may change
  • Initial candidates: encoder templates A and B, carried inside the Eval run's 32 initial rollout configs

Toolkit: one tool

evaluate_candidate — full definition below.

  • Input: one proposed encoder user prompt template
  • Runs it on all 4 pairs at ratio 0.5 under a cheaper Score Node Eval Config: 20 tasks, 1 seed
  • Capacity: 20 calls

Budget, guidance, selection

  • Optimization budget = the tool's capacity: 20 calls ≈ 3,200 LLM calls, ~10% of the official pass
  • Guidance: "return your best 3 encoder user prompt templates, ordered"
  • Selection: official-evaluate the top 2 templates

Designer

Freezes the experiment: 32 initial rollout configs from the Eval generator, mutation surface = encoder {user_prompt_template}, the evaluate_candidate tool definition (capacity 20, 20-task subset, 1 seed, ratio 0.5, all 4 pairs), guidance "return your best 3 encoder user prompt templates, ordered", selection = official-evaluate top 2, and the official eval config — identical to the Eval run's. Everything serialized and hashed.

Evaluation harness

The agent exits; the harness captures the gpt-5.6 conversation's token usage as control cost. This run is now comparable to a COPRO or GEPA run on total-resources-consumed axes: tool-call cost itemized per rollout, control cost from the harness.

Evaluation harness

Diff check: all returned rollout configs match their initial counterparts everywhere outside the encoder {user_prompt_template} — consistent with the mutation surface. Evaluator

Submits 15,744 jobs — one per (rollout config, task, seed) — as a dr-platform campaign, exactly as in the Eval run: per-model selector capacity, failure taxonomy, retries. Failures land as rollout results with a cause, never gaps.

Aggregates from official rollout results only: per rollout config, Average Binary Test Pass Rate and mean measured compression. What lands on the plot: two proposed templates against the A/B baselines

The two winning templates land as 8 candidates — (template, provider, model) — each a distribution of 4 rollout aggregates across the budget ratios: Average Binary Test Pass Rate vs mean measured compression. Because the official eval config hash is identical to the Eval run's, the template A/B baselines overlay directly — no re-running, the existing official rollout results already carry the same keys. The natural readings: proposed vs initial templates at matched model and ratio, and each proposal's budget-response curve. The agent's self-ordering sits in provenance, available to compare against the official ordering — but it never touches the plot.

Additional Notes

The contract crosses the boundary unchanged

  • Contracts are serializable schemas first: COPRO, MIPROv2, and GEPA call them in-process; here the same bytes cross a process boundary
  • Request out as a prompt plus files; result back as a JSON file parsed against the schema; tools over a bridge
  • Nothing downstream changes: the diff check, selection, Evaluator, and analysis cannot tell the transport apart

Guidance, not enforcement

  • Constraints live in tool definitions: the capacity is what the tool is, not a policy layer around the agent
  • The requested count ("best 3") is guidance; over-return is advisory provenance about the agent
  • A different experiment can issue a 5-call tool, an unlimited tool, or two tools with different task subsets — tool design is experimental design

Cheap evaluation vs official evaluation

  • The tool's rollouts run under a cheaper Score Node Eval Config — smaller task set, 1 seed, same preprocess and metrics extraction — carried inside those rollout configs' hashes
  • Only the harness's official evaluation produces official rollout results; plots read only those
  • The agent may evaluate however its tools allow — "evaluate" alone is the general verb

Budget and cost accounting

  • Optimization budget = tool capacity: 20 calls × 80 rollouts ≈ 3,200 LLM calls vs ≈ 31,500 in the official pass
  • Control cost (agent tokens, wall clock) is captured at exit — advisory metadata for comparing optimizers, never a score

Sample holdout is the designer's call

  • Whether the tool's 20-task subset may contain official-evaluation tasks is left to experimental designers — the question is sharpest in this doc, and this design does not litigate it
  • The hashes keep the two measurement regimes separate either way
