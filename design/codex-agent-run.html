Vocab & Defs Concrete Changes Eval COPRO MIPROv2 GEPA Codex CLI agent Quick Test

One Optimization Run contains exactly one opaque, long-running, tool-using Optimization Step. Its serializable Optimization Step Request crosses into the Codex CLI process; one Optimization Step Result returns the final proposals and evidence. Subsequent official evaluation remains authoritative.

Purple marks what this run adds or changes relative to the Eval run; everything unmarked matches it.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • bridge — the transport that exposes the toolkit to a process we don't own and carries tool calls and results across the process boundary (MCP in this example). Needed because this is the only doc where the contract leaves our process.
  • control cost — the optimizer's own resource use (agent tokens, wall clock), reported when the agent exits. Advisory metadata for comparing optimizers, never a score. Needed because an agentic optimizer has overhead its tool calls don't show.

Mutation surface & initial candidates

  • Mutation surface: the encoder {user_prompt_template} — nothing else may change
  • Starting Experiment Candidates: templates A and B across four Model Routes; fixed ratio axes remain harness-owned

Toolkit: one tool

evaluate_candidate — full definition below.

  • Input: one named Model Route plus one proposed encoder user prompt template
  • Runs that route at budget_ratio=0.5 under a cheaper internal Eval Config: internal_task_count=20 pinned tasks and internal_repeat_count=1
  • Tool Capacity: max_evaluation_calls=20 accepted unique call identities maximum; the agent may stop after any count from 0 through 20

Limit, output contract, selection

  • max_evaluation_calls=20 limits accepted unique identities; it does not require the agent to call the tool. Derived ceilings are max_internal_rollout_count=400 and max_graph_lm_call_count=800
  • Output contract: returned_proposal_count=4 ordered Optimization Proposals, one per Model Route
  • Selection: official-evaluate all 4 valid Optimization Proposals

Full Run Walkthrough: One Step with observed_tool_call_count=18

Designer

Freezes candidates, axes, mutation surface, output contract, process configuration, and one concrete Tool Spec referencing the versioned evaluate_candidate Definition. The Spec owns internal measurement, Reward, endpoint, ledger/refusal policy, and max_evaluation_calls=20.

Evaluation harness

Creates the Run's only Optimization Step Request and spawns Codex CLI. The rendered Request crosses the process boundary with the serialized Tool Spec, while the bridge exposes the non-serializable Runtime Tool Handle bound to its durable Tool Call Ledger.

Illustrative observations inside the one opaque Step. These are sequential evidence summaries, not phases configured by Whetstone.

Optimizer

Illustrative observed probe: drafts six differently shaped encoder templates — terse spec, pseudocode, invariants-first — and calls evaluate_candidate on each to map the space. The post-run label and observed_probe_call_count=6 are evidence summaries, not an exposed phase or parameter.

output6 × 20 internal Rollout Results referenced by Tool Results; 6 accepted unique identities of the 20 maximum
Optimizer

Illustrative observed refinement: iterates on two promising directions — tightening wording, trading description length against decoder success — with observed_refine_call_count=9 more accepted unique calls. How it chose these variants is invisible to, and irrelevant to, the contract.

output9 more calls' Rollout Results; 15 accepted unique identities cumulatively
Optimizer

Illustrative observed confirmation: re-runs final Optimization Proposals with new call IDs, then elects to stop after observed_confirm_call_count=3 more calls. The ledger returns completed Results for identical keys, would refuse a 21st accepted unique identity, and restores accounting after bridge restart. Two available slots remain unused by agent choice.

Optimizer

Writes the one Optimization Step Result with exactly returned_proposal_count=4 ordered Optimization Proposals. Each has a stable proposal ID and binds one of the four Model Routes plus its proposed template; no route is duplicated or omitted. Annotations are advisory provenance, never Reward or an official Objective.

outputserialized Step Result with final proposals and Tool Call/Result references, parsed against the shared schema
Evaluation harness

The agent exits; the harness validates and commits the immutable Step Result, attaches agent conversation evidence and token usage as control cost, and terminalizes the one-Step Optimization Run. This run is now comparable to COPRO or GEPA on total resources consumed.

outputcontrol cost: agent tokens + wall clock; 18 Tool Calls' costs already itemized
Evaluation harness

Validates all returned_proposal_count=4 Optimization Proposals as distinct allowed bases with template-only mutations, preserves agent order, then expands them to official_materialization_count=16 Graph Config/Materialization Record pairs. The evaluate-all Selection Policy selects all 16 ordered record slots, deduplicates the selected records by full Graph Hash, and admits D≤16 graphs. Official Evaluation Records and Official Plot Manifests MUST preserve the ordered mapping from every selected Materialization Record reference through graph_hash to the shared planned/result-key set and aggregate reference, including when multiple selected records share one graph and aggregate.

Evaluator

Submits exactly official_rollout_count=D×official_task_count×official_repeat_count Rollout Work Items with no duplicate Rollout Execution Keys; the illustrative nonconvergent example has D=16 and derives 7,872. Every logical provider attempt runs in one @DBOS.step(retries_allowed=False) with native retries zero; DBOS checkpoints completed typed observations and replay reconstructs Node Outcomes and the Graph Run Result. dr-store persists one immutable Rollout Result containing that nested evidence, not an attempt ledger or a separate Graph Run Result, and atomic binding remains authoritative.

outputauthoritative bound refs or explicit FAILED Stages; dr-platform does not own uniqueness
Evaluation harness

Aggregates complete official Rollout Results only. The illustrative observed_internal_rollout_count=360 internal Results remain queryable but lack official authority. Exhausted infrastructure failures keep aggregates incomplete; deliberate re-evaluation needs a new context or Repeat.

outputexactly D Rollout Aggregates by distinct Graph Hash; 16 in the explicit nonconvergent example

Additional Notes

The single-Step contract crosses the boundary

  • The Optimization Run contains exactly one opaque, long-running, tool-using Optimization Step; no intermediate Step boundary is inferred from Tool Calls
  • Optimization Step Request out as a prompt plus files; serialized Optimization Step Result with final proposals/evidence back as JSON; tools over a bridge
  • Nothing downstream changes: the diff check, selection, Evaluator, and analysis cannot tell the transport apart

Tool limit and output enforcement

  • Tool Definition defines interface; Tool Spec configures policy; Tool Call Ledger owns call/Tool Capacity state; Runtime Tool Handle binds services
  • max_evaluation_calls=20 allows 0 through 20 accepted unique identities; it is a ceiling, not a target. Same-ID replay consumes no new slot, and the 21st new identity is refused
  • The required shape is returned_proposal_count=4 base-bound proposals; schema validation rejects duplicate/missing bases, while qualitative strategy remains opaque
  • The observed 6/9/3 probe/refine/confirm split is only an illustrative post-run interpretation, never a state machine or hyperparameter

Internal Evaluation vs Official Evaluation

  • The tool's Rollouts bind a cheaper Eval Config — smaller task set, 1 repeat, same facts-first Evaluation Procedure Config. The shared procedure assignment is in Graph Config identity, so matching internal and official variants retain the same graph_hash
  • Only the authorized official Evaluation Context/write path confers officialness; equal hashes never promote internal Rollout Results
  • The agent may evaluate however its tools allow — "evaluate" alone is the general verb

Maximums, observations, and cost

  • max_internal_rollout_count=400 and max_graph_lm_call_count=800 are derived ceilings, not promised usage; this walkthrough observes observed_tool_call_count=18 and observed_internal_rollout_count=360
  • official_rollout_count=D×official_task_count×official_repeat_count; the illustrative nonconvergent case has D=16 and derives 7,872 official Rollouts
  • Control cost (agent tokens, wall clock) is captured at exit — advisory metadata for comparing optimizers, never a score

Sample holdout is the designer's call

  • Whether the tool's internal_task_count=20 subset may contain official-evaluation tasks is left to experimental designers — the question is sharpest in this doc, and this design does not litigate it
  • The hashes keep the two measurement regimes separate either way
