This configuration: one seed, one tool, an opaque loop

The experiment optimizes one seed treatment's encoder prompt. The designer issues a single tool, fully specified as data: evaluate_prompt runs a candidate spec over a 10-task sample with a cheaper proxy decoder model than certification uses — deliberately different from the experiment's own dataset and models, because the tool's design is itself part of the experimental design. Guidance asks the agent to return its best 3, ranked; selection certifies the top 2 as ranked under Ccert. Everything between receiving the request and writing the result is the agent's business.

The 20-call capacity is not budget enforcement. Nothing is enforced on the agent — the designer simply provided a tool that holds 20 evaluations, the same way a different experiment might provide a 5-call tool, an unlimited tool, or two tools with different proxies. Call 21 receives a structured refusal because that is what this tool is, not because a policy layer intervened. Constraints live in tool design; guidance lives in the request; the experiment decides both.

Full run walkthrough: 18 observed calls around an unobserved loop

Designer

Freezes the experiment config: seed spec, mutation surface = encoder prompt, the evaluate_prompt tool spec (capacity 20, 10-task proxy sample, cheap decoder model), guidance "return your best 3, ranked", selection = certify top 2 as ranked, Ccert. Everything serialized and digested.

ledgerexperiment config, seed treatment digest, and tool-config digest registered
Harness

Spawns the codex CLI with the gpt-5.6 agent. The OptimizationRequest is rendered as a prompt plus files — seed spec JSON, mutation-surface declaration, guidance — and the MCP bridge is connected, exposing exactly one tool.

Agent's inner loop — structure unknown, calls observed
Agent

Probe (inferred): drafts six differently-shaped encoder prompts — terse spec, pseudocode, invariants-first — and calls evaluate_prompt on each to map the space.

ledger6 proxy records tagged with the tool-config digest; capacity 6/20
Agent

Refine (inferred): iterates on the two most promising directions — tightening wording, trading description length against decoder success — spending nine more calls. How it chose these variants is invisible and irrelevant to the contract.

ledger9 proxy records; capacity 15/20
Agent

Confirm (inferred): re-runs its final three candidates on the sample to settle its ranking. Stops at 18 of 20 calls — under capacity, by its own choice. A 21st call would have received the tool's structured refusal.

ledger3 proxy records; capacity 18/20, final
Agent

Writes the OptimizationResult JSON file: four ordered candidate specs — one more than the guidance asked — plus annotations: two untested variants it considered, its rationale, and its self-ranking. The over-return is not an error; it is recorded as data about this agent.

ledgerresult ingested; over-return noted; annotations stored as provenance, never as scores
Harness

Agent exits; the harness reports the gpt-5.6 conversation's token usage. This run is now comparable to a COPRO or GEPA run on total-resources-consumed axes: proxy eval cost from the ledger, control-plane cost from the harness.

ledgercontrol-plane cost recorded: agent tokens + 18 tool-call costs already itemized
Environment

Diff-validates all four returned specs against the seed: only the encoder prompt parameter differs — consistent with the mutation surface. Selection applies the recorded policy: top 2 as ranked. (A different experiment could select using the ledger's proxy scores, or the compression axis — free to compute on all four descriptions — without a new contract.)

Certifier

Runs the two selected candidates × 40 tasks under Ccert — full decoder model, the experiment's own dataset. Failures classified by the whetstone taxonomy; transient ones retry.

ledger80 certified per-task records tagged (Ti, Ccert): pass/fail, compressed bytes, ground-truth bytes, tokens, provider cost
Environment

Aggregates from certified records only and emits the report. The 18 proxy records remain in the ledger — queryable, cost-accounted, and excluded from the plot by construction because they carry the tool-config digest, not Ccert's.

ledger2 certified aggregates; pareto contribution derived as a view

What lands on the plot: two certified points from an unowned loop

Two certified points join the correctness–compression plot, each traceable through the ledger to the run that produced them: 18 proxy evaluations under one tool-config digest, one opaque control plane with its token cost on record, one selection decision applied as configured. The agent's self-assessments and untested variants sit in provenance — available to seed a future run's proposals, structurally incapable of contaminating the plot.

This configuration is the reason the contracts are data. Nothing here required new machinery — only a new transport (a bridge instead of a function call) and a designer-specified tool. The same rails carry the distant-future control plane that edits graph structure: widen the mutation surface declaration, and the diff validator, selection, certification, and ledger already know what to do.
