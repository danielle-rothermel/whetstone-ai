The same serializable Optimization Request and Optimization Result cross a process boundary to an unobserved Codex CLI agent loop, while official evaluation remains authoritative.

Purple marks what this run adds or changes relative to the Eval run; everything unmarked matches it.

Algorithm-local vocabulary — defined here because these terms are specific to this optimizer, not shared Whetstone ontology terms:
  • bridge — the transport that exposes the toolkit to a process we don't own and carries tool calls and results across the process boundary (MCP in this example). Needed because this is the only doc where the contract leaves our process.
  • tool capacity — the Tool Spec configures the limit; the durable Whetstone Tool Call Ledger owns acceptance/refusal/completion and accounting, while the runtime Tool Handle is only a non-serializable binding.
  • control cost — the optimizer's own resource use (agent tokens, wall clock), reported when the agent exits. Advisory metadata for comparing optimizers, never a score. Needed because an agentic optimizer has overhead its tool calls don't show.

Reused from the Eval run

  • Same dr-graph Graph Definition used by Whetstone in the Rollout Definition role: LLM Call Node → LLM Call Node → Eval Node
  • Same 4 Model Routes, displayed as (Provider Service, requested Model) while complete output-affecting Provider Call Config refs/hashes contribute to Rollout Variant identity
  • Variance source: budget_ratio = 0.25, 0.5, 0.75, 1.0
  • HumanEval+ (164 tasks) @ pinned revision · 3 repeats · zstd-19
  • Same Official Eval Config

Mutation surface & initial candidates

  • Mutation surface: the encoder {user_prompt_template} — nothing else may change
  • Starting Experiment Candidates: templates A and B across four Model Routes; fixed ratio axes remain harness-owned

Toolkit: one tool

evaluate_candidate — full definition below.

  • Input: one named Model Route plus one proposed encoder user prompt template
  • Runs that route at ratio 0.5 under a cheaper internal Eval Config: 20 pinned tasks, 1 repeat
  • Capacity: 20 calls

Budget, guidance, selection

  • Optimization budget = the Tool Handle's capacity: 20 calls = 400 Rollouts ≈ 800 graph LLM calls
  • Guidance: "return exactly 4 ordered proposals, one per Model Route"
  • Selection: official-evaluate all 4 valid proposals

The agent exits; the harness captures the gpt-5.6 conversation's token usage as control cost. This run is now comparable to a COPRO or GEPA run on total-resources-consumed axes: tool-call cost itemized per rollout, control cost from the harness.

Evaluation harness

Validates all four proposals as distinct allowed bases with template-only mutations, preserves agent order, then expands every valid proposal over four ratios → 16 Graph Config outputs and 16 Materialization Records. Evaluate-all selects all 16 ordered record slots, deduplicates the selected records by full Graph Hash, and admits D ≤ 16 graphs. Official Evaluation Records and Official Plot Manifests MUST preserve the ordered mapping from every selected Materialization Record reference through graph_hash to the shared planned/result-key set and aggregate reference, including when multiple selected records share one graph and aggregate.

Evaluator

Submits exactly D × 164 × 3 Work Items with no duplicate Rollout Execution Keys; the explicit nonconvergent example has D=16 and 7,872. Every logical provider attempt runs in one @DBOS.step(retries_allowed=False) with native retries zero; DBOS checkpoints completed typed observations and replay reconstructs Node Outcomes and the Graph Run Result. dr-store persists one immutable Rollout Result containing that nested evidence, not an attempt ledger or a separate Graph Run Result, and atomic binding remains authoritative.

outputauthoritative bound refs or explicit FAILED Stages; dr-platform does not own uniqueness
Evaluation harness

Aggregates complete official results only. The 360 internal results remain queryable but lack official authority. Exhausted infrastructure failures keep aggregates incomplete; deliberate re-evaluation needs a new context or Repeat.

Additional Notes

The contract crosses the boundary unchanged

  • Contracts are serializable schemas first: COPRO, MIPROv2, and GEPA call them in-process; here the same bytes cross a process boundary
  • Request out as a prompt plus files; result back as a JSON file parsed against the schema; tools over a bridge
  • Nothing downstream changes: the diff check, selection, Evaluator, and analysis cannot tell the transport apart

Guidance, not enforcement

  • Tool Definition defines interface; Tool Spec configures policy; Tool Call Ledger owns call/capacity state; runtime Handle binds services
  • The requested shape is exactly four base-bound proposals; schema validation rejects duplicate/missing bases, while qualitative strategy remains guidance
  • A different experiment can issue a 5-call tool, an unlimited tool, or two tools with different task subsets — tool design is experimental design

Internal Evaluation vs Official Evaluation

  • The tool's Rollouts bind a cheaper Eval Config — smaller task set, 1 repeat, same facts-first Evaluation Procedure Config. The shared procedure assignment is in Graph Config identity, so matching internal and official variants retain the same graph_hash
  • Only the authorized official Evaluation Context/write path confers officialness; equal hashes never promote internal results
  • The agent may evaluate however its tools allow — "evaluate" alone is the general verb

Budget and cost accounting

  • Optimization budget = Tool Handle capacity: 20 calls × 20 Rollouts × 2 graph LM calls ≈ 800 LLM calls vs D × 164 × 3 Rollouts in the official pass; the explicit nonconvergent example has D=16 and 7,872
  • Control cost (agent tokens, wall clock) is captured at exit — advisory metadata for comparing optimizers, never a score

Sample holdout is the designer's call

  • Whether the tool's 20-task subset may contain official-evaluation tasks is left to experimental designers — the question is sharpest in this doc, and this design does not litigate it
  • The hashes keep the two measurement regimes separate either way
