Quick-Test Task Rubric

whetstone-ai · 2026-07-21
Vocab & Defs Concrete Changes Eval COPRO MIPROv2 GEPA Codex CLI agent Quick Test

Criteria for choosing a small, cheap task family and pinned dataset that validates the optimizer implementations and shared execution contract before the full HumanEval compression experiment. This is a distinct lightweight Eval Definition, not a weakened mode of the code-generation Eval Definition.

Goal

Pick one task that every run design — Eval (identity), COPRO, MIPROv2, GEPA, and the Codex CLI agent — can execute end to end for cents per run, in minutes, and that produces an unambiguous signal about whether each optimizer works and which hyperparameters are reasonable.

  • Validate the full contract path: Optimization Request with Tool Specs → runtime Handles/Tool Call Ledgers → ordered Optimization Proposals → validate all proposals → expand every valid proposal over fixed axes → materialize every complete slot → Selection Policy chooses an ordered record subset → full-graph_hash deduplication → task/repeat expansion → authorized evaluation.
  • Support hyperparameter selection: breadth/depth, minibatch size, promotion cadence, iteration count.
  • Serve as the debugging target before any full-scale run spends real money.
Minimum concrete definition. Pin a generated task-family schema, generator version, dataset manifest and split identities. Declare one dr-graph Graph Definition, used by Whetstone in the Rollout Definition role—not a second artifact or identity—with one LLM Call Node (Config) and one Eval Node (Config). Each complete output-affecting assignment and axis selection emits one concrete Graph Config plus one Materialization Record. The Graph Config is one Rollout Variant, identified solely by graph_hash. The Eval Node statically references an Evaluation Procedure Config included in the Graph Config Identity Document; every Eval Config remains Sampling + Evaluation Procedure + Aggregation and must reference that identical Procedure Config. Internal and official evals using the same procedure retain the same graph_hash for matching Graph Configs: changing Procedure Config changes both graph_hash and eval_config_hash, while changing only Sampling/Aggregation changes eval_config_hash. Provider Call Config is output-affecting and contributes to Graph Config identity; Provider Call Request references it plus Transcript and materializes native LlmRequest without duplicate controls. Static Config references are not runtime Node Input Sources. Only Whetstone-accepted nonblank response text becomes Generation. Task values enter as Graph External Inputs and remain outside graph identity; an Evaluation Context binds authority and realized-environment attestation.

Rationale

  • The full experiment is ≈15,744 rollouts and ≈31,500 LLM calls per pass, plus sandboxed test execution per rollout — too expensive and slow to debug against.
  • Its two-model graph and noisy scores make it hard to tell a working optimizer from a lucky one, and impossible to compare hyperparameters cheaply.
  • A cheap task makes optimization runs iterable: many runs per day, so implementation bugs and bad hyperparameters surface early.

Core criteria

The first three make the test cheap. The fourth makes it informative — it is what separates a working optimizer from a broken one, and good hyperparameters from bad ones.

1 · The graph is one LLM call node plus one eval node

  • No encoder → decoder chain; one LLM Call Node's accepted nonblank native response becomes a Generation/Submission Text, and one Eval Node records Metric Facts/Records before reduction. The simple Node is semantically a Config; node_id is graph-local and execution addresses are (graph_hash, node_id), with no separate Node hash.
  • Each Node Input Source is exactly a Graph External Input or an upstream Node Output. Each Node Output is a validated declared output object.
  • The Eval Node is the graph's exactly one sink and has the terminal role, not a special Node Type. Its ordinary Node Output becomes Graph Output unchanged on success; there is no special terminal-output reference or shape.
  • Halves the LLM calls per rollout and removes one model's noise from the signal.

Why. The contract machinery — mutation surface, tools, diff check, selection — does not depend on graph depth. The smallest graph that exercises it is enough.

2 · Scoring is an exact check by a simple function

  • Correctness is a deterministic comparison — e.g. two numbers match, or the answer equals the labeled choice.
  • No sandbox, code-test suite, or candidate-extraction preprocessing. Each example's exact rule returns 1 or 0; there is no per-example partial credit.

Why. Removes the most expensive and failure-prone parts of evaluation, and makes every score trustworthy — a wrong aggregate can only come from the machinery under test.

3 · Token cost per rollout is small and bounded

  • Short inputs; outputs constrained to multiple choice or a few words.
  • Per-rollout cost is predictable, so run budgets can be set tightly.

Why. Bounded output length caps the worst-case spend. Cheap rollouts are what allow many optimization runs, larger task pools, and repeat runs for comparing hyperparameters.

4 · Default prompts score low, and better prompts close the gap in steps

  • Models' default behavior sits well below the ceiling, so there is room to improve.
  • Prompt changes improve the score incrementally — the best prompt is not one-shot guessable.

Why. If any reasonable prompt hits the ceiling, every optimizer looks identical and the run validates nothing. A gradual path is what lets runs rank optimizers and hyperparameters.

Additional criteria

5 · Runs are deterministic, so prompt differences resolve at small sample sizes

  • Decode at temperature 0 (or the provider's closest equivalent); pin sampled task identities and record any sampler RNG seed as metadata.
  • Per-task behavior is then as close to deterministic given a prompt as the provider allows.
  • The effect size of a meaningful prompt change (≥10 points) exceeds the residual noise on a 10–20 task internal eval.
  • Distinct repeat_id values still run (e.g. 3) to validate the plumbing; with determinism they should largely agree. Retry attempt numbers never substitute for repeats.

Why. Optimizers rank proposals with tiny internal evaluations (3–20 tasks, 1 repeat). If noise swamps the effect, internal rankings are garbage and hyperparameter comparisons are meaningless. Determinism is a deliberate choice: score differences should come from prompts, not sampling.

6 · Internal rewards predict official objectives

  • Check the rank correlation between Reward values from internal Evaluation Contexts and official Objective aggregates for the same materialized Graph Hashes.

Why. Optimizers may steer by a Reward Policy while official selection reads Objectives. The quick test is the first cheap chance to measure whether that proxy is predictive.

7 · The task pool is large and generated with known ground truth

  • Synthetic, parameterized tasks generated by a pinned program, each with an exact label and stable task identity.
  • Hundreds of tasks: disjoint minibatch / subset / official splits, per-iteration minibatch sampling (GEPA), demo bootstrapping (MIPROv2).

Why. A big pool exercises the task-set and holdout machinery for real, and generation puts the difficulty under the designer's control.

8 · Models cannot already know the answers

  • Randomized synthetic instances rather than a published benchmark — resistant to data contamination.

Why. Memorized answers erase the headroom criterion 4 depends on, and do so unevenly across models — the test would measure recall, not optimization.

9 · The ceiling, floor, and a reference prompt are known in advance

  • The designer can write a prompt that scores ≈100%, and knows what naive prompts score.
  • Runs get a falsifiable success criterion: e.g. "recover ≥90% of the known gap within 8 iterations."

Why. Without a known ceiling, "the score went up" is the only conclusion available. With one, a run can pass or fail.

10 · Difficulty decomposes into independent rules

  • The task is governed by several latent rules; each rule a prompt captures adds score.
  • Which rules a returned prompt discovered is inspectable after the run.

Why. This is the mechanism behind criterion 4's incremental path: partial knowledge earns partial score. It also shows what each optimizer found, not just how much.

11 · Failures are diagnostic enough to guide reflection

  • Exact-match evaluation yields "expected X, got Y"; the error pattern should reveal which rule was violated.

Why. GEPA's mechanism is a reflection LM reading rollout traces. Interpretable failures test that mechanism; opaque failures test only its plumbing.

12 · Few-shot examples measurably help

  • Input → output pairs are compact enough to include cheaply, and some latent rules are learnable from examples.

Why. Demo sets are one axis of MIPROv2's grid. If examples cannot help, that axis is dead and MIPROv2 degenerates to instruction search.

13 · Failure paths get exercised on purpose

  • The output format is occasionally violated in practice, or failures are injected deliberately.
  • The Evaluation Procedure Config assigned on the Eval Node derives each exact 0/1 observation; the Eval Config references that identical Procedure Config and Aggregation only reduces. Exhausted infrastructure-failure Results make official aggregation incomplete, never silently zero.
  • Optimization Request serializes Tool Specs, not Tool Definitions or Handles. Each Spec references a versioned Tool Definition and configures endpoint, internal Eval Config, Reward Policy, capacity/timeout/policies, ledger namespace, idempotency and refusal.
  • A non-serializable runtime Handle binds the Spec. The durable Whetstone Tool Call Ledger keyed by tool_spec_hash + call_id owns acceptance/refusal/completion and capacity across retry/restart.

Why. The failure taxonomy and retry paths are part of what needs validation. A task that is too clean never touches them before the expensive run does.

14 · Runs finish in minutes, not just cheaply

  • Small model, short outputs, no sandbox — and no rate-limit bottleneck at the intended concurrency.
  • One Platform Campaign per Evaluation Campaign/context uses top-up Submission Batches. Rollout Execution Key adds evaluation_context_id to Rollout Key and uniquely owns its Work Item, execute-rollout Stage and authoritative Rollout Result.
  • Invoke one @DBOS.step(retries_allowed=False) per logical provider attempt with native dr-providers retries zero. Expected provider success/failure returns a serializable typed observation; unexpected programming or infrastructure errors raise.
  • Assert each completed observation carries logical call identity, attempt number, execution-policy identity, timing/latency, raw request and response or failure evidence, normalized usage/cost/diagnostics, and typed outcome.
  • Force replay after a checkpoint and verify DBOS reuses the completed Provider Call Attempt observation and deterministically reconstructs Node Outcomes and the Graph Run Result. Verify attempt numbering, step order, branching, durable sleep and backoff remain deterministic, with no custom Whetstone attempt ledger or fence.
  • Force a crash after provider effect but before checkpoint and accept that a later execution may duplicate the physical call or billing. Assert only the checkpointed physical outcome is retained and no exactly-once or every-billed-call evidence guarantee is claimed.
  • Assert the atomic bind contract: unbound + ref binds; the same ref replays idempotently; a different ref raises identity conflict and never overwrites the authoritative binding.
  • A Stage becomes SUCCEEDED only after binding resolves to the one persisted immutable Rollout Result Object Reference. Failure to produce or persist that Result, or a binding/storage failure leaving the key unbound, raises FAILED. A distinct Platform Stage Attempt/operator retry proceeds only through Whetstone's FAILED+unbound gateway and executor recheck. A different-ref conflict fails the conflicting attempt and never authorizes overwrite or retry replacement. Deliberate re-evaluation uses a new context or Repeat.
  • dr-platform transports the authoritative bound Object Reference as output_reference; it does not own result uniqueness.
  • A Graph Run Result is the ordinary logical Result for Graph Run. It contains Graph Run identity, Graph External Inputs, terminal Outcome, relevant Provider Call Attempt records or references and provenance, and per-Node Outcomes, execution order and evidence. It is not a summarization or transformation, contains no Platform Stage or Stage Attempt state, is nested evidence inside—not the source of truth for—the authoritative Rollout Result, and has no separate persistence path.
  • On success and failure, retain the least-processed request and response or failure bodies plus normalized usage/cost/diagnostics. Never persist authorization headers or credential material; never silently truncate, and surface any backend limit encountered.
  • Provider Concurrency Control uses dr-platform selector capacity to bound Stage occupancy; the canonical tuple, accepted inaccuracies and non-goals are below. Provider Execution Policy owns timeout, credentials and native retry zero.

Why. Wall-clock is what makes hyperparameter sweeps iterable. A cheap run that takes hours still blocks the debug loop.

Provider Concurrency Control. Provider Quota Identity is best-effort and exactly (provider, protocol, model), with no credential, account, or override. Whetstone derives collision-free labels, applies every relevant label to multi-route work, and configures mandatory empty-selector plus exact-label capacities. Every matching generic dr-platform control holds capacity for the full ADMITTED Stage, including backoff. Verify independent accounts may be conservatively grouped and shared quotas exposed under different tuples may be under-controlled. This is not exact call-boundary concurrency, RPM/TPM or token reservation, global/cross-stage/cross-version quota, strict fairness, adaptive limiting, or credential-aware identity.
Aggregation must cross strata. Include enough examples from each latent-rule stratum that exact 0/1 example scores can be averaged first by repeat within task, then across every pinned task and stratum. Improvements should come from solving more complete examples across strata, never from awarding partial credit inside an example.

What the quick test does not cover

Criteria 1–3 deliberately cut scope. Each cut item needs its own targeted check before the full experiment:

  • Multi-node graph routing — the encoder → decoder chain and its input/output wiring.
  • Sandbox execution — test-suite runs, semantic image/version pinning in Evaluation Procedure Config, timeouts, and realized-environment attestation in Evaluation Context.
  • dr-code preprocessing — extraction, Absent handling, metric extraction from traces.
  • Production compression semantics — zstd measurement over the encoder representation and its HumanEval reference denominator.
  • Repeat averaging under real variance — determinism (criterion 5) means the 3 repeats mostly agree, so the aggregation path runs but its math is never stressed by disagreeing scores.
Mandatory when claiming GEPA or two-objective coverage. Add output length in UTF-8 bytes as an explicitly distinct second Objective, with a fixed length-budget materialization axis. Pin the vector direction as exact-match rate ↑ and mean output bytes ↓, exercise Pareto archive/update rules, and report dispersion across repeats. Without this metric and variance check, the quick test validates only single-objective optimizer plumbing and must say so.
Proposal and authority check. Every optimizer returns ordered Optimization Proposals. The harness validates all proposals first, expands every valid proposal over every fixed-axis selection, and emits one concrete Graph Config plus one Materialization Record for each complete proposal/axis slot. The Selection Policy then selects an ordered subset of Materialization Record slots for official evaluation. Unselected valid slots retain their Graph Configs and Materialization Records, are not officially evaluated, and require no selected official-lineage mapping. Execution planning deduplicates only the selected/admitted records by full graph_hash, then expands the D distinct admitted Graph Hashes over tasks and repeats to create exactly D × tasks × repeats Work Items and Rollouts, D aggregates, and no duplicate Rollout Execution Keys. The acceptance fixture selects two Materialization Records with the same Graph Hash and requires one shared work matrix and one shared aggregate. Official Evaluation Records and Official Plot Manifests MUST preserve the ordered mapping from every selected Materialization Record reference through graph_hash to that shared planned/result-key set and aggregate reference, preserving separately attributable slots when multiple selected records share one graph and aggregate. Record refs remain outside Rollout Keys, Work Requests, Results, and aggregate identity. Internal and official Eval Configs must reference the identical Evaluation Procedure Config before their executions can share graph_hash; their contexts remain separately authorized, and official reuse requires matching Rollout Execution evidence, attestation and compatible realized environment, never hashes alone.
