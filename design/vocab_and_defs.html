This page is the target Whetstone contract: one normative vocabulary, the object and identity rules that join the repositories, a non-normative native-API crosswalk, and a concrete Eval example.

Normative Vocabulary

Terms are presented as core lifecycle vocabulary, shared cross-repository vocabulary, first-party dependency-repository vocabulary, external runtime vocabulary, and Whetstone-specific roles and contracts. All tables in this section are normative; Whetstone role-qualified specializations stay in the Whetstone-specific table even when implemented atop a dependency.

Implementation status: Implemented · Planned in Concrete Changes · Implementation gap not captured in Concrete Changes

Core lifecycle

TermDefinition
DefinitionVersioned description of an object's shape and its unset variables.
VariableNamed slot whose allowed values and identity effect are declared by a definition.
Set (Verb)Assign a validated value to a variable.
ConfigA definition with every required variable set.
HashDeterministic fixed-length digest computed by a declared algorithm over exact declared input bytes; its specific role and semantics come from the qualified term or domain contract.
InputRuntime value supplied to a run and recorded by identity or immutable reference.
OutputRuntime value produced by a successful logical operation or Run and recorded by identity or immutable reference when retained.
RunOne logical execution of a config with concrete inputs, potentially spanning retry attempts.
OutcomeTyped terminal value/state of one logical operation. Success carries its Output or Outputs; typed non-success carries causal failure, refusal, or dependency-block evidence. It contains neither attempt history nor provenance and is not itself a Result.

First-party dependency-repository vocabulary

Each table groups terms by the first-party dependency repository that owns their normative semantics.

dr-serialize

TermDefinition

dr-graph

TermDefinition
GraphUmbrella structural concept for a directed acyclic composition of Nodes whose connections route outputs to inputs.
Graph DefinitionVersioned, variable-bearing directed acyclic graph shape represented as a dr-graph Definition. It declares Node Definitions, Variables, structure, input and output contracts, and exactly one terminal Node, and materializes one or more Graph Configs.
Graph ConfigConcrete fully set and validated Config materialized from a Graph Definition and represented by the target native dr-graph GraphConfig. Graph RunDeterministic topological interpretation of one Graph Config with Graph External Inputs and a validated node runner.

It is not a summarized transformation, contains no Platform Stage state, is nested within Rollout Result, and has no separate authoritative persistence path.

Configured and executed Node addressing is exactly (graph_hash, node_id).
Node Input SourceRuntime wiring that maps one declared Node input to exactly one Graph External Input or one upstream Node Output. It is part of Graph Config identity. Static config references such as Evaluation Procedure Config are Variable assignments, not Node Input Sources.
Node OutputValidated output object whose fields are declared by the Node Definition and emitted by a successful Node invocation.
Node OutcomeNode role of Outcome: one node's successful, failed, or dependency-blocked terminal state with the corresponding output or causal evidence. This semantic mapping does not require a shared exported schema.
[NodeRole] Terminal NodeStructural role of the Graph's unique sink Node, not a Node Type. On success it produces an ordinary Node Output that becomes the Graph Output unchanged.

Target System Contract

Identity protocol

For every Whetstone identity, the owning domain deliberately selects and validates a finite-JSON Identity Payload containing exactly its identity-bearing fields. That payload appears inside the complete self-describing, versioned Identity Document {"schema": "...", "schema_version": 1, "payload": {...}}, whose shape is exact. dr-serialize MUST validate finite JSON and the document shape, reject non-string object keys, non-finite numbers, implicit object conversion, and unordered collections without schema-declared order, render Canonical Identity JSON as UTF-8 from the complete Identity Document, and compute its full lowercase 64-character SHA-256 Identity Hash. dr-serialize MUST NOT select domain fields. Diagnostic Serializer.to_jsonable output, Identity Hash Prefixes, dependency-native shortened hashes, and Content Hashes are not Identity Payloads or substitutes for Identity Hashes.

End-to-end sequence

  1. Each owning domain selects and validates the Identity Payload for the Graph Definition (used by Whetstone in the Rollout Definition role), Eval Config, Tool Definitions, Tool Specs, Selection Policy, and authority records; dr-serialize validates each complete, exact Identity Document, renders Canonical Identity JSON from it, and computes its full Identity Hash.
  2. Issue an Optimization Request containing Initial Candidates before fixed-axis and Variance Source expansion, explicit bases, Mutation Surface, serialized Tool Specs, budgets, and non-binding guidance; construct Runtime Tool Handles only at the execution boundary.
  3. For each Tool Call, atomically accept or refuse (tool_spec_hash, call_id) in the Tool Call Ledger; an accepted call binds the Tool Spec's Internal Eval Config through a non-authoritative Evaluation Context and its Reward Policy maps returned aggregates or objectives into the typed Tool Result.
  4. The Optimization Result returns ordered Optimization Proposals; the harness validates and Diff Checks every proposal against its explicit base, rejecting invalid proposals before expansion.
  5. The harness expands every valid candidate or proposal assignment across all fixed axes and Variance Sources. For every resulting complete slot, materialization validates the closed Node Definitions, unique Node IDs, every Node Input Source, every static Variable assignment, and exactly one terminal/sink Node; emits one concrete Graph Config and one Materialization Record; and computes the full Graph Hash from the Graph Config Identity Document. Each Graph Config is one Rollout Variant; route, template, temperature, Character Budget ratio, or Evaluation Procedure Config changes therefore change its identity. Different materializations may converge to one graph_hash, so materialization count may exceed distinct variant count.
  6. Selection Policy selects an ordered subset of the complete Materialization Record slots for evaluation; unselected valid records remain retained and require no selected-lineage mapping. Execution planning maps each selected record to its full graph_hash, deduplicates selected/admitted records by that hash before Task/Repeat expansion, and defines D as the number of distinct admitted Graph Hashes. The evaluator binds the composite Eval Config through an Evaluation Context, validates that its Evaluation Procedure Config identity exactly matches the Eval Node/Graph Config reference, expands its Sampling Config's Task Set and Repeat plan, and creates exactly D × task_count × repeat_count Rollout Keys, Rollout Execution Keys, Work Items, and immutable Rollout Work Requests. Task values enter those requests as Graph External Inputs, evaluation inputs, or reference material and never enter Graph Config or Rollout Variant identity; Materialization Record references remain outside Work Requests and keys.
  7. One Evaluation Campaign is submitted in one or more Submission Batches; each Rollout Work Item is unique by Rollout Execution Key and carries collision-free labels for every applicable (provider, protocol, model) Provider Quota Identity. Whetstone configures the required default empty-selector capacity and every exact-label capacity; dr-platform applies all matching controls for the full admitted Stage lifetime and leaves work unadmitted when the default is absent.
  8. The Whetstone Node runner invokes one @DBOS.step(retries_allowed=False) per logical provider attempt with native dr-providers retries fixed at zero. Expected success or classified failure returns a serializable Provider Call Attempt observation; unexpected programming or infrastructure errors raise. Whetstone deterministically classifies checkpointed outcomes, durably sleeps and backs off when policy permits, and invokes the next bounded logical attempt. DBOS replay reconstructs Node Outcomes and the Graph Run Result from those checkpoints; Whetstone does not inject completed Node Outputs across DBOS workflow identities.
  9. The unique terminal Node produces an ordinary Node Output; on Graph Run success that unchanged object is the Graph Output. The executor nests the Graph Run Result in one complete Rollout Result, immutably puts that terminal Rollout Result into dr-store, receives a typed Object Reference carrying its Content Hash in content_hash, and binds that reference under the Rollout Execution Key through the authoritative Result Store compare-and-set. Persisted and bound success and exhausted-semantic-failure Results both make the Platform Stage Execution operationally SUCCEEDED. Only inability to produce, persist, bind, or confirm that exact terminal Result makes the Stage FAILED. Whetstone's operator gateway permits retry_stage only for a platform-FAILED Stage whose key is unbound, and the retried executor rechecks because the gateway precheck can race. A different-reference conflict preserves the existing binding and never authorizes overwrite or retry replacement.
  10. The evaluation adapter turns decoder Generation into Submission Text, native TextArtifact, Preprocessing Trace, Code Candidate Set, compiled Code Artifacts, auxiliary traces, Metric Records, Metric Facts, and named Scores. Encoder Generation remains the compression representation used to compute Compressed Description Length.
  11. The aggregator requires the configured complete matrix and derives Rollout Aggregates and direction-bearing Objectives without Reward; the Evaluation Authority writes Official Evaluation records and Official Plot Manifests only through the official path.

  • dr-store Object Store puts MUST be immutable and append-only, and gets MUST verify the Content Hash declared by the Object Reference.
  • Every dr-store atomic key-to-reference binding MUST obey one rule: an unbound key plus reference binds; replay with the same reference is idempotent success; a different reference conflicts; no binding is overwritten.

  • Each persisted terminal Rollout Result MUST be immutable and final, and the Result Store MUST preserve one binding from its Rollout Execution Key to a typed Rollout Result Object Reference: the same reference replays successfully, a different reference conflicts, and neither path overwrites.
  • Whetstone alone MUST own and bound provider semantic retries inside a Stage.
