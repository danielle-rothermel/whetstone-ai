Normative Vocabulary

Generic shape vocabulary comes first, followed by domain-specific terms.

Rollout ResultResult containing the configured measurements and metadata from one rollout. Failures remain explicit rollout results with causes, never gaps.
Experiment definition
CandidateAssignment of values to variables the experiment compares; optimizer proposals may change only permitted candidate variables.
Initial CandidatesCandidate values supplied to the optimizer within initial rollout configs as proposal starting points.
Variance SourceSetting intentionally swept within a candidate, producing distinct rollout configs and results.
TaskSingle dataset element evaluated by a rollout.
SeedRepeat of an otherwise identical configuration used to measure evaluation randomness; an RNG seed when fully controlled.
Reference LengthNormalization basis for setting relative budgets and reporting compression.
Measurement and aggregation
Binary Test Pass ScoreCorrectness score for one rollout (config × task × seed): exactly 1 when the reconstructed code passes the task's full test suite; otherwise 0, with no partial credit.
Average Binary Test Pass RateAggregate correctness of one rollout config: mean binary test pass score across seeds per task, then unweighted mean across tasks.
Rollout AggregateAggregate of the rollout results produced for one rollout config under one eval config.
Authority and integrity
Official EvaluationHarness's authoritative execution context, which supplies an official eval config; only this context produces official rollout results used in plots. Config contents and hash do not confer officialness. Optimizer tool evaluations are non-official.
HashContent hash identifying a serialized object (rollout config, rollout definition, eval config): identical bytes yield the same hash; any change yields a new hash. It also keys rollout results. OCI sandbox images retain the term "image digest".
Diff CheckHarness comparison of returned rollout configs with initial configs, confirming only permitted changes.

Draft dr-code Vocabulary

This section records dr-code-owned contracts for integration with Whetstone. It is a first-pass draft for discussion, is not yet Whetstone normative vocabulary, and does not rename any dr-code API.

The concrete vocabulary follows dr-code's public types and behavior: it uses Definition, Runner, and Trace rather than “Preprocess Flow,” because configuration, bound execution, and retained runtime evidence are separate contracts; likewise, “Metric Extraction” names the behavior of extract_metrics, not a nonexistent “Metric Extraction Flow” API.

TermDraft definition
Inputs and artifacts
Submission TextThe runtime role of the raw string supplied for dr-code processing from an upstream Whetstone LLM generation; the generation itself remains a Whetstone generation, and the string is not yet an artifact.
Text ArtifactA dr-code TextArtifact wrapper carrying arbitrary text, with no claim that the text is code, constructed from submission text before preprocessing.
Python SourceA raw string interpreted as Python source, without by itself asserting that compilation succeeds.
Code ArtifactA dr-code CodeArtifact carrying a source string under the stronger producer contract that it has passed an upstream compilation check.
Code CandidateOne Python source string at an ordered position in a code candidate set artifact, distinct from Whetstone's experiment-level candidate.
Code Candidate Set ArtifactA dr-code CodeCandidateSetArtifact containing zero or more ordered code candidate source strings and optionally aligned lineage; an empty set is a present artifact, not absent.
AbsentA dr-code Absent causal sentinel for a step that failed while processing a present input, retaining failure and propagation lineage; a missing upstream submission produces no preprocessing trace, while a missing trace key is a wiring error.
Preprocessing
Preprocessing DefinitionA dr-code PreprocessingDefinition, the fully concrete, immutable, versioned specification of ordered named steps and settings, which therefore behaves like a Whetstone config despite its existing class name.
Bound Preprocessing RunnerA reusable BoundPreprocessingRunner produced by resolving a preprocessing definition's steps, validating settings and artifact-kind wiring, and pinning producer identity before it accepts and kind-checks input artifacts.
Preprocessing TraceThe dr-code Trace returned for one input, generically retaining artifact-or-absent values, step facts, and producer identity; the named HumanEval function-candidates definition specifically ends in Absent or one CodeCandidateSetArtifact, not a list of CodeArtifacts.
Metric extraction
Metric FamilyA stable kind of metric question identified by dr-code's MetricName, such as code_test or compressed_length, and implemented by a versioned operator.
Metric QuestionA dr-code MetricQuestion declaring one metric family, one trace key to evaluate, and the operator settings that determine its semantics.
Metrics DefinitionA dr-code MetricsDefinition, the fully concrete, immutable, versioned specification of an ordered collection of metric questions, which therefore behaves like a Whetstone config despite its existing class name.
Metric ExtractionThe extract_metrics operation that evaluates one dr-code Trace once per metric question in declaration order and returns ordered metric records.
Metric RecordA self-describing dr-code MetricRecord answering one declared question with one or more named metric values when measured, or with explicit not_applicable or operator_failure state.
Metric ValueOne named scalar fact in a measured metric record's values mapping, not a Whetstone score or objective unless Whetstone derives one.
Code-Test FactsThe code_test record values describing case counts, pass/fail/error/timeout counts, coverage completeness, function count, and selected function name, not Whetstone's binary test pass score.
Compressed-Length FactsThe compressed_length record values for UTF-8 representation bytes, compressed bytes, and optional reference ratio and percent reduction under pinned codec, level, and reference-source semantics, not automatically Whetstone's chosen compression objective.

Current-behavior caveat. dr-code's exhaustive evaluate_preprocessing_candidates currently evaluates raw candidate sources and directly constructs MetricRecords, bypassing the generic per-candidate trace and extract_metrics path; the diagram is a target boundary, not a description of that evaluator's current API behavior.

Proposed Whetstone integration boundary. dr-code owns artifact, trace, metric-operator, and metric-fact contracts; Whetstone owns the adapter, config selection, candidate reduction, score and objective derivation, aggregation, plotting, and official authority. The legacy dr-code score_humaneval_submission currently performs candidate reduction and scoring, while this target facts-first integration assigns that policy to Whetstone.

Running Example: Eval Experiment

ConceptConcrete detail in the Eval experiment
Rollout
Rollout configs8 candidates × 4 budget ratios produce 32 rollout configs, identified by rollout_config_hash.
Rollout graphEach rollout executes Encoder LLM Call Node → Decoder LLM Call Node → Eval Node for one (rollout config, task, seed).
Rollout ResultEach Official Evaluation result is keyed by (rollout config hash, official eval config hash, task, seed) and contains Binary Test Pass Score, measured zstd bytes, and metadata for tokens, cost, and latency. Its Official Evaluation context — not hash identity — makes it official. A failed rollout still produces a result with its cause — never a gap.
Inference resources
Providers and modelsThe system currently supports OpenAI and OpenRouter. For example, gpt-5.4-nano is provided by the OpenAI provider, while openai/gpt-5.4-nano is provided by OpenRouter.
Experiment definition
CandidateA candidate is an (encoder user template, provider, model) tuple.
Variance and curvesThe only variance source is budget_ratio. Its four settings distinguish the rollout configs and trace each candidate's curve. Tasks and seeds are not variance sources: both are averaged into each rollout aggregate.
TasksIn the HumanEval task mapping, each task supplies a ground-truth code sample and tests for the use case. The encoder's input and reference code sample are task.gt_code_wo_comments; the task set defines the quantity the result estimates.
SeedsGeneration randomness comes from API calls to the model providers, so repeats do not change a controlled rng seed; they repeat the calls. Repeats capture generation stochasticity and buy confidence in the result (see the eval doc’s Additional Notes).
Reference length and compressionReference length is the length of task.gt_code_wo_comments. The budget is a fraction of that length, and official compression is the description's measured zstd bytes divided by it.
Eval ConfigsThe shared shape is {task_set}, {seeds}, {reference_length}, {codec}, {preprocess_config}, {metrics_extract_config}, and {aggregation_method}. Task set, seeds, and aggregation drive the surrounding sweep; the eval node uses the remaining evaluation settings. For this Eval setup, internal_eval_config_hash == official_eval_config_hash. Optimization normally uses a cheaper, different internal eval config with the same preprocessing and metric extraction but fewer tasks and seeds, so the hashes differ.
Measurement and aggregation
Metrics and aggregationPer rollout, Binary Test Pass Score is 1 only when the reconstructed code passes the task's full test suite, otherwise 0; it is never a fraction of tests passed. Per rollout config, Average Binary Test Pass Rate is the per-task mean over seeds followed by the unweighted mean over the task set, and measured compression is averaged.
Rollout AggregateEach rollout aggregate combines one rollout config's official rollout results into (Average Binary Test Pass Rate, mean measured compression). The experiment produces 32 aggregates — 4 per candidate — for the plot.
Authority and integrity
Hashes and image digestSerialized Rollout Definitions, rollout configs, and eval configs use content hashes as identity: the same bytes give the same hash and any change produces a new one. Hashes also key rollout results, but they do not assign eval-config roles or confer officialness. The sandbox image retains the OCI term “image digest.”
