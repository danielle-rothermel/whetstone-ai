The shared vocabulary of the whetstone design-doc family. Terms defined here carry these exact meanings across every design doc.
TermDefinition
ProviderA service that provides an api endpoint that we can hit with a request for a specific model and a prompt/history to get the llm generation. Currently our system supports OpenAI and OpenRouter.
ModelA specific model available from a given provider, eg. gpt-5.4-nano is provided by the openai provider and openai/gpt-5.4-nano is provided by openrouter.
ArchitectureA graph shape with each node's inputs, variables, and outputs named, but no values set.
CandidateA choice of values for the candidate variables — the variables whose settings the experiment compares, one candidate per plotted line. In the eval experiment, a candidate is an (encoder user template, provider, model) tuple. An optimizer proposes new candidates: new values for the candidate variables it is allowed to change.
Rollout configThe architecture with every variable value set — candidate variables, variance-source variables, and fixed values. The executable unit of the system, identified by its hash (graph_hash). In the eval experiment: 8 candidates × 4 budget ratios = 32 rollout configs.
RolloutOne execution of a rollout config on a (task, seed): encoder → decoder → score, producing one rollout result. Evaluation collects rollouts and aggregates their scores.
Rollout resultThe output of one rollout, keyed (rollout config hash, official eval config hash, task, seed): Binary Test Pass Score, measured zstd bytes, and metadata (tokens, cost, latency). Failures are rollout results too, with a cause — never gaps.
Rollout config resultThe aggregate of one rollout config's official rollout results: (Average Binary Test Pass Rate, mean measured compression). 32 in the eval experiment — 4 per candidate; what lands on the plot.
Variance sourceThe settings swept within one candidate to turn it into a distribution of rollout config results on the plot axes — in the eval experiment, just budget_ratio, which distinguishes rollout configs and traces the candidate's curve. Seeds and tasks are not variance sources: both are averaged into each rollout config result. Seed captures generation stochasticity and buys confidence in the result; the task set defines the quantity the result estimates (see the eval doc’s Additional Notes).
Initial candidatesThe candidate values an optimization run starts with — carried inside the initial rollout configs handed to the optimizer as the basis for its proposals.
TaskA single element of the dataset which should provide a way to get (1) a ground truth code sample and (2) tests for this usecase.
SeedA repeat of the exact same configuration to capture randomness in the evaluation. In fully controlled experiments this would be defined by a rng seed but in our experiments the randomness is coming from the API calls to the model providers so we don't change the seeds we just repeat the calls.
HashThe content hash of a serialized object (rollout config, architecture, eval config), used as its identity: same bytes give the same hash, any change makes a new one. Also keys the rollout results. The one exception in wording: the sandbox image keeps the OCI term "image digest".
Official evaluationThe evaluation harness's authoritative measurement pass: collecting rollouts under the official eval config produces the official rollout results that feed the plots. "Evaluate" alone is the general verb — an optimizer may evaluate however its tools allow, but only official evaluation produces official rollout results.
Score Node Eval ConfigThe eval config the rollout config itself carries, as Score Node variables: {task_set}, {seeds}, {reference_length}, {codec}, {preprocess_config}, {metrics_extract_config}, {aggregation_method} — all inside the rollout config hash. In an Eval run it matches the Official Eval Config — that is the point. In another setting an optimizer might set a cheaper one while it iterates: same preprocess and metrics extraction, but a smaller task subset and fewer seeds. The cross-run fields (task set, seeds, aggregation) direct whoever drives the sweep; a single rollout uses the rest.
Binary Test Pass ScoreThe correctness score of a single rollout (rollout config, task, seed): 1 if the reconstructed code passes the task's full test suite, 0 otherwise — never a fraction of tests passed.
Average Binary Test Pass RateThe aggregate correctness of one rollout config: per-task mean Binary Test Pass Score over the seeds, then unweighted mean over the eval config's task set — the correctness axis of that rollout config's result.
Diff checkThe evaluation harness's comparison of each returned rollout config against the initial ones, confirming only permitted changes were made.
Reference lengthThe length of the code sample the encoder is asked to compress (task.gt_code_wo_comments in the eval experiment): the budget is set as a fraction of it, and official compression is the description's compressed bytes divided by it.
