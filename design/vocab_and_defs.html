This page is the target Whetstone contract: one normative vocabulary, the object and identity rules that join the repositories, a non-normative native-API crosswalk, and a concrete Eval example.

Target System Contract

Identity protocol

For every Whetstone identity, the owning domain deliberately selects and validates a finite-JSON Identity Payload containing exactly its identity-bearing fields. That payload appears inside the complete self-describing, versioned Identity Document {"schema": "...", "schema_version": 1, "payload": {...}}, whose shape is exact. dr-serialize MUST validate finite JSON and the document shape, reject non-string object keys, non-finite numbers, implicit object conversion, and unordered collections without schema-declared order, render Canonical Identity JSON as UTF-8 from the complete Identity Document, and compute its full lowercase 64-character SHA-256 Identity Hash. dr-serialize MUST NOT select domain fields. Diagnostic Serializer.to_jsonable output, Identity Hash Prefixes, dependency-native shortened digests, and Content Hashes are not Identity Payloads or substitutes for Identity Hashes.

End-to-end sequence

  1. Each owning domain selects and validates the Identity Payload for the Rollout Definition, Eval Config, Tool Definitions, Tool Specs, Selection Policy, and authority records; dr-serialize validates each complete, exact Identity Document, renders Canonical Identity JSON from it, and computes its full Identity Hash.
  2. Issue an Optimization Request containing Initial Candidates, explicit bases, Mutation Surface, serialized Tool Specs, budgets, and non-binding guidance; construct Runtime Tool Handles only at the execution boundary.
  3. For each Tool Call, atomically accept or refuse (tool_spec_hash, call_id) in the Tool Call Ledger; an accepted call binds the Tool Spec's Internal Eval Config through a non-authoritative Evaluation Context and its Reward Policy maps returned aggregates or objectives into the typed Tool Result.
  4. The Optimization Result returns ordered Optimization Proposals; the harness diff-checks each proposal against its base, applies the Selection Policy, and materializes selected proposals across fixed axes and Variance Sources.
  5. Materialization validates the closed node operation schemas, emits exactly one concrete Graph Spec per fully assigned Rollout Config, computes its full Graph Digest, and computes the config Identity Hash from definition, assignments, and digest; route, template, temperature, or ratio changes therefore change the spec and digest.
  6. The evaluator binds the independent Eval Config through an Evaluation Context, expands its Task Set and Repeat plan, and creates one Rollout Key, Rollout Execution Key, and immutable Rollout Work Request per observation.
  7. One Evaluation Campaign is submitted in one or more Submission Batches; each Rollout Work Item is unique by Rollout Execution Key, and each Platform Stage Attempt may contain multiple Stage Invocations as DBOS replays its application body.
  8. The Node Attempt coordinator validates or resumes node outputs, invokes Provider Clients under a Provider Execution Policy with native retries zero, records every semantic Node and physical Provider Call Attempt, and exhausts automatic retries before constructing a terminal successful or failed Rollout Result.
  9. The executor immutably puts the complete terminal Result into dr-store, receives a typed Object Reference carrying its Content Hash in content_hash, and binds that reference under the Rollout Execution Key through the Result Store. The Platform Stage Execution becomes operationally SUCCEEDED only after the Result Store resolves the key to that exact reference, either through a new binding or same-reference idempotent replay. If the executor cannot produce or persist a terminal Result, or cannot bind or confirm that exact reference, it raises and the stage becomes FAILED; operator retry is then allowed if and only if the key remains unbound, including when an orphan reference was persisted before binding failed. A different-reference conflict preserves the existing authoritative binding and never permits overwrite or retry replacement.
  10. The evaluation adapter turns Generation into Submission Text, native TextArtifact, Preprocessing Trace, Code Candidate Set, compiled Code Artifacts, auxiliary traces, Metric Records, Metric Facts, and named Scores.
  11. The aggregator requires the configured complete matrix and derives Rollout Aggregates and direction-bearing Objectives without Reward; the Evaluation Authority writes Official Evaluation records and Official Plot Manifests only through the official path.
Critical invariants and MUST implications.
  • The owning domain MUST deliberately select every Identity Payload field; dr-serialize MUST validate the finite JSON and exact Identity Document, derive Canonical Identity JSON from the complete document, and compute the full Identity Hash, but MUST NOT select or infer domain fields.
  • Identity Hashes MUST be complete 64-character lowercase SHA-256 values over Canonical Identity JSON. Identity Hash Prefixes MUST remain display-only and MUST NOT establish equality, storage keys, or references.
  • A Content Hash MUST cover the complete canonical persisted record and MUST remain distinct from any Identity Hash over Canonical Identity JSON derived from an Identity Document; every Object Reference MUST be typed, carry the hash in content_hash, and resolve one exact immutable record through Content Hash verification.
  • dr-store Object Store puts MUST be immutable and append-only, and gets MUST verify the Content Hash declared by the Object Reference.
  • Every dr-store atomic key-to-reference binding MUST obey one rule: an unbound key plus reference binds; replay with the same reference is idempotent success; a different reference conflicts; no binding is overwritten.
  • Treatment, measurement, and authority identities MUST remain separate: rollout_config_hash excludes Eval Config; eval_config_hash excludes treatment; officialness comes only from authority provenance.
  • The same treatment Graph Spec MUST run under cheap internal and full official Eval Configs without changing graph_digest or rollout_config_hash.
  • Two executions claim the same Eval Config identity only when their complete Identity Hash values in eval_config_hash are equal under the same identity schema and version.
  • Internal and Official Eval Configs MAY have equal Identity Hashes because they are roles of one type, but an internal result MUST NOT be relabeled or copied into the official namespace.
  • Any setting capable of changing treatment execution, including every Provider Call Config field, belongs in Rollout Config; Eval Config owns sampling, per-observation measurement and Score derivation, reduction, and declared Objective directions, while Reward Policy belongs only to Tool Spec.
  • Graph Digest and Rollout Config Identity Hash MUST be full independent identities and MUST NOT be collapsed into one generic graph-identity field.
  • One Rollout Definition MAY materialize many Graph Specs, but each fully assigned Rollout Config MUST identify exactly one Graph Spec and any route, template, temperature, ratio, or other treatment change MUST affect that spec and digest.
  • Rollout Key MUST identify the semantic measurement cell, while Rollout Execution Key MUST add Evaluation Context identity and serve as the Work Item and terminal-index uniqueness key.
  • Each persisted terminal Rollout Result MUST be immutable and final, and the Result Store MUST preserve one binding from its Rollout Execution Key to a typed Rollout Result Object Reference: the same reference replays successfully, a different reference conflicts, and neither path overwrites.
  • Automatic Node retries MUST exhaust before terminal Result construction; the platform Stage Execution MUST become operationally SUCCEEDED only after either a terminal success or exhausted-failure Result is immutably persisted and the Result Store resolves the Rollout Execution Key to that exact typed Object Reference through a new binding or same-reference idempotent replay.
  • If an executor cannot produce or immutably persist a terminal Rollout Result, or cannot bind or confirm that exact Object Reference, it MUST raise. The failed stage MAY receive explicit operator retry_stage if and only if the Rollout Execution Key remains unbound, even when the failed execution persisted an orphan Object Reference; a different-reference conflict MUST preserve the existing authoritative binding and MUST NOT permit overwrite or retry replacement.
  • Deliberate re-evaluation after a terminal Result MUST use a new Evaluation Context ID or Repeat ID and MUST NOT overwrite or supersede the existing Result Store binding.
  • Raw provider bodies and credentials MUST NOT be durable result metadata by default; any exception requires explicit bounded redaction and retention policy.
  • Provider Call Request MUST reference one complete identity-bearing Provider Call Config without duplicate controls, while timeout, credentials, redaction, retention, and native retry zero MUST come only from Provider Execution Policy.
  • Native provider transport retries MUST be zero under durable execution; only the Whetstone Node Attempt coordinator may authorize classified automatic semantic retries.
  • DBOS replay MUST preserve the same Platform Stage Attempt identity and MAY add Stage Invocation evidence, while operator retry MAY create a new Platform Stage Attempt after failure if and only if the Result Store leaves the Rollout Execution Key unbound, regardless of any orphan Object Reference persisted before binding failed; an authoritative binding, including one exposed by a different-reference conflict, MUST prevent retry replacement.
  • Completed node output reuse MUST pass Resume Validation before values enter dr-graph.
  • Repeat ID MUST identify a planned observation; no retry attempt number may substitute for it, and a controlled RNG seed remains optional data.
  • The Eval driver, Eval Node, and aggregator MUST consume only their declared Eval Config slices; official evaluation MUST derive no Reward.
  • Optimization Request MUST serialize Tool Specs rather than Runtime Tool Handles, and mutable tool status and capacity consumption MUST live only in the Tool Call Ledger keyed by Tool Spec Identity Hash and Tool Call ID, spelled (tool_spec_hash, call_id).
  • Candidate correctness MUST examine all candidates and use any-passing semantics; infrastructure-unknown MUST fail the rollout instead of becoming Binary Test Pass Score 0.
  • No input, no trace, missing trace key, Preprocessing Failure, empty candidate set, compile failure, definitive test failure, and infrastructure failure MUST remain distinguishable.
  • Compression MUST use zstd-19 bytes of exact encoder Generation UTF-8 over UTF-8 bytes of task.gt_code_wo_comments, reject a zero denominator, and keep character budget separate.
  • Metric Records and Metric Facts MUST precede named Scores; Objectives MUST declare direction; Reward MUST cite a Reward Policy; anonymous scoring and score-metadata fields are not allowed.
  • Official aggregation MUST account for every planned Rollout Execution Key under its configured failure policy; missing or failed rows MUST NOT be silently dropped.
  • One Platform Campaign MUST represent one Evaluation Campaign, one Work Item MUST represent one Rollout Execution Key, and Pipeline Stage MUST never be used as a synonym for Node.
  • Selector Capacity MUST be treated only as concurrent admitted occupancy; Provider Call Limiter owns provider-native request and token limits.
  • Published official plots MUST resolve through an Official Plot Manifest and provenance-preserving official result or attestation references.

Non-normative Dependency Crosswalk

This section describes current native APIs where they exist, required target dependency contracts, and adapter boundaries. It does not rename dependency APIs, add normative synonyms, or duplicate the Whetstone ontology above.

Running Example: Eval Experiment

Contract stepConcrete Eval experiment
Treatment and proposal materialization
Rollout DefinitionThe variable-bearing treatment template is Encoder LLM Call Node → Decoder LLM Call Node → Eval Node; the Eval Node declares a context-bound procedure input, so no Eval Config value enters Graph Spec identity.
Experiment CandidateEach of 8 candidates assigns encoder user template and service-scoped Model Routes; budget_ratio is the fixed four-value Variance Source.
Optimizer tool boundaryThe Optimization Request serializes Tool Specs, not clients; each runtime call uses a constructed Runtime Tool Handle and an atomic Tool Call Ledger entry keyed by Tool Spec Identity Hash and call ID, while that spec's Reward Policy maps internal aggregates or objectives to optimizer Reward.
Proposal MaterializationAn optimizer returns ordered proposals containing a base and permitted prompt-template assignment; after Diff Check and Selection Policy, the harness expands each selected proposal across all four budget ratios.
Rollout ConfigThe baseline expands 8 candidates × 4 budget ratios into 32 fully assigned treatment configs and therefore 32 concrete Graph Specs. The Rollout Config domain selects {rollout_definition_hash, assignments, graph_digest}, including complete Provider Call Config Identity Hashes, as its Identity Payload; dr-serialize validates the complete versioned Identity Document and computes its full Identity Hash from Canonical Identity JSON. Any route, template, temperature, or ratio change changes the spec and Graph Digest.
Evaluation and identities
Eval ConfigThe Official Eval Config independently composes the pinned 164-task Task Set and three-repeat plan; preprocessing, all-candidate correctness, per-observation Score derivation, metric extraction, sandbox image, and zstd-19/reference semantics; plus complete unweighted aggregation and declared correctness-maximizing and compression-minimizing Objectives.
Evaluation ContextThe official harness binds that Eval Config, Evaluation Authority, campaign, execution policy, dependency lock, source revisions, sandbox image digest, and environment provenance at execution time; optimizer tools bind cheaper Eval Configs through separate internal contexts without changing treatment Identity Hashes.
Rollout KeyEach observation is keyed by (rollout_config_hash, eval_config_hash, task_identity, repeat_id), where both config fields carry full Identity Hashes and the referenced Eval Config has the official role in this context; the optional RNG seed is absent because provider randomness is uncontrolled, and retries never change repeat_id.
Rollout Execution KeyThe official Evaluation Context ID joins the Rollout Key to form the Work Item and terminal-index identity; a deliberate later re-evaluation must use a new context ID or Repeat ID.
Rollout ResultEach immutable final result records the Rollout Execution Key, both config Identity Hashes, Evaluation Context and authority identities, Graph Run Summary, facts and named scores or exhausted causal failure, and Node, Provider Call, Platform Stage Attempt, and retained Stage Invocation evidence. Its full canonical persisted record receives a Content Hash and typed Rollout Result Object Reference carrying that hash in content_hash.
Platform mapping and execution
Campaign and submissionOne Whetstone Evaluation Campaign maps to one Platform Campaign; each submission group is one Submission Batch and native Platform Submission Run.
Rollout Work ItemOne Work Item maps to one Rollout Execution Key; its native input_ref carries a typed Object Reference with content_hash for one exact immutable Rollout Work Request, and a succeeded stage's output_reference does the same for the terminal Rollout Result.
Attempts and terminalizationA Platform Stage Attempt is the native attempt-row/DBOS workflow identity and may contain multiple replayed Stage Invocations. The Node Attempt coordinator exhausts semantic retries with native provider retries zero, then dr-store immutably puts the success or exhausted-failure Result. The stage succeeds only when the Result Store resolves the Rollout Execution Key to that exact typed reference through a new binding or same-reference idempotent replay. Failure to produce or persist the Result, or to bind or confirm its exact reference, raises; operator retry_stage is allowed after failure if and only if the key remains unbound, even if the put left an orphan reference. A different-reference conflict preserves the existing authoritative binding and never permits overwrite or retry replacement.
Facts-first code evaluation
Artifact pipelineExact encoder Generation becomes Submission Text, native TextArtifact, Preprocessing Trace, and Code Candidate Set; the Whetstone adapter fans out every candidate, compile-checks and wraps Code Artifacts, constructs auxiliary traces, and extracts Metric Records.
Binary Test Pass ScoreCode-test Metric Facts derive score 1 when any fully evaluated Code Artifact passes the full task suite and 0 only when all fully evaluated Code Artifacts definitively fail; infrastructure uncertainty terminates the rollout as failed.
Compression RatioCompressed Description Length is the integer zstd level 19 byte count of exact encoder Generation UTF-8, divided by nonzero UTF-8 bytes of task.gt_code_wo_comments; the prompt budget separately uses character count.
Rollout AggregateFor each of the 32 treatment configs, the complete 164 × 3 planned matrix yields Average Binary Test Pass Rate and Mean Compression Ratio; failures and missing observations follow declared policy and are never silently dropped.
Official authority and publication
Official EvaluationThe authorized write path issues an Official Evaluation Record over planned Rollout Execution Keys, official result references, complete aggregates, and direction-bearing Objectives but computes no Reward; equal internal and official config Identity Hashes permit identity comparison but never relabel an internal result.
Official Plot ManifestThe published four-point curve per candidate resolves through a manifest naming Official Evaluation IDs, 32 complete aggregates, Selection Policy, source and dataset revisions, dependency lock, and environment identity.
