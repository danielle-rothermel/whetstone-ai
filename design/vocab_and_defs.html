What do the shared terms mean?

normative vocabulary

Terms are presented as core lifecycle vocabulary, shared cross-repository vocabulary, first-party dependency-repository vocabulary, external runtime vocabulary, and Whetstone-specific roles and contracts. All tables in this section are normative; Whetstone role-qualified specializations stay in the Whetstone-specific table even when implemented atop a dependency.

What identity and object rules must hold?

target system contract

Identity protocol

For every Whetstone identity, the owning domain deliberately selects and validates a finite-JSON Identity Payload containing exactly its identity-bearing fields. That payload appears inside the complete self-describing, versioned Identity Document {"schema": "...", "schema_version": 1, "payload": {...}}, whose shape is exact. dr-serialize MUST validate finite JSON and the document shape, reject non-string object keys, non-finite numbers, implicit object conversion, and unordered collections without schema-declared order, render Canonical Identity JSON as UTF-8 from the complete Identity Document, and compute its full lowercase 64-character SHA-256 Identity Hash. dr-serialize MUST NOT select domain fields. Diagnostic Serializer.to_jsonable output, Identity Hash Prefixes, dependency-native shortened hashes, and Content Hashes are not Identity Payloads or substitutes for Identity Hashes.

End-to-end sequence

  1. Each owning domain selects and validates the Identity Payload for the Graph Definition (used by Whetstone in the Rollout Definition role), Eval Config, Tool Definitions, Tool Configs, Selection Policy, and authority records; every Definition or Config is carried by its typed reference plus Identity Hash. dr-serialize validates each complete, exact Identity Document, renders Canonical Identity JSON from it, and computes its full Identity Hash.
  2. Start one optimization run and issue its first immutable optimization step request. Each later Step Request carries only immutable prior Step Result, state, and history Object References plus Content Hashes, the next candidates/pools, hyperparameters, budget state, output contract, and optional serialized Tool Configs with typed Tool Definition references plus Identity Hashes. Runtime Tool Handles are never serialized or persisted.
  3. For a proposal-only Optimization Step, COPRO or MIPROv2 returns candidates plus immutable Evaluation Intents; Whetstone evaluates each candidate outside the optimizer invocation under the Intent's exact target Eval Config and declared Evaluation Context role/policy, records the resolution evidence, and only then persists the immutable Step Result. Evaluation under another Eval Config does not resolve the Intent.
  4. For a tool-using optimization step, GEPA or Codex receives serialized Tool Configs and constructs Runtime Tool Handles at the execution boundary. For each Tool Call, Whetstone atomically transitions (tool_config_hash, call_id) from absent to accepted or refused in the Tool Call Store, debits capacity exactly once on acceptance, and later transitions accepted to completed. Same-transition replay is idempotent and divergent state conflicts. An accepted call binds the Tool Config's ordinary Eval Config through an Evaluation Context whose Evaluation Role is internal, and the immutable Step Result references every terminal Tool Result Object Reference plus Content Hash and Tool Call Store Entry used by the Step.
  5. Persist exactly one immutable Optimization Step Result with budget accounting and continue, complete, or failed status before advancing or terminalizing. When the ordered Steps finish, persist one Optimization Result containing ordered Optimization Proposals, every Step Result reference, record-local provenance, and cost; the harness validates and Diff Checks every proposal against its explicit base before expansion, and official evaluation follows separately.
  6. The harness expands every valid candidate or proposal assignment across all fixed axes and Variance Sources. For every resulting complete slot, materialization validates the closed Node Definitions, unique Node IDs, every Node Input Source, every static Variable assignment, and exactly one terminal/sink Node; emits one concrete Graph Config and one Materialization Record; and computes the full Graph Hash from the Graph Config Identity Document. Each Graph Config is one Rollout Variant; route, template, temperature, Character Budget ratio, or Evaluation Procedure Config changes therefore change its identity. Different materializations may converge to one graph_hash, so materialization count may exceed distinct variant count.
  7. Selection Policy selects an ordered subset of the complete Materialization Record slots for evaluation; unselected valid records remain retained and require no selected-lineage mapping. Execution planning maps each selected record to its full graph_hash, deduplicates selected/admitted records by that hash before Task Set/Repeat Plan expansion, and defines D as the number of distinct admitted Graph Hashes. The evaluator binds the composite Eval Config through an Evaluation Context, validates that its Evaluation Procedure Config identity exactly matches the Eval Node/Graph Config reference, expands its Sampling Config's Task Set and Repeat Plan, and creates exactly D × task_count × repeat_count Rollout Keys, Rollout Execution Keys, Work Items, and immutable Rollout Work Requests. Dataset-task values enter those requests as Graph External Inputs, evaluation inputs, or reference material and never enter Graph Config or Rollout Variant identity; Materialization Record references remain outside Work Requests and keys.
  8. One Evaluation Campaign is submitted in one or more Submission Batches; each Rollout Work Item is unique by Rollout Execution Key and carries collision-free labels for every applicable (provider, protocol, model) Provider Quota Identity. Whetstone configures the required default empty-selector capacity and every exact-label capacity; dr-platform applies all matching controls for the full admitted Stage lifetime and leaves work unadmitted when the default is absent.
  9. The Whetstone Node runner invokes one @DBOS.step(retries_allowed=False) per logical provider attempt. dr-providers executes the Provider Call Request under Provider Transport Policy with native retries zero and returns a typed no-throw expected Provider Transport Outcome plus stable Provider Invocation Evidence. Whetstone wraps that evidence in a Provider Call Attempt, applies semantic classification, durably sleeps and backs off when Provider Execution Policy permits, and invokes the next bounded logical attempt. Unexpected programming or infrastructure errors raise. DBOS replay reconstructs Node Outcomes and the Graph Run Result from checkpoints; Whetstone does not inject completed Node Outputs across workflow identities.
  10. The unique terminal Node produces an ordinary Node Output; on Graph Run success that unchanged object is the Graph Output. A different-reference conflict preserves the existing binding and never authorizes overwrite or retry replacement.

Rollout Variant and proposal materialization
Rollout DefinitionWhetstone uses one dr-graph Graph Definition in the Rollout Definition role; this does not create another type, schema, artifact, or identity. Its variable-bearing shape is Encoder LLM Call Node → Decoder LLM Call Node → Eval Node, with the Eval Node as the unique terminal Node and its Evaluation Procedure Config reference declared as a static Variable assignment.
Experiment CandidateEach of 8 candidates assigns the encoder user template and service-scoped Model Routes before fixed-axis expansion; budget_ratio is the fixed four-value Variance Source, so each candidate produces four Graph Config materializations and four Materialization Records.
Optimization Step protocolThe identity optimizer executes one pure optimization step: its immutable step request has no Tool Configs, and its immutable step result returns unchanged candidates with no evaluation intent because this transition needs no optimizer-internal evaluation. The terminal optimization result references that sole Step Result; official evaluation remains downstream.
Proposal MaterializationAn optimizer returns ordered proposals containing a base and permitted prompt-template assignment. The harness validates and Diff Checks every proposal, expands every valid assignment across all four budget ratios, and emits one Graph Config plus one Materialization Record for every complete slot. Selection Policy then selects an ordered subset of those record slots for evaluation; the stated baseline selects all 32 slots.
Graph ConfigThe baseline performs 8 valid candidates × 4 budget ratios = 32 materializations and therefore emits 32 Graph Config outputs plus 32 Materialization Records before selection. The stated Selection Policy selects all 32 slots. In the stated nonconvergent baseline those selected outputs have 32 distinct Graph Hashes and identify 32 Rollout Variants; if assignments converge, multiple selected records may share one graph_hash and the deduplicated variant count is lower. The dr-graph domain selects the complete static identity-bearing payload, including Provider Call Config fields/references, budget_ratio, Evaluation Procedure Config reference, Node Input Sources, declared outputs, and unique terminal Node. Any route, template, temperature, ratio, or Procedure Config change that changes the concrete graph changes its Graph Hash.
Evaluation and identities
Eval ConfigThat ordinary Eval Config is bound under the Evaluation Role official and materialized from one Eval Definition. That Config composes the pinned 164-task Sampling Config and three-repeat plan; an Evaluation Procedure Config fixing preprocessing, all-candidate correctness, per-observation Score derivation, metric extraction, semantic sandbox image/version, and zstd-19/reference semantics; and an Aggregation Config fixing complete unweighted reductions. The Whetstone experiment separately declares correctness-maximizing and compression-minimizing Objectives over those outputs. The Procedure Config identity MUST exactly match the Eval Node/Graph Config assignment. Each Task supplies Graph External Inputs, evaluation inputs, and reference material excluded from Rollout Variant identity.
Evaluation ContextThe official harness binds that Eval Config, Evaluation Authority, campaign, execution policy, released dependency package versions, source revisions, and realized environment attestation and provenance at execution time. Optimizer tools may bind cheaper Eval Configs through separate internal contexts without changing Graph Hashes only when they preserve the exact Evaluation Procedure Config identity; changing only Sampling or Aggregation changes eval_config_hash, while changing Procedure Config changes both hashes.
Rollout KeyEach observation is keyed by (graph_hash, eval_config_hash, task_identity, repeat_id), where both hash fields carry full Identity Hashes and the referenced Eval Config has the official role in this context; the optional RNG seed is absent because provider randomness is uncontrolled, retries never change repeat_id, and Materialization Record references are excluded.
Rollout Execution KeyThe official Evaluation Context ID joins the Rollout Key to form the Work Item and terminal-index identity; a deliberate later re-evaluation must use a new context ID or Repeat ID.
Rollout ResultEach immutable final result records the Rollout Execution Key, Graph Hash and Eval Config Identity Hash, Evaluation Context and authority identities, a nested Graph Run Result, facts and named scores or exhausted causal failure, checkpointed Provider Call Attempt observations, and Platform Stage Attempt plus Durability Replay/checkpoint evidence. The Graph Run Result contains no Platform Stage state and has no separate authoritative persistence path. It carries no Materialization Record reference. Its full canonical persisted record receives a Content Hash and typed Rollout Result Object Reference carrying that hash in content_hash.
Platform mapping and execution
Campaign and submissionOne Whetstone Evaluation Campaign maps to one Platform Campaign; each submission group is one Submission Batch and native Platform Submission Run.
Rollout Work ItemOne Work Item maps to one Rollout Execution Key; its native input_ref carries a typed Object Reference with content_hash for one exact immutable Rollout Work Request, and a succeeded stage's output_reference does the same for the terminal Rollout Result.
Provider Concurrency ControlWhetstone configures the default empty-selector Stage capacity and exact-label capacities for every encoder and decoder (provider, protocol, model) identity; each Work Item carries every applicable collision-free label. Admission applies all matching controls for the full Stage lifetime, including backoff; without the default control, work remains unadmitted.
Attempts and terminalizationEach bounded logical provider attempt runs as one retry-disabled DBOS step and returns a serializable expected-outcome observation; DBOS checkpoints completed observations and Whetstone deterministically sleeps, backs off, branches, and reconstructs Node Outcomes and the Graph Run Result on replay. Whetstone injects no completed Node Output across workflow identities. The Eval terminal Node's ordinary output becomes Graph Output unchanged. One immutable Rollout Result containing the Graph Run Result and completed provider observations is persisted and atomically bound; there is no custom execution ledger. A persisted and bound success or exhausted-failure Result makes the Stage succeed; the FAILED-plus-unbound operator gateway and executor recheck remain authoritative.
Facts-first code evaluation
Artifact pipelineThe Whetstone adapter projects exact decoder Generation into the Submission Text role of native TextArtifact and supplies the experiment-specific Task inputs and Evaluation Procedure Config. The dr-code kernel performs preprocessing fan-out, emits the Preprocessing Trace and typed Code Candidate Set, compile-checks valid Code Artifacts, and extracts Metric Facts and Scores. Binary Test Pass ScoreCode-test Metric Facts derive score 1 when any fully evaluated Code Artifact passes the full task suite and 0 only when all fully evaluated Code Artifacts definitively fail; infrastructure uncertainty terminates the rollout as failed.
Compression RatioCompressed Description Length is the integer zstd level 19 byte count of exact encoder Generation UTF-8, divided by nonzero UTF-8 bytes of task.gt_code_wo_comments. The output-affecting budget_ratio is a static Graph Config assignment, while the concrete Task-derived Character Budget is runtime input excluded from Rollout Variant identity; the prompt budget uses character count, separate from the compression byte denominator.
Rollout AggregateIn the stated nonconvergent baseline, D = 32 distinct Graph Hashes and each complete 164 × 3 planned matrix yields one Average Binary Test Pass Rate and Mean Compression Ratio aggregate; failures and missing observations follow declared policy and are never silently dropped.
