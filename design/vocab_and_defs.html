Draft dr-serialize Vocabulary

This section is a first-pass draft for discussion, is not Whetstone normative vocabulary, and does not rename the dr-serialize API. dr-serialize owns normalization policy, canonical JSON rendering, and a digest primitive; it does not own domain schemas, artifacts, references, decoding, or Whetstone definition, config, run, and result contracts.

TermDraft definition
JSON-Safe NormalizationThe potentially lossy Serializer.to_jsonable conversion of an arbitrary Python value into a JSONable value under explicit limits and handlers.
JSONable ValueA value in the recursive Jsonable union of JSON-shaped scalars, lists, and string-keyed dictionaries; the type and normalization admit non-finite floats, but canonicalization rejects them.
SerializerA frozen reusable Serializer binding one serialization limits value to an ordered, instance-local tuple of JSONable handlers.
Serialization LimitsA frozen SerializationLimits value specifying normalization depth, payload-byte, and optional storage hard ceilings.
JSONable HandlerAn instance-local JsonableHandler returning a JsonableHandle and recursing through ConversionContext, with no global handler registry.
dr-serialize Canonical JSON TextThe canonical_json rendering of an already-Jsonable value under dr-serialize's pinned compact, sorted-key Python JSON profile, without claiming RFC 8785 conformance.
SHA-256 JSON DigestThe full hexadecimal or consumer-selected prefix returned by sha256_json_digest over dr-serialize canonical JSON text, becoming a domain hash only when a consumer assigns identity semantics.
JSON PathA JsonPath tuple of string keys and integer indexes locating a value within normalization or encoding diagnostics.
Serialization Error DiagnosticThe error-specific, persistence-safe mapping returned by a SerializationError.diagnostics implementation, including its JSON path and bounded detail.
Open integration questions.
  • What protocol or version guarantee pins canonical text and normalization behavior?
  • Should normalization and canonicalization use one consistent non-finite-float policy?
  • What deterministic policy, if any, should govern sets and non-string mapping keys?
  • Which consumer owns digest length and the decision that a digest establishes identity?

Draft dr-providers Vocabulary

This section is a first-pass draft for discussion, is not Whetstone normative vocabulary, and does not rename the dr-providers API. dr-providers owns provider-call configuration, transcripts, wire translation, success normalization, telemetry, failure classification, and local transport retry; it does not own graph or run identity, persistence, latency, scoring, officialness, or streaming.

Native dr-providers Provider means the executor protocol represented below as provider client; Whetstone provider means the external service, so matching words do not establish a normalized role.

TermDraft definition
Provider ServiceThe external vendor or service family identified by ProviderKind, distinct from both its endpoint protocol and the native Provider executor protocol.
Provider ClientA synchronous executor implementing native Provider.complete, such as HttpProvider or ScriptedProvider, that returns LlmResponse on success or raises a provider exception.
Provider Call ConfigA concrete request-carried ProviderConfig describing service, endpoint, model, credentials source, supported controls, and wire shape, with no separate Provider Definition or bound-client lifecycle.
Requested Model NameThe model string in ProviderConfig.model sent on the wire for a provider call.
Reported Model NameThe response-side LlmResponse.model, populated from the provider body when present and otherwise from the requested config by built-in parsers.
Prompt MessageA frozen PromptMessage pairing one MessageRole with a content string.
TranscriptThe ordered, possibly empty LlmRequest.messages tuple of prompt messages, for which dr-providers defines no separate History type.
Provider Call RequestA frozen LlmRequest containing one provider call config, a transcript, explicit generation controls, optional extra body, and optional idempotency key.
Provider Call ResponseAn LlmResponse success envelope containing normalized text and optional telemetry, warnings, identifiers, diagnostics, and raw metadata, but neither its request nor a failure and therefore not a generic Whetstone result.
GenerationA possible Whetstone role mapping for accepted nonblank LlmResponse.text; dr-providers exposes the text field but no native Generation type.
Token UsageOptional TokenUsage counters for prompt, completion, total, and reasoning tokens normalized from a provider response.
Provider CostThe optional LlmResponse.cost, when present, is a CostInfo containing a required numeric total parsed from supported response-body fields plus a currency field that currently defaults to USD; it is not a Whetstone reward or objective.
Provider Response DiagnosticsContent-free, allowlisted ResponsesDiagnostics observations from an OpenAI Responses body, distinct from unbounded raw provider response metadata.
Provider Conformance WarningAn LlmWarning recording observed response or request-response conformance evidence while leaving fatality policy to the caller.
Provider FailureA frozen ProviderFailure classified by FailureClass and carried by typed exceptions, with dr-providers classifying the condition while callers own broader recovery policy.
Transport Retry PolicyA local TransportPolicy for timeout and bounded retry/backoff inside HttpProvider, separate from any Whetstone or durable-workflow retry layer.
Raw Provider ResponseThe unbounded provider-specific response body retained in LlmResponse.provider_metadata, not the content-free diagnostic projection.
Open integration questions.
  • Does model identity include endpoint or protocol as well as provider service and model name?
  • Should an LLM node output generation text or a projection of the full provider response?
  • Which layer owns latency measurement?
  • How do transport retries compose with Whetstone and durable-workflow retries?
  • Can raw provider metadata be stored durably, and under what retention and redaction contract?

Draft dr-graph Vocabulary

This section is a first-pass draft for discussion, is not Whetstone normative vocabulary, and does not rename the dr-graph API. dr-graph owns concrete flat graph specifications, routing validation, a sequential interpreter, runner-local summaries, and graph digests; it does not own variable-bearing Whetstone rollout definitions, node semantics, provider or evaluation behavior, durability, or a complete rollout result.

GraphSpec and NodeConfig are already concrete rather than Whetstone definitions with unset variables, and native GraphRunResult is intentionally narrower than Whetstone's generic result contract.

TermDraft definition
Graph SpecA concrete validated flat DAG in GraphSpec, containing an ordered non-empty tuple of nodes and one terminal node identifier, with declared node order included in its digest payload.
Node SpecA NodeSpec pairing a validated node identifier with an open operation name and concrete node configuration.
Node OperationThe open string NodeSpec.op interpreted by the caller's execution callback, with no registry or built-in LLM or Eval meaning in dr-graph.
Node ConfigurationA concrete NodeConfig containing field declarations, input bindings, an output field, and parameters, with no Definition lifecycle or unset variables.
Node FieldA FieldSpec declaration of name, input/output role, type-name string, and optional description, without runtime validation of the declared type_name.
Binding RefA string-serialized BindingRef naming either an external field or an upstream node output and thereby deriving a graph connection.
External InputA value supplied through the reserved task.<field> binding namespace, whose “task” spelling does not require the value to be a Whetstone task.
Node ParametersThe fully assigned, digest-covered NodeConfig.parameters mapping passed as part of the concrete spec, not a set of unset Whetstone variables.
Terminal NodeThe node named by GraphSpec.terminal_node_id whose configured output or terminal error becomes the graph run's terminal summary.
Graph DigestThe consumer-length SHA-256 prefix returned by graph_digest over a dr-serialize-canonical graph payload, assigned content-identity meaning by dr-graph.
Inlined SubgraphA subgraph transformed by inline_subgraph into prefixed and rewired parent nodes at build time, leaving no subgraph boundary at runtime.
Graph ExecutionAn execute_graph interpretation that deterministically visits nodes in topological order and routes values while the supplied RunNode callback owns node effects and semantics.
Node OutputA NodeOutput mapping of named values plus optional metadata returned or validated for a successful node invocation.
Node OutcomeA NodeOutcome recording one node's SUCCESS, ERROR, or derived BLOCKED state with the corresponding output, error snapshot, or blocking dependencies.
Graph Run ResultA runner-local GraphRunResult containing node outcomes, execution order, status, and terminal summary, but omitting config identity, original inputs, and measurements and therefore not a Whetstone rollout result.
Open integration questions.
  • Does GraphSpec sit inside an enclosing Whetstone Rollout Config, and which values remain outside it?
  • What concrete parameter and output schemas identify LLM and Eval node operations?
  • What wrapper supplies complete inputs, config identity, measurements, and failure context around GraphRunResult?
  • Should deep immutability and declared node-order identity become explicit contracts?
  • What digest length and versioning rule is stable across persisted graph identities?

Draft dr-code Vocabulary

This section records dr-code-owned contracts for integration with Whetstone. It is a first-pass draft for discussion, is not yet Whetstone normative vocabulary, and does not rename any dr-code API.

The concrete vocabulary follows dr-code's public types and behavior: it uses Definition, Runner, and Trace rather than “Preprocess Flow,” because configuration, bound execution, and retained runtime evidence are separate contracts; likewise, “Metric Extraction” names the behavior of extract_metrics, not a nonexistent “Metric Extraction Flow” API.

TermDraft definition
Inputs and artifacts
Submission TextThe runtime role of the raw string supplied for dr-code processing from an upstream Whetstone LLM generation; the generation itself remains a Whetstone generation, and the string is not yet an artifact.
Text ArtifactA dr-code TextArtifact wrapper carrying arbitrary text, with no claim that the text is code, constructed from submission text before preprocessing.
Python SourceA raw string interpreted as Python source, without by itself asserting that compilation succeeds.
Code ArtifactA dr-code CodeArtifact carrying a source string under the stronger producer contract that it has passed an upstream compilation check.
Code CandidateOne Python source string at an ordered position in a code candidate set artifact, distinct from Whetstone's experiment-level candidate.
Code Candidate Set ArtifactA dr-code CodeCandidateSetArtifact containing zero or more ordered code candidate source strings and optionally aligned lineage; an empty set is a present artifact, not absent.
AbsentA dr-code Absent causal sentinel for a step that failed while processing a present input, retaining failure and propagation lineage; a missing upstream submission produces no preprocessing trace, while a missing trace key is a wiring error.
Preprocessing
Preprocessing DefinitionA dr-code PreprocessingDefinition, the fully concrete, immutable, versioned specification of ordered named steps and settings, which therefore behaves like a Whetstone config despite its existing class name.
Bound Preprocessing RunnerA reusable BoundPreprocessingRunner produced by resolving a preprocessing definition's steps, validating settings and artifact-kind wiring, and pinning producer identity before it accepts and kind-checks input artifacts.
Preprocessing TraceThe dr-code Trace returned for one input, generically retaining artifact-or-absent values, step facts, and producer identity; the named HumanEval function-candidates definition specifically ends in Absent or one CodeCandidateSetArtifact, not a list of CodeArtifacts.
Metric extraction
Metric FamilyA stable kind of metric question identified by dr-code's MetricName, such as code_test or compressed_length, and implemented by a versioned operator.
Metric QuestionA dr-code MetricQuestion declaring one metric family, one trace key to evaluate, and the operator settings that determine its semantics.
Metrics DefinitionA dr-code MetricsDefinition, the fully concrete, immutable, versioned specification of an ordered collection of metric questions, which therefore behaves like a Whetstone config despite its existing class name.
Metric ExtractionThe extract_metrics operation that evaluates one dr-code Trace once per metric question in declaration order and returns ordered metric records.
Metric RecordA self-describing dr-code MetricRecord answering one declared question with one or more named metric values when measured, or with explicit not_applicable or operator_failure state.
Metric ValueOne named scalar fact in a measured metric record's values mapping, not a Whetstone score or objective unless Whetstone derives one.
Code-Test FactsThe code_test record values describing case counts, pass/fail/error/timeout counts, coverage completeness, function count, and selected function name, not Whetstone's binary test pass score.
Compressed-Length FactsThe compressed_length record values for UTF-8 representation bytes, compressed bytes, and optional reference ratio and percent reduction under pinned codec, level, and reference-source semantics, not automatically Whetstone's chosen compression objective.

Draft dr-platform Vocabulary

This section is a first-pass draft for discussion, is not Whetstone normative vocabulary, and does not rename the dr-platform API. dr-platform owns durable work identity, submission, stage state, admission, selector capacity, controls, retry and cancellation intent, and inspection; it does not own durable workflow execution and recovery, which DBOS owns, or domain input and output resolution, configuration, stage logic, results, and officialness, which the application owns.

Native PipelineDefinition is concrete and linear, a StageDefinition is not a Whetstone node, operation DTOs whose names end in Result are not generic Whetstone results, references remain opaque and application-owned, and deterministic DBOS workflow IDs are recipes rather than content hashes.

TermDraft definition
Platform CampaignThe application namespace identified by CampaignKey for related logical work, without defining that work's domain meaning.
Platform Submission RunOne immutable pipeline submission invocation identified by RunKey and persisted as PipelineRunRecord, not the generic Whetstone run.
Platform Work ItemLogical work identified by CampaignWorkIdentity and persisted with an origin run, opaque input reference, and labels, whose mapping to a Whetstone rollout unit remains undecided.
Pipeline DefinitionA concrete, immutable, versioned PipelineDefinition containing an ordered non-empty linear sequence of stages, not a Whetstone definition or graph.
Pipeline IdentityThe PipelineIdentity pair of PipelineKey and positive version, assigned by declaration rather than derived as a content hash.
Stage DefinitionOne ordered StageDefinition naming a stage key, DBOS queue, application callable, and argument adapter in a linear pipeline, not a dr-graph or Whetstone node.
Pipeline RegistryAn in-memory PipelineRegistry mapping pipeline identity to the registered authoritative PipelineDefinition; production dispatcher wiring separately requires wrapped definitions, while submission-only registries may contain unwrapped definitions.
Stage ExecutionThe single logical StageExecutionRecord for one work-item stage across all of its retries.
Stage AttemptOne physical attempt in an append-only numbered sequence for a stage execution; its persisted row advances through controlled admission and terminal fields, while StageAttemptRecord is a frozen snapshot DTO.
Stage StateThe logical StageExecutionState value READY, ADMITTED, SUCCEEDED, FAILED, or CANCELLED; there is no RUNNING state.
AdmissionThe transactional act that selects eligible READY work under controls, appends or admits its attempt, transitions the logical stage to ADMITTED, and enqueues the DBOS workflow.
Input ReferenceA non-empty opaque application-owned string stored as WorkInput.input_ref and WorkItemRecord.input_reference, whose target and resolution protocol dr-platform does not interpret.
Output ReferenceA non-empty opaque application-owned string returned by successful stage logic and stored on StageExecutionRecord.output_reference, not the output value itself.
Work LabelsThe immutable string mapping on WorkInput.labels and WorkItemRecord.labels used to match selector-scoped controls without defining domain semantics.
Stage ControlA persisted StageControlRecord scoped by pipeline identity, stage, and exact label selector, carrying capacity and pause settings.
Selector CapacityThe maximum concurrent ADMITTED occupancy allowed by a matching stage control, not a provider request-rate or token limit.
Terminal SummaryThe optional diagnostic JSON mapping in StageAttemptRecord.terminal_summary describing a physical attempt's terminal outcome, not a generic Whetstone result.
Open integration questions.
  • Is a platform work item a rollout, a task/config pair, or another scheduling unit?
  • What exactly does execution_config_reference identify and who resolves it?
  • Which result and reference store resolves domain inputs, outputs, and complete Whetstone results?
  • How do provider, graph, platform, and application retry layers coordinate?
  • Where does concurrent selector occupancy end and provider rate limiting begin?
  • Which Whetstone lifecycle or authority boundary, if any, should a campaign represent?

Running Example: Eval Experiment

ConceptConcrete detail in the Eval experiment
Rollout
Rollout configs8 candidates × 4 budget ratios produce 32 rollout configs, identified by rollout_config_hash.
Rollout graphEach rollout executes Encoder LLM Call Node → Decoder LLM Call Node → Eval Node for one (rollout config, task, seed).
Rollout ResultEach Official Evaluation result is keyed by (rollout config hash, official eval config hash, task, seed) and contains Binary Test Pass Score, measured zstd bytes, and metadata for tokens, cost, and latency. Its Official Evaluation context — not hash identity — makes it official. A failed rollout still produces a result with its cause — never a gap.
Inference resources
Providers and modelsThe system currently supports OpenAI and OpenRouter. For example, gpt-5.4-nano is provided by the OpenAI provider, while openai/gpt-5.4-nano is provided by OpenRouter.
Experiment definition
CandidateA candidate is an (encoder user template, provider, model) tuple.
Variance and curvesThe only variance source is budget_ratio. Its four settings distinguish the rollout configs and trace each candidate's curve. Tasks and seeds are not variance sources: both are averaged into each rollout aggregate.
TasksIn the HumanEval task mapping, each task supplies a ground-truth code sample and tests for the use case. The encoder's input and reference code sample are task.gt_code_wo_comments; the task set defines the quantity the result estimates.
SeedsGeneration randomness comes from API calls to the model providers, so repeats do not change a controlled rng seed; they repeat the calls. Repeats capture generation stochasticity and buy confidence in the result (see the eval doc’s Additional Notes).
Reference length and compressionReference length is the length of task.gt_code_wo_comments. The budget is a fraction of that length, and official compression is the description's measured zstd bytes divided by it.
Eval ConfigsThe shared shape is {task_set}, {seeds}, {reference_length}, {codec}, {preprocess_config}, {metrics_extract_config}, and {aggregation_method}. Task set, seeds, and aggregation drive the surrounding sweep; the eval node uses the remaining evaluation settings. For this Eval setup, internal_eval_config_hash == official_eval_config_hash. Optimization normally uses a cheaper, different internal eval config with the same preprocessing and metric extraction but fewer tasks and seeds, so the hashes differ.
Measurement and aggregation
Metrics and aggregationPer rollout, Binary Test Pass Score is 1 only when the reconstructed code passes the task's full test suite, otherwise 0; it is never a fraction of tests passed. Per rollout config, Average Binary Test Pass Rate is the per-task mean over seeds followed by the unweighted mean over the task set, and measured compression is averaged.
Rollout AggregateEach rollout aggregate combines one rollout config's official rollout results into (Average Binary Test Pass Rate, mean measured compression). The experiment produces 32 aggregates — 4 per candidate — for the plot.
Authority and integrity
Hashes and image digestSerialized Rollout Definitions, rollout configs, and eval configs use content hashes as identity: the same bytes give the same hash and any change produces a new one. Hashes also key rollout results, but they do not assign eval-config roles or confer officialness. The sandbox image retains the OCI term “image digest.”
