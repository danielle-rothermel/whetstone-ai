This page defines the shared terms used consistently throughout the whetstone design-doc family.

Normative Vocabulary

Terms follow the experiment lifecycle, from system building blocks through authority and integrity.

TermDefinition
System building blocks
ProviderService providing API endpoints that accept a model plus prompt/history and return LLM generations.
ModelSpecific model available from a provider.
ArchitectureGraph shape naming each node's inputs, variables, and outputs without assigning values.
Experiment definition
CandidateAssignment of values to variables the experiment compares; optimizer proposals may change only permitted candidate variables.
Initial candidatesCandidate values supplied to the optimizer within initial rollout configs as proposal starting points.
Variance sourceSetting intentionally swept within a candidate, producing distinct rollout configs and results.
TaskSingle dataset element evaluated by a rollout.
SeedRepeat of an otherwise identical configuration used to measure evaluation randomness; an RNG seed when fully controlled.
Reference lengthNormalization basis for setting relative budgets and reporting compression.
Score Node Eval ConfigEval config carried as Score Node variables in the rollout config: {task_set}, {seeds}, {reference_length}, {codec}, {preprocess_config}, {metrics_extract_config}, and {aggregation_method} — all included in the rollout config hash. Cross-run fields (task set, seeds, aggregation) direct the sweep driver; each rollout uses the remaining fields.
Rollout configFully bound architecture — candidate, variance-source, and fixed values — forming an executable unit identified by graph_hash.
Execution
RolloutOne rollout config executed on one (task, seed), producing one rollout result.
Measurement and results
Binary Test Pass ScoreCorrectness score for one rollout (config × task × seed): exactly 1 when the reconstructed code passes the task's full test suite; otherwise 0, with no partial credit.
Rollout resultConfigured measurements and metadata from one rollout. Failures remain explicit results with causes, never gaps.
Average Binary Test Pass RateAggregate correctness of one rollout config: mean Binary Test Pass Score across seeds per task, then unweighted mean across tasks.
Rollout config resultAggregate of one rollout config's official rollout results.
Authority and integrity
Official evaluationHarness's authoritative pass under the official eval config; alone produces official rollout results used in plots. Optimizer tool evaluations are non-official.
HashContent hash identifying a serialized object (rollout config, architecture, eval config): identical bytes yield the same hash; any change yields a new hash. It also keys rollout results. OCI sandbox images retain the term "image digest".
Diff checkHarness comparison of returned rollout configs with initial configs, confirming only permitted changes.

Running Example: Eval Experiment

ConceptConcrete detail in the Eval experiment
System building blocks
Providers and modelsThe system currently supports OpenAI and OpenRouter. For example, gpt-5.4-nano is provided by the OpenAI provider, while openai/gpt-5.4-nano is provided by OpenRouter.
Experiment definition
CandidateA candidate is an (encoder user template, provider, model) tuple.
Variance and curvesThe only variance source is budget_ratio. Its four settings distinguish the rollout configs and trace each candidate's curve. Tasks and seeds are not variance sources: both are averaged into each rollout config result.
TasksIn the HumanEval task mapping, each task supplies a ground-truth code sample and tests for the use case. The encoder's input and reference code sample are task.gt_code_wo_comments; the task set defines the quantity the result estimates.
SeedsGeneration randomness comes from API calls to the model providers, so repeats do not change a controlled rng seed; they repeat the calls. Repeats capture generation stochasticity and buy confidence in the result (see the eval doc’s Additional Notes).
Reference length and compressionReference length is the length of task.gt_code_wo_comments. The budget is a fraction of that length, and official compression is the description's measured zstd bytes divided by it.
Score Node Eval ConfigFor an Eval run, the Score Node Eval Config matches the Official Eval Config — that is the point. While iterating in another setting, an optimizer may use a cheaper config with the same preprocessing and metrics extraction but a smaller task subset and fewer seeds.
Rollout configs8 candidates × 4 budget ratios produce 32 rollout configs, identified by graph_hash.
Execution
Rollout graphEach rollout executes encoder → decoder → score for one (rollout config, task, seed).
Measurement and results
Rollout resultEach result is keyed by (rollout config hash, official eval config hash, task, seed) and contains Binary Test Pass Score, measured zstd bytes, and metadata for tokens, cost, and latency. A failed rollout still produces a result with its cause — never a gap.
Metrics and aggregationPer rollout, Binary Test Pass Score is 1 only when the reconstructed code passes the task's full test suite, otherwise 0; it is never a fraction of tests passed. Per rollout config, Average Binary Test Pass Rate is the per-task mean over seeds followed by the unweighted mean over the task set, and measured compression is averaged.
Rollout config resultEach result aggregates one rollout config's official rollout results into (Average Binary Test Pass Rate, mean measured compression). The experiment produces 32 such results — 4 per candidate — for the plot.
Authority and integrity
Hashes and image digestSerialized architectures, rollout configs, and eval configs use content hashes as identity: the same bytes give the same hash and any change produces a new one. Hashes also key rollout results. The sandbox image retains the OCI term “image digest.”
