Run Configurations

The canonical Eval Definition defines the measurement-policy shape. dr-graph's versioned, variable-bearing DAG Graph Definition is used by Whetstone in the role of Rollout Definition; those names do not denote separate artifacts or identities. Each complete output-affecting assignment and axis selection materializes one concrete Graph Config plus one Materialization Record; the Graph Config is one Rollout Variant and graph_hash is its sole identity. A separately identified Eval Config is bound through an Evaluation Context when that variant is measured, but it must reference the identical Evaluation Procedure Config statically assigned to the Graph's Eval Node.

Designer

Freezes one Graph Definition used in the Rollout Definition role, 8 starting Experiment Candidates and four ratios. Materialization produces 32 fully assigned Graph Config outputs and 32 Materialization Records. The official harness independently binds the Eval Config in an authorized Evaluation Context; that Eval Config references the same Evaluation Procedure Config assigned to every materialized Eval Node.

Evaluation harness

Issues the OptimizationRequest to the identity optimizer: 8 starting Experiment Candidates, fixed materialization axes, mutation surface ∅, toolkit ∅, guidance ∅.

Optimizer

Returns 8 ordered Optimization Proposals unchanged — zero tool calls, zero LM calls, no extras.

Evaluation harness

Validates all 8 proposal payloads against their starting candidates, then expands every valid proposal over four fixed ratios to emit 32 Graph Config/Materialization Record pairs. Evaluate-all selects all 32 ordered record slots, deduplicates the selected records by full Graph Hash, and admits D distinct graphs. Official Evaluation Records and Official Plot Manifests MUST preserve the ordered mapping from every selected Materialization Record reference through graph_hash to the shared planned/result-key set and aggregate reference, including when multiple selected records share one graph and aggregate.

Evaluator

Creates one Platform Campaign for the Evaluation Campaign/context and submits exactly D × 164 × 3 Work Items, unique by Rollout Execution Key. Here D=32, so the explicit nonconvergent example has 15,744. Whetstone derives collision-free labels from every exact (provider, protocol, model) identity and applies every relevant label to multi-route work. Provider Concurrency Control requires the default empty-selector capacity plus every exact-label capacity; all matching controls hold coarse ADMITTED-Stage capacity through backoff, not exact call-boundary concurrency.

Evaluator

DBOS checkpoints every completed retry-disabled Provider Call Attempt observation. Whetstone deterministically reconstructs Node Outcomes and the Graph Run Result on replay, nests that evidence in one Rollout Result, then immutably persists and atomically binds that authoritative Result by Rollout Execution Key; there is no separate Graph Run Result persistence path. An unbound key binds and same-ref replay succeeds; only then does dr-platform return the authoritative ref and mark the Stage SUCCEEDED. Failure leaving the key unbound raises FAILED; operator retry proceeds only through Whetstone's FAILED+unbound gateway and executor recheck. A different-ref conflict preserves the winner. A crash after provider effect but before checkpoint may duplicate physical calls or billing; only the checkpointed physical outcome is retained.

outputthe authoritative bound Object Reference for each completed key; otherwise an explicit FAILED Stage — never replacement of an existing binding
Evaluation harness

Aggregates only complete, officially attested results: exactly one aggregate per distinct admitted Graph Hash, hence D aggregates. Each uses per-task mean Binary Test Pass Score over 3 repeats, then unweighted mean over all 164 tasks, plus mean Compressed Description Length ratio. An exhausted infrastructure failure keeps the aggregate incomplete; deliberate re-evaluation needs a new context or Repeat. The explicit nonconvergent baseline has D=32.

Each selected candidate retains four separately attributable materialization slots across ratios, while converged slots share the planned/result-key set and Rollout Aggregate for their Graph Hash. Additional Notes

INPUT_CODE, the budget, and the reference length

  • {INPUT_CODE} is populated from the pinned task's task.gt_code_wo_comments; the Graph Config declares the input port and its Node Input Source as a Graph External Input, while the concrete task value belongs to the Rollout Key and Evaluation Context, not Graph Config identity
  • Compressed Description Length = integer zstd_bytes(encoded_code_rep) at level 19; Compression Ratio divides it by UTF-8 reference bytes
  • MAX_BUDGET = round(BUDGET_RATIO × chars(task.gt_code_wo_comments)), derived per task — at ratio 1.0 the description may be as long as the reference
  • Guidance, not enforcement: the budget steers, nothing clips; rollouts that blow it land further right on the plot

Candidates and the variance source

  • 8 candidates = encoder user template × Model Route — one plotted line each; the route label is display shorthand, while complete output-affecting Provider Call Config refs/hashes contribute to Rollout Variant identity
  • Variance source = budget_ratio — spreads each candidate into its distribution of rollout aggregates
  • Repeat IDs are repetition, not retry attempt numbers: averaged into each rollout aggregate for confidence. An RNG seed may be recorded as optional metadata but is not the repeat's identity
  • 8 × 4 ratios = 32 Graph Config materializations and 32 Materialization Records; in this nonconvergent baseline, 32 distinct Graph Hashes × 164 tasks × 3 repeats = 15,744 Rollouts

Tasks are not a variance source

  • The task set defines the quantity each rollout aggregate estimates: the average score over it
  • Fewer or sampled tasks → a noisier estimate of the same average
  • A different task set (train / validation / test) → a different quantity: a different eval config

Identity: Graph Config vs candidate

  • graph_hash, eval_config_hash, context IDs, and result IDs are separate, full schema-versioned SHA-256 identities; Graph Hash is the sole Rollout Variant config identity and is not a Content Hash
  • graph_hash covers the complete static output-affecting Graph Config, including Provider Call Config fields, templates, ratio, derivation rule, and the Eval Node's Evaluation Procedure Config ref/hash; it excludes Eval Sampling, Aggregation, Evaluation Context, and runtime task-derived Graph External Input values
  • Candidate identity is only the candidate variables' values
  • Changing the Evaluation Procedure Config changes both graph_hash and eval_config_hash; changing only Sampling or Aggregation changes eval_config_hash while preserving graph_hash, producing different Rollout Keys

Contract fit

  • The full experiment lands in existing boxes: Graph Config, Materialization Record, Eval Definition and Eval Config roles, or rollout-time derivation
  • One new mechanism: the per-task budget derivation inside the encoder stage
