Note on "budget" computation and usage: MAX_BUDGET = round(ratio × chars(gt_stripped)), where gt_stripped is dr-code's HumanEvalTask.ground_truth_code_without_comments. At ratio 1.0 the description may be as long as the stripped code itself — the calibration anchor. Consistent with our guidance-not-enforcement stance, MAX_BUDGET steers the encoder but nothing clips its output: the measured objective is the actual description's compressed bytes, so budget-blowing candidates simply land further right on the plot, and a budget-compliance ratio (chars_used / MAX_BUDGET) is recorded per run.

Notes on the architecture

  • The shape and every generator-filled slot value are part of each candidate's digest.
  • The task-input slots ({code}, {tests}) are filled per task at run time — the one kind of slot the generator never fills and the digest never covers.
  • The candidate generator fills the remaining slots to produce the 32 candidates — including {codec} = zstd-19, which is why the score node is red (see the break note below).
  • {prompt_kvs} keeps the architecture template-agnostic: which keys exist, and any derived values (budget_ratio → MAX_BUDGET), are a convention between the template text and its kvs — template implementation detail, not architecture. The derivation code is versioned with the encoder op so identical kvs always mean identical prompts.
  • Correctness-scoring config (preprocessing version, sandbox image, timeout) comes from Ccert, not from any slot.
  • A future control plane that edits the shape itself is architecture search — the same contract, with a wider mutation surface. An architecture digest (shape with slots blanked) is the natural plot grouping key. viz
the generator fills the slots — {codec} = zstd-19 for all 32 — over the 4 × 2 × 4 grid ▼

Candidate

The architecture with its slots bound — the system being measured. Digest-covered: any change here is a new candidate with a new graph_digest.

  • provider + model — split slots on both LM nodes; this run binds matched encoder/decoder pairs
  • Encoder user_prompt_template — A | B, id + full text; system_prompt_template bound empty this run
  • Encoder {prompt_kvs} — {"budget_ratio": 0.25 | 0.5 | 0.75 | 1.0}; decoder kvs empty
  • Decoder user_prompt_template — constant across all 32, still digested; system empty
  • temperature: 1.0 on both nodes, pinned
  • {codec} = zstd-19 on the score node — measurement config inside the candidate, by decision

Same-model pairing is a rule of the candidate generator, not the schema — mixed pairs need no contract change.

Ccert eval config

How we measure. Digest-tagged on every record: changing it is a new measurement, never a new candidate.

  • Task set — full HumanEval+, 164 tasks @ pinned revision
  • Repetitions — 3 per (candidate, task)
  • Correctness scoring — {preprocessing} = v1 hash, {sandbox_image} = pinned digest, {timeout}
  • Compression denominator — stripped-GT utf-8 bytes
  • Aggregation — per-task mean over reps, then task mean

The plot reads only records carrying this digest.

Run-time derivation

Computed per task during execution. Appears on records, never in specs.

  • MAX_BUDGET = round(ratio × chars(gt_stripped))
  • The instantiated encoder prompt (template + budget + sample)
  • Description length + budget compliance ratio
  • Measured zstd bytes, pass/fail — per rep

The formula is a candidate-plane rule; the values are data about one run.

Recorded break in the identity model settled: {codec} is a score-node slot bound by the generator, so compression-measurement config now lives inside the candidate digest. This breaks the rule the record keying was built on — "changing Ccert is a new measurement, never a new candidate": re-measuring under a different codec would mint new candidates rather than new records for the same candidates, and "same system, different measurement" is no longer expressible as a (candidate digest, protocol digest) pair. Decided with the break understood; the identity model and its ledger implications will be revisited when there's enough context to redo them properly. The red box above registers my objection for the record.

Full run walkthrough: one pass, 15,744 executions

Designer

Freezes the experiment: the generator expands the grid into 32 GraphSpecs (each carrying provider preset, template text + id, ratio, temperature, decoder prompt); Ccert, selection policy, and the empty surface/toolkit are serialized alongside.

ledger32 candidate digests + experiment and Ccert config digests registered
Environment

Issues the OptimizationRequest to the identity optimizer: 32 seeds, mutation surface ∅, toolkit ∅, guidance ∅.

Optimizer

Identity policy returns the ordered list [T1…T32] unchanged, no annotations.

ledgeroptimizer step: 0 tool calls, 0 tokens
Environment

Diff validation: 32/32 byte-identical to seeds, consistent with the empty surface. Selection admits all 32 to certification.

Certifier

Submits 15,744 jobs — one per (candidate, task, rep) — as a dr-platform campaign with labels {model, template, ratio}. Per-model selector capacity keeps a rate-limited provider from stalling the other three; whetstone's failure taxonomy classifies errors, retrying transient/rate-limited ones and recording permanent ones without aborting the sweep.

ledgerwork ledger opens: 15,744 items, idempotent top-up if the run is interrupted and resubmitted
Certifier

One run, concretely — T = (deepseek-v4-flash, template B, ratio 0.5), task HumanEval/36, rep 2: gt_stripped is 412 chars → MAX_BUDGET = 206. Encoder emits a 198-char description (compliance 0.96) → zstd-19 → 121 bytes → measured compression 121/412 ≈ 0.29. Decoder returns a fenced code block → preprocessing extracts one compilable candidate → sandbox runs the HumanEval+ suite → all tests pass.

ledgerone record: (T-digest, Ccert-digest, HumanEval/36, rep 2) → passed=true, desc_chars=198, budget=206, zstd_bytes=121, gt_bytes=412, tokens, provider cost, latency
Certifier

The campaign drains: 15,744 records accumulate, including classified failures (an empty generation here, a provider timeout there) — failures are records, not gaps.

ledgercompletion ledger reconciles: every (candidate, task, rep) accounted for, passed, failed, or permanently-failed-with-cause
Environment

Aggregates per candidate from certified records only: per-task pass fraction over 3 reps → unweighted mean over 164 tasks; same aggregation for measured compression. Emits 32 certified points and derives the front.

ledger32 certified aggregates; pareto front derived as a view, never stored as truth

What lands on the plot: 32 points, eight curves, one front

Each candidate contributes one certified point: mean pass rate vs mean measured compression. The natural readings: eight budget-response curves (4 models × 2 templates, each traced across the four ratios), the template A vs B comparison at matched model and ratio, and the overall front. Because the x-axis is measured compression, the budget knob's honesty is itself visible: a model that ignores MAX_BUDGET drifts right of its ratio cohort, and the recorded compliance ratio quantifies it per run.

What this concreteness proved about the contracts: the entire experiment specification landed in existing boxes — candidate spec, eval config, or run-time derivation — with exactly one new mechanism needed: the per-task budget derivation inside the encoder stage. The two choices it surfaced along the way (stripped-vs-full sample in the prompt; the plot denominator) were experiment-design decisions, not contract gaps, and both are now settled: stripped text everywhere, zstd-19 over stripped-GT bytes.
