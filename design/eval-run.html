Prompt Template Values

Some variables are defined by the task and some derived from a swept value.

  • INPUT_CODE comes from task inputs
  • BUDGET_RATIO = 0.25, 0.5, 0.75, 1.0
  • MAX_BUDGET = BUDGET_RATIO * num_chars(INPUT_CODE)

Tasks and Seeds

  • Tasks: Full HumanEval+, 164 tasks
  • Seeds: 3 rollouts per (rollout config, task)

Full run walkthrough: one pass, 15,744 rollouts

Designer

Freezes the experiment: the generator expands 8 candidates × 4 budget ratios into 32 rollout configs, each carrying its provider preset, template id + text, budget ratio, temperature, decoder prompt, and Score Node Eval Config (matching the official one). The official eval config, the selection policy, and the empty mutation surface and toolkit are serialized alongside.

Evaluation harness

Issues the OptimizationRequest to the identity optimizer: 32 initial rollout configs, mutation surface ∅, toolkit ∅, guidance ∅.

Optimizer

Returns the ordered list of 32 rollout configs unchanged — zero tool calls, zero LM calls, no extras.

Evaluation harness

Diff check: 32/32 byte-identical to the initial rollout configs, consistent with the empty mutation surface. Selection (evaluate all) admits all 32 to official evaluation.

Evaluator

Submits 15,744 jobs — one per (rollout config, task, seed) — as a dr-platform campaign with labels {model, template, ratio}. Per-model selector capacity keeps a rate-limited provider from stalling the other three; whetstone's failure taxonomy retries transient errors and keeps permanent failures as rollout results without aborting the sweep.

Evaluator

One rollout, concretely — candidate (template B, deepseek-v4-flash) at budget_ratio 0.5, task HumanEval/36, seed 2: task.gt_code_wo_comments is 412 chars → MAX_BUDGET = 206. The encoder emits a 198-char encoded_code_rep → zstd-19 → 121 bytes → measured compression 121/412 ≈ 0.29. The decoder returns a fenced code block → preprocessing extracts one compilable function → the sandbox runs the HumanEval+ suite → all tests pass: Binary Test Pass Score = 1.

Evaluator

The campaign drains: 15,744 rollout results accumulate, including classified failures (an empty generation here, a provider timeout there) — failures are rollout results, not gaps.

outputevery (rollout config, task, seed) ends with a rollout result — passed, failed, or permanently-failed-with-cause
Evaluation harness

Aggregates from official rollout results only: for each rollout config, per-task mean Binary Test Pass Score over 3 seeds, then unweighted mean over the 164 tasks — the Average Binary Test Pass Rate — and the same aggregation for measured compression. Emits 32 rollout aggregates: 8 candidates × 4 ratios.

Each candidate lands as its own distribution of 4 rollout aggregates — the variance source unrolled across the budget ratios — each aggregate one rollout config averaged over 164 tasks × 3 seeds: Average Binary Test Pass Rate vs mean measured compression. Because the x-axis is measured compression, the budget knob's honesty is itself visible: a model that ignores MAX_BUDGET drifts right of its ratio cohort.

Additional Notes

INPUT_CODE, the budget, and the reference length

  • {INPUT_CODE} is set to task.gt_code_wo_comments — stripped ground truth, dr-code's HumanEvalTask.ground_truth_code_without_comments; that choice is part of the rollout config, not the architecture
  • Reference length = utf8_bytes(task.gt_code_wo_comments); measured compression = zstd_bytes(encoded_code_rep) ÷ reference length, zstd-19
  • MAX_BUDGET = round(BUDGET_RATIO × chars(task.gt_code_wo_comments)), derived per task — at ratio 1.0 the description may be as long as the reference
  • Guidance, not enforcement: the budget steers, nothing clips; rollouts that blow it land further right on the plot

Candidates and the variance source

  • 8 candidates = (encoder user template, provider, model) — one plotted line each
  • Variance source = budget_ratio — spreads each candidate into its distribution of rollout aggregates
  • Seeds are repetition, not variance: averaged into each rollout aggregate for confidence — large even at temperature 0, provider generations aren't deterministic
  • 8 × 4 ratios = 32 rollout configs; × 164 tasks × 3 seeds = 15,744 rollouts

Tasks are not a variance source

  • The task set defines the quantity each rollout aggregate estimates: the average score over it
  • Fewer or sampled tasks → a noisier estimate of the same average
  • A different task set (train / validation / test) → a different quantity: a different eval config

Identity: rollout config vs candidate

  • The rollout config hash covers every variable, including the Score Node Eval Config — it shapes the reward an optimizer steers by
  • Candidate identity is only the candidate variables' values
  • Same candidate, different measurement = two rollout configs sharing candidate values

Contract fit

  • The full experiment lands in existing boxes: rollout config, official eval config, or rollout-time derivation
  • One new mechanism: the per-task budget derivation inside the encoder stage
