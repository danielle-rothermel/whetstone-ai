The identity optimizer executes one pure Optimization Step and returns unchanged Optimization Proposals; the terminal Optimization Result references that sole immutable Step Result. The harness then materializes fixed Graph Configs plus Materialization Records and evaluates the graphs through an authorized official context, establishing the baseline for optimization runs.

Algorithm-local vocabulary — defined here because these terms describe this baseline experiment rather than shared Whetstone ontology:
  • identity optimizer — the optimizer that performs exactly one pure Optimization Step and returns one unchanged Optimization Proposal for each starting Experiment Candidate. Its transition needs no Tool Config, tool or LM call, or Evaluation Intent.
  • per-task budget derivation — the rollout-time calculation MAX_BUDGET = round(budget_ratio × chars(input_code)) from the static ratio/rule and concrete Task input.

What does the Step Request serialize?

run configurations

The canonical Eval Definition defines the measurement-policy shape. dr-graph's versioned, variable-bearing DAG Graph Definition is used by Whetstone in the role of Rollout Definition; those names do not denote separate artifacts or identities. Each complete output-affecting assignment and axis selection materializes one concrete Graph Config plus one Materialization Record; the Graph Config is one Rollout Variant and graph_hash is its sole identity. A separately identified Eval Config is bound through an Evaluation Context when that variant is measured, but it must reference the identical Evaluation Procedure Config statically assigned to the Graph's Eval Node.

What happens across D × 164 × 3 rollouts?

Designer

Freezes one Graph Definition used in the Rollout Definition role, 8 starting Experiment Candidates, and four ratios. The official harness independently binds the Eval Config in an authorized Evaluation Context; that Eval Config references the same Evaluation Procedure Config assigned to every later materialized Eval Node.

Evaluation harness

Starts one Optimization Run and persists its sole immutable Optimization Step Request: run/Step/config identities, kind identity, index 0, 8 starting Experiment Candidates, mutation surface ∅, tools ∅, prior state/history ∅, identity output contract, and complete budget.

Optimizer

Performs the pure identity transition and returns 8 ordered unchanged candidates — zero Tool Calls, zero LM calls, no Evaluation Intent, and no extras.

Optimization harness

Persists exactly one immutable Optimization Step Result with the Step Request reference, 8 proposed and accepted unchanged candidates, empty evaluation/tool evidence, immutable empty state/history delta, zero consumed budget, remaining budget, and complete status. It then persists one terminal Optimization Result containing 8 ordered unchanged Optimization Proposals, the sole Step Result reference, and zero-cost record-local provenance. No tool or Evaluation Intent is required for this identity transition, and official evaluation has not yet occurred.

Evaluation harness

Validates all 8 Optimization Proposal payloads against their starting Experiment Candidates, then expands every valid Optimization Proposal over four fixed ratios to emit 32 Graph Config/Materialization Record pairs. The evaluate-all Selection Policy selects all 32 ordered record slots, deduplicates the selected records by full Graph Hash, and admits D distinct graphs. Official Evaluation Records and Official Plot Manifests MUST preserve the ordered mapping from every selected Materialization Record reference through graph_hash to the shared planned/result-key set and aggregate reference, including when multiple selected records share one graph and aggregate.

Evaluator

Creates one Platform Campaign for the Evaluation Campaign/context and submits exactly D × 164 × 3 Rollout Work Items, unique by Rollout Execution Key. Here D=32, so the explicit nonconvergent example has 15,744. Whetstone derives collision-free labels from every exact (provider, protocol, model) identity and applies every relevant label to multi-route work. Provider Concurrency Control requires the default empty-selector capacity plus every exact-label capacity; all matching controls hold coarse ADMITTED-Stage capacity through backoff, not exact call-boundary concurrency.

Evaluator

Within one Platform Stage Attempt, DBOS checkpoints every completed retry-disabled Provider Call Attempt observation. Durability Replay reuses checkpoint evidence while Whetstone deterministically reconstructs Node Outcomes and the Graph Run Result, nests that evidence in one Rollout Result, then immutably persists and atomically binds that authoritative Result by Rollout Execution Key; there is no separate Graph Run Result persistence path. An unbound key binds and same-ref replay succeeds; only then does dr-platform return the authoritative ref and mark the Stage SUCCEEDED. Failure leaving the key unbound raises FAILED; operator retry appends a new Platform Stage Attempt only through Whetstone's FAILED+unbound gateway and executor recheck. A different-ref conflict preserves the winner. A crash after provider effect but before checkpoint may duplicate physical calls or billing; only the checkpointed physical outcome is retained.

outputthe authoritative bound Object Reference for each completed key; otherwise an explicit FAILED Stage — never replacement of an existing binding
Evaluation harness

Aggregates only complete ordinary results referenced and certified by the Official Evaluation Record: exactly one aggregate per distinct admitted Graph Hash, hence D aggregates. Each uses per-task mean Binary Test Pass Score over 3 repeats, then unweighted mean over all 164 tasks, plus mean Compressed Description Length ratio. An exhausted infrastructure failure keeps the aggregate incomplete; deliberate re-evaluation needs a new context or Repeat. The explicit nonconvergent baseline has D=32.

What edge cases and guarantees should you keep in mind?

additional notes

INPUT_CODE, the budget, and the reference length

  • {INPUT_CODE} is populated from the pinned task's task.gt_code_wo_comments; the Graph Config declares the input port and its Node Input Source as a Graph External Input, while the concrete task value belongs to the Rollout Key and Evaluation Context, not Graph Config identity
  • Compressed Description Length = integer zstd_bytes(encoded_code_rep) at level 19; Compression Ratio divides it by UTF-8 reference bytes
  • MAX_BUDGET = round(BUDGET_RATIO × chars(task.gt_code_wo_comments)) follows the per-task budget derivation — at ratio 1.0 the description may be as long as the reference
  • Guidance, not enforcement: the budget steers, nothing clips; rollouts that blow it land further right on the plot

Candidates and the variance source

  • 8 Experiment Candidates = encoder user template × Model Route — one plotted lineage each; the route label is display shorthand, while complete output-affecting Provider Call Config refs/hashes contribute to Rollout Variant identity
  • Variance source = budget_ratio — spreads each candidate into its distribution of rollout aggregates
  • Repeat IDs identify Repeats, not retry attempt numbers: averaged into each Rollout Aggregate for confidence. An RNG seed may be recorded as optional metadata but is not the Repeat's identity
  • 8 × 4 ratios = 32 Graph Config materializations and 32 Materialization Records; in this nonconvergent baseline, 32 distinct Graph Hashes × 164 tasks × 3 repeats = 15,744 Rollouts

Tasks are not a variance source

  • The task set defines the quantity each rollout aggregate estimates: the average score over it
  • Fewer or sampled tasks → a noisier estimate of the same average
  • A different task set (train / validation / test) → a different quantity: a different eval config

Identity: Graph Config vs candidate

  • graph_hash, eval_config_hash, context IDs, and result IDs are separate, full schema-versioned SHA-256 identities; Graph Hash is the sole Rollout Variant config identity and is not a Content Hash
  • graph_hash covers the complete static output-affecting Graph Config, including Provider Call Config fields, templates, ratio, derivation rule, and the Eval Node's Evaluation Procedure Config ref/hash; it excludes Eval Sampling, Aggregation, Evaluation Context, and runtime task-derived Graph External Input values
  • Candidate identity is only the candidate variables' values
  • Changing the Evaluation Procedure Config changes both graph_hash and eval_config_hash; changing only Sampling or Aggregation changes eval_config_hash while preserving graph_hash, producing different Rollout Keys

Contract fit

  • The full experiment lands in existing boxes: Graph Config, Materialization Record, Eval Definition and Eval Config roles, or rollout-time derivation
  • One new mechanism: the per-task budget derivation inside the encoder stage
