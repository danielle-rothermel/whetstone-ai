Prompt Template Values

Some variables are defined by the task and some derived from a swept value.

  • BUDGET_RATIO = 0.25, 0.5, 0.75, 1.0

Full run walkthrough: one pass, 15,744 rollouts

Designer

Freezes one Rollout Definition/graph shape, 8 starting Experiment Candidates and four ratios. Materialization produces one fully assigned Graph Spec per Rollout Config. The official harness independently binds the Eval Config in an authorized Evaluation Context.

Evaluation harness

Issues the OptimizationRequest to the identity optimizer: 8 starting Experiment Candidates, fixed materialization axes, mutation surface ∅, toolkit ∅, guidance ∅.

Optimizer

Returns 8 ordered Optimization Proposals unchanged — zero tool calls, zero LM calls, no extras.

Evaluation harness

Diff-checks the 8 proposal payloads against their starting candidates, then applies the four fixed ratio values to materialize 32 concrete Rollout Configs. Selection (evaluate all) admits all 32 to official evaluation.

Evaluator

Creates one Platform Campaign for the Evaluation Campaign/context and submits top-up Submission Batches. Its 15,744 Work Items are unique by Rollout Execution Key: Rollout Key plus evaluation_context_id. Selector Capacity controls Rollouts; Provider Call Limiters gate calls. A Platform Stage Attempt may contain multiple replayed Stage Invocations.

Evaluator

One rollout, concretely — at ratio 0.5, HumanEval/36, repeat 2, the 412-character reference gives MAX_BUDGET=206. The encoder representation's integer Compressed Description Length is 121 zstd-19 bytes. Submission Text preprocesses to ordered Code Artifacts; every candidate is tested and any full pass yields score 1. Mean Compression Ratio uses 121/412 ≈ 0.29.

outputone immutable dr-store Rollout Result Object Reference bound by Rollout Execution Key → Preprocessing Trace, Code Artifact refs, Metric Facts/Records, binary score=1, Compressed Description Length=121 bytes, reference_bytes=412, cost, latency and provenance
Evaluator

After automatic Node retries, Whetstone immutably persists the terminal Result to dr-store and atomically binds its Rollout Execution Key in the Whetstone Result Store. An unbound key binds; replay of the same ref succeeds idempotently; only then does dr-platform return that authoritative ref as output_reference and mark the Stage SUCCEEDED. Failure to produce or persist the Result, or a binding/storage failure leaving the key unbound, raises FAILED and permits operator retry only while it remains unbound. A different-ref conflict preserves the existing binding, fails the conflicting attempt, and never authorizes overwrite or retry replacement. Deliberate re-evaluation instead uses a new Evaluation Context or Repeat.

outputthe authoritative bound Object Reference for each completed key; otherwise an explicit FAILED Stage — never replacement of an existing binding
Evaluation harness

Aggregates only complete, officially attested results: for each Rollout Config, per-task mean Binary Test Pass Score over 3 repeats, then unweighted mean over all 164 tasks, plus mean Compressed Description Length ratio. An exhausted infrastructure failure keeps the aggregate incomplete; deliberate re-evaluation needs a new context or Repeat. Emits 32 Rollout Aggregates.

output32 rollout aggregates — each candidate's distribution of 4 across the ratios; the pareto front is derived from them

What lands on the plot: eight candidates, 32 rollout aggregates, one front

Each candidate yields 4 Rollout Aggregates across ratios, each averaged over 164 tasks × 3 repeats: Average Binary Test Pass Rate vs Mean Compression Ratio derived from integer Compressed Description Length. These form eight budget-response curves and the official Pareto front.

Additional Notes

INPUT_CODE, the budget, and the reference length

  • {INPUT_CODE} is populated from the pinned task's task.gt_code_wo_comments; the Graph Spec declares the input port while the task binding belongs to the Rollout Key and Evaluation Context, not the Rollout Config hash
  • Compressed Description Length = integer zstd_bytes(encoded_code_rep) at level 19; Compression Ratio divides it by UTF-8 reference bytes
  • MAX_BUDGET = round(BUDGET_RATIO × chars(task.gt_code_wo_comments)), derived per task — at ratio 1.0 the description may be as long as the reference
  • Guidance, not enforcement: the budget steers, nothing clips; rollouts that blow it land further right on the plot

Candidates and the variance source

  • 8 candidates = encoder user template × Model Route — one plotted line each; the route label is display shorthand, while complete Provider Call Config refs/hashes define treatment identity
  • Variance source = budget_ratio — spreads each candidate into its distribution of rollout aggregates
  • Repeat IDs are repetition, not retry attempt numbers: averaged into each rollout aggregate for confidence. An RNG seed may be recorded as optional metadata but is not the repeat's identity
  • 8 × 4 ratios = 32 Rollout Configs; × 164 tasks × 3 repeats = 15,744 Rollouts

Tasks are not a variance source

  • The task set defines the quantity each rollout aggregate estimates: the average score over it
  • Fewer or sampled tasks → a noisier estimate of the same average
  • A different task set (train / validation / test) → a different quantity: a different eval config

Identity: rollout config vs candidate

  • graph_digest, rollout_config_hash, eval_config_hash, context IDs, and result IDs are separate, full schema-versioned SHA-256 identities
  • rollout_config_hash covers only generation treatment inputs and the Graph Spec reference; it never includes an Eval Config
  • Candidate identity is only the candidate variables' values
  • Same Rollout Config under a different measurement policy has the same rollout_config_hash and a different eval_config_hash, producing different Rollout Keys

Contract fit

  • The full experiment lands in existing boxes: Rollout Config, Eval Definition and Eval Config roles, or rollout-time derivation
  • One new mechanism: the per-task budget derivation inside the encoder stage
