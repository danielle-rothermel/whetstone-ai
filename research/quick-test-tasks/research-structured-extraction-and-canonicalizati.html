House-style canonicalizers fit the quick-test rubric better than any published benchmark

Twelve structured-extraction candidates, scored for fit; two invented-convention designs win outright.

How does structured extraction map onto the rubric?

A breadth-focused scan of the structured-extraction and canonicalization domain for the whetstone-ai quick-test task. Every candidate is checked against the quick-test rubric: a single LLM call plus exact deterministic programmatic scoring (no sandbox), output bounded to MC or a few words, default prompts scoring well below a known ceiling, a gap that closes incrementally, difficulty that decomposes into independent latent rules, a synthetic and generatable pool with exact ground truth (hundreds of contamination-resistant instances), diagnostic failures, few-shot demos that help, and prompt differences resolvable on 10-20 evals at temperature 0.

House-style canonicalizers are an unusually good fit. Canonical JSON, slugs, phone/date normalizers, checksum validators, and CSV/fixed-width serializers share one property: the ground truth is produced by a deterministic reference function the designer writes, so exact-string scoring is free and the ceiling is provably 100%. The transformation is governed by a ladder of independent normalization rules (sort keys, strip whitespace, lowercase, transliterate, escape, zero-pad, trim hyphens, compute a checksum) — exactly the "difficulty decomposes into independent latent rules" mechanism behind incremental gap-closing. Contamination is dodged by inventing a house convention that exists nowhere in pretraining. The two strongest direct candidates are the house-style normalizer family (#12) and invented-convention canonical JSON (#1); FollowBench (#6) and IFEval (#5) are the best mechanism donors. This domain corresponds to cluster G in the Landscape.

If you want per-field partial credit you must design the scoring metric carefully — for the quick test, a single exact-match on the whole canonical string (or on one bounded field) sidesteps that entirely; use rule-count only as post-hoc analysis of which latent rules a prompt recovered, never as the training signal. Separately: several arXiv IDs surfaced by search carried implausible future-year prefixes (2602-2607) in this environment, flagged [date unverified] below — treat them as leads to confirm, not settled citations. The well-established works (RFC 8785, IFEval, FollowBench, FoFo, JSONSchemaBench, StructEval, StructText, EDC, PolyNorm) are solidly dated.

Which benchmarks are the closest direct analog?

1 · RFC 8785 JSON Canonicalization Scheme (JCS)

Canonical JSON as the task — reference spec, not a benchmark

What it is. An IETF spec (informational, 2020) defining a deterministic byte-exact JSON serialization: I-JSON subset, lexicographic sorting of object keys by UTF-16 code units, ECMAScript number formatting, minimal string escaping, no insignificant whitespace. Independent implementations produce byte-identical output — the whole point (hash/sign stability).

Adaptation. Weaponize the spec into a generator: emit random small JSON objects, ask the model to return the JCS canonical form, score exact-string against a reference canonicalizer — the near-exact analog of the project's own dr-serialize work. Replace the RFC's rules with an invented house convention (sort keys by length-then-reverse-alpha, a bespoke escape table, a custom number format) so no pretrained JCS knowledge shortcuts the task.

Rubric fit — excellent. Deterministic exact scoring; provable 100% ceiling; independent, stackable rules (sorting, whitespace, escaping, number format, unicode) give a clean incremental ladder and inspectable "which rule did the prompt discover." Weakness: genuine JCS is a plausible pretraining hit, so the convention must be invented; full canonical JSON can grow long — bound object size or score one field.

RFC 8785, JSON Canonicalization Scheme (JCS), IETF, 2020 — rfc-editor.org · datatracker

2 · StructEval — 18-format structured-output generation

Format taxonomy and below-ceiling evidence

What it is. Benchmark evaluating generation of renderable (HTML, React, SVG, Mermaid) and non-renderable (JSON, YAML, CSV, TOML) formats over 44 task types, scored by Syntax/Keyword-Matching/VQA. Even o1-mini averages ≈75.58; open models run ~10 pts lower.

Adaptation. Use only the non-renderable, programmatically-checkable slice (JSON/YAML/CSV/TOML) and swap its LLM-judge/VQA scoring for exact-string equality against a synthetically-generated target. Pick one format, invent a house style, generate instances.

Rubric fit — good as a source of formats and difficulty evidence. Weakness: published scoring is Keyword/VQA/LLM-judge (not exact-match), some formats need a sandbox (disqualified by criterion 2), and it is a published set — regenerate for contamination.

arXiv:2505.20139 (May 2025) — abs

3 · JSONSchemaBench — 10K schemas for constrained decoding

Difficulty-graded schema pool

What it is. ~10K JSON schemas spanning constraint complexity, used to compare six constrained-decoding stacks (Guidance, Outlines, llama.cpp, XGrammar, OpenAI, Gemini) on efficiency, coverage, and output quality; frameworks fail notably on GitHub-Hard / JSON Schema Store.

Adaptation. Not a plug-in task — it measures decoder compliance, not a canonicalization gap — but a rich, graded schema pool: sample a schema, generate a valid instance, task = "return the canonical instance conforming to schema S", scored exact-match.

Rubric fit — moderate. Great graded schema corpus. Weakness: the benchmark's own objective is constrained decoding, not prompt-optimizable canonicalization; you borrow the schemas, not the task; regenerate for contamination.

Geng et al., arXiv:2501.10868 (Jan 2025) — abs · code

4 · EDC (Extract, Define, Canonicalize)

Schema canonicalization pipeline — conceptual heart of the domain

What it is. Three-phase LLM knowledge-graph framework: open extraction → schema definition → post-hoc canonicalization of triples to a fixed vocabulary, with a trained retriever for relevant schema elements.

Adaptation. Distill the canonicalize step into a bounded task: give a surface entity/relation string plus a small fixed target vocabulary, ask for the exact canonical label. Latent rules = mapping conventions (casing, synonym collapse, unit normalization).

Rubric fit — good conceptual template; bounded output is natural. Weakness: real KG canonicalization is semantic/fuzzy (many "correct" mappings), which fights determinism — must constrain to a designer-owned invented vocabulary with one correct answer per instance.

Zhang & Soh, EMNLP 2024 — aclanthology · arXiv:2404.03868 · code

Which benchmarks donate a scoring mechanism rather than a task?

5 · IFEval

Verifiable format/lexical instruction following

~500 prompts, 25 types of programmatically verifiable instructions (word/keyword counts, casing, JSON-only output, bullet counts), scored strict/loose — no LLM judge. Borrow the verifier library but compose many independent constraints per instance to build a latent-rule ladder, and generate synthetic combos so instances are fresh. Rubric fit — strong on determinism and diagnostic failures. Weakness: single-constraint instances are near one-shot guessable; fix only by stacking constraints. Published prompts need regeneration.

Zhou et al., arXiv:2311.07911 (Nov 2023) — abs

6 · FollowBench

Multi-level incremental-constraint following

Starts from a base instruction and incrementally adds constraints (levels 1-5) across Content/Situation/Style/Format/Example — the rubric's "incremental gap that closes in steps" made explicit. Adopt the level-ladder pattern: level N = N house rules active; replace the strong-LLM judge with programmatic exact-match. Rubric fit — excellent conceptual donor. Weakness: as published it leans on LLM-as-judge for open-ended constraints — keep the design, not the scoring.

Jiang et al., ACL 2024 — aclanthology · arXiv:2310.20410

7 · FoFo

Domain-specific format-following

494 instructions requiring adherence to real-world domain formats (medical reports, LaTeX), built via AI-human collaboration; demonstrates format-following is hard and below ceiling. Evidence source and format catalog, not directly usable: scoring is LLM-as-judge, formats are open-ended, and it is published. Rubric fit — weak as a direct task, useful as floor-evidence and format-idea donor.

Xia et al., arXiv:2402.18667 (Feb 2024) — abs

Which synthetic-generation methods build a usable instance pool?

8 · StructText — run the table→text generator backwards

Starts from tabular ground truth and generates natural-language passages via a plan-then-execute pipeline, yielding key-value extraction benchmarks at scale with exact cell ground truth. Run it backwards as your generator: since the table cells are known ground truth, the quick-test task = "extract field X and return it in canonical form", scored exact-match on that one bounded field. Rubric fit — good pool-generation recipe. Weakness: its own metrics are multi-dimensional/soft — use the generator, not the metrics.

Shirai et al., arXiv:2507.21340; VLDB 2025 TaDA workshop — abs · code

9 · DTBench [date unverified]

Synthetic document→table extraction

Synthetic benchmark for document-to-table extraction with programmatically known ground truth. Similar adaptation to StructText: use the generator for input docs with exact ground-truth cells, reduce to a single bounded canonical-field extraction. Rubric fit — moderate; promising but must be verified (future-dated arXiv ID). Table outputs are wide — bound to one field.

arXiv:2602.13812 (confirm date/venue) — pdf

10 · ExtractBench [date unverified]

Complex extraction, exact & tolerance metrics

Benchmark and methodology for complex structured extraction; specifies exact-match and tolerance-based numeric comparison plus semantic array alignment. Mine the metric design to justify the quick-test's whole-string exact-match choice, and borrow task shapes. Rubric fit — moderate, primarily a metric-design reference; verify provenance before citing.

arXiv:2602.12247 (confirm date/venue) — pdf

11 · LLMStructBench [date unverified]

Extraction across prompting strategies

Benchmarks many LLMs on structured extraction across ~five prompting strategies; finds prompting strategy matters more than model size and semantic errors persist even when structural validity holds — direct evidence that prompt optimization has real headroom on structured tasks. Rubric fit — moderate as a task source, strong as motivation evidence. Verify provenance.

arXiv:2602.14743 (confirm date/venue) — html

What is the single best all-around fit in this domain?

12 · House-style normalizer family — slugify / phone / date / checksum

Best all-around fit · a family you build, not a published benchmark

What it is. A synthetic family built on well-defined deterministic normalization pipelines: slugification (Unicode NFKD → transliterate → lowercase → strip to [a-z0-9-] → collapse/trim hyphens → dedupe suffix), ISO-8601 date normalization, E.164 phone formatting, and checksum validators/formatters (Luhn, ISBN-13, IBAN mod-97). Each pipeline is an ordered stack of small, independent, exactly-checkable rules.

Adaptation. Pick one family (slugify is the strongest single choice) and invent a house convention that diverges from any public library (custom transliteration table, non-standard hyphen rules, a bespoke check digit) so pretraining can't shortcut it. Generate hundreds of random inputs; the reference function is the exact ground truth; score whole-output exact-match. Failures are diagnostic ("expected cafe-2, got café_2" reveals the transliteration+separator rule was missed).

Rubric fit — best all-around. Provable 100% ceiling (you own the function); fully synthetic, infinite, and contamination-proof; output is one short slug/number (bounded); deterministic at temp 0; a long independent rule ladder gives a clean incremental gap and inspectable rule discovery; few-shot measurably helps; checksum/format violations give natural, injectable failure cases; a length-budget knob (max slug length) is the vestigial second metric.

Weakness. Rules must be tuned so the naive-prompt floor is genuinely low but the ladder is long enough to avoid one-shot guessing — stack 5-8 non-obvious rules, not one obvious "lowercase and hyphenate" step. Per-field partial credit needs care; a single whole-string exact-match avoids it.

Rule sources: python-slugify conventions and Unicode NFKD/transliteration (peasydesign.com guide); Luhn/ISBN-13 EAN check digit/IBAN mod-97 (ISO 7064) standard checksum definitions; PolyNorm — Few-Shot LLM-Based Text Normalization for TTS, arXiv:2511.03080 (2025), evidence that date/phone/number normalization is a live LLM few-shot task — html.

What carries across every candidate?

  • Strongest direct candidates: #12 (house-style slugify/normalizer family) and #1 (invented-convention canonical JSON) — both give a designer-owned 100% ceiling, exact scoring, an independent-rule ladder, and trivial contamination-proofing. #6 (FollowBench level-ladder) and #5 (IFEval verifier library) are the best design/mechanism donors.
  • Recurring caveat. Most published structured-output/format benchmarks (StructEval, FoFo, FollowBench, ExtractBench) lean on LLM-judge or soft/multi-dimensional scoring and published prompts. They contribute task shapes, difficulty evidence, and rule inventories, but the actual quick-test should be synthetically regenerated with a programmatic exact-match checker.
  • Partial-credit caveat. Per-field scoring invites tolerance/alignment metrics (ExtractBench) that break determinism-friendly exact-match. Prefer a single bounded output scored by exact equality; use rule-count only as post-hoc analysis, never as the training signal.
  • Provenance flags. #9 DTBench, #10 ExtractBench, #11 LLMStructBench surfaced with future-year arXiv IDs in this environment; confirm exact dates/venues before citing. The established works (RFC 8785, IFEval, FollowBench, FoFo, EDC, JSONSchemaBench, StructEval, StructText, PolyNorm) are reliably dated.
