c11 Baseline Spec: Headroom Before Complexity

Draft
whetstone-ai · 2026-07-22
This spec is a draft awaiting owner decisions. Every number below is a proposal, not a frozen parameter. The model set (§4), the decision-rule thresholds (§3), and six other open items (§7) are explicitly owner-gated — see the open decisions list before treating anything here as final.

This experiment measures how much headroom exists between a naive canonicalization prompt and a best-effort ceiling prompt on plain RFC 8785 (JCS) canonical JSON — before any build effort goes into c11's invented, adversarially-deviating house-rule layer. It is a go/no-go instrument, not the benchmark itself: plain JCS sits in the crowded "public-standard-conformance" cell the related-work synthesis maps out, so a large gain there is a materially weaker result than a gain on the invented-rule task — but it is the cheapest, lowest-risk first measurement of whether the task shape has headroom at all before house rules are designed and validated.

Fixed constraints — not open for this experiment
  • Plain RFC 8785 (JCS) canonical JSON, no invented deviations yet. That added complexity is exactly what this experiment tests the necessity of, so it cannot appear in the baseline.
  • Fresh seeds, never a published instance. Inputs are sampled from pinned json-schema-faker schemas; no seed may coincide with a previously published instance (contamination resistance, rubric criterion 8).
  • Independent oracle — the trailofbits rfc8785-py reference implementation (rfc8785.dumps), used unmodified. It is a pure function of the input Python object, computed from RFC rules rather than from how the input was built (oracle independence).
  • Scoring: 0/1 whole-string exact match of the returned canonical JSON string against the oracle. One trivial input normalization is permitted before the compare: strip surrounding whitespace / code fences. Nothing else.
  • Temperature 0. Repeats = 3 (repeat_id in {0,1,2}) to exercise the aggregation plumbing.

1. Strata design

Stratum axis: which JCS sub-rule the instance stresses

The capability synthesis is unambiguous: structural JSON validity is near-saturated; the failures live in the canonical details — key sort order (UTF-16 code-unit comparison), number canonicalization (ECMAScript shortest-round-trip, flagged as the single hardest sub-rule), exact escaping, and zero-insignificant-whitespace. Schema content swings the score more than model choice. So the stratum axis is realized as schema-content shape, not model or prompt variation.

Five strata proposed

#StratumWhat the schema forces
S1Flat / shallowFew keys, all-ASCII strings, ints only, no sort tension
S2Key-sort stressMany keys whose insertion order ≠ UTF-16 sorted order; nested objects
S3Number-canonicalizationFloats needing ECMA shortest round-trip; large ints; −0; exponent forms
S4Unicode / escapingStrings needing \u escapes, control chars, exact escape policy
S5Mixed / deepNested combination of S2+S3+S4 in one instance

Every instance is generated adversarially so it actually exercises its stratum's sub-rule (e.g. S2 instances are rejected if their keys are already sorted, since messy and canonical forms would then coincide).

200 official + 200 held-out = 400 pinned instances proposed

Design driver: rubric criterion 5 resolvability. A meaningful prompt change (≥10 points) must resolve above residual noise on a 10–20 task internal eval. The internal-eval slice is ≥2 tasks per stratum × 5 strata = 10 tasks minimum, 20 tasks target, exactly the 10–20 band criterion 5 names; at temperature 0 a 10-point effect is ≥1 flipped task out of 10, resolvable given the near-determinism the rubric assumes.

Official/calibration pool: 40 instances per stratum × 5 strata = 200, sized so a per-stratum accuracy is stable to a binomial standard error of ≈±8pp at p=0.5, while keeping a full baseline run in the cents-per-run / minutes envelope (§5) and satisfying the "hundreds of tasks" pool (rubric criterion 7). Held-out fresh-seed split: a further 200 disjoint-seed instances, generated and frozen now but not scored in this baseline — reserved so any later optimizer claim is verified on seeds never seen.

Evidence note: the output-bytes objective is provably degenerate here (correct canonical strings have fixed length), so this baseline validates single-objective optimization plus two-objective plumbing only, not a real multi-objective result — it must say so wherever it is reported.

2. The two probe prompts (drafted verbatim)

Both probes are prompt-only — no constrained decoding, which is a decoding-engine intervention unavailable in this baseline. The model receives the messy input JSON ({input}) and must return only the canonical string. These operationalize rubric criterion 4 (default prompts score low; better prompts close the gap) and criterion 9 (floor, ceiling, and a reference prompt known in advance).

2.1 Probe (a) — deliberately underspecified naive prompt

Canonicalize this JSON.

{input}

Design intent: names the operation but states none of the JCS sub-rules. It is the "single instruction, no worked examples" floor the capability synthesis predicts at ~5–30% (best guess ~15%), and the operationalization of criterion 4's claim that models' default behavior sits well below the ceiling.

2.2 Probe (b) — best-effort ceiling prompt (all JCS rules + worked examples)

Convert the JSON below into its RFC 8785 (JCS) canonical form and return ONLY the
resulting string, with no code fences, no commentary, and no trailing newline.

Apply every one of these rules exactly:

1. Object keys are sorted by their UTF-16 code units (compare the raw UTF-16 code-unit
   sequences of the key strings, ascending). Sort keys at every nesting level.
2. No insignificant whitespace: no spaces or newlines anywhere except inside string
   values. `{"a":1,"b":[2,3]}`, never `{ "a": 1 }`.
3. Strings use the shortest JSON escaping: escape only " \ and the control characters
   U+0000..U+001F. Use the two-character forms \" \\ \b \f \n \r \t where they exist;
   otherwise \u00XX (lowercase hex). Do not escape forward slash or any other character.
4. Numbers use the ECMAScript Number-to-string (shortest round-trip) form: integers with
   no decimal point or exponent; no leading zeros; no "+" on exponents; "-0" becomes "0";
   the minimal digit sequence that round-trips to the same IEEE-754 double.
5. Literals are exactly true, false, null. Arrays preserve their element order.

Worked examples:

Input:  {"b": 2, "a": 1}
Output: {"a":1,"b":2}

Input:  {"x": 1.0, "y": 1e2, "z": -0}
Output: {"x":1,"y":100,"z":0}

Input:  {"s": "line1\nline2", "t": "π"}
Output: {"s":"line1\nline2","t":"π"}

Input:  {"nested": {"d": 4, "c": 3}, "arr": [true, null]}
Output: {"arr":[true,null],"nested":{"c":3,"d":4}}

Now canonicalize:

{input}

Design intent: states every standard convention of the existing task and shows four worked examples spanning sort / number / escape / whitespace — the "several worked examples" the capability synthesis conditions its ceiling prediction on, ~40–75% (best guess ~55%). The examples are illustrative covering, not full-table coverage (no rule table exists in plain JCS), so no leakage concern applies to this baseline; the leakage guard becomes live only for the eventual invented-rule task.

Evidence note (thin): the Output strings in the worked examples above are hand-written and must be regenerated by rfc8785.dumps on the exact inputs and pasted verbatim before freezing — do not trust the hand-typed forms (the π/escape example in particular needs confirming against the oracle's literal-UTF-8-vs-\u-escape behavior). Flagged in §7.5.

3. Three-outcome decision rule

Let F = naive-probe score, C = ceiling-probe score, each official-pool exact-match, averaged per repeat then across strata. Thresholds are inferences from the capability synthesis (predicted naive floor ~5–30%, best ~15%; predicted ceiling ~40–75%, best ~55%), with an explicit upside risk that plain JCS is memorizable public knowledge and the naive floor could exceed 40%. All bands are DRAFT and must be re-pinned once calibration numbers land.

Proposed numeric anchors all owner-adjustable: HIGH ≥ ~80%; MID ≈ ~40–75%; LOW ≤ ~30%; "≈" (F approximately equals C) means within ~10 points (the criterion-5 resolution grain); "≪" (F much less than C) means C − F ≥ ~20 points (2× the resolution grain, so a "prompt-closable" verdict clears noise with margin).

Outcome (a) · No headroom

F ≈ C ≈ HIGH

|F − C| < 10 and both ≥ ~80%

Plain JCS is too easy or memorized — the synthesis's "JSON is saturated" upside risk realized. Any reasonable prompt hits the ceiling; every optimizer would look identical (rubric criterion 4 fails).

Decision: proceed to the candidate's added-complexity design (invented house-rule deviations, post-diet ≤6 single-pass rules). The task shape has no discriminative gap without them.
Outcome (b) · Headroom, not prompt-closable

F ≈ C ≈ MID/LOW

|F − C| < 10 and both < ~75%

The ceiling prompt states every rule and still can't lift the score — a raw capability deficit (the persistent number-canonicalization slip that does not go away with prompting).

Decision: pursue the added-complexity design and keep base difficulty shallow. If the model can't execute plain JCS even when told every rule, invented deviations layered on top will floor the score. Bias strata toward S1/S2, drop or cap S3 number-heavy content.
Outcome (c) · Prompt-closable headroom

F ≪ C

C − F ≥ ~20 points, C not near ceiling-saturated

Naive is low, and stating the rules substantially lifts the score — the gap is prompt-shaped, the criterion-4 ideal: room to improve, and prompts close it.

Decision: direct prompt optimization is viable on plain JCS; run COPRO/MIPROv2/GEPA on it first. Added complexity is optional, not required. Caveat: plain JCS still sits in the crowded public-standard-conformance cell — "viable for the quick-test instrument" ≠ "sufficient for the paper" (§7.8).
Where the risk concentrates (§7.2, §7.8): outcome (a) is the synthesis's primary predicted risk for plain JCS — run the naive floor specifically on the number-heavy (S3) and unicode-heavy (S4) strata, since the corpus says that is where breakage, and therefore discriminative signal, lives. If even S3/S4 come back HIGH, outcome (a) is firm. All thresholds must be frozen from the measured calibration numbers, not the predictions, before any optimizer run.

4.2 Candidate slate to choose ≥2 from not finalized

The quick test runs at the cheap tier (rubric criteria 3, 14: cents/run, minutes), spanning ≥2 small models so the result isn't single-model under-powered:

  • Gemini-2.5-Flash — cheap frontier-family, fast, high concurrency.
  • GPT-5-mini — "-mini" closed tier the StructEval/PolyNorm proxies most resemble.
  • Claude Haiku 4.5 — cheap closed, strong instruction-following.
  • DeepSeek-Chat — cheap, and the DeepSeek family is the one with published tiny-model structured-output numbers (proxy anchor).

If any Claude/Anthropic model is on the final slate, confirm exact model IDs and pricing before finalizing §5 cost — do not rely on memory.

4.3 Tradeoffs for the owner

  • One model minimizes §5 cost and answers the headroom go/no-go, but produces a result reviewers flag as under-powered (field norm is 12–22 models).
  • ≥2 small models costs ~2× but lets an eventual optimizer claim argue generality across the cheap tier; cheap-tier scores swing widely, so a single model's number may not generalize.
  • Reasoning model? No decisive edge on structured-output proxies but did help most on the novel-constraint axis (IFBench o3). Plain JCS is not novel, so likely unnecessary for this baseline but may matter for the eventual invented-rule task.
  • Model choice matters less than schema content here (synthesis, stated twice) — argues for spending budget on strata coverage over model count if forced to trade.

5. Cost & wall-clock estimate per full baseline run

Formula: instances × probes × models × repeats = LLM calls; calls × tokens/call = tokens; tokens × price = cost. Fixed factors: 200 official instances (the 200 held-out are generated once and frozen, adding no per-run token cost); 2 probes (naive, ceiling); repeats = 3.

Per-instance token estimate proposed

ComponentNaive probeCeiling probe
Input JSON instance (S1 shallow ~50 → S5 deep ~300)~50–300~50–300
Prompt overhead (naive: bare instruction; ceiling: rules + 4 worked examples)~10~450
Output canonical string (roughly the input size)~50–300~50–300
Blended total tokens/call~350~800

Calls, tokens, and dollar cost

Calls = 200 official × 2 probes × models(M) × 3 repeats.

M=1: 1,200 calls × ~600 tokens/call blended ≈ ~720K tokens. M=2: 2,400 calls ≈ ~1.44M tokens. M=4: 4,800 calls ≈ ~2.88M tokens.

Cost (order-of-magnitude, cheap-tier blended ~$0.5–$1.5 per 1M tokens; confirm live pricing before running): M=2 ≈ $1–$2; M=4 ≈ $2–$4 per full baseline run — squarely in the rubric's cents-to-a-few-dollars band (criteria 3, 14).

Wall-clock (proposed): at a modest sustained ~5 calls/sec (temp 0, short outputs, cheap-tier concurrency), M=2's 2,400 calls ≈ ~8 minutes; M=4's 4,800 calls ≈ ~16 minutes — "minutes, not hours" holds. Generation time is negligible: json-schema-faker produced its full suite in ~870ms in a repo run, so generating all 400 instances is sub-second.

Evidence note (thin): the ceiling-probe token estimate folds in a fixed ~450-token rule-and-example overhead; actual output length varies with schema depth and is a guess pending a pilot. A small pilot per model should replace these estimates before committing the full run (§7.4).

7. Open decisions (awaiting the owner)

Nothing below is frozen — all eight items require an explicit owner call before this spec can run.

  1. Model slate (§4) — which models, and how many. Recommendation to choose from: Gemini-2.5-Flash / GPT-5-mini / Claude Haiku 4.5 / DeepSeek-Chat; ≥2 for generality vs 1 for cost. Include a reasoning model now or defer to the invented-rule task? Drives §5 cost directly.
  2. Decision-rule thresholds (§3) — the HIGH/MID/LOW bands and the ~20-point "prompt-closable" gap are draft inferences from the synthesis. Confirm or re-pin after calibration numbers land.
  3. N per stratum (§1) — 40 official / 40 held-out per stratum (200/200 total) proposed. Owner may want more official-pool instances if per-stratum error bars (~±8pp) are too loose for the go/no-go, or fewer to cut cost.
  4. Internal-eval slice size (§1) — 10 (min) vs 20 (target) tasks; criterion 5 permits either. Owner sets whether it is a disjoint split or a labeled subset of the official 200.
  5. Ceiling-probe example set (§2) — four worked examples proposed; owner may add or remove. Regardless, all example outputs must be regenerated by the real oracle and pasted verbatim before freezing.
  6. Schema pin (§1) — exact json-schema-faker schemas per stratum are not yet written. Owner approves the schema set and confirms date formats are excluded or minDateTime/maxDateTime pinned (the one wall-clock nondeterminism source in the generator).
  7. Version pins — json-schema-faker version (v0.6.2 reviewed), rfc8785-py version, and confirmation the oracle is used strictly unmodified for the plain-JCS baseline.
  8. What outcome (a) triggers (§3) — if plain JCS is HIGH, the design jumps to the invented-rule build. Owner confirms that branch is in scope and budgeted before this baseline runs, so the result is actionable either way.
Evidence-thinness disclosures. No paper runs c11's exact task (byte-exact canonical JSON vs a deterministic oracle); every floor/ceiling number above is an inference with wide error bars. The cheap tier c11 runs at is not directly tested by any paper (§4). The serialization-convention ablation — the exact axis c11 varies — has zero measured effect size anywhere in the corpus; the §1 stratum-difficulty ordering is inferred from adjacent break-point evidence, not measured. The naive-floor prediction pulls two ways: novel-constraint evidence argues for a low floor, but plain JCS is a memorizable public standard, so the floor could exceed 40% — this is the uncertainty the baseline exists to resolve.
