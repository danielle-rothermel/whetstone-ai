Measure the vanilla Minigrid shape's headroom before building c19's complexity

Draft

A cheap go/no-go probe: does the naive floor already sit near the ceiling on vanilla dynamics?

whetstone-ai · 2026-07-22

This experiment measures how much headroom exists between a naive prompt and a best-effort ceiling prompt on the vanilla Minigrid task shape — before any build effort goes into the invented-command / wrap-vs-clamp complexity layer. It is a go/no-go instrument, not the benchmark itself: c19's biggest early-kill risk is a naive floor that already sits near the ceiling (LLM-BabyBench's Predict split shows 85–97.5% on Easy standard-semantics). If that risk is real on the vanilla task, this baseline finds it cheaply, before the net-new simulation logic that is c19's real cost gets built.

This spec is a draft awaiting owner decisions. Every number below is a proposal, not a frozen parameter. The model set (§4), the numeric thresholds (§3), and nine other open items (§7) are explicitly owner-gated — see the open decisions list before treating anything here as final.
Fixed constraints — not open for this experiment
  • Vanilla Minigrid dynamics only. Standard actions, standard movement semantics. No invented commands, no wrap-vs-clamp variants — that added complexity is exactly what this experiment tests the necessity of, so it cannot appear in the baseline.
  • Fresh seeds, never a published instance. No seed may coincide with one published by any of the five related-work papers (contamination resistance, rubric criterion 8).
  • Stochastic-layout envs only — Fetch, SimpleCrossing, FourRooms, Empty-Random. Fixed-layout Empty is excluded: it does not vary by seed (Empty-8x8 is identical for seeds 7 and 8), so it cannot supply distinct instances.
  • Single derived-fact question, scored exact 0/1 match against an independent oracle (the Minigrid object-model walk).
  • Temperature 0. Repeats = 3 (repeat_id in {0,1,2}) to exercise the aggregation plumbing.

How is the instance pool stratified, and how big is it?

§1 · Strata

Three crossed axes: env id, grid size, fact type

Env id (4 levels) — Fetch, SimpleCrossing, FourRooms, Empty-Random; all four are run-verified as seed-threaded and byte-reproducible. Grid size (2 levels) — small ~5×5 and medium ~8×8 (env-native size variants where available), because the related-work synthesis shows monotone decline 5×5→9×9 with 7×7 already halving visual-grid accuracy. Fact type (4 levels) — final coordinate, heading, carrying-flag, what-is-in-front, read directly from the serialized grid and agent state by independent oracle glue. Coordinate-after-a-sequence is the hardest (most spatial); heading/carry are easier (fewer classes, less spatial) — the axis spreads difficulty and guards against a single near-guessable fact.

Applicability matrix proposed

Not every (env × fact) pair is meaningful: carrying-flag and what-is-in-front are only informative on envs with fetchable/placed objects.

Carrying-flag needs pickup-able objects plus a command sequence that may pick one up; only Fetch reliably supplies that under vanilla dynamics. Open: the owner may extend carrying to other envs if the glue places objects (§7.5).

N per stratum = 15 proposed; 390 instances total

Design driver: rubric criterion 5 resolvability. Optimizers rank proposals on tiny internal evals (3–20 tasks, 1 repeat); a meaningful prompt change (≥10 points) must exceed residual noise on a 10–20 task internal eval. A 10-point effect on a 10-task eval is one task flipping, so N=10 per internal slice is the quantization floor. Setting per-stratum N to 15 keeps every stratum at or above that band so a single stratum can serve as an internal-eval slice and still resolve a 10-point effect (1.5 tasks).

Env × size × {coordinate, heading, what-is-in-front} = 4 × 2 × 3 = 24 strata. Env × size × {carrying-flag}, Fetch only = 1 × 2 × 1 = 2 strata. 26 strata × 15 = 390 instances in the pinned pool — comfortably in the "hundreds of tasks" band rubric criterion 7 wants for disjoint minibatch/subset/official splits. Seeds are the instance index within a stratum, drawn from a fresh reserved range no paper has published.

Evidence note (thin): no related-work paper measured per-stratum variance on this exact (single-fact, ASCII, vanilla, temp-0) cell. N=15 comes from the rubric's resolvability arithmetic, not a measured variance estimate. If a pilot shows temp-0 repeat disagreement is non-trivial (it should be near-zero), N must rise — flagged in §7.8.

What exactly do the naive and ceiling prompts say?

§2 · Probes

Both probes are shown for the final-coordinate fact; the harness substitutes the fact-specific question line for the other three fact types. {GRID} is filled with pprint_grid() output; {COMMAND} is the vanilla command string (standard Minigrid actions only — turn-left, turn-right, forward, pickup, drop, toggle — rendered as single letters L R F P D T). These operationalize rubric criterion 4 (default prompts score low; better prompts close the gap) and criterion 9 (floor, ceiling, and a reference prompt known in advance).

2.1 Probe (a) — deliberately naive prompt

Here is a grid and a sequence of moves for the robot.

{GRID}

Moves: {COMMAND}

Where does the robot end up? Answer with just the final coordinate.

Fact-line variants: heading — "Which direction is the robot facing at the end? Answer with one letter." · carrying-flag — "Is the robot carrying an object at the end? Answer yes or no." · what-is-in-front — "What is directly in front of the robot at the end? Answer with one word."

Design intent: gives the grid, the command, and the question — nothing else. No coordinate convention, no glyph legend, no statement of move semantics, no origin. It leans entirely on the model's memorized "standard grid semantics." This is exactly where the too-high-floor risk lives: a capable model may infer the standard conventions unaided, which is precisely what this probe is designed to measure.

2.2 Probe (b) — best-effort ceiling prompt (all standard conventions stated)

You are simulating a robot on a 2D grid. Follow these rules EXACTLY.

COORDINATES: cells are written (row, col). Row 0 is the TOP row; col 0 is the LEFTMOST
column. Rows increase downward, columns increase rightward.

GLYPHS in the grid below (two characters per cell):
  - a period "." is an empty floor cell the robot may enter.
  - "#" or "WG"/"WB" etc. (a wall glyph) is a wall; the robot CANNOT enter or pass through it.
  - a two-letter object glyph (e.g. "KY" = yellow key, "BR" = red ball, "GG" = green goal)
    is an object occupying that cell.
  - the robot is shown by a direction arrow: ">" faces right (east), "<" faces left (west),
    "^" faces up (north / toward row 0), "V" faces down (south).

HEADINGS: the robot has a facing direction, one of E (east/right), W (west/left),
N (north/up), S (south/down).

MOVES (apply in order, left to right):
  - L = turn left 90 degrees in place (does not move): E->N->W->S->E.
  - R = turn right 90 degrees in place: E->S->W->N->E.
  - F = step ONE cell forward in the current facing direction. If the cell directly ahead
    is a wall or is off the edge of the grid, the robot does NOT move (it stays put); it does
    not wrap around and does not pass through walls.
  - P = pick up the object in the cell directly ahead, if any, and only if not already
    carrying something; the robot then carries it.
  - D = drop the carried object into the cell directly ahead, if that cell is empty.
  - T = toggle (open/close) the object directly ahead; does not change position or heading.

Work step by step: track the robot's (row, col) position and its heading after EACH move,
then report only the final answer.

{GRID}

Moves: {COMMAND}

QUESTION: What is the robot's final coordinate? Answer on the last line as: row,col

Fact-line variants (replace the QUESTION line): heading — "Answer on the last line as one of: E W N S" · carrying-flag — "Answer on the last line as: yes or no" · what-is-in-front — "Answer on the last line with one word (e.g. wall, empty, key, ball, goal)."

Design intent: states every standard Minigrid convention the naive prompt withholds — origin and indexing, glyph legend, exact move semantics including the no-wrap/no-pass-through edge rule, and an explicit step-by-step instruction. It includes a lightweight CoT nudge ("work step by step") because the capability synthesis shows CoT is the single load-bearing lever for the cheap tier (0%→74–86%); a ceiling prompt that omitted it would understate the true ceiling and inflate apparent headroom.

Evidence note: the capability synthesis shows the ASCII rendered-grid representation is the weak format (visual-grid 34% vs adjacency 86% at 5×5 for Gemini-CoT). The ceiling prompt stays on ASCII by constraint (the task shape is fixed), so this is the best-prompt-on-ASCII ceiling — estimated ~55–75% — not the ~80–90% best-prompt-if-restructured ceiling. The decision rule in §3 is calibrated to the ASCII ceiling.

What does each measured outcome tell the project to do next?

§3 · Decision rule

Let naive = mean exact-match under probe (a), ceiling = mean under probe (b), each averaged repeat-within-task then across strata, per model, then aggregated across the model set (owner picks aggregate: mean or min — §7.2). Thresholds are proposals grounded in the capability synthesis (central estimates: naive ~30–45%, ceiling ~65–80% on ASCII; too-high-floor is the dominant risk) and the positioning early-warning gate ("if naive floor is high (>~80%) OR ceiling barely beats naive, STOP").

Proposed numeric anchors all owner-adjustable: HIGH band = score ≥ 75%; MID/LOW band = score < 75%; "≈" (naive approximately equals ceiling) means |naive − ceiling| < 10 points (the criterion-5 resolvability unit); "≪" (naive much less than ceiling) means ceiling − naive ≥ 20 points (two resolvability units).

Outcome (a) · No headroom

naive ≈ ceiling ≈ HIGH

|naive − ceiling| < 10 and both ≥ 75%

The vanilla task is already near-solved regardless of prompt; there is no gap for an optimizer to close. This is the P2-Easy scenario (85–97.5% standard semantics) flagged as the biggest early-kill risk.

Decision: the vanilla task is inadequate as-is — proceed to build the invented-command / wrap-vs-clamp complexity layer. That net-new difficulty is now justified by measurement, not assumed.
Outcome (b) · Headroom, not prompt-closable

naive ≈ ceiling ≈ MID/LOW

|naive − ceiling| < 10 and both < 75%

The task is hard, but stating all standard conventions doesn't help — the deficit is raw execution/state-tracking, not missing convention knowledge. Models can have intact local reasoning yet fail to integrate a sequence within one forward pass.

Decision: build the invented-rule layer (there is difficulty to exploit) and keep execution depth shallow — short command sequences, small grids — so the discriminating difficulty comes from convention rules (which prompts can surface) rather than deep multi-step execution (which they can't).
Outcome (c) · Prompt-closable headroom

naive ≪ ceiling

ceiling − naive ≥ 20 points

Naive under-performs, and simply stating the standard conventions recovers a large, real gap — exactly the substrate an optimizer benchmark needs, and the same signature as the CoT/convention-statement swing documented at the cheap tier.

Decision: direct prompt optimization is viable on the vanilla task; the invented-rule build may be deferred or skipped for the quick-test purpose. Caveat: the publishability case still needs the convention layer to differentiate from prior work — "viable for the quick-test instrument" ≠ "sufficient for the paper" (§7.10).
Boundary cases the owner must rule on (§7.4): mixed bands (naive MID, ceiling HIGH, gap 10–20 points — falls between (b) and (c)); large per-model divergence (one model shows (c), another (a)), given the documented GPT-5-mini non-conformance. Proposed default: use the min across models for the HIGH test (conservative against a single lucky model) and the max gap across models for the ≪ test (any model showing prompt-closable headroom counts).

4.2 Proposed candidate set not finalized

The quick test runs at the cheap tier. Proposed set = the exact P5 (2604.10690) roster, because it is the only paper whose entire model set is the cheap tier and it tested the closest task shape:

  • Gemini-2.5-Flash
  • Claude-Haiku-4.5
  • DeepSeek-Chat
  • GPT-5-mini — included specifically because it is the documented non-conformer; it stress-tests the cross-model divergence path in §3's decision rule.

4.3 Tradeoffs for the owner

  • Pro: directly comparable to the most on-point published numbers; spans providers; cheap enough for many repeats; includes a known outlier to exercise the divergence logic.
  • Con: all four are cheap-tier; adding a frontier anchor (Claude 3.7 Sonnet or an O-series model) would test whether the too-high-floor risk is worse at frontier, but raises cost.
  • Con: GPT-5-mini's non-conformance can dominate a min-aggregate; model choice must be paired with the aggregate choice in §3.
  • Con: a 4-provider set has 4 independent rate limits; wall-clock is gated by the slowest provider.
  • Note: exact model versions/snapshots and provider routing are output-affecting and must be pinned once selected (§7.1).

What does one full baseline run cost in dollars and minutes?

§5 · Cost

Formula: instances × probes × models × repeats = LLM calls; calls × tokens/call = tokens; tokens × price = cost. Fixed factors: N = 390 instances; 2 probes (naive, ceiling); repeats = 3.

Per-instance token estimate proposed

ComponentNaive probeCeiling probe
Input tokens (grid ~64 cells ×2 chars + command + scaffold)~250~750
Output tokens (short answer vs. CoT trace + answer)~30~400
Total tokens/call~280~1,150

Calls, tokens, and dollar cost

Calls per model per full run: 390 × 2 probes × 3 repeats = 2,340 calls/model (1,170 naive + 1,170 ceiling).

Tokens per model per full run: (1,170 × 280) + (1,170 × 1,150) = 327,600 + 1,345,500 ≈ 1.67M tokens/model.

For the proposed 4-model set: 4 × 2,340 = 9,360 calls; 4 × 1.67M ≈ 6.7M tokens per full baseline run.

Cost (order-of-magnitude, cheap-tier blended ~$0.50–$1.50 per 1M tokens): roughly $3–$10 per full baseline run — squarely in the cents-to-single-dollars band the rubric requires.

Wall-clock (proposed): at modest concurrency (~20 concurrent calls/provider, ~2–4s/call for cheap-tier CoT outputs), 9,360 calls / (20 × 4 providers) ≈ 117 waves × ~3s ≈ 6–10 minutes, dominated by the slowest provider and its rate limit. Comfortably "minutes, not hours."

Evidence note (thin): the ceiling-probe output-token estimate (~400 CoT tokens) is a guess; CoT length is model-dependent and no source paper reports token counts. A 10-instance pilot per model should replace these estimates before committing the full run (§7.8).

What must the owner decide before this spec can run?

§7 · Open decisions

Nothing below is frozen — all ten items require an explicit owner call before this spec can run.

  1. Model set (§4) — final roster; whether to add a frontier anchor; exact model versions/snapshots and provider routing (all output-affecting, must be pinned).
  2. Aggregate function for the decision rule (§3) — mean vs min across models; pairing with the GPT-5-mini non-conformer.
  3. Decision thresholds (§3) — the 75% HIGH band, the 10-point "≈" unit, the 20-point "≪" unit. All are proposals derived from the capability synthesis's central estimates, not from measured c19 data.
  4. Boundary-case handling (§3) — mixed bands (gap 10–20 points) and per-model divergence resolution.
  5. N per stratum / stratum weights (§1) — proposed uniform N=15; owner may reweight (e.g. more N on the coordinate-after-sequence stratum flagged as most discriminating, less on near-guessable carry/heading). Whether to expand carrying-flag beyond Fetch.
  6. Grid sizes (§1) — the small/medium levels (~5×5 / ~8×8) and env-native size variants to use.
  7. Seed ranges / split boundaries (§1) — exact reserved ranges guaranteeing disjoint train/minibatch/official splits and zero overlap with any published instance.
  8. Pilot before full run (§5) — run 10 instances/model to replace the token/CoT-length estimates and confirm temp-0 repeat agreement before committing the ~$3–10 full run; if repeats disagree, raise N.
  9. Answer normalization policy — how the exact-match scorer canonicalizes model output (whitespace, row,col vs (row,col), E vs east, yes vs y) before the 0/1 compare. Loose normalization inflates scores; strict normalization may penalize format-only misses.
  10. Quick-test vs paper scope (§3 outcome c) — even if outcome (c) says the vanilla task is a viable optimizer-benchmark instrument, the positioning synthesis says the paper still needs the invented-command layer to differentiate from prior work. Owner decides whether "viable for the quick test" is sufficient or the build proceeds regardless for the publication case.
