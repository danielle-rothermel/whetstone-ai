c18 Baseline Spec: Headroom Before Complexity

Draft
whetstone-ai · 2026-07-22
This spec is a draft awaiting owner decisions. Every number below is a proposal, not a frozen parameter. The model set (§4), the numeric thresholds (§3), and seven other open items (§7) are explicitly owner-gated. The top open decision is O1 — whether the codebase's native distractor setting (--distractors) runs ON or OFF for this baseline — because it is the single biggest swing factor on whether any headroom exists to measure. See the open decisions list before treating anything here as final.

This experiment measures how much headroom exists between a naive prompt and a best-effort ceiling prompt on the existing PrOntoQA task shape — before building any of c18's added-complexity axes (the Unknown/open-world label, the constraint-puzzle stratum, soft-rule strata, or the independent forward-chaining oracle as a design element). It is a go/no-go instrument, not the benchmark itself: c18's biggest early-kill risk is that without distractors, a naive prompt already answers almost perfectly by string-matching the queried property against context (PrOntoQA: "InstructGPT is able to predict the label almost perfectly" with no distractors). If that risk is real on the reseed-only task, this baseline finds it cheaply, before any added-complexity build.

Fixed constraints — not open for this experiment
  • Instances regenerated as-is from asaparov/prontoqa (run_experiment.py --model-name json), fresh --seed per split, fresh --ontology fictional nonce ontologies. Never reuse a published instance (rubric criterion 8; contamination resistance).
  • Existing hop-depth settings and native question format. No new axes: no added Unknown-label handling beyond what the codebase natively emits, no distractor or soft-rule strata added by us — the codebase's own --distractors setting is a separate matter, left as Open Decision O1.
  • Output is True/False, exact match against the generator's stored answer gold, scored 0/1 by a simple function against an independent oracle (forward-chaining fixpoint re-derivation, ~30–50 lines). No partial credit.
  • Temperature 0. Repeats = 3 (repeat_id distinct) to validate the aggregation plumbing; determinism means repeats should largely agree (rubric criterion 5).

1. Strata design

One stratification axis: hop depth. Ontology type held constant.

Hop depth (4 levels) — D1, D2, D3, D5, via the native --min-hops/--max-hops loop. This is the single most robust accuracy dial in the literature: monotonic decline with depth across every dossier paper. It gives an incremental ladder with a designer-known ceiling. Ontology type is fixed to fictional for contamination-resistance (criterion 8); true/false ontologies are excluded because real-world priors create a ceiling effect — PrOntoQA shows the true ontology does not degrade D3→D5, which would confound the headroom read. Ontology type is therefore held constant, not a stratum axis, in this baseline — a deliberate simplification consistent with "measure headroom on the existing shape."

Depth levels D1, D2, D3, D5 proposed

D1 is the near-solved anchor (expected ceiling-adjacent); D2/D3 are the predicted mid-band; D5 is the hard tail where cheap models approach the two-way floor. D4 is dropped to keep N bounded; D0 is dropped as trivially saturated. Open: the owner may re-add either (§7, O2).

N = 150 per depth stratum proposed × 4 strata = 600 instances total

Design driver: rubric criterion 5 resolvability. A meaningful prompt change must produce a ≥10-point effect that exceeds residual noise on a 10–20 task internal eval. Because decoding is temperature 0, per-task behavior is near-deterministic given a prompt, so residual noise on a 10–20 task internal eval is small and a 10-point effect (≥1–2 of 10–20 tasks flipping) resolves cleanly. The pool must be large enough that each depth stratum can supply disjoint internal-eval subsets (≥10–20 tasks) and a held-out official split (criterion 7) without reusing tasks, and that per-stratum aggregate accuracy has a tight enough CI to read a ≥10-point floor-vs-ceiling gap as real.

At N=150/stratum a Wilson 95% CI half-width is ≈±7–8 points near p=0.7 — enough to resolve a 10-point gap per stratum, and comfortably enough to carve disjoint 10–20-task internal-eval subsets plus a ≥100-task official split. 150 × 4 = 600 instances in the baseline pool, keeping a single full run cheap (§5) while exceeding the "hundreds of tasks" pool requirement (criterion 7). Aggregation crosses strata by averaging exact 0/1 scores first by repeat within task, then across tasks within stratum, then reporting per-depth and a depth-weighted overall.

Evidence note: PrOntoQA itself used 400/condition with 95% Wilson intervals — the reviewer-expected standard for a publishable number. Owner may prefer 400/stratum to match that regime exactly (§7, O3); 150 is the cheap-iteration floor, 400 is the publication-grade target.

2. The two probe prompts (drafted verbatim)

Both prompts take the generator's native question (facts + rules) and query concatenated, and must elicit a single True/False token for exact match. These operationalize rubric criterion 4 (default prompts score low; better prompts close the gap) and criterion 9 (floor, ceiling, and a reference prompt known in advance).

2.1 Probe (a) — deliberately naive prompt

{question}

{query}

Answer True or False.

Design intent: no chain-of-thought, no statement of the closed-world convention, no statement that only the listed rules may be used, no output-format discipline beyond "True or False." This is the intended floor — it invites surface-plausibility answering, which the candidate doc's own worked example shows misses a two-hop chain. Per the capability synthesis this should land materially above the 50% two-way floor (clean fictional surface, easy A≈2 rule regime) but well below the ceiling prompt.

2.2 Probe (b) — best-effort ceiling prompt (all standard conventions stated)

You are a careful deductive reasoner. You are given a set of facts and if-then
rules, followed by a single query statement. Every predicate is fictional and
carries no real-world meaning — rely ONLY on the facts and rules given, never on
outside knowledge or surface plausibility.

Determine whether the query statement is entailed by the facts and rules under
the closed-world assumption: a statement is True if it can be derived by
chaining the given rules from the given facts, and False otherwise. Apply the
rules step by step, following each "every X is a Y" / "X are (not) Z" rule in
order, until you reach the queried property.

Facts and rules:
{question}

Query:
{query}

Reason step by step through the chain of rules, then end your reply with exactly
one word on its own final line: either
True
or
False

Design intent — each clause maps to a lever the literature shows works: "rely ONLY on the facts and rules… never on outside knowledge or surface plausibility" blocks the property-string-match shortcut and the real-world-prior shortcut. "closed-world assumption… True if derivable… False otherwise" states the native 2-way convention precisely — the CWA/Unknown ambiguity risk flagged elsewhere is sidestepped here because the native format is 2-way, with no Unknown to mishandle. "Reason step by step" (CoT) is the single largest reliable lever for non-reasoning cheap models: +8 to +12 points across JustLogic. "following each rule in order" names the reasoning form and ordering, a measured lever. "exactly one word on its own final line" makes exact-match extraction clean and guards the format-violation failure path (rubric criterion 13).

Evidence note (thin): no dossier paper ran a CoT-vs-naive ablation on PrOntoQA label-only for a modern cheap model — the +8–12 CoT lift is extrapolated from JustLogic's harder natural-language surface (uncertainty "moderate"). Treat the predicted gap magnitude as a hypothesis this run tests, not a known quantity.

3. Three-outcome decision rule

Let N = naive-prompt aggregate accuracy, C = ceiling-prompt aggregate accuracy, both depth-weighted over the four strata, cheap tier, temperature 0. Thresholds are grounded in the capability synthesis, which predicts (cheap tier, 2-way native, distractors ON): naive floor ~60–78%, ceiling ~75–90%, with the strong caveat that if distractors are OFF the naive prompt may already score 85–95% by shortcut-matching.

Proposed numeric anchors all owner-adjustable: HIGH ≥ 85%, MID = 65–85%, LOW < 65% (bands apply to the ceiling score C); gap G = C − N, with "small" G < 10 points (below the criterion-5 resolvability threshold) and "large" G ≥ 15 points.

Outcome (a) · No headroom

N ≈ C, both HIGH

C ≥ 85% and G < 10

The task is near-saturated even naively; likely the distractor-shortcut regime (§1's O1 risk). No gap for an optimizer to close.

Decision: proceed to the candidate's added-complexity design (distractors on/deeper, the Unknown axis) — the existing shape is too easy to discriminate.
Outcome (b) · Headroom, not prompt-closable

N ≈ C, both MID/LOW

C < 85% and G < 10

Both prompts stay stuck below ceiling: a raw capability deficit at depth, not a prompt gap. Matches the depth-collapse finding — CoT does not remove depth degradation.

Decision: proceed to added-complexity design and keep base difficulty shallow — deep hops are unsolvable by prompting, so the discriminating band is shallow-to-mid depth.
Outcome (c) · Prompt-closable headroom

N ≪ C

G ≥ 15 points, C in MID/HIGH

The ceiling prompt's CoT plus convention-stating closes a real gap — the gap is prompt-shaped, not a raw capability wall.

Decision: direct prompt optimization is viable on the existing shape — run COPRO/MIPROv2/GEPA on the reseed-only task; this is the publishable path identified in the positioning synthesis.
Threshold justification: the 85% HIGH cut sits at the top of the predicted ceiling band (75–90%) and inside the distractor-off shortcut band (85–95%) — a naive score at/above it is the saturation signature. The 65% LOW cut sits just below the predicted naive floor (60–78%) — a ceiling score below it means even best-effort prompting cannot lift the aggregate, i.e. deep-hop capability collapse dominates the mix. The G ≥ 15 "large gap" cut sits comfortably above the criterion-5 resolvability floor (10 points), consistent with the CoT lift magnitude (+8–12 at the JustLogic surface, expected somewhat larger at PrOntoQA's cleaner surface). Per-depth override: even when the aggregate reads (a), inspect the D3/D5 strata — if the gap concentrates there while D1/D2 saturate, that is effectively outcome (c) restricted to the hard tail and argues for prompt optimization scoped to deep instances. Report the per-depth gap regardless.
Evidence-thinness caveat: the exact band boundaries are extrapolations — no dossier tests this precise (cheap-2026-tier × 2-way-native × PrOntoQA-label-only) cell (uncertainty "moderate-to-high"). The rule is robust to being off on absolute levels because it keys on the shape (N≈C vs N≪C, and where C lands), not on hitting a precise number.

4.2 Proposed candidate set not finalized

The 2026 cheap tier named as the quick-test target (no dossier tests these directly):

  • Gemini-2.5-Flash
  • GPT-5-mini
  • Claude Haiku 4.5
  • DeepSeek-Chat

Proposed baseline: run 2 of these 4 for the first headroom pass (one to anchor cost estimates, a second to check cross-model agreement of the go/no-go read), expanding to all 4 only if the two disagree on the outcome bucket.

4.3 Tradeoffs for the owner

  • More models — stronger evidence the headroom read is not model-specific, and cross-model transfer becomes checkable (a reviewer demand); but linear cost/wall-clock growth.
  • A single cheap model — cheapest, fastest iteration, but a single-model read is explicitly not publishable and risks reading model-idiosyncrasy as task property.
  • Include one mid/frontier anchor (e.g. Claude Sonnet or DeepSeek-V3) — bounds the ceiling and shows whether the gap is cheap-tier-specific; adds cost. The quick test's whole point is cheap iteration, so a frontier anchor is optional and probably a later ablation.
  • Reasoning vs non-reasoning cheap models — GPT-5-mini / DeepSeek-Chat behave differently on depth than pure chat models; mixing them tests robustness of the go/no-go read but muddies a clean single-tier number.

5. Cost & wall-clock estimate per full baseline run

Formula: instances (N) × probes × models (M) × repeats = LLM calls; calls × tokens/call = tokens; tokens × price = cost. Fixed factors: N = 600 instances (150 × 4 depth strata); 2 probes (naive, ceiling); repeats = 3.

Per-instance token estimate proposed

Native PrOntoQA fictional instance surface is short (facts+rules paragraph + one-line query); a single theory+query without in-context demos is small. Deeper hops (D5) grow the facts/rules block, but demo-free even D5 stays well under ~600 input tokens — use ~500 input as a safe upper bound.

ComponentNaive probeCeiling probe (CoT)
Input tokens (instance + prompt scaffold)~250~400
Output tokens~3 (True/False)~250 (CoT chain + label)
Total tokens/instance~253~650

Calls, tokens, and dollar cost

Calls per model per full run: 600 × 2 probes × 3 repeats = 3,600 calls/model.

Tokens per model per full run: using a blended ~450 tok/call (half naive ~253, half ceiling ~650): 3,600 × 450 ≈ 1.62M tokens/model (≈0.9M input + 0.7M output).

By model count (M): 1 model ≈ 1.6M tokens, est. $0.5–1; 2 models (proposed) ≈ 3.2M tokens, est. $1–2; 4 models ≈ 6.5M tokens, est. $2–4 — at a representative cheap-tier blended rate of ~$0.30–0.60 per 1M tokens.

Wall-clock (proposed): 3,600×M short calls at temp 0. At modest concurrency (~20 in flight) and ~1–3s/call, one model's 3,600 calls ≈ 3–9 minutes; 2 models ≈ 6–18 minutes; 4 models ≈ 12–36 minutes — assuming no rate-limit bottleneck. Ceiling-prompt CoT calls dominate wall-clock (longer outputs). Comfortably "minutes, not hours."

Evidence note (thin): the per-instance token estimates are stated assumptions, not measured — confirm against a real reseeded batch, especially D5 input size demo-free, before committing the full run (§7, O5). Provider price sheet for the chosen cheap tier is also unconfirmed (§7, O6); concurrency/rate-limit headroom per provider is unverified locally (§7, O7).

7. Open decisions (awaiting the owner)

Nothing below is frozen — all nine items require an explicit owner call before this spec can run.

  1. O1 — Distractors ON or OFF in the reseed-only baseline (codebase --distractors {none,relevant,irrelevant}). The single biggest swing factor: distractors OFF risks naive-prompt saturation at 85–95% via property-string-matching, collapsing all headroom. Must verify the generator's default in the codebase before running. Note: "no distractor strata added by us" does not itself decide whether to leave the native distractor knob on. Default if unspecified: ON (relevant) — required to avoid the trivial shortcut and keep the headroom read meaningful. Flagged as the top thing to confirm.
  2. O2 — Depth levels (§1): proposed {D1, D2, D3, D5}. Add D4? Add D0? Extend beyond D5? Determines where the informative mid-band sits and whether the hard tail is represented. Default: {D1, D2, D3, D5}.
  3. O3 — N per stratum (§1): 150 (cheap-iteration) vs 400 (PrOntoQA CI-grade). Publication-grade CIs (Wilson, 400/condition) vs fast iteration. Default: 150 for first pass, 400 for any publishable run.
  4. O4 — Model set (§4): which of {Gemini-2.5-Flash, GPT-5-mini, Haiku 4.5, DeepSeek-Chat}, how many, and whether to add a mid/frontier anchor. Cost/wall-clock vs evidence strength; a single model is not publishable. Default: 2 cheap models first pass.
  5. O5 — Confirm per-instance token estimates (§5) against a real reseeded batch, especially D5 input size demo-free. Affects cost/wall-clock accuracy. Default: ~500 input / ~250 CoT output upper bound.
  6. O6 — Provider price sheet for the chosen cheap tier (§5). Turns the token estimate into a dollar figure. Default: ~$0.3–0.6 / 1M tokens blended.
  7. O7 — Concurrency / rate-limit headroom per provider at intended parallelism (§5). The wall-clock claim (criterion 14) assumes no rate-limit bottleneck. Default: 20 in flight, assumed unthrottled.
  8. O8 — Independent-oracle scope: full forward-chaining fixpoint (30–50 LOC) vs spot-check subset. The native label is definitional (a negation flag), not a prover verdict — a generation bug is invisible in the label alone. Reviewers demand a verifier-soundness check. Default: full fixpoint check on 100% of instances (cheap for A≈2 chaining).
  9. O9 — Ceiling-prompt exact wording (§2): freeze after 2–3 pre-freeze gate reads? The candidate doc names 2–3 pre-freeze gates for CWA/Unknown wording; even at 2-way, the "closed-world True/False" phrasing should be sanity-checked so it doesn't itself leak the answer or bias toward one label. Default: freeze §2(b) after one gate read.

Appendix: grounding notes on floor/ceiling expectations

  • Floor (naive), predicted ~60–78% aggregate — extrapolated from JustLogic 3-way zero-shot ~53% for cheap tier, adjusted up for 2-way format, cleaner fictional surface, easier A≈2 rules, and a stronger 2026 tier. Uncertainty moderate-to-high; distractors OFF could push it to 85–95% (O1).
  • Ceiling (best-effort CoT), predicted ~75–90% aggregate — floor + CoT (+8–12) + convention-naming, but the depth ceiling is real and prompting does not remove deep-hop degradation, so the aggregate stays short of ~95%. Uncertainty moderate.
  • Thin-evidence flag: no dossier measures the exact (2026-cheap × 2-way native × PrOntoQA label-only) cell; the closest PrOntoQA numbers are proof accuracy on a 2022 model. The decision rule (§3) is deliberately shape-based to survive absolute-level error.
