Measure the existing task's headroom before building c23's complexity

Draft

A cheap go/no-go probe: how large is the naive-vs-ceiling prompt gap, and is it prompt-closable?

whetstone-ai · 2026-07-22

This experiment measures how much headroom exists between a naive prompt and a best-effort ceiling prompt on the existing InductionBench-style single-rule ISL/OSL task shape — before any build effort goes into c23's added complexity (the composition ladder, invented multi-rule stacking, the MC variant, multi-family strata). It is a go/no-go instrument, not the benchmark itself: InductionBench's own headline finding is that frontier models collapse to 0.00–5.69% compatibility at moderate difficulty. If the existing shape is already unreachable — or already trivial — at the cheap tier, this baseline finds that cheaply, before the composition-ladder work that is c23's real cost gets built.

This spec is a draft awaiting owner decisions. Every number below is a proposal, not a frozen parameter. The model set (§4), the decision-rule thresholds (§3), the k=3 stratum keep/drop call, and five other open items are explicitly owner-gated — see the open decisions list before treating anything here as final.
Fixed constraints — not open for this experiment
  • InductionBench-style subregular (ISL/OSL) instances only. No composition ladder, no invented multi-rule stacking, no MC variant, no multi-family strata — that added complexity is exactly what this experiment tests the necessity of, so it cannot appear in the baseline.
  • Fresh seeds, never a published InductionBench instance — regenerated with the vendored + patched codebase (config.py stub, drop the translate_fewshot_input_output_pairs import, thread a real seed param, replace list(set(...)) hash-order nondeterminism with sorted(...) at lines 53/61/107/132/208/244 rather than pinning PYTHONHASHSEED). Contamination resistance, rubric criterion 8.
  • Single-rule instances only. --number_of_rules = 1 is fixed under this constraint.
  • Existing I/O-demos-plus-query format; short-string exact-match output.
  • Exact 0/1 scoring vs. the independent oracle (apply_ISL_rule / apply_L_OSL_rule / apply_R_OSL_rule re-applied to the held-out query, compared for string equality).
  • Temperature 0. Repeats = 3 to exercise the aggregation plumbing.

How is the instance pool stratified, and how big is it?

§1 · Strata

Three live axes: type, k (context window), vocab size — k is primary

The candidate and repo expose four generator knobs on the ISL/OSL path: --type (ISL / L-OSL / R-OSL), --k (context window), --vocab_size (|Σ|), and --number_of_rules. Under the single-rule constraint, --number_of_rules = 1 is fixed, leaving three live axes. k is the single largest difficulty driver (InductionBench Sec 5.2): k=2→k=4 collapses most models from partial success to ~0 compatibility, so it is the primary stratum axis. --type (ISL / L-OSL / R-OSL) is the secondary axis — it changes the transducer family via one generation diff and all three share one oracle codepath, so covering them is cheap. --vocab_size (|Σ|) matters less than k, so it is held small and fixed (|Σ|=4) to keep the axis count down and land in the partial-success regime; it becomes a curation knob only if the pilot shows the band collapsing.

Why the k ladder stops at 3, not 4: the synthesis shows k=4 sends everything except o3-mini to ~0 (IB Table 1). A baseline whose hardest stratum is unreachable violates criterion 9 (a known-reachable ceiling) and criterion 4 (room to improve). k∈{2,3} keeps all strata inside the partial-success band. Open: the pilot must confirm S4 (k=3) is reachable by the ceiling prompt; if it sits near floor for all models, drop it and hold k=2 across the board (§7.3). k=4 is reserved only as a curation lever if k=2 turns out one-shot-guessable.

N per stratum = 50 proposed; 200 instances total

Design driver: rubric criterion 5 resolvability — a meaningful prompt change (≥10 points) must exceed residual noise on a 10–20 task internal eval. Exact-match is per-instance binary, so N is sized so a 10-point (0.10) shift is resolvable and per-stratum averages are stable. At p≈0.5 (worst-case binomial variance), the standard error of a single-stratum exact-match rate at N=50 is √(0.25/50) ≈ 0.071; a 10-point true difference is ~1.4 SE — marginal alone. The go/no-go decision (§3) reads the pooled naive-vs-ceiling contrast across all four strata, not one stratum, which amplifies resolution.

4 strata × N=50 = 200 instances in the pinned pool. Pooled across strata, the SE of the overall exact-match rate at p≈0.5 is √(0.25/200) ≈ 0.035; the naive-vs-ceiling difference SE (two independent 200-instance measurements) is √2 × 0.035 ≈ 0.050. A 10-point pooled gap is ~2 SE (resolvable at ~95%); the synthesis-predicted 20–40-point gap is 4–8 SE (unambiguous). This satisfies criterion 5's resolvability requirement at the decision level. Repeats=3 at temp 0 validate the aggregation plumbing (criterion 5), not variance reduction — they should largely agree.

Evidence note: N=50/stratum is a proposal balancing criterion-5 resolution against §5's cost. If the owner wants each individual stratum (not just the pool) to resolve a 10-point effect at ~95%, raise to N≈100–200/stratum — the pooled-decision framing is why 50 is proposed as the floor, not the target (§7.2).

What exactly do the naive and ceiling prompts say?

§2 · Probes

Both probes are run verbatim, temperature 0, repeats=3, against the same pinned instance pool; the only difference between them is the instruction text — same demos, same query, same extraction. These operationalize rubric criterion 4 (default prompts score low; better prompts close the gap) and criterion 9 (floor, ceiling, and a reference prompt known in advance). Output extraction for both: take the text after the last Output: line, strip surrounding whitespace and markdown fences, compare for exact string equality to the oracle output. No CoT is forced in either prompt — the synthesis shows no-CoT beats forced-CoT unanimously for the cheap tier; forcing it would confound the ceiling.

2.1 Probe (a) — deliberately naive prompt (floor probe)

Here are some examples:

vez fam qor -> VEZ vez fam qor
qor luz tam -> QOR tam luz luz
fam vez luz -> FAM vez luz luz

luz qor vez ->

Verbatim instruction string used to build the prompt: <demos as "IN -> OUT" lines>\n<query IN> -> . That is the entire prompt — no system prompt beyond the provider default.

Design intent: gives the demonstration pairs and the query — nothing else. No task framing, no statement that a hidden rule exists, no output-format instruction, no delimiter guidance. This is the "repeat the transformation shown" level of scaffolding; it leans entirely on the model's own ability to notice a pattern exists and infer it unaided, which is precisely where the too-easy-or-too-hard floor risk lives.

2.2 Probe (b) — best-effort ceiling prompt (states all standard conventions)

SYSTEM:
You are solving a hidden-rule string-transformation puzzle. Each puzzle has one fixed,
deterministic transformation rule that maps an input string to an output string. The rule
depends only on the tokens in the input (their identity, their length, their position, and
the characters at the ends of each token) — it never uses outside knowledge, randomness, or
context beyond the examples. Tokens are the whitespace-separated words. The same rule was
applied to every example below. Your job: infer that single rule from the examples, then
apply exactly the same rule to the final query.

Rules of the format:
- Read all the demonstration pairs. Each is written "INPUT -> OUTPUT".
- The transformation is a length- and position- and suffix/prefix-sensitive edit over the
  tokens: tokens may be duplicated, reordered, re-cased, or rewritten based on their local
  context (the characters immediately around each token position, up to a small window).
- Determine the exact rule that makes ALL demonstrations correct simultaneously. If more than
  one rule fits the demonstrations, choose the simplest one that fits every pair.
- Do not explain your reasoning. Output only the transformed string for the query.
- Preserve spacing and casing exactly as the rule dictates. Do not add quotes, punctuation,
  or commentary.

USER:
Demonstrations:
vez fam qor -> VEZ vez fam qor
qor luz tam -> QOR tam luz luz
fam vez luz -> FAM vez luz luz

Query:
luz qor vez ->

Output only the transformed string, on a single line prefixed with "Output:".

Design intent: states every standard convention of the existing task family — single deterministic rule, tokens = whitespace-separated words, local/suffix/positional sensitivity matching the ISL/OSL substrate, the "fits all demos / simplest rule" induction principle, direct-answer / no-CoT, and a strict Output: format for robust extraction.

Why this is a legitimate ceiling, not cheating: it states the conventions and inductive bias of the task family (what kind of rule, how to read the format, how to answer) but does not reveal the specific latent rule of any instance — the model still has to induce it from the demos. This is the "designer can write a prompt that scores ≈100% because they know the conventions" sense of criterion 9, adapted to the reality (synthesis) that ≈100% is not reachable at the cheap tier even with perfect conventions stated. Open: the owner should confirm the wording does not over-hint and inflate the ceiling (§7.6).

What does each measured outcome tell the project to do next?

§3 · Decision rule

Read the pooled naive rate A and pooled ceiling rate C (across all 200 instances × 3 repeats × chosen models, exact-match). Thresholds are proposals grounded in the capability synthesis, whose headline predictions are: naive floor ~5–25%, ceiling ~35–65%, gap ~20–40 points, and no cheap model saturates (ceiling well below 100%).

Proposed numeric anchors all owner-adjustable: HIGH band = pooled rate ≥ 70% (near the top of the synthesis ceiling range or above); MID/LOW band = pooled rate ≤ 45% (at or below the synthesis-predicted ceiling midpoint); "gap significant" means C − A ≥ 15 points (≈3 SE of the pooled difference, comfortably above criterion 5's 10-point resolvability floor).

Outcome (a) · No headroom

A ≈ C, both HIGH

gap < 15 pts and both ≥ 70%

The existing shape is already near-ceiling; there is nothing for an optimizer to close. Hitting HIGH would itself be a genuine surprise — the synthesis predicts no cheap model saturates, so this would indicate the strata are too easy (one-shot-guessable, a criterion 4 failure).

Decision: proceed to the candidate's added-complexity design (composition ladder / multi-family strata) — the baseline shape is too easy to validate optimizers.
Outcome (b) · Headroom, not prompt-closable

A ≈ C, both MID/LOW

gap < 15 pts and both ≤ 45%

Raw capability deficit: even best-effort conventions don't lift the cheap tier. This is InductionBench's own "even the simplest complexity class fails" thesis showing up again.

Decision: build the added-complexity design and keep base difficulty shallow — hold k=2, small vocab; the ladder must add decomposable difficulty on top of a reachable floor, not push deeper into the collapse regime.
Outcome (c) · Prompt-closable headroom

C − A ≥ 15, A in MID/LOW

gap significant, ceiling meaningfully higher

The naive prompt leaves recoverable points on the table that stating conventions closes — exactly the substrate an optimizer benchmark needs.

Decision: direct prompt optimization is viable on the existing shape — run COPRO/MIPROv2/GEPA on this baseline substrate; no added complexity needed for a usable optimizer signal.
Justification of the numeric thresholds (§7.5): the 15-point gap threshold sits below the synthesis-predicted 20–40-point gap (so a real effect clears it) but above criterion 5's 10-point noise floor and the ~5-point pooled difference-SE band (so noise doesn't trip it). The 70% HIGH band is deliberately above the synthesis ceiling top (~65%). The 45% MID/LOW band is the synthesis ceiling midpoint. Format-refusal caveat: Claude-Haiku-class and Gemini-Flash-class models can crater to single digits purely from misreading the I/O block (MIR refusal behavior; WILT Haiku 1/50, Flash-8b 0/50). If one specific model shows A≈C≈near-zero while others show a healthy gap, that is not outcome (b) — it is a per-model representation problem. Report per-model rates, not only the pool, and treat a single model's floor-crater as a format bug to fix before concluding.

4.2 Proposed candidate set not finalized

A 3-model set balancing coverage, cost, and the synthesis's warnings:

  • GPT-5-mini — cleanest GPT-4o proxy; GPT-4o is the best-behaved cheap datapoint on-target.
  • DeepSeek-Chat — cheapest, and the synthesis's DeepSeek proxies are the least refusal-prone of the cheap tier.
  • Claude Haiku 4.5 — deliberately included because it is the format-refusal risk; if the baseline is going to break on I/O-block comprehension, better to see it here than in the expensive run (serves criterion 13 — failure paths exercised on purpose).

Optional 4th: Gemini-2.5-Flash (second refusal-risk data point) or a Claude-3.5-Sonnet-class upper anchor (to see how much of the gap is tier-limited vs. task-limited).

4.3 Tradeoffs for the owner

  • Pro (more models): better separation of "task headroom" from "this one model," and satisfies the ≥2–3-model positioning demand — but multiplies §5 cost linearly.
  • Pro/Con (refusal-risk models): including Haiku/Flash surfaces the representation failure early (good for the debug loop) but risks a per-model floor-crater that muddies the pooled decision rule — mitigated by reading per-model rates (§3 caveat).
  • Con (a reasoning model): o1/o3-mini class would show the ceiling of the substrate but is off-tier and expensive; probably out of scope for a baseline headroom measurement.

What does one full baseline run cost in dollars and minutes?

§5 · Cost

Formula: instances × prompts × models × repeats = LLM calls; calls × tokens/call = tokens; tokens × price = cost. Fixed factors: N = 200 instances; 2 prompts (naive, ceiling); repeats = 3.

Per-instance token estimate proposed

ComponentPrompt A (naive)Prompt B (ceiling)
Input tokens (short strings, small vocab, few demos)~120~400
Output tokens (short transformed-string answer)~20~20
Blended average~300 in + ~20 out ≈ 320 tok/instance/prompt

Calls, tokens, and dollar cost

Calls per full baseline run: 200 instances × 2 prompts × 3 models × 3 repeats = 3,600 LLM calls.

Token volume: input ≈ 3,600 × ~260 avg input tok ≈ 0.94M input tokens; output ≈ 3,600 × ~20 output tok ≈ 0.07M output tokens.

Cost (order-of-magnitude, cheap-tier pricing ~$0.15–$0.60/M input, ~$0.60–$2.40/M output, 3 models averaged): input ≈ 0.94M × ~$0.40/M ≈ $0.38; output ≈ 0.07M × ~$1.50/M ≈ $0.11; per full baseline run ≈ $0.50 — well under the "cents-to-a-dollar" band rubric criterion 3 wants.

Wall-clock (proposed): 3,600 calls at short output length. At a modest 10 concurrent requests and ~1.5s latency/call, ≈ 3,600 × 1.5 / 10 ≈ ~9 minutes; at 20 concurrent, ~4.5 minutes. Comfortably inside criterion 14's "minutes, not hours," assuming no rate-limit bottleneck.

Evidence note: costs scale linearly with model count and N. Adding a 4th model → ~$0.67/run, ~12 min. Raising N to 100–200/stratum doubles both. All figures are order-of-magnitude; pin real per-model pricing before the owner signs off.

What must the owner decide before this spec can run?

§7 · Open decisions

Nothing below is frozen — all eight items require an explicit owner call before this spec can run.

  1. Model set (§4) — finalize the 3-model set (proposed: GPT-5-mini, DeepSeek-Chat, Claude Haiku 4.5); decide whether to add a 4th refusal-risk model (Gemini-2.5-Flash) and/or an upper anchor (Sonnet-class).
  2. N per stratum (§1.3) — proposed 50 (pooled-decision resolution). Owner may raise to 100–200 if per-stratum (not just pooled) 10-point resolution is required, at linear cost.
  3. Keep or drop the k=3 stratum (S4) (§1.2) — contingent on the pilot: if S4 sits near floor for all models, drop it and hold k=2. Owner sets the reachability bar for keeping it.
  4. Vocab size (§1.1) — fixed at |Σ|=4 in this draft; owner confirms, or authorizes it as a curation lever if k=2 proves one-shot-guessable.
  5. Decision-rule thresholds (§3) — confirm the 15-point gap / 70% HIGH / 45% MID-LOW bands, or adjust to the owner's risk tolerance.
  6. Prompt B wording (§2.2) — the verbatim ceiling prompt states conventions but not the specific rule; owner should confirm it doesn't over-hint (which would inflate the ceiling and mask headroom).
  7. CoT axis (§2) — this draft fixes both prompts to direct-answer / no-CoT per the synthesis (no-CoT beats forced-CoT unanimously at the cheap tier). Owner confirms the baseline does not sweep CoT (it is a phase-2 optimizer confound to control, not a baseline variable).
  8. Pilot gate (§1.2, repo Step 5) — a small pilot must confirm reachability before freezing the pool. Owner approves the pilot-first sequencing and the go/no-go it feeds.
