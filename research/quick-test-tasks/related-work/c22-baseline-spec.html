Measure the reseed-only IFEval shape's headroom before building c22's complexity

Draft

A cheap go/no-go probe: is the naive-vs-ceiling gap wide but shallow — closable by one prompt?

whetstone-ai · 2026-07-22

This experiment measures how much headroom exists between a naive prompt and a best-effort ceiling prompt on c22's reseed-only baseline — standard IFEval checker atoms only, before any build effort goes into the added-complexity layer (conditional Selection/discovery atoms, IFBench OOD atoms, seeded-generator hardening). It is a go/no-go instrument, not the benchmark itself: c22's related-work synthesis shows the reseed-only baseline strips out exactly the two mechanisms (Selection/discovery, OOD atoms) that carry c22's entire anti-enumeration story, so a naive-to-ceiling gap here may be wide but shallow — closable by a single well-drafted enumerate-the-atoms prompt. If that risk is real, this baseline finds it cheaply, before the Selection/OOD/generator work that is c22's real cost gets built.

This spec is a draft awaiting owner decisions. Every number below is a proposal, not a frozen parameter. The model set (§4), the decision thresholds (§3), and six other open items (§7) are explicitly owner-gated — see the open decisions list before treating anything here as final.
Fixed constraints — not open for this experiment
  • Standard IFEval checker atoms only. Reuse the google-research IFEval checker library as-is (instructions.py / instructions_util.py / instructions_registry.py, INSTRUCTION_DICT), each class's check_following as the 0/1 atom oracle. Scoring glue reuses evaluation_lib.test_instruction_following_strict.
  • No OOD atoms, no conditional Selection constraints. Those are exactly the added-complexity design this experiment gates — they cannot appear in the baseline.
  • 3–5 atoms sampled fresh per instance over trivial micro-tasks; all-checkers-pass strict 0/1 (all(check_following(...))), no partial credit.
  • Fresh seeds only — never a published instance. Instances synthesized per random.seed(instance_seed) before each build_description, seed recorded as metadata (rubric criterion 8, contamination resistance). The shipped config.seed is a decoy wired only to the LLM payload; do not rely on it.
  • Independent oracle, exact 0/1 vs. the deterministic checker — never an LLM judge.
  • Temperature 0. Repeats = 3 (validate the aggregation plumbing; determinism means they should largely agree).

How is the instance pool stratified, and how big is it?

§1 · Strata

Two crossed axes: constraint count, atom-type mix

Constraint count (3 levels) — n ∈ {3, 4, 5}, the candidate's own stated difficulty dial; strict-AND compounds failure per added atom. Atom-type mix (2 levels) — easy-skewed (all atoms from the easy pool: casing, format-wrapper, keyword-presence, start/end-token — per-atom pass ~0.80–0.90) vs. mixed (≥1 hard atom included: exact-word-count-equals-N, forbidden-letter across the whole output — per-atom pass ~0.55–0.75). Atom-type mix is the single largest score driver identified in the related-work synthesis. Honor INSTRUCTION_CONFLICTS (instructions_registry.py:79-176) when sampling a stack so atoms cannot contradict.

Deliberately excluded from the grid

Language-detection atoms (nondeterministic edge) and any atom requiring the nltk punkt tokenizer unless the runtime download is pinned — both are determinism risks (repo red flag) that would undercut criterion 5's resolvability requirement. Prefer regex/count-only checkers for the easy pool.

N per stratum = 20 proposed; 120 instances total; 6 strata

Design driver: rubric criterion 5 resolvability. At temperature 0 with pinned seeds the per-task score is near-deterministic, so residual noise across the 3 repeats should be small; the binding constraint is instead the binomial resolution of a per-stratum rate estimate. At an all-pass rate near 0.3–0.7, the standard error of a 20-instance rate estimate is ≈ 0.10–0.11, so a ≥10-point (0.10) effect sits at the ~1 SE margin per single stratum and is comfortably resolvable when pooled across the aggregate. N=20 satisfies criterion 5's "10–20 task internal eval" band directly.

Constraint count × atom-type mix = 3 × 2 = 6 strata × 20 = 120 instances in the pinned pool, × 3 repeats = 360 rollouts per (model, prompt) condition. Aggregation: average exact 0/1 first by repeat within task, then across tasks within stratum, then across strata. Report per-stratum rates AND the pooled all-pass rate. Improvements must come from solving more complete instances, never partial credit inside an instance.

Evidence note (thin): N=20/stratum trades resolvability against cost. If the owner needs a ≥10-pt effect resolvable within a single stratum at high confidence, bump to N=40 (flagged in §7.2). The pooled 120-instance aggregate resolves a 10-pt shift robustly regardless.

What exactly do the naive and ceiling prompts say?

§2 · Probes

Both prompts receive the same generated instance: a base micro-task line plus the concatenated per-atom constraint descriptions emitted by build_description. These operationalize rubric criterion 4 (default prompts score low; better prompts close the gap) and criterion 9 (floor, ceiling, and a reference prompt known in advance).

2.1 Probe (a) — deliberately minimal naive prompt

{BASE_TASK_AND_CONCATENATED_CONSTRAINTS}

Answer:

Design intent: that is the entire prompt — the raw generated instance text, a blank line, and Answer:. No restatement of the constraints, no enumeration, no format hygiene, no emphasis, no worked reasoning. This is the "prompt lists only the stated atoms" miss case from the candidate's own example — it operationalizes the criterion-4 expectation that default behavior sits well below ceiling.

2.2 Probe (b) — best-effort ceiling prompt (all standard conventions stated)

You must produce a response that satisfies EVERY constraint below. The response
is scored 1 only if all constraints pass a deterministic checker; missing even
one constraint scores 0. There is no partial credit.

Constraints (all must hold simultaneously):
{BASE_TASK_AND_CONCATENATED_CONSTRAINTS}

Follow these conventions of this task exactly:
- Every constraint is stated explicitly above. Nothing is hidden; you do not need
  to infer any unstated rule. Enumerate each constraint and satisfy it directly.
- Output ONLY the answer text itself. Do not add a preamble, explanation,
  restatement, label, quotation marks, or trailing commentary — extra text can
  itself violate a length, word-count, casing, or forbidden-token constraint.
- Do NOT use markdown, bold, bullets, or headers unless a constraint explicitly
  requires them; stray formatting characters count against exact-match checks.
- For any word-count or length constraint, count exactly and match the stated
  number precisely (exact means exact, not "about").
- For any forbidden-letter or forbidden-word constraint, scan your whole answer
  and confirm the letter/word appears nowhere.
- For any casing, start-token, or end-token constraint, verify the first/last
  characters or tokens literally match what is required.
- Before finalizing, silently check each constraint one more time; if any fails,
  revise until all pass. Then output only the final answer.

Answer:

Design intent: states every standard/default convention of the reseed-only task — strict all-pass semantics, output-only hygiene (the strict-vs-loose false-negative mitigation from the synthesis, since c22 inherits IFEval checkers without the loose metric), per-atom-type checking hints, and a self-verify pass. Because nothing is hidden in the reseed-only baseline, this prompt can in principle enumerate and satisfy all stated atoms — it is the true ceiling probe.

Evidence note: the related-work synthesis is the direct source of the "nothing hidden" framing — every documented Selection operator in the corpus (ComplexBench, IF-RewardBench) states its branch condition in the prompt, and the reseed-only baseline has no Selection layer at all yet. The ceiling prompt's enumerate-and-verify design is calibrated to that fully-stated baseline, not to the later Selection/OOD layer.

What does each measured outcome tell the project to do next?

§3 · Decision rule

Let naive = pooled all-pass rate under probe (a), ceiling = pooled all-pass rate under probe (b), gap = ceiling − naive. Thresholds are anchored to the related-work synthesis's point estimates for the expected default (3–5 mixed atoms, cheap tier, temp 0, strict all-pass): naive floor central guess ≈ 0.30 (range 0.25–0.40), ceiling central guess ≈ 0.70 (range 0.60–0.80). "High" is defined relative to the ceiling-prompt expectation, not 1.0.

Proposed numeric anchors all owner-adjustable: "≈" (naive approximately equals ceiling) means gap < 0.10 (the criterion-5 resolvability unit); "≪" (naive much less than ceiling) means gap ≥ 0.20 (two resolvability units); HIGH band = rate ≥ 0.75; MID/LOW band = rate < 0.55 (especially < 0.40).

Outcome (a) · No headroom

naive ≈ ceiling ≈ HIGH

gap < 0.10 and both ≥ 0.75

Both prompts land near-ceiling; the task is trivially solved either way, regardless of prompt engineering effort.

Decision: proceed to the candidate's added-complexity design (Selection/OOD atoms). The existing reseed-only shape is exhausted as an optimizer target.
Outcome (b) · Headroom, not prompt-closable

naive ≈ ceiling ≈ MID/LOW

gap < 0.10 and both < 0.55 (esp. < 0.40)

Even the ceiling prompt cannot close the gap to high — a raw capability deficit under strict stacking, not a prompting problem.

Decision: proceed to the added-complexity design and keep base difficulty shallow — fewer atoms, easy-skewed strata — so the optimizer signal is not floored.
Outcome (c) · Prompt-closable headroom

naive ≪ ceiling

gap ≥ 0.20

A single engineered prompt recovers most of the gap — the anti-enumeration mechanisms are absent from this baseline, so the gap is wide-but-shallow.

Decision: direct prompt optimization is viable on the existing reseed-only shape for the quick-test purpose. Caveat: the synthesis warns this gap is likely one-shot-closable, so sustained optimization difficulty may still require the Selection/OOD complexity layer for a durable optimizer target (§7.4).
Ambiguous middle (§7.4): 0.10 ≤ gap < 0.20, or both prompts land ~0.55–0.75. Proposed default: treat as a soft (c)-lean but flag for owner review — synthesis uncertainty is HIGH and asymmetric here (floor could plausibly be 0.10–0.50), so a borderline gap should trigger a targeted re-run (e.g. split easy-skewed vs. mixed strata) before committing a branch. Read per-stratum too: the decision may differ by stratum — easy-skewed/3-atom strata may show outcome (a) while mixed/5-atom strata show (b) or (c); if so, the shallow-base recommendation in (b) is directly actionable.

4.2 Proposed candidate set not finalized

The quick-test contract wants a small, cheap model (rubric goal: cents per run, minutes). Proposed:

  • Primary: one cheap-tier hosted model — candidates: Gemini-2.5-Flash, GPT-5-mini, Claude Haiku 4.5, or DeepSeek-Chat. This is the decisive band for c22 and, critically, the band with essentially zero direct evidence in the related-work corpus — measuring it is the point.
  • Optional second (frontier anchor): one frontier model (e.g. a GPT-4-class or Claude Sonnet-class model) to bracket the ceiling and confirm the strict-AND compression direction, matching the reviewer demand for multiple base models.

4.3 Tradeoffs for the owner

  • Pro: the cheap tier is where the answer matters but where the corpus is flying blind — the floor/ceiling estimates (§3) are interpolations, not measurements; the actual numbers can genuinely inform or surprise.
  • Con: a single cheap model is cheapest and fastest but a single-model result "won't generalize"; adding a frontier anchor roughly doubles cost but pre-empts that reviewer objection.
  • Con: model choice interacts with §3 thresholds — a stronger cheap model shifts both naive and ceiling upward, making outcome (a) more likely; a weaker one pushes toward (b). The thresholds are set for the cheap-tier band and may need per-model adjustment.
  • Note: exact model versions/snapshots are output-affecting and must be pinned once selected (§7.1).

What does one full baseline run cost in dollars and minutes?

§5 · Cost

Formula: N × repeats × prompts = rollouts/model; rollouts × tokens/rollout = tokens; tokens × price = cost. Fixed factors: N = 120 instances (6 strata × 20); 2 prompts (naive, ceiling); repeats = 3.

Per-instance token estimate proposed

ComponentNaive probeCeiling probe
Input tokens (base task + concatenated constraints, +convention block for ceiling)~150~400
Output tokens (micro-task, a few words)~20~20
Total tokens/call~170~420

The ceiling prompt's self-verify step is instructed to stay silent, so output stays tiny; if a model emits visible reasoning, cap via max_tokens ≈ 64.

Calls, tokens, and dollar cost

Rollouts per model per full run: 120 × 2 prompts × 3 repeats = 720 rollouts/model (360 naive + 360 ceiling).

Tokens per model per full run: naive 360 × (150 in + 20 out) = 54k in + 7.2k out; ceiling 360 × (400 in + 20 out) = 144k in + 7.2k out — ≈198k input + 14.4k output ≈ 213k tokens/model.

Cost (illustrative placeholder rates, ~$0.15/M input, ~$0.60/M output for cheap tier): per model ≈ (0.198M×$0.15) + (0.0144M×$0.60) ≈ $0.04 per full baseline run. Adding a frontier anchor (~$3/M in, $15/M out): frontier ≈ $0.81 — combined ≈ $0.85 per full run. Both satisfy the rubric's "cents per run" goal.

Wall-clock (proposed): 720 rollouts, ~20 output tokens each, no sandbox, no chained calls. At modest concurrency (~20 in flight) and ~1–2s/call, ≈ 1–2 minutes per model, rate-limits permitting. Generator and scoring are local and negligible. Comfortably "minutes, not hours."

Evidence note (thin): the §5 dollar rates are illustrative placeholders, not confirmed provider pricing. Confirm actual rates for the chosen models before committing the full run (§7.7).

What must the owner decide before this spec can run?

§7 · Open decisions

Nothing below is frozen — all eight items require an explicit owner call before this spec can run.

  1. Model set (§4) — which cheap-tier model as primary; whether to include a frontier anchor (doubles cost, brackets the ceiling, pre-empts the single-model objection).
  2. N per stratum (§1) — proposed 20 (satisfies criterion 5, cheap). Bump to 40 if a ≥10-pt effect must resolve within a single stratum at high confidence?
  3. Atom pool membership (§1) — confirm the exact IFEval instruction_id set assigned to the easy vs. hard pools, and confirm exclusion of language-detection / nltk-punkt-dependent atoms (determinism risk). Owner may want a specific curated list.
  4. Decision thresholds / ambiguity band (§3) — accept gap ≥ 0.20 = prompt-closable and gap < 0.10 = no gap, or tighten/loosen given the HIGH synthesis uncertainty on the floor (0.10–0.50)?
  5. Ceiling-prompt wording (§2.2) — approve the drafted convention block, or trim/extend the hygiene hints? It inherits IFEval-strict false-negative risk; the hygiene block is the mitigation.
  6. Loose-metric side report — the synthesis notes strict-only costs a few points to formatting false-negatives. Optionally also log per-atom (loose-style) rates alongside strict all-pass, as reviewers will later demand — cheap to add now.
  7. Dollar rates (§5) — the cost estimates use illustrative placeholder prices; confirm actual provider pricing for the chosen models.
  8. Seed-plumbing implementation — confirm the glue seeds the module-global random (or holds a random.Random(seed)) before each build_description, and passes explicit kwargs for nonce vocab; the repo's config.seed is a decoy.
