# c17 · Invented-Gate Circuit Evaluation

**Status:** RANKED #3 of 20 (round-2 rescore)

**You know this as:** a synthetic boolean-circuit / invented-connective evaluator — a small DAG of gates whose truth tables are invented and randomized per run, closest in spirit to CIRCUIT-style logic-gate reasoning but built, not borrowed.

## Verdict

**Officially ranked #3 of 20 after the implementation-aware rescore — the podium finish out of the 27-tier.** This candidate carries the best desk profile in the whole pool: its second oracle, stratification, seeding discipline, and skeptic verdict all scored the maximum, which is what lifts it above two run-verified generators that beat it only on codebase reuse. The reason it did not go higher is entirely on the build side. ## What it proposes

### Task: evaluate a randomized-truth-table gate DAG and name the value at a queried node

The generator builds a small directed acyclic graph of four to six **gate types** whose **truth tables are invented and randomized per run**; a formula-syntax variant instead uses invented **connective glyphs** with nonstandard precedence, undefined-atom propagation, and ill-formedness detection. Each instance supplies the **primary inputs** and asks for the value at a **named node** as a single bit, or an MC verdict (true / false / undefined / ill-formed). Output is one token, scored by **exact match** against evaluating the DAG or formula directly. The point is that each gate or connective table is an independent, cleanly demo-learnable latent rule: no fixed instruction can encode a randomized truth table without showing demonstrations, which is exactly what makes the optimizer's demo axis load-bearing. Wrong-gate errors leave diagnostic patterns, seeded semantics are contamination-proof, and boolean arithmetic gives strong temperature-0 determinism.

### Example instance (illustrative)

Seeded conditions: gate `G1` is assigned a randomized 2-input truth table this run, `G2` a NAND-like table; primary inputs `A=1, B=0, C=1`; query node is `N3`.

```
Gate tables (this seed):
  G1(x,y):  00->1  01->0  10->0  11->1
  G2(x,y):  00->1  01->1  10->1  11->0

Circuit:
  N1 = G1(A, B)
  N2 = G2(B, C)
  N3 = G1(N1, N2)

Inputs: A=1, B=0, C=1
Query:  value at N3 = ?

Gold: N1 = G1(1,0) = 0 ; N2 = G2(0,1) = 1 ; N3 = G1(0,1) = 0  ->  0

Naive-prompt miss: a model that assumes G1 is XOR/AND (a standard gate) computes
the wrong bit — the table is invented this run and only recoverable from demos.
```

## Academic anchor & lineage

### Literature parents

- **CIRCUIT (Skelic et al., arXiv 2025)** — https://arxiv.org/abs/2502.07980 — the corpus's primary anchor: a 510-pair benchmark for LLM circuit interpretation and reasoning. **Semantic mismatch (verified):** CIRCUIT benchmarks reasoning over *analog electrical* circuits (transistor / op-amp QA), not digital boolean-gate DAG evaluation with invented truth tables. It is a naming- and domain-adjacent anchor, not a direct precedent for the c17 generator.
- **A Implies B: Circuit Analysis in LLMs for Propositional Logical Reasoning (Hong et al., arXiv 2024)** — https://arxiv.org/abs/2411.04105 — the corpus's "propositional circuit-analysis 2024" cite. **Semantic mismatch (verified):** this is a mechanistic-interpretability paper analyzing the internal computational "circuits" of transformers during propositional reasoning — *not* evaluation of boolean logic-gate circuits. The label is technically accurate but easily conflated with the c17 gate-DAG task; the word "circuit" means two different things.

Both cited papers resolve to real, confirmed arXiv entries with matching titles and authors — there is no fabricated-provenance issue here. The provenance gap is confined to semantic adjacency: c17's "merged from literature" list is thin (two sources, both adjacent rather than exact), and the research file itself flags that no off-the-shelf synthetic gate-eval benchmark was found. This is a synthetic-family candidate you would build, essentially unanchored in a published benchmark.

### Brainstorm parents

- Marble-Track / Logic-Gate Signal Propagator — seeded the "evaluate a wired network under an input assignment" framing.
- Micro-Logic Notation Truth Evaluation — seeded the formula-syntax variant with invented connective glyphs and nonstandard precedence.
- Merged entry: [candidates-merged c17](../candidates-merged.md) — origin "both" (brainstorm + literature), consolidating the DAG and formula variants into one candidate.

### Dead branches: dropped near-relatives

- **D4 · CRUXEval-style code-output prediction** — dropped: the published set is heavily contaminated, and regeneration needs a code DSL with unconstrained return types that fight exact-match bounding. Execution flavor is covered elsewhere.
- **D5 · Grounded grid compositional (gSCAN / ReaSCAN)** — dropped: grid rendering plus action-sequence outputs are the most expensive to bound, and the reviewer judged it "not worth the adaptation cost." Grid flavor is covered by c19.
- **Refuted cousin c03 · Keyed Custom-Operator Expression Evaluation** — not dropped, but its natural reuse (BIG-bench multistep_arithmetic) was refuted in round 2: the oracle is `eval()` of the generator's own string, which is tautological and invalid for invented operators, so the load-bearing oracle becomes new code. c17's oracle-independence (a separate re-parse-and-evaluate path) is precisely what c03's off-the-shelf codebase lacked.

## Perspectives

### Round-1 rubric fit
- **Supports:** arguably the purest embodiment of "demos are load-bearing" — randomized truth tables cannot be encoded by any fixed instruction, so the demo axis is structurally required. Seeded semantics are contamination-proof and boolean arithmetic gives strong temp-0 determinism.
- **Against:** the single-token boolean output gives slightly coarser strata than the 28-tier tasks, and a bare one-bit output carries a 50% guess floor unless multiple outputs or deeper circuits are used — which is exactly why it sat at 27, just below the finalist cut.

### Optimization-path skeptic
- **Supports:** each gate/connective table is an independent, cleanly demo-learnable latent rule, so difficulty decomposes and rule capture is incremental across strata.
- **Against:** c17 sits below the 28-tier cut, so it never received a dedicated round-1 optimization-path skeptic pass — the finalist-tier adversarial lenses examined c21/c03/c05/c15 and their peers, not this candidate. The "incremental ladder survives" claim is inferred from its rubric, not adversarially stress-tested.

### Measurement skeptic
- **Supports:** exact single-token match against an independent DAG evaluation is an ideal deterministic oracle; wrong-gate errors produce diagnostic patterns a reflection LM can learn from.
- **Against:** likewise never given a dedicated round-1 measurement skeptic pass (below the 28 tier). The known measurement soft spot — a coarse boolean output with a 50% floor on the binary variant — is flagged in the landscape but was not adversarially resolved.

### Implementation lens (round 2)
- **Supports:** the best desk profile in the pool. Second oracle, stratification, seeding, and skeptic verdict all scored the maximum; the BeyondBench generator is run-verified deterministic with fifty of fifty instances validated by an independent recursive-descent evaluator.
- **Against:** the invented-gate / random-truth-table core is a small rewrite, not config — the repo hardcodes standard ops via `eval` — and it ships a confirmed XOR-dropping bug plus module-global RNG state that must both be handled. That is what caps codebase reuse below the top.

### Research currency
- **Supports:** the generate-boolean-instances-with-a-difficulty-knob pattern is live (SATBench, EMNLP 2025), and gate/circuit reasoning is an active LLM-evaluation theme.
- **Against:** both named anchors are only domain-adjacent (analog-circuit QA; mechanistic interpretability), so the task is essentially unanchored in a published gate-evaluation benchmark — it is a build, not an adaptation of a current dataset.

## Implementation: steps & risks

1. **Vendor the generator core.** Copy `_make_expr` / `_eval_expr` (~25 lines) from BeyondBench's `logical_operations_task.py` into an external `c17_gates.py`; do not run BeyondBench's engine (instantiation is coupled to a model_handler plus disk I/O).
2. **Fix the verified XOR-dropping bug.** Reorder the chained string replaces (or use word-boundary regex) so `OR` inside `XOR` is not replaced first — the shipped code emits zero XOR instances out of two hundred and falls back to trivial expressions.
3. **Write the invented-gate core (the real work).** Replace the fixed AND/OR/XOR set with N generated gate symbols each backed by a random truth table, and evaluate via table lookup instead of Python `eval` (~40 new lines) — this is genuine new logic, not glue.
4. **Thread a per-instance RNG.** Use an explicit `random.Random(seed)` per instance instead of the module-global `random.seed`, removing the repo's global-RNG red flag on our side.
5. **Reuse prompt + scoring glue.** Keep the `\boxed{}` prompt shape and `parse_logical_answer` for exact 0/1 match; add strata whose query nodes cover different gates, and add multi-output or deeper circuits to escape the 50% single-bit floor.

**Risks:**
- The gate-evaluation core is a rewrite, not config: the repo evaluates standard ops via `eval`, so table-driven invented-gate evaluation is net-new logic that must be authored and tested.
- The shipped XOR bug is confirmed live (0/200 XOR instances) and the repo's own tests do not catch it — the fix is one-line but is a hard prerequisite, and module-global RNG discipline must be enforced per instance.
- Coarse single-token boolean output carries a 50% guess floor on the binary variant; strata must use multiple outputs / deeper circuits or an MC verdict to keep the effect size resolvable on 10-20 tasks.

**Repo review & run verification:**
- **Codebase:** [BeyondBench](https://github.com/ctrl-gaurav/BeyondBench) (Apache-2.0) — deep-read notes at [../repos/beyondbench.md](../repos/beyondbench.md). I1 = 2.
- **Run verification:** VERIFIED (2026-07-21). Ran the real `LogicalOperationsTask.generate_data`: same seed twice was byte-identical (matching md5), a different seed diverged substantially, and an independent recursive-descent evaluator matched all 50 of 50 emitted instances. The XOR-dropping bug was reproduced live (0/50 XOR instances) — the one substantive defect, confirmed at runtime and required-fix before reuse.
