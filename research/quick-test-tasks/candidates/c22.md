# c22 · Stacked Verifiable-Constraint Micro-Generation

**Status: RANKED #9 of 20 (round-2 rescore)**

You know this as: **the IFEval/IFBench verifiable-checker family, restacked** — a trivial micro-task wrapped in three-to-five composed, deterministically-checkable constraints scored strict all-pass.

Domain: instruction following / constrained generation.

---

## Verdict

**Officially ranked #9 of 20 after the implementation-aware rescore, up from the 26-tier in round 1.** This candidate rises because its oracle is not a promise but a shipped, unit-tested library: the IFBench checker suite passes its full test set verbatim, so the deterministic 0/1 scoring half of the task exists as reusable code on day one. What holds it below the podium is that the *generator* — the layer that seeds a base task, samples a constraint subset, composes the descriptions, and threads a reproducible RNG — is net-new work the repo never shipped, and the round-1 design earns its non-one-shot-guessable credit almost entirely from the conditional Selection constraints, a single mechanism carrying the whole anti-enumeration load. It is a strong mid-pack pick: the cleanest oracle reuse in the field, paired with a modest but real build burden and a narrow structural defence.

## What it proposes

### A trivial micro-task gated by a stack of independently-checkable constraints, scored all-or-nothing

The generator builds a deliberately trivial base task (produce a short answer or micro-description) and layers three-to-five **composed constraints** drawn from a deterministic **checker library** — exact word count, forbidden letter, required keyword, casing rule, end-with token — including conditional **Selection constraints** of the form "if the input has property P obey rule A, else rule B" that the prompt must **discover rather than copy**. The output is a few words. It scores 1 if and only if **every** checker passes (strict 0/1, no partial credit), while each per-checker verdict is logged as a diagnostic fact for reflection. Constraint atoms and base tasks are generated synthetically from a COLLIE-style grammar, giving an infinite, contamination-proof pool; the **constraint count and level** are the difficulty dial, and IFBench-style OOD atoms resist one-shot enumeration.

### Example instance (illustrative — constructed from the merged summary)

Seed note: one base micro-task, four stacked atoms, one of them conditional on a property of the input the prompt is not told to look for.

```
BASE TASK: Name a color.
CONSTRAINTS (all must hold):
  1. Response is exactly 2 words.
  2. Response contains the letter "z".
  3. Response ends with the token "END".
  4. SELECTION: if the base task's noun has an even letter-count,
     the answer must be lowercase; else Title Case.
     ("color" = 5 letters -> odd -> Title Case)

GOLD (one satisfying answer): "Azure END"
NAIVE-PROMPT MISS: a prompt that lists only the stated atoms
  emits "hazy" (1 word, no END, ignores the hidden casing branch)
  -> a checker verdict per atom shows exactly which rule was missed.
```

## Academic anchor & lineage

### Literature parents

- **IFEval (Zhou et al., arXiv 2311.07911, Nov 2023)** — https://arxiv.org/abs/2311.07911 — the canonical verifiable-instruction benchmark; c22 reuses its deterministic checker library (25 instruction types) as the constraint-atom source.
- **COLLIE (Yao et al., ICLR 2024; arXiv 2307.08689, 2023)** — https://arxiv.org/abs/2307.08689 — the grammar-based synthetic constrained-generation generator; the COLLIE-style checker library/grammar that defines c22's core generation approach.
- **IFBench / IF-RLVR (Pyatkin et al., arXiv 2507.02833, 2025)** — https://arxiv.org/abs/2507.02833 — 58 OOD verifiable constraints plus verifier functions; supplies c22's non-one-shot-guessable OOD atoms, and is GEPA's headline IF task.
- **ComplexBench (Wen et al., NeurIPS 2024 D&B; arXiv 2407.03978)** — https://arxiv.org/abs/2407.03978 — the And/Chain/Selection/Nested composition operators; c22's conditional Selection constraints ("discover, not copy") come from here.
- **FollowBench (Jiang et al., ACL 2024; arXiv 2310.20410)** — https://arxiv.org/abs/2310.20410 — multi-level incremental constraint stacking (1→5); c22 borrows the level-ladder mechanic for its incremental difficulty dial.
- **InFoBench / DRFR (Qin et al., Findings of ACL 2024; arXiv 2401.03601)** — https://arxiv.org/abs/2401.03601 — Decomposed Requirements Following Ratio; the per-atom partial-credit scoring model c22 logs as diagnostic verdicts.
- **VFF: Verifiable Format Control (Wang et al., Findings of NAACL 2025; arXiv 2502.04498)** — https://arxiv.org/abs/2502.04498 — 64 fully Python-checkable format meta-constraints; documents the low small-model floor cited by c22's adaptation notes.
- **LIFEBench (Zhang et al., NeurIPS 2025; arXiv 2505.16234)** — https://arxiv.org/abs/2505.16234 — length-instruction benchmark documenting a low small-model floor; c22's length atom. *(Corpus mis-cites the first author as "Wu"; verified first author is Wei Zhang — see corrections.)*

### Brainstorm parents

- **Constrained Micro-Description** (brainstorm) — the seed idea of gating a tiny generation on a stack of exact checks; merged origin "both."
- **c22 entry, candidates-merged.md** — the merge that folded FoFo's format-following content (dropped cluster D8) into this candidate alongside c11.

### Dead branches: dropped near-relatives

- **FoFo format-following (dropped cluster D8)** — LLM-as-judge scoring fails the deterministic-exact-check criterion; its format-idea content was absorbed into c11 and c22 rather than kept as its own task.
- **ComplexBench native scoring** — the source benchmark augments its rules with LLM evaluators and dependency-structure aggregation, which fails the pure-rule / no-partial-credit criteria; c22 keeps only the composition algebra (Selection), not the scorer.
- **Free-generation instruction-following at large (Theme H)** — attractive for research currency but scoring wants a deterministic checker library and strict all-pass 0/1; without the "discover the hidden constraint" framing it is one-shot enumerable, which is why the broad theme lands at 26 rather than in the finalist tier.

## Perspectives

### Round-1 rubric fit
- **Supports:** single-call, exact deterministic 0/1 scoring, and a synthetic infinite pool make it a clean fit on the core criteria; the constraint count is a genuine incremental difficulty dial.
- **Against:** it scores 26 — below the finalist tier — because its non-one-shot-guessable credit rests narrowly on the conditional Selection constraints; strip those and a single prompt enumerates the stated atoms.

### Optimization-path skeptic
- **Supports:** stratified constraint combos give a smooth climb as prompts internalize more atom types, and IFBench-style OOD atoms keep the ladder from being closed by one clever sentence.
- **Against:** no skeptic pass examined this candidate below the 28-tier — the round-1 adversarial lenses ran on the finalist set, so the optimization-path story here is asserted from the rubric, not stress-tested.

### Measurement skeptic
- **Supports:** every atom has an independent, non-tautological checker (ground truth reads only the target value and the response), so the ceiling is exactly writable and errors are per-atom diagnosable.
- **Against:** strict all-pass 0/1 over four-to-five atoms compresses signal harshly, and no measurement-lens pass was run below the 28-tier to confirm the pool score behaves monotonically under the chosen strata.

### Implementation lens (round 2)
- **Supports:** the oracle is shipped and unit-tested — the IFBench checker suite runs its full 59-test set green verbatim, so the scoring half is reusable almost unchanged; that anchors I1 and I2 at 2 and I6 at 2 (no repair flags).
- **Against:** the seeded stacking generator, RNG plumbing, and composition layer are net-new (~120–200 LOC outside the repo), and the shipped `config.seed` is a decoy wired only to the LLM payload — hence I3, I4, I5 all at 1 and the repo capped at a 2, not a 3.

### Research currency
- **Supports:** the IFEval → IFBench → IF-RLVR lineage is the live verifiable-instruction line, and IFBench is GEPA's headline IF task, so the family is squarely current through 2025-2026.
- **Against:** verifiable-constraint checking is a crowded, fast-moving area (IF-RewardBench, DeonticBench, the Instruction Hierarchy line), so a restacked-atoms task has a modest novelty margin over the benchmarks it borrows from.

## Implementation: steps & risks

1. **Vendor the IFBench oracle read-only.** Import `INSTRUCTION_DICT` from `instructions.py` / `instructions_util.py` / `instructions_registry.py` and use each class's `check_following` as the 0/1 atom oracle; optionally strip the unused `spacy` / `langdetect` imports and the import-time `.nltk_data` env mutation. Roughly zero lines changed.
2. **Write the seeded stacking generator (~60–100 LOC).** Given a seed and a chosen `instruction_id` subset (the latent-rule strata), instantiate checkers and call `build_description` with an explicit kwargs dict derived from a `random.Random(seed)` — passing values in rather than relying on the module-global unseeded `random` — then concatenate the per-constraint descriptions onto a base/nonce prompt.
3. **Add the Selection / discovery layer and serializer (~40 LOC).** Encode the conditional "if input has property P obey A else B" atoms so the branch is not stated in the prompt, feed nonce keyword lists as explicit kwargs for contamination-proofing, and emit records in the `{key, prompt, instruction_id_list, kwargs}` shape the evaluator already consumes.
4. **Wire strict all-pass scoring (~30 LOC).** Single LLM call, then `all(check_following(resp))` across the stack, reusing `evaluation_lib.test_instruction_following_strict` nearly verbatim; log each per-atom verdict as a diagnostic fact.

### Risks
- **Unseeded module-global `random` in every `build_description` (19 sites plus `generate_keywords`)** means generation is nondeterministic and non-re-seedable as shipped; if the explicit-kwargs plumbing is incomplete, instances are not reproducible.
- **The anti-one-shot defence is narrow.** With the Selection/discovery atoms removed or weakly seeded, a single prompt enumerates the stated constraints and the incremental gap collapses (the criterion-4 failure mode the whole design guards against).
- **Strict all-pass over many atoms can floor a small model near zero**, flattening the optimizer signal unless the stratified combos are tuned so easy strata are reliably reachable.

### Repo review & run verification
- **Codebase:** allenai/IFBench (https://github.com/allenai/IFBench), Apache-2.0 code / ODC-BY-1.0 data — deep-read notes at [../repos/allenai-ifbench.md](../repos/allenai-ifbench.md). Secondary: google-research IFEval — notes at [../repos/google-research-instruction-following-eval-ifeval.md](../repos/google-research-instruction-following-eval-ifeval.md).
- **Run verification: PARTIAL (oracle only).** The reviewer ran the IFBench checker unit tests in a fresh venv and reports `Ran 59 tests ... OK`, confirming the oracle half is reusable and correct. The generator/sampler does not exist in the repo (frozen 299-row dataset), so the end-to-end seeded-generation path was never run-verified — it is the net-new code that keeps this a provisional I1 = 2.
