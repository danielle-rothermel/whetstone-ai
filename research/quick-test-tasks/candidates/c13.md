## Verdict

**Officially ranked #17 of 20 after the implementation-aware rescore.** This is the archetypal "independent-rules-with-override" task, and on the round-1 rubric it scored well — the first-match ordering gives genuine rule interaction that resists one-shot specification, each rule owns a case slice so partial rule knowledge earns partial pool score, and a feature-combinatorial seeded pool is contamination-proof by construction. What sank it in round 2 is entirely on the implementation axis: discovery found no plausibly reusable codebase, so every hour of the build is from-scratch. Its desk profile is otherwise fine — a second oracle is trivial (exhaustively close the feature grid), calibration is light, and round 1 left no repair flags — but the double-weighted reuse dimension scoring zero drops it into the bottom tier alongside the other no-repo tasks. It is a good task with no code to stand on.

## What it proposes

### Task: label one record by executing a latent first-match rule list with overrides

The generator samples a short **ordered decision list** (first-match semantics) over four to six categorical or numeric **features**, plus a set of **override rules** — `security=true` forces P0, a senior veto, a regional-cutoff shift, a paying-customer never-SUPPRESS floor — arranged in an explicit **precedence chain**. Each instance is one record (or vote set, or event); the model emits one **label** from a roughly five-way set; scoring is exact match against executing the list top-to-bottom and returning the first rule that fires. Because each rule governs a distinct case slice, partial rule coverage yields partial pool score and failures are diagnostic: "all `security=true` cases wrong" localizes precisely to the override that was missed. Boolean-concept complexity — rule count, feature arity, override-chain depth — is the principled difficulty dial.

### Example instance (illustrative — constructed from the merged summary)

Seeded conditions: a five-way label set relabeled with opaque symbols to defeat priors, a first-match list where a security override precedes a region downgrade which precedes a paying-customer floor.

```
Rules (latent, in precedence order):
  R1: if security = true            -> P0
  R2: if severity >= 4              -> P1
  R3: if region = APAC              -> downgrade one level
  R4: if plan = paying              -> never SUPPRESS
  R5: otherwise                     -> P3

Record:  severity=4, security=false, region=APAC, plan=free
Gold:    P2      (R2 fires -> P1, then R3 downgrades -> P2)

Naive-prompt miss: a model told only "classify the incident severity"
returns P1 (applies R2, ignores the latent APAC downgrade in R3).
```

## Perspectives

### Round-1 rubric fit
- **Supports:** Excellent on the decomposition criteria — synthetic and contamination-proof, a known 100% ceiling (state every rule), cleanly decomposable independent rules, first-match precedence for genuine interaction, and diagnostic "which rule fired" failures. Scored 27 with a full tie-breaker of 2.
- **Against:** Its rule families (security override, region downgrade, paying floor) are less contamination-novel than the nonce-graph or opaque-codebook designs, which is exactly why it was edged out of the 28-tier finalist cut.

### Optimization-path skeptic
- **Supports:** The first-match ordering makes rule interactions non-trivial and not one-shot guessable, and relabeling the classes with opaque symbols defeats any residual triage prior — so the gap is a real learnable one across prompts.
- **Against:** Over-simple feature encodings can be one-shot solved by a competent prompt engineer; difficulty must be actively controlled through rule count, feature arity, and override-chain depth or criterion 4 collapses.

### Measurement skeptic
- **Supports:** Boolean-concept complexity is a principled difficulty dial, and each rule owning a distinct case slice means the pool accuracy climbs rule-by-rule with an inspectable ladder.
- **Against:** No dedicated skeptic pass examined this candidate below the 28-tier finalists — the measurement critique here is inferred from the domain deep-dive and the merged summary, not from a candidate-specific adversarial review.

### Implementation lens (round 2)
- **Supports:** The desk profile is genuinely good — a trivial second oracle via exhaustive feature-grid closure (I2=2), light calibration burden of two to three gates (I4=1), and no repair debt from round 1 (I6=2).
- **Against:** No reusable repo exists, so I1 is a firm zero and, being double-weighted, it dominates the total; first-match ordering partially couples the rules, so a change can force more than a single-stratum regeneration (I5=1), and a shadowed-rule audit is needed since a later rule may never fire (I3=1).

### Research currency
- **Supports:** Sits on live 2025-2026 research — RuleArena (ACL 2025) and the 2026 DeonticBench both frame rule-guided reasoning as an open problem, and LLMTabBench (2026) proposes a generator in exactly this shape.
- **Against:** The core rule-list machinery is classical (CORELS is 2017), and the 2026 anchors are very recent with artifacts not independently corroborated, so the currency rests partly on unverified sources.

## Implementation: steps & risks

1. **Build the rule-list sampler.** Programmatically sample a short ordered decision list over four to six categorical/numeric features with an explicit precedence chain of override rules — from scratch, since no reusable codebase was found.
2. **Write the first-match executor and the closure oracle.** Execute the list top-to-bottom returning the first rule that fires; independently verify each label by exhaustively closing the finite feature grid (the trivial second oracle giving I2=2).
3. **Add a shadowed-rule audit.** Under first-match semantics a later rule may be fully shadowed by an earlier one and never fire; check coverage so every rule owns a live case slice and the diagnostic ladder stays honest.
4. **Relabel classes with opaque symbols and balance strata.** Defeat the pretrained triage prior with opaque label symbols, and balance the strata so 10-20-task internal evals resolve prompt differences.
5. **Pin the reference ceiling prompt and floor gates.** Author a state-all-rules ceiling prompt that provably reaches ~100% at temp 0; measure per-stratum floors; confirm no single fixed convention statement covers the pool (two to three pre-freeze gates).

### Risks
- **No codebase means every hour is from-scratch** — the double-weighted I1=0 is the single load-bearing reason this candidate ranks #17 despite a strong task design; there is no repo to inherit determinism, serialization, or an oracle from.
- **Shadowed rules silently break the ladder** — if a later rule never fires under first-match, its stratum is dead, inflating apparent accuracy and mis-localizing failures; the audit in step 3 is mandatory.
- **First-match ordering couples the rules** — because precedence links rules, changing one rule can perturb which cases other rules govern, so a change may force more than a one-stratum regeneration (I5=1).
- **Over-simple encodings are one-shot solvable** — without enough rule count, feature arity, and override depth, a competent prompt engineer closes the gap in one prompt.
