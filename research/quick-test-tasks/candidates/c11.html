c11House-Convention Canonicalizer

RANKED #8
You know this as: canonical-JSON / house-style normalization — a deterministic serializer whose invented house rules deviate from every public standard (RFC 8785 JCS, slugify, ISO dates), scored by whole-string exact match.

Officially ranked #8 of 20 after the implementation-aware rescore, a slip from its round-1 28-tier finalist standing. The candidate's distinctive strength is project alignment: canonical-JSON house rules directly rehearse the dr-serialize intuitions the main experiment cares about, a transfer bonus no other finalist carries. Its generator story is genuinely reusable — json-schema-faker is a clean, seed-threaded instance producer that ran green in a fresh environment — but the reusable half is only the input generator; the canonicalizer and its oracle are new code we author, which is why the codebase cell settles at reusable-but-incomplete rather than drop-in glue. The rescore also inherits round 1's live obligations: the merged rule set must go on a diet (the check-digit and long-list-sort strata are things a small model cannot execute in one constrained forward pass, making the true ceiling both lower and unknowable), the rule tables must stay wider than any demo set or GEPA minibatch to block full-coverage copy-from-context, and the second objective is degenerate — correct canonical strings have fixed length, so output-bytes validates single-objective optimization plus two-objective plumbing only. Those are bounded fixes, but they are why it sits below the run-verified generators rather than beside them.

What it proposes

Normalize a messy value to one canonical string under invented house rules

The generator picks ONE deterministic normalization pipeline — canonical JSON or slug are the preferred families — governed by five to six invented house rules that deliberately deviate from every public standard: length-then-codepoint key sort, a bespoke float and escape policy, invented month abbreviations and grouping tables, a custom slug separator and stopword list. Each instance is a messy input value; the model must emit the canonical string; scoring is 0/1 whole-string exact match against a roughly forty-line reference function that is the ceiling definition. Every rule owns a stratum of instances, so pool accuracy climbs rule by rule and the expected/got diff names precisely which rule was violated. Random inputs give an infinite, contamination-proof pool, and because the conventions are invented they are not one-shot guessable.

Example instance

Illustrative, from the round-1 corpus (slug family; house rules: lowercase; token separator is ~ not -; drop words in an invented stopword list {the, of, an}; ordinals map through an invented table so 2nd → twyx; keep at most 4 tokens, truncating from the end).

Input title:  The Quick   Brown Fox (2nd Edition)!
Gold:         quick~brown~fox~twyx

Naive slugify emits `the-quick-brown-fox-2nd-edition` → 0.

Academic anchor & lineage

Literature parents

  • RFC 8785 JSON Canonicalization Scheme / JCS (Rundgren, Jordan & Erdtman, IETF Informational 2020) — the named task donor: canonical JSON is the generatable task, and c11 invents house rules that deliberately deviate from JCS rather than reproduce it.
  • PolyNorm: Few-Shot LLM-Based Text Normalization for TTS (Wong et al., arXiv 2025) — evidence that date/phone/number normalization is a live LLM few-shot task; the slugify/checksum "family" itself is practitioner/standards material (python-slugify, Luhn/ISBN-13/IBAN mod-97), not a single academic paper.
  • StructEval (Yang et al., arXiv 2025) — its non-renderable slice (JSON/YAML/CSV/TOML) supplies the format taxonomy and below-ceiling floor evidence.
  • JSONSchemaBench (Geng et al., arXiv 2025) — the schema-pool donor: a graded schema corpus for calibrating canonicalization difficulty, contamination-proofed by regeneration.
  • IFEval (Zhou et al., arXiv 2023) — verifier-library design donor: ready-made deterministic exact checkers and the verifiable-instruction paradigm c11's scoring reuses.
  • EDC: Extract, Define, Canonicalize (Zhang & Soh, EMNLP 2024) — a canonicalize-step reference for the domain; primarily c12's parent, but it anchors the canonicalization framing c11 shares.

Brainstorm parents

  • Canonical JSON Serializer & the normalizer family — the seed cluster: slug/URL canonicalizer, identifier case-and-affix normalizer, phone/number canonicalizer, date/time fictional locale, whitespace/quote linter, CSV reserializer, and the Signed-Checksum Field Validator.
  • Merged entry (origin: both) — where the ten brainstorm normalizer variants plus the four literature donors were consolidated into one "pick ONE pipeline, 5–8 invented rules" task.

Dead branches: dropped near-relatives

  • D8 · FoFo format-following (FoFo (Xia et al., 2024)) — its LLM-as-judge scoring fails the deterministic exact-check criterion; the format-following content was folded into c11 and c22 rather than kept as its own cluster.
  • Signed-Checksum Field Validator (brainstorm sibling, absorbed then diet-flagged) — the arithmetic check digit it contributes is exactly the computation-heavy rule the measurement lens says a small model cannot execute in one forward pass; the plan drops it or reduces it to an enumerable ≤3-digit lookup.
  • JSONSchemaBench as a standalone cluster — not counted as a cluster; folded into c11's source list as the schema-pool donor rather than a task in its own right.

Related active research (2025–2026)

  • IFBench / Generalizing Verifiable Instruction Following (Pyatkin et al., arXiv 2025) — 58 new verifiable out-of-domain constraints with programmatic exact checks; directly extends the constraint-following surge c11's tie-breaker names.
  • StructText (Kashyap et al., arXiv 2025) — a synthetic table-to-text generator reversible into a canonicalization instance source; cited for c12 but applies to c11's synthetic-pool design.
  • c11 related work — deep-read dossiers + capability/positioning synthesis, 2026-07-22.
  • c11 baseline spec — draft baseline experiment spec — headroom before complexity.

Implementation: steps & risks

Put the rule set on a diet

Restrict rules to single-pass-executable transformations — table lookups (invented month/stopword/ordinal tables), character-class rewrites, fixed replacements, whitespace collapsing, truncation, sorts of a few short keys. Drop the arithmetic check digit and long-list sorts (or reduce the check digit to a fully enumerable short lookup). Cap the rule count.

Enforce table-width > coverage

Table-valued rules get more entries than any demo set or GEPA minibatch, so rules must accrete over iterations. Strike "cover all cases" from demo curation.

Make the generator adversarial per instance

Every instance actively distinguishes a house rule from its public standard (reject inputs where they coincide); global formatting rules form an explicit first-rung stratum. Reuse json-schema-faker as the seeded input producer, pinning date bounds to sidestep its one wall-clock nondeterminism source.

Build the canonicalizer + oracle as new external code

Write the deterministic house-rule pipeline that is both the reference the model reproduces and the ground truth; optionally fork rfc8785.py and edit its escape table, key-sort comparator, and float format for the canonical-JSON family.

Pre-freeze certification

Reference prompt near-perfect with agreeing temp-0 repeats on the pinned split; derive floor/gap targets from measured numbers; freeze exactly one trivial normalization in the Eval Config; run the "infer every rule from the examples" one-shot with the largest permitted demo set and treat a high score as a rule-mix bug; document that the output-bytes objective is degenerate.

Risks

  • Ceiling unknowability if the diet is skipped: leaving computation-heavy rules in place makes the true ceiling a band no one can predict in advance, with residual failures indistinguishable from temp-0 provider nondeterminism — the measurement lens's refutation.
  • Full-coverage demo leakage: if demo sets or minibatches cover every table entry, lookup rules degrade into copy-from-context and the gap the task measures collapses; table-width policing is the only guard.
  • Degenerate second objective: output-bytes is fixed for correct canonical strings, so this candidate validates single-objective optimization plus two-objective plumbing only — a real limitation versus c21/c14 that must be declared per the rubric callout.
