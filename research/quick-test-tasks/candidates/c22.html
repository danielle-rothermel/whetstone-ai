c22Stacked Verifiable-Constraint Micro-Generation

RANKED #9

A trivial micro-task wrapped in composed, deterministically-checkable constraints, scored strict all-pass.

whetstone-ai · 2026-07-22

Officially ranked #9 of 20 after the implementation-aware rescore, up from the 26-tier in round 1. This candidate rises because its oracle is not a promise but a shipped, unit-tested library: the IFBench checker suite passes its full test set verbatim, so the deterministic scoring half of the task exists as reusable code on day one. What holds it below the podium is that the generator — the layer that seeds a base task, samples a constraint subset, composes the descriptions, and threads a reproducible RNG — is net-new work the repo never shipped, and the round-1 design earns its non-one-shot-guessable credit almost entirely from the conditional Selection constraints, a single mechanism carrying the whole anti-enumeration load. It is a strong mid-pack pick: the cleanest oracle reuse in the field, paired with a modest but real build burden and a narrow structural defence.

What does it propose?

A trivial micro-task gated by a stack of independently-checkable constraints, scored all-or-nothing

The generator builds a deliberately trivial base task (produce a short answer or micro-description) and layers three-to-five composed constraints drawn from a deterministic checker library — exact word count, forbidden letter, required keyword, casing rule, end-with token — including conditional Selection constraints of the form "if the input has property P obey rule A, else rule B" that the prompt must discover rather than copy. The output is a few words. It scores 1 iff every checker passes (strict 0/1, no partial credit), while each per-checker verdict is logged as a diagnostic fact. Atoms and base tasks are generated synthetically from a COLLIE-style grammar, giving an infinite, contamination-proof pool; the constraint count and level are the difficulty dial, and IFBench-style OOD atoms resist one-shot enumeration.

Example instance

Illustrative — constructed from the merged summary: one base micro-task, four stacked atoms, one conditional on a property of the input the prompt is not told to look for.

BASE TASK: Name a color.
CONSTRAINTS (all must hold):
  1. Response is exactly 2 words.
  2. Response contains the letter "z".
  3. Response ends with the token "END".
  4. SELECTION: if the base task's noun has an even
     letter-count, the answer must be lowercase; else
     Title Case.  ("color" = 5 letters -> odd -> Title)

GOLD (one satisfying answer): "Azure END"
NAIVE-PROMPT MISS: a prompt listing only the stated atoms
  emits "hazy" (1 word, no END, ignores hidden casing branch)
  -> per-atom verdicts show exactly which rule was missed.

Where does it come from?

Literature parents

  • IFEval (Zhou et al., arXiv 2311.07911, Nov 2023) — the canonical verifiable-instruction benchmark; c22 reuses its deterministic checker library (25 instruction types) as the constraint-atom source.
  • COLLIE (Yao et al., ICLR 2024; arXiv 2307.08689, 2023) — the grammar-based synthetic constrained-generation generator that defines c22's core generation approach.
  • IFBench / IF-RLVR (Pyatkin et al., arXiv 2507.02833, 2025) — 58 OOD verifiable constraints plus verifier functions; supplies c22's non-one-shot-guessable OOD atoms, and is GEPA's headline IF task.
  • ComplexBench (Wen et al., NeurIPS 2024 D&B; arXiv 2407.03978) — the And/Chain/Selection/Nested composition operators; c22's conditional Selection constraints ("discover, not copy") come from here.
  • FollowBench (Jiang et al., ACL 2024; arXiv 2310.20410) — multi-level incremental constraint stacking (1→5); c22 borrows the level-ladder mechanic for its difficulty dial.
  • InFoBench / DRFR (Qin et al., Findings of ACL 2024; arXiv 2401.03601) — Decomposed Requirements Following Ratio; the per-atom partial-credit model c22 logs as diagnostic verdicts.
  • VFF: Verifiable Format Control (Wang et al., Findings of NAACL 2025; arXiv 2502.04498) — 64 fully Python-checkable format meta-constraints; documents the low small-model floor cited by c22's adaptation notes.
  • LIFEBench (Zhang et al., NeurIPS 2025; arXiv 2505.16234) — length-instruction benchmark documenting a low small-model floor; c22's length atom. Corpus mis-cites the first author as "Wu"; verified as Wei Zhang (see corrections).

Brainstorm parents

  • Constrained Micro-Description — the seed idea of gating a tiny generation on a stack of exact checks; merged origin "both".
  • c22 entry, candidates merged — the merge that folded FoFo's format-following content (dropped cluster D8) into this candidate alongside c11.

Dead branches: dropped near-relatives

  • FoFo format-following (dropped cluster D8) — LLM-as-judge scoring fails the deterministic-exact-check criterion; its format-idea content was absorbed into c11 and c22 rather than kept as its own task.
  • ComplexBench native scoring — the source benchmark augments its rules with LLM evaluators and dependency-structure aggregation, failing the pure-rule / no-partial-credit criteria; c22 keeps only the Selection composition algebra, not the scorer.
  • Free-generation instruction-following at large (Theme H) — attractive for research currency but scoring wants a deterministic checker library and strict all-pass 0/1; without the "discover the hidden constraint" framing it is one-shot enumerable, so the broad theme lands at 26.

What's the current research context?

  • IF-RewardBench (thu-coai, arXiv 2603.04738, 2026, ACL 2026) — 842 IF instructions with single-turn / multi-turn / system-prompt-steerability constraint coverage; extends the verifiable-constraint atom pool. The corpus cites it once under RewardBench 2.
  • DeonticBench (Dou et al., arXiv 2604.04443, 2026) — 6,232 rule-reasoning tasks with an optional Prolog verifier and explicit rule traces; a 2026 successor to the RuleArena rule-application family adjacent to c22.
  • The Instruction Hierarchy (Wallace et al., arXiv 2404.13208, 2024) — the foundational conflicting-instruction / privileged-instruction framework underlying IHEval and c22's authority-priority variant; the corpus cites it only via an OpenReview id, not the arXiv page.
  • c22 related work — deep-read dossiers + capability/positioning synthesis, 2026-07-22.
  • c22 baseline spec — draft baseline experiment spec — headroom before complexity.

How do different lenses see it?

Round-1 rubric fit

Supports
  • Single-call, exact deterministic 0/1 scoring, and a synthetic infinite pool make it a clean fit on the core criteria; the constraint count is a genuine incremental difficulty dial
Against
  • It scores 26 — below the finalist tier — because its non-one-shot-guessable credit rests narrowly on the Selection constraints; strip those and a single prompt enumerates the stated atoms

Optimization-path skeptic

Supports
  • Stratified constraint combos give a smooth climb as prompts internalize more atom types, and IFBench-style OOD atoms keep the ladder from being closed by one clever sentence
Against
  • No skeptic pass examined this candidate below the 28-tier — the round-1 adversarial lenses ran only on the finalist set, so the optimization-path story here is asserted from the rubric, not stress-tested

Measurement skeptic

Supports
  • Every atom has an independent, non-tautological checker (ground truth reads only the target value and the response), so the ceiling is exactly writable and errors are per-atom diagnosable
Against
  • Strict all-pass 0/1 over four-to-five atoms compresses signal harshly, and no measurement-lens pass ran below the 28-tier to confirm the pool score behaves monotonically under the chosen strata

Implementation lens (round 2)

Supports
  • The oracle is shipped and unit-tested — the IFBench checker suite runs its full 59-test set green verbatim, so the scoring half is reusable almost unchanged; anchors I1/I2 at 2 and I6 at 2 (no repair flags)
Against
  • The seeded stacking generator, RNG plumbing, and composition layer are net-new (~120–200 LOC outside the repo), and the shipped config.seed is a decoy wired only to the LLM payload — hence I3/I4/I5 at 1 and the repo capped at 2

Research currency

Supports
  • The IFEval → IFBench → IF-RLVR lineage is the live verifiable-instruction line, and IFBench is GEPA's headline IF task, so the family is squarely current through 2025-2026
Against
  • Verifiable-constraint checking is a crowded, fast-moving area (IF-RewardBench, DeonticBench, the Instruction Hierarchy line), so a restacked-atoms task has a modest novelty margin over the benchmarks it borrows from

How would we build it, and what could go wrong?

Vendor the IFBench oracle read-only

Import INSTRUCTION_DICT from instructions.py / instructions_util.py / instructions_registry.py and use each class's check_following as the 0/1 atom oracle; optionally strip the unused spacy/langdetect imports and the import-time .nltk_data env mutation. Roughly zero lines changed.

Write the seeded stacking generator (~60–100 LOC)

Given a seed and a chosen instruction_id subset (latent-rule strata), instantiate checkers and call build_description with an explicit kwargs dict derived from a random.Random(seed) — passing values in rather than relying on the module-global unseeded random — then concatenate per-constraint descriptions onto a base/nonce prompt.

Add the Selection / discovery layer and serializer (~40 LOC)

Encode the conditional "if input has property P obey A else B" atoms so the branch is not stated in the prompt, feed nonce keyword lists as explicit kwargs for contamination-proofing, and emit records in the {key, prompt, instruction_id_list, kwargs} shape the evaluator already consumes.

Wire strict all-pass scoring (~30 LOC)

Single LLM call, then all(check_following(resp)) across the stack, reusing evaluation_lib.test_instruction_following_strict nearly verbatim; log each per-atom verdict as a diagnostic fact.

Risks

  • Unseeded module-global random in every build_description (19 sites plus generate_keywords) means generation is nondeterministic and non-re-seedable as shipped; if the explicit-kwargs plumbing is incomplete, instances are not reproducible.
  • The anti-one-shot defence is narrow — with the Selection/discovery atoms removed or weakly seeded, a single prompt enumerates the stated constraints and the incremental gap collapses (the criterion-4 failure mode the whole design guards against).
  • Strict all-pass over many atoms can floor a small model near zero, flattening the optimizer signal unless the stratified combos are tuned so easy strata are reliably reachable.

  • Run verification: partial — the reviewer ran the IFBench checker unit tests in a fresh venv and reports Ran 59 tests ... OK, confirming the oracle half is reusable and correct.
