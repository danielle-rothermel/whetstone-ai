# c03 · Keyed Custom-Operator Expression Evaluation

**Status: RANKED #10** (round-2 rescore) · Round-1 backup

**You know this as:** a BIG-bench multistep-arithmetic generator with invented infix operators and a hidden precedence/wrap convention — apply a per-instance operator table, not PEMDAS, and output one integer.

## Verdict

**Round-1 backup, demoted to the middle of the round-2 rescore.** This was the round-1 co-recommendation with the adopted winner — the only other candidate both adversarial skeptics left unrefuted — and its design is genuinely clean: a per-instance symbol table, a tiny reference evaluator, and per-instance random parameters that make it the most straightforwardly contamination-proof candidate in the pool. What sank it in the implementation-aware rescore is the codebase. The BIG-bench multistep-arithmetic generator that would seed it computes its ground truth by running `eval()` on the generator's own string, which is a fine oracle for standard `+ - *` but is tautological for invented operators Python does not know. The load-bearing oracle is therefore new code we must write, not config, and the round-1-mandated foil-bank plus per-rung effect-size machinery keeps the moving-part count and calibration burden up — so its total lands mid-pack rather than near the top. Its weakest round-1 criterion (a legible-flag one-shot exposure) survived attack only on the strength of fixes that are cheap but load-bearing, and the wrap rule as first framed was measured to be nearly invisible.

## What it proposes

### Apply an invented operator table under a hidden house precedence, output one integer

Each instance opens with a **per-instance symbol table** defining invented infix operators as closed forms (for example `(a·k + b) mod m`), plus **decoy symbols** that are never used, an **opaque mode token**, and a short chained expression. The model must apply the table and the house conventions and emit **one integer** (or a four-way multiple-choice letter over candidate results), scored strict 0/1 exact match against a roughly ten-line reference evaluator. The headroom lives in **family-constant latent rules** the instance does not state or actively mis-states: modular wrap into residues 1..m rather than 0..m−1, the precedence and associativity carried by the opaque mode token, a threshold post-processing step, and a recurrence stratum with an index term. Naive prompts assume PEMDAS and standard semantics; the ceiling prompt states the general read-table-then-evaluate procedure and should reach roughly full accuracy.

### Example instance

Family-latent conventions the instance never states: `⊞` binds tighter than `◇`, and every result wraps into residues 1..m. Drawn from the round-1 recommendation writeup.

```
Definitions:
  x ◇ y = (3x + 2y) mod 6
  x ⊞ y = |x − y| + 1
  x ⨂ y = x            (decoy, never used)
  mode: K3
Evaluate:  2 ◇ 4 ⊞ 3

Gold: 4
  house precedence: 4 ⊞ 3 = 2, then 2 ◇ 2 = 10 mod 6 = 4.
Naive PEMDAS / left-to-right:
  2 ◇ 4 = 14 mod 6 = 2, then 2 ⊞ 3 = 2  → answers 2  → scored 0.
```

## Perspectives

### Round-1 rubric fit

- **Supports:** Textbook single-call, exact-integer-match graph, and per-instance random parameters make it the most straightforwardly contamination-proof candidate; the reference evaluator gives an exact checkable ceiling. Errors are diagnostic: an off-by-one at the residue boundary flags the wrap rule, order errors flag associativity, magnitude errors flag precedence.
- **Against:** Criterion 4 scored a bare pass — a generic "ignore PEMDAS, read the table literally, obey the flags, output only the integer" prompt captures everything the instance states legibly, so the headroom lives entirely in unstated conventions.

### Optimization-path skeptic

- **Supports:** Not refuted at major severity; the incremental ladder survives contingent on fixes that are concrete and cheap (opaque convention tokens, foil-bank rejection sampling).
- **Against:** A score-only instruction rephraser (COPRO) cannot produce per-run hidden conventions, so it validates plumbing and generic-instruction gains only, not deep hyperparameter comparison; the design deliberately keeps the anti-PEMDAS rung reachable so a broken optimizer stays distinguishable from a stumped one.

### Measurement skeptic

- **Supports:** Also not refuted; the ceiling is exactly computable from the reference evaluator, and per-rung effect sizes are verifiable at calibration with ablated reference prompts.
- **Against:** Simulation in the verdict found the wrap rule nearly invisible: naive 0..m−1 and house 1..m answers coincide on the large majority of unforced instances, so learning it would be worth only a couple of points on a fifteen-task eval — far below the resolvability bar unless boundary cases are forced.

### Implementation lens (round 2)

- **Supports:** The BIG-bench generation skeleton is genuinely reusable — the seeded, re-seedable `RandomState` recursion lifts cleanly (about forty lines), and oracle independence becomes a second oracle we control.
- **Against:** The `eval()`-of-the-generator-string oracle is tautological for invented operators, so the load-bearing evaluator is new code, not config; the foil-bank, forced boundary cases, and per-latent effect-size gates raise both the moving-part count and the calibration burden, keeping the total at ten.

### Research currency

- **Supports:** Sits in the live in-context rule-learning and novel-operator arithmetic line — In-Context Algebra, algorithmic generalization, and the OPRO/EvoPrompt-on-BBH lineage — a small but publishable community.
- **Against:** A smaller community than instruction-following, and the keyed-operator design is a synthetic family with no single canonical benchmark, so it opens no established wave of its own.

## Implementation: steps & risks

1. **Lift the generator, write a real oracle.** Copy the roughly forty lines of seeded recursion (`generate_content`/`generate_subparenthesis`/`generate_string` plus the `RandomState(seed)` plumbing) from the BIG-bench task, swap in per-instance nonce operator symbols and a sampled symbol table, then replace the `eval()` ground truth with a new roughly forty-to-seventy-line evaluator honoring the per-instance operator semantics and the wrap/precedence conventions.
2. **Enforce the observability partition.** Print per-instance-random conventions in the instance, but carry the headroom only in family-constant latents (1..m wrap, opaque mode-token semantics, threshold, recurrence); make every rule toggle an opaque token whose meaning is never stated, and keep the `mod m` surface notation deliberately misleading versus the 1..m ground truth.
3. **Foil-bank rejection sampling and forced boundary cases.** Evaluate each candidate instance under wrong-wrap, wrong-associativity, PEMDAS, no-threshold, and no-index foils; accept only instances where each active stratum's foil answer differs from truth; force at least one intermediate ≡ 0 mod m in every wrap-stratum instance and at least three operands so associativity errors always surface; draw MC distractors from the deduplicated foil outputs.
4. **Calibrate effect sizes and freeze.** Verify each rung is worth at least ten points on a ten-to-twenty-task eval using ablated reference prompts (one per latent rule); empirically confirm the ceiling is near full and the naive floor is low on the pinned model; scope few-shot demos by stratum so no single demo set reveals more than one hidden convention; add a strict integer/MC parser (non-numeric maps to 0).

### Risks

- One legibility slip collapses a rung: since all headroom lives in unstated conventions, any instance that inadvertently states or reveals a family-latent flag turns a hidden rule into a one-shot freebie.
- The wrap rule is nearly invisible without the forced ≡ 0 mod m boundary case, so skipping that generator constraint yields a stratum with a real but immeasurably small effect size.
- Fewer rungs than the richer candidates (roughly four or five family-constant latents) mean coarser hyperparameter resolution, and the new custom oracle carries an injection surface if any adaptation ever admits external strings into the `eval`-style path.
