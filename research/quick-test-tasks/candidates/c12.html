The implementation-aware rescore is what sank it. What does it actually ask the model to do?

What it proposes

Extract one house-normalized field from a record in an invented markup

The generator emits synthetic records written in invented markups — sigil-tagged fields, fixed-width columns with implied decimals, custom-escaped delimiters, or one of several dialects signaled by a header token — and each instance asks for exactly ONE field, returned in normalized house form (integer-cents to decimal, dotted enum code via most-specific-wins). Output is a single short token, scored by exact match against the generator's own record, with strictly one field per instance so no intra-example partial credit leaks in. The latent rules — sigil-to-field map, per-dialect conventions, column widths, escape handling, per-field normalization, enum precedence — are each stratified across the pool, so pool accuracy is meant to climb as a prompt captures more rules. Sigils, widths, delimiters, and enum vocabularies are randomized per run, making the pool contamination-proof and resistant to one-shot guessing.

Example instance

Seeded conditions (from the round-1 corpus): dialect VEL; ᚠ marks unit price with the implied decimal at 3 places this run; ᚷ marks a date; a second money-shaped field ᚢ is a same-type distractor.

Record: ⟦hdr:VEL⟧ ᚠ7203 ᚢ1150 ᚷ0925
Query:  unit price, in house form.

Gold:   7.203
        — the sigil map picks ᚠ over the same-type
          distractor ᚢ; this run's implied-decimal
          position is 3.

Naive extraction answers 7203 or 72.03  →  0.

Where does the design come from?

Academic anchor & lineage

Literature parents

  • EDC: Extract, Define, Canonicalize (Zhang & Soh, EMNLP 2024) — its post-hoc canonicalize step is the model for c12's per-field normalization into a fixed house form.
  • StructText (Kashyap et al., arXiv 2025) — a synthetic table-to-text benchmark generator; c12 reverses it, generating records from structured data so field extraction can be scored against ground truth.
  • StructEval (Yang et al., arXiv 2025) — its non-renderable (JSON/YAML/CSV) slice is the format-fidelity reference for the house-form output; the round-1 corpus mis-attributed this slice to StructText, corrected here.

Brainstorm parents

  • Glyph-Ledger / Fixed-Width / Multi-Schema Router extraction seeds — the merge (origin: both) folded in seven brainstorm variants: Glyph-Ledger Field Extraction, Positional Fixed-Width Decoder, Escaped-Delimiter CSV, Multi-Schema Router Extraction, Enum-Canonicalization With Hierarchy, Unit-and-Sign Normalizer, Protocol Bitmask.
  • candidates-merged · c12 — rewrote every multi-field pipe-output variant as a single-field query and balanced the dialect mix so per-dialect rule capture shows as pool-level steps.

Dead branches: dropped near-relatives

  • D2 · Interval-Overlap Field Selection — only 2–3 boundary/tie conventions, enumerable in one prompt; its boundary-convention flavor was absorbed into c12/c13 strata rather than kept as a cluster.
  • Checksum VALID/INVALID stratum (dropped in adaptation) — broke three criteria at once: a 50% guess floor, zero-bit binary failures a reflection LM cannot learn from, and an arithmetic ceiling a cheap model cannot reach; replaced by a non-arithmetic comparison/lookup rule.
  • Character-offset fixed-width strata (redesigned) — exact character counting fights BPE tokenization, so the fix rebuilt these around field order plus an invented padding glyph.
  • DTBench / ExtractBench / LLMStructBench (literature, motivation-only) — listed as motivation but provenance-unverified; no canonical venue could be located, and the LLMStructBench-style "prompting beats model size" finding is flagged not-to-be-cited publicly — treat as placeholders.

Is the surrounding research area still active?

Related active research (2025–2026)
  • StructEval: Benchmarking LLMs' Capabilities to Generate Structural Outputs (Yang et al., arXiv 2025) — a major 2025 structured-output benchmark showing even o1-mini reaches only about three-quarters accuracy, corroborating c12's headroom claim for format-constrained extraction.
  • JSONSchemaBench (Geng et al., arXiv 2025) — a rigorous 2025 structured-output/constrained-decoding benchmark over ten thousand schemas; directly relevant to c12's schema-routed single-field extraction and exact-match scoring (its primary home in this atlas is the sibling candidate c11).

Does the case hold up from every angle?

Perspectives

Round-1 rubric fit

Supports
  • Earned a top rubric score and stood among the seven verified finalists; per-run randomized sigils/widths/vocabularies are contamination-proof and resist one-shot prompts, and one field per instance keeps scoring cleanly binary.
Against
  • The top score masked a measurement story refuted as specified — the raw spec included checksum, character-offset, and fixed-decimal strata that failed the criteria and had to be surgically removed.

Optimization-path skeptic

Supports
  • Survived at major-but-not-refuted: the spec-dump reference prompt defines a reachable ceiling and the latent rules genuinely stratify, so a gap-recovery ladder exists in principle.
Against
  • A semantic type-matching prompt solves the sigil stratum for free unless every record carries ≥2 same-type distractors, and COPRO's score-only proposer cannot reach per-run randomized rules, so the optimizer axis must be scoped to plumbing validation.

Measurement skeptic

Supports
  • Once the offending strata are removed, the single-token exact-match design is deterministic and hand-verifiable, and the example shows a clean naive-miss versus gold separation.
Against
  • Refuted as specified at major severity — cents-to-decimal at a fixed position is world knowledge, not latent headroom, and single-token failures are the least individually diagnostic in the set: diagnosis reads at the batch level, not the example.

Implementation lens (round 2)

Supports
  • The target task shape is simple (one sampler, one short-token oracle) and the exact-match scorer is trivial once the record format is fixed.
Against
  • No reusable codebase exists — every discovered repo was frozen, LLM-driven, or the wrong shape — so the whole generator is a from-scratch build, and it carried the most stratum surgery of the finalists, landing its total at the bottom.

Research currency

Supports
  • Structured extraction and format-constrained output are live 2025 topics — StructEval and JSONSchemaBench both show real headroom even for strong models, so the phenomenon is current and publishable.
Against
  • The currency lives mostly in sibling candidates (JSONSchemaBench's home is c11), and three of c12's motivating benchmarks are provenance-unverified placeholders that cannot be cited, thinning the citable support specific to this candidate.

What would it take to build, and what could go wrong?

Implementation: steps & risks

Build the record generator from scratch

No reusable repo exists, so the seeded generator over sigil-tagged, fixed-width, escaped-delimiter, and header-dialect record families is entirely new code — the load-bearing and most expensive component.

Excise and replace the refuted strata

Drop or replace the checksum stratum with a non-arithmetic comparison/lookup rule; redesign fixed-width strata around field order plus an invented padding glyph (short ≤4-char widths, distinctive pad runs) instead of character counting.

Harden distractors and normalization

Mandate ≥2 same-type distractor fields per record so type-matching cannot short-circuit the sigil stratum, and randomize the implied-decimal position per run (2/3/4 places) so normalization is an inducible rule, not a fixed convention.

Add the pre-freeze calibration gates

Require the spec-dump reference prompt to clear a high per-stratum ceiling on the pinned model and seed before a stratum enters the pool, and record per-stratum reference/floor scores in the manifest so gap-recovery is falsifiable per stratum.

Fix the output surface and provenance

Stratified splits with even dialect/normalization mixes, a single-token output with a published exact-match normalizer, and pinned generator plus seed metadata.

Risks

  • From-scratch build with no anchor code — the double-weighted reuse axis is zero, so every hour is greenfield generator and oracle work with no maintained repo to validate against.
  • Batch-level-only diagnosis — single-token exact-match failures carry almost no per-instance signal, so reflection/gap-recovery must operate on aggregated per-stratum scores, weakening the diagnostic loop the task is meant to exercise.
  • Prompt shortcut on the sigil stratum — without ≥2 same-type distractors per record, a generic type-matching prompt collapses the intended headroom to near-ceiling.
