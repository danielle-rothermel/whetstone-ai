c20Invented Micro-Game Turn Resolution

RANKED #16

A per-run invented card/resource micro-game, resolved to one token — strong design, expensive to build.

whetstone-ai · 2026-07-22

Ranked in the lower-middle of the pool after the implementation-aware rescore. c20 carries the strongest design story in the family — a per-run invented game with several independently discoverable rules is contamination-proof by construction and gives a low naive-prompt floor because models default to real-card-game priors — but that same richness is what costs it on implementation. Round 1's own reflection flagged it as the highest design effort in the list, and the reason is legible: every invented rule set must be validated unambiguous against a reference near-perfect prompt before it can be frozen, a heavy calibration burden with many moving parts. The best reusable codebase, gg-bench, is a frozen corpus of LLM-generated Gym micro-games whose per-turn step() triple is a clean oracle, so the diff would be pure config-plus-scoring glue; but the corpus is LLM-generated-game shaped, not a rule-engine that can be pinned — its ground truth is "whatever the generated code implements," which may diverge from the natural-language rulebook, its seeding is inconsistent across envs, and it ships zero tests. The repo review's provisional reuse score was accepted as read rather than run-verified, so the codebase strength on this candidate is unconfirmed. The net is a task that is theoretically excellent but expensive and unpinned to build — which is why a strong round-1 design slides to the lower-middle once implementation is priced in.

What does it propose?

Resolve one turn of a per-run invented game, exact-match on a single token

A seeded generator invents a tiny deterministic game for each run: a nonstandard rank order (for example seven-high), trump or override cards, resource exchange rates, caps with overflow discard, and a counterintuitive action-resolution priority. An instance is a game state plus a trick or move sequence; the model outputs exactly one token — a winning card id, a final token count, or WIN/LOSE — scored by exact match against the reference resolver. Each invented rule flips a distinct subset of instances, so rule capture is incremental and a miss is diagnostic: "used real-world ace-high" is a signature a reflection LM can read.

Strong priors toward real card games guarantee a low default floor, invented rules per run are contamination-proof, and few-shot worked turns measurably teach the rank order and the resolution priority. The design-defining requirement — flagged in round 1 — is that the rules must be validated unambiguous with a reference near-perfect prompt before use, and that injected illegal moves exercise the rule-violation failure path via a reject token.

Example instance

Illustrative. Seeded conditions: a four-card trick-taking micro-game with rank order 7-high (7 > 6 > … > 2 > A), suit moss is trump this run, follow-suit required, no override cards active.

Rules (given by prompt): rank order is 7 high then 6,5,4,3,2, ace low; must follow the
led suit if able; trump suit = moss beats any non-moss regardless of rank; highest card
of the led suit wins if no trump is played.

Trick (in play order):
  P1 leads  bloom-6
  P2 plays  bloom-A   (followed suit)
  P3 plays  moss-2    (could not follow, played trump)
  P4 plays  bloom-7   (followed suit)

Query: which card id wins the trick?
Gold: moss-2

Naive-prompt miss: a model applying real-world ace-high and ignoring the invented
trump picks bloom-A (highest bloom by standard rank), or applying 7-high but missing
trump picks bloom-7 — each wrong-rule signature is legible to a reflection LM.

Where does it come from?

Literature parents

  • TextArena (Guertler et al., arXiv 2025) — the core source: 50-plus text games and the rule-encoding infrastructure that the invented-game family imitates.
  • ZeroSumEval (Khan et al., arXiv 2025) — an inter-model competition framework cited as synthetic / infinite game-generation evidence.
  • Game Reasoning Arena (Cipolina-Kun et al., arXiv 2025) — a benchmark assessing LLM reasoning through game play, cited for the rule-encoding patterns c20 reuses.
  • GVGAI-LLM (Li et al., arXiv 2025) — cited in the c20 research notes as infinite / synthetic game generation, supporting the contamination-proof invented-game framing.

Brainstorm parents

  • Invented Card Trick-Taking Scorer — invented suits/ranks, trump precedence, follow-suit eligibility, non-standard rank order, override cards, tie-break, combo bonus; the trick-taking half of the merge, chosen for the strong real-card-game prior that opens headroom.
  • Dice/Token Resource Micro-Game Resolver — invented exchange rates, resource caps/overflow, action priority/resolution order, payment-before-gain, win threshold; the resource half of the merge, contributing the token-count and WIN/LOSE output flavor.
  • c20 in candidates-merged — origin "both"; the merge fused the two brainstorm micro-games with the TextArena-style invented-game literature into one turn-resolution task.

Dead branches: dropped near-relatives

  • D5 · Grounded grid compositional (gSCAN / ReaSCAN) — dropped: grid rendering plus action-sequence outputs are the most expensive things in the pool to bound, and the reviewer judged them "not worth the adaptation cost." A sibling deterministic-simulation branch killed on output-bounding grounds; grid coverage went to c19, compositional splits to c23.
  • D4 · CRUXEval-style code-output prediction — dropped: the published set is heavily contaminated and regeneration needs a code DSL with unconstrained return types that fight exact-match bounding. The execution-simulation cousin of c20, killed because it could not stay single-token; execution flavor went to c05, rule-induction flavor to c23.
  • Scheduler / protocol-validator execution (research candidate 12, folded into c05) — a near-relative deterministic-resolution idea (transition table plus event trace → final state) that was routed into the state-tracking/execution line rather than kept as a separate invented-game candidate.

How do different lenses see it?

Round-1 rubric fit

Supports
  • Invented rules per run are contamination-proof by construction; strong real-card-game priors guarantee a low naive-prompt floor; each rule flips a distinct instance subset so rule capture is incremental and misses are diagnostic; injected illegal moves cleanly exercise the rule-violation failure path.
Against
  • It is the highest design effort in the pool — rules must be authored and validated unambiguous against a reference near-perfect prompt, and it is easy to accidentally make a rule ambiguous, which would break exact-match.

Optimization-path skeptic

Supports
  • Decisive knowledge (rank order, trump precedence, resolution priority) is re-invented per run and cannot be captured by score-only instruction rephrasing without actually stating the rules, and worked-turn demos are load-bearing.
Against
  • No dedicated round-1 skeptic pass examined c20 — round-1 recommendations.md analyzes only the five finalists (c21, c03, c14, c11, c12), so this lens is inferred from the merged summary, not an adversarial verdict.

Measurement skeptic

Supports
  • Ground truth is the game rules as code — the reference resolver's own transition, independent of any generation-time answer key and, for a small trick, hand-verifiable in under a minute.
Against
  • With the gg-bench corpus the oracle is LLM-generated code, so step() may not faithfully implement the natural-language rulebook; "ground truth" means "whatever the generated rules implement," and no test asserts rule-versus-description agreement. Never given a formal round-1 measurement-skeptic pass either.

Implementation lens (round 2)

Supports
  • If reuse holds, the diff is pure config-plus-scoring glue (roughly 120-200 lines outside the repo) over a frozen corpus whose per-turn oracle needs no LLM call to produce ground truth; the pool is large and pre-filtered.
Against
  • The reuse score was an unverified provisional accepted as read, never run-verified; seeding is inconsistent (about 142 of 1000 envs use module-global np.random that ignores the seed), the corpus ships zero tests and no CI, and it is a game-generator corpus, not a rule-engine that can be pinned — reflected in a bottom I3 and low I2/I5/I6.

Research currency

Supports
  • Invented-game evaluation is live in 2025 — TextArena, Game Reasoning Arena, ZeroSumEval, GVGAI-LLM, and gg-bench are all 2025 artifacts, so a quick test built here sits in an active area.
Against
  • The field is crowded and fast-moving, and the specific contribution is a careful single-token reframing of an existing invented-game genre rather than a new problem class; the authoring cost may outrun the marginal novelty.

How would we build it, and what could go wrong?

Vendor the frozen corpus, not the pipeline

Blobless-clone gg-bench and sparse-checkout only gg_bench/data/{envs,descriptions,actions,splits} (the RL models/ tree of ~2128 files is off-path). Add gymnasium + numpy only; deliberately skip stable-baselines3 / sbx / torch. Write a small loader.py that importlib-loads env_{id}.py by path, reusing the exact instantiate → valid_moves() → step() → render() pattern from the repo's scripts/filter/execution.py.

Own the RNG and build the instance generator

In instance_generator.py, global-seed (np.random.seed(s); random.seed(s)) before env.reset(seed=s) to defeat the unthreaded-seed envs, replay a seeded prefix of moves via valid_moves(), capture render() as board text plus sorted legal moves, and emit an instance of the description, actions, board, legal moves, and the c20 question. Restrict the pool to filtering.json valid envs intersected with the deterministic / self-np_random envs for reproducibility.

Wire the oracle, strata, and validation gates

oracle.py calls env.step(candidate) on a fresh replayed copy for the all-or-nothing exact-match target. Map easy/medium/hard splits to latent-rule strata. Then the design-defining gate: for each rule set, run a reference near-perfect prompt and confirm the rules are unambiguous (near-ceiling) before freezing, add an illegal-move stratum requiring a reject token, and audit for description-versus-code mismatch so the code oracle is trusted knowingly.

Risks

  • Unthreaded seeding in roughly 142 of 1000 envs (module-global np.random, e.g. env_1.py:30-31) plus 14 more on unseeded stdlib random means reproducibility depends on the harness's own global-seeding or restricting the pool — must be handled, not assumed.
  • The corpus is LLM-generated, so an env's step() may not match its descriptions/{id}.txt; treating the code as authoritative accepts a minority of rule/description mismatches, and there are zero tests or CI asserting any env is correct.
  • The unambiguous-rules validation gate is heavy and per-run — the single most expensive calibration burden in the pool — and a missed ambiguity silently breaks exact-match rather than failing loudly.
  • The reuse strength was accepted as an unverified provisional, so the codebase advantage is unconfirmed by a fresh run and could degrade on contact.
