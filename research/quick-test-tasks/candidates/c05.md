# c05 · Deterministic Execution & State Tracking (FSM / protocol / pointer)

**Status:** GATED (round-2 rescore) — unranked, refuted without concrete rescue

You know this as: **state-machine simulation** — run a small deterministic FSM, message protocol, tag-system, or pointer chain and report one derived final token.

## Verdict

**Gated: removed from contention regardless of a decent implementation score.** c05 topped the mechanical round-1 sort on rubric total, and its implementation story is genuinely middling-to-workable — a run-verifiable permutation-composition generator plus a running-sum generator both exist as small, permissive, single-file libraries. But standing here is decided by a gate, not a score: both round-1 skeptic lenses refuted the design at major severity and neither refutation had a concrete rescue that kept the task the same task. The named latent rules — AUTH-before-DATA ordering, undefined-transition fallback, monotone sequence numbers — are textbook protocol and interpreter conventions that a competent prompt engineer states in one shot, which collapses the incremental gap the rubric depends on. The difficulty knob makes it worse: the candidate's own cited literature shows a trace-length dilemma where long traces put the ceiling out of reach of a single no-scratchpad call, while short traces plus chain-of-thought leave only a two-or-three-rung ladder that a single demo bootstrap flattens. The proposed fixes exist, but they converge on arbitrary counter-intuitive conventions — which is to say they redesign c05 into c21 or c03. The gate removes it from the recommended set even though its implementation total would otherwise place it mid-pack.

## What it proposes

### The task: run one small deterministic machine, report one derived token

A seeded generator emits one small deterministic machine and asks a **single bounded question**. The machine is one of a family: an **FSM** given as a transition table plus an event trace (answer = final state label); a **message protocol** with exactly one injected rule violation (answer = index of the first violating message); a **device-command or tag-system** sequence (answer = final device-state letter); or a **key-to-value store with pointer chains** (answer = the fully dereferenced value or a final running sum). Output is a single token, scored by exact match against the reference machine. The **latent rules** decompose cleanly — per-transition entries, undefined-transition fallback, ordering constraints such as AUTH-before-DATA, monotone sequence fields, cycle-to-sentinel, undefined-to-default, a max-depth cap, append-versus-delete ordering — and each injected violation maps a failure back to a **known rule**, which is what made this the most diagnostic cluster in the round-1 pool. Randomized tables and traces make it contamination-proof by construction.

### Example instance (illustrative)

*Seed note: constructed from the merged summary; a four-state FSM with one undefined-transition fallback rule and a self-loop, queried for the final state after a short event trace.*

```
Transition table (state, event -> next state):
  S0, a -> S1     S1, a -> S2     S2, a -> S2 (self-loop)
  S0, b -> S0     S1, b -> S0     S2, b -> S3
  (any state, undefined event) -> ERR   [undefined-transition fallback]

Start: S0
Event trace: a a a b

Query: final state label?
Gold: S3
Naive-prompt miss: a model that ignores the S2 self-loop on 'a' (treating the
third 'a' as advancing S2->S3 directly, then 'b' as ERR) reports ERR instead
of walking S0->S1->S2->S2->S3. The wrong-rule signature is legible.
```

### Literature parents

- [Computational Reasoning of Large Language Models (TMBench)](https://arxiv.org/abs/2504.20771) (Wu et al., arXiv 2025) — the core Turing-machine / m-tag step-simulation source; exact-match scoring, documented floor and ceiling, difficulty tuned by step count. (Note: the round-1 corpus cited this as "Turing Machine Evaluation for Large Language Model"; the actual paper title is "Computational Reasoning of Large Language Models" — the benchmark inside is TMBench.)
- [The Illusion of Diminishing Returns: Measuring Long Horizon Execution in LLMs](https://arxiv.org/abs/2509.09677) (Sinha et al., ICLR 2026) — the retrieve-and-sum execution source; isolates execution from planning, introduces H0.5 and the self-conditioning effect, and is the long-trace-ceiling evidence cited against c05's own difficulty knob.
- [(How) Do Language Models Track State?](https://arxiv.org/abs/2503.02854) (Li et al., arXiv 2025) — FSM-family mechanistic support via permutation-composition state tracking; cited in the recommendations for the no-CoT ceiling dilemma.
- [Finite State Automata Inside Transformers with Chain-of-Thought](https://arxiv.org/abs/2502.20129) (Zhang et al., arXiv 2025) — mechanistic study showing transformers implement finite-state automata for state tracking and that CoT helps; grounds the scheduler/protocol-validator FSM family.
- [Exploring State Tracking Capabilities of LLMs](https://arxiv.org/abs/2511.10457) (Rezaee et al., arXiv 2025) — empirical LLM state-tracking evaluation documenting degradation over steps and the benefit of CoT.

## Perspectives

### Round-1 rubric fit
- **Supports:** Topped the mechanical sort on rubric total (28/2) — bounded single-token output, exact deterministic scoring, and the most diagnostic per-rule violation injection in the pool, since each injected violation names the rule it breaks.
- **Against:** The 28-tier scorecard did not price in the skeptic findings; the raw total overstated the task because it did not test whether the gap survives a competent prompt.

### Optimization-path skeptic
- **Supports:** In principle each transition entry and ordering constraint is an independent, discoverable latent rule, so an optimizer that surfaces one more rule should lift its stratum.
- **Against:** Refuted at major severity — the named rules (AUTH-before-DATA, undefined-transition fallback, monotone sequence numbers) are textbook protocol/interpreter conventions a competent prompt states one-shot, so the incremental gap collapses and the only fix redesigns the task into c21/c03.

### Measurement skeptic
- **Supports:** Ground truth is read by running the reference machine, and the injected-violation design makes failures maximally diagnostic; the oracle is independent and hand-checkable.
- **Against:** Refuted at major severity via the trace-length dilemma — the candidate's own citations (arXiv:2503.02854, arXiv:2509.09677) document that long traces put the ceiling out of a single no-scratchpad call while short traces plus CoT leave only a two-or-three-rung ladder one demo bootstrap flattens, so no trace length holds both a reachable ceiling and a non-trivial floor.

### Implementation lens (round 2)
- **Supports:** Two small, permissive, run-verifiable generators exist — belindal/state-tracking (permutation composition, MIT) and long-horizon-execution/measuring-execution (running sum) — each a single-file library reusable as config plus scoring glue, giving a workable I1=2.
- **Against:** Eligibility is moot: the gate binds regardless of implementation, so I4=0 (unreachable/dilemma-bound ceiling calibration) and I6=0 (a structural-redesign repair flag, not a bounded one) sink the total to 8/16 and the task cannot be recommended.

### Research currency
- **Supports:** State tracking and long-horizon execution are a hot 2025-2026 cluster — TMBench, the ICLR 2026 long-horizon-execution paper, and the FSM-in-transformers mechanistic line all appeared in this window, so a test built here could grow into its own direction.
- **Against:** That same currency supplies the refutation: the long-horizon and no-CoT ceiling results are precisely the evidence that the difficulty knob cannot be tuned to hold a reachable ceiling and a real floor at once.

## Implementation: steps & risks

1. **Lift a seeded generator core.** Reuse belindal/state-tracking's `permutation_task.py` (~180 LOC, MIT) as a library, or lift long-horizon-execution's `dict_sum_util.py` generator core (~130 LOC); in both cases inject seeding on the active path before use, since neither plumbs a seed by default.
2. **Write the exact-match scoring glue outside the repo.** ~60-90 LOC: roll out K actions, capture the final state (or running sum / parity) as the 0/1 oracle recomputed independently, render the prompt, single LLM call, exact string match. No oracle edits.
3. **Attempt the gate-clearing redesign (the load-bearing, unresolved step).** Replace textbook protocol conventions with arbitrary counter-intuitive nonce conventions and pick a trace length that holds a reachable ceiling and a non-trivial floor simultaneously — the step round 1 judged converges on c21/c03, i.e. a different task.

### Risks
- **The gate has no bounded fix (I6=0).** Making the conventions non-one-shot means inventing arbitrary rules, and escaping the trace-length dilemma means changing the difficulty mechanism; either way c05 stops being c05, so the "fix" is a redesign into an already-adopted candidate.
- **Trace-length dilemma is a hard constraint, not a tuning nuisance (I4=0).** No single trace length simultaneously keeps the ceiling reachable in one no-scratchpad call and the floor below it; the candidate's own citations establish this, so ceiling calibration is not merely unfinished but structurally blocked.
- **Seed plumbing absent in both repos.** belindal/state-tracking has no `random.seed`/`--seed` on the active path; long-horizon-execution reseeds the module-global `random` at a call site and shares one un-reset stream across instances, so per-instance reproducibility requires our own seeding fix.
