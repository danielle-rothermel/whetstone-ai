Instruction Following & Constrained Generation as Quick-Test Task Candidates

whetstone-ai · 2026-07-21
Quick-test rubric Recommendations Landscape Instruction following

This surveys IFEval-style instruction-following and constrained-generation benchmarks for use or adaptation as the cheap quick-test task that validates prompt optimizers (COPRO, MIPROv2, GEPA, agentic) against the quick-test rubric. Breadth over depth: 12 candidates, not ranked.

The domain is a strong structural match, with one fixed adaptation tax. Verifiable instructions are checked by deterministic Python functions (no judge LLM, no sandbox), and each instruction is an independent latent rule that adds score when the prompt captures it (criterion 10). The recommended shape is a synthetic multi-atom constrained micro-generation: base task "produce a short answer," plus 3-4 stacked verifiable atoms, scored as fraction-of-atoms-satisfied. Every candidate needs the same three fixes: bound outputs to short strings (most literature scores paragraphs, violating criterion 3), restrict to the code-verifiable subset (many benchmarks lean on an LLM judge, disqualified by criterion 2), and regenerate synthetically to dodge contamination (criterion 8).

Why this domain fits the rubric, and where it strains

Fit

The rubric wants: single LLM call + deterministic programmatic scoring (no sandbox), output bounded to multiple-choice-or-a-few-words, default prompts far below a known ceiling, an incremental gap that closes as prompts capture independent latent rules, a synthetic/generatable pool with exact ground truth, contamination resistance, diagnostic failures, few-shot demos that help, and effect sizes resolvable on 10-20 tasks at temp 0.

  • Verifiable instructions are checked by deterministic Python functions — no judge LLM, no sandbox (criterion 2).
  • Each instruction is an independent latent rule that adds score as the prompt captures it (criterion 10).

Strain (the recurring adaptation moves)

  • Output length. Most of this literature scores paragraphs/essays, violating criterion 3. Shrink the target to a short constrained string / MCQ-like token.
  • Atom selection. Keep only constraints checkable on short outputs: keyword presence, casing, ending token, exact word count, forbidden letters.
  • Contamination. Regenerate instances synthetically to dodge criterion 8.
  • LLM judges. Several benchmarks lean on an LLM judge for "soft" constraints; those variants fail criterion 2 and must be restricted to the programmatically-verifiable subset.

Off-the-shelf generators: synthetic, exact, compositional

1 · IFEval — verifiable instructions, deterministic checker

What it is: ~541 prompts, each with 1-3 verifiable instructions (25 instruction types across 9 categories: keyword frequency, length in words/sentences/paragraphs, exact casing, JSON/format, start/end tokens, bullet counts, placeholders). Each instruction checked by a deterministic program — no judge. The canonical reference for the whole domain.

Citation: Zhou et al., "Instruction-Following Evaluation for Large Language Models," arXiv:2311.07911 (Nov 2023). arxiv.org/abs/2311.07911 · leaderboard llm-stats.com/benchmarks/ifeval

Adaptation: Reuse the checker library (the instructions_registry), not the fixed 541 prompts (contaminated — criterion 8). Generate synthetic instances by sampling a base micro-task ("describe X in a few words") plus N constraint atoms drawn from the verifiable set; keep only atoms verifiable on short outputs (forbidden word, must-include keyword, all-caps, end-with token, exact word count) to satisfy criterion 3. Ground truth = checker verdict per atom; score = fraction of atoms satisfied → naturally incremental (criterion 4/10).

Rubric fit / weaknesses: Excellent on criteria 2, 7, 8, 10, 11, 13 (unparseable/empty outputs land as verifiable failures). Strong models already near-ceiling on single constraints, so use a small model + 3-4 stacked constraints to push the floor down and create headroom. Few-shot demos plausibly help (criterion 12) since format compliance is demonstrable. Weakness: original prompts want long free-form text; you must engineer the short-output variant yourself. "Loose" vs "strict" accuracy modes exist — pick strict for determinism.

2 · COLLIE — grammar-based synthetic constraint generator

What it is: A grammar/framework for programmatically constructing constrained-generation tasks with an automatic extractor that mines instances from a corpus (Gutenberg, Wikipedia). Constraints compose from Levels (char/word/sentence/passage) × Transformations (Count, Position, ForEach) × Relations (==, in, comparisons) × Logic (And/Or/All) × Reductions. Ships COLLIE-v1 (1,132 instances, 13 constraint structures). Scoring is constraint.check(text, target) → boolean, fully deterministic.

Citation: Yao et al., "COLLIE: Systematic Construction of Constrained Text Generation Tasks," ICLR 2024. arXiv:2307.08689 arxiv.org/abs/2307.08689 · code github.com/princeton-nlp/Collie · site collie-benchmark.github.io

Adaptation: Arguably the best-fit generator in the domain — designed to be synthetic, parameterized, exact, and compositional. Restrict the constraint grammar to short-output-checkable structures (word count == k, must contain target words, sentence position rules on a 1-2 sentence output) to hit criterion 3. Each composed atom is an independent latent rule (criterion 10); the And/Or logic gives graded difficulty. Regenerate with your own seed/targets for contamination resistance (criterion 8, built-in).

Rubric fit / weaknesses: Hits criteria 2, 7, 8, 9, 10, 11 squarely — the deterministic checker + programmatic generation is exactly what criteria 7-9 ask for. Weakness: some COLLIE constraints require reasoning/counting over longer passages (planning-hard, long outputs) — curate the sub-grammar down to the short/cheap ones. Its natural-language rendering of constraints is fixed; for prompt-optimization you want the constraint held constant while the system prompt varies, so treat the rendered instruction as part of the task input, not the mutable prompt.

3 · VFF — Verifiable Format Following, fully Python-checkable

What it is: Dataset of 64 unique meta-constraints for output format control (JSON structure, field presence, casing, delimiters, ordering), where every sample is validated by a Python function — explicitly built to avoid GPT-4-as-judge bias. Progressive single→multi-constraint structure. Targets the finding that ~7B models struggle badly with fine-grained format compliance (built-in low floor).

Citation: Wang, Zhaoyang et al., "Verifiable Format Control for Large Language Model Generations," Findings of NAACL 2025. arXiv:2502.04498 arxiv.org/abs/2502.04498 · data huggingface.co/datasets/jinqij/VFF

Adaptation: Bound outputs to small JSON / delimited strings (a few fields) so criterion 3 holds. Use the 64 meta-constraints as the latent-rule library; stack 2-4 per instance for incremental scoring. Regenerate field names/values synthetically for contamination resistance. Ground truth = per-constraint Python verdict.

Rubric fit / weaknesses: Very strong on criteria 1, 2, 3 (format outputs are naturally short), 7, 8, 10, 11, 13 (malformed JSON is a clean, diagnostic failure — great for the parse-error machinery in criterion 13). The paper documents small-model failure directly, supporting criterion 4's low floor. Weakness: format constraints can be one-shot guessable by a competent prompt engineer ("output valid JSON with fields a,b,c") — mitigate by adding non-obvious interacting constraints (ordering + casing + a forbidden character) so the gap closes in steps, not one jump.

4 · IFBench + IF-RLVR — out-of-distribution verifiable constraints

What it is: 58 new, diverse, challenging verifiable out-of-domain constraints (hard: length/format/keyword; plus content/style), released with 29 hand-annotated training constraints + verification functions and RLVR training prompts. Key finding: models overfit the small IFEval constraint set and fail to generalize — i.e., there is real, non-guessable headroom on unseen constraints.

Citation: Pyatkin et al., "Generalizing Verifiable Instruction Following," 2025. arXiv:2507.02833 arxiv.org/abs/2507.02833 · wiki ai.miraheze.org/wiki/IFBench

Adaptation: Harvest the verification functions as your latent-rule checker set; the "OOD" constraints are precisely the ones not one-shot guessable (directly serves criterion 4). Generate synthetic base tasks and stack these constraints; bound output length by choosing the short-checkable subset. Contamination-resistant by design (constraints are novel).

Rubric fit / weaknesses: Best-in-class for criterion 4 (documented incremental, non-guessable gap) and criterion 8. Deterministic verifiers cover the hard constraints (criterion 2). Weakness: the "soft" content/style constraints rely on model/LLM judging — exclude those; keep only the code-verifiable hard constraints. Some constraints assume essay-length outputs; filter to short-output-compatible ones.

Design mechanics to borrow (adopt method, not scorer)

5 · FollowBench — multi-level incremental constraint stacking

What it is: Multi-level benchmark that incrementally adds constraints (1→5) to a base instruction across 5 categories (Content, Situation, Style, Format, Example), letting you pinpoint the level at which a model fails. This "level" mechanic is a near-perfect analog for the rubric's incremental-gap requirement.

Citation: Jiang et al., "FollowBench: A Multi-level Fine-grained Constraints Following Benchmark," ACL 2024. arXiv:2310.20410 arxiv.org/abs/2310.20410 · code github.com/YJiangcm/FollowBench · data huggingface.co/datasets/YuxinJiang/FollowBench

Adaptation: Borrow the multi-level construction recipe (start from one constraint, add one per level) but instantiate only with programmatically-checkable atoms and short outputs. The level index gives a built-in difficulty curve and a natural per-rule score (criterion 10). Regenerate synthetically (criterion 8).

Rubric fit / weaknesses: The incremental-level design maps directly onto criteria 4 and 10 — arguably its biggest selling point. Weakness: FollowBench's own evaluation uses strong LLMs as judges for open-ended constraints (fails criterion 2). You must reimplement scoring as deterministic checkers over a restricted constraint set; you're adopting the methodology, not the scorer.

6 · InFoBench / DRFR — decomposed requirement-following ratio

What it is: 500 instructions decomposed into 2,250 atomic yes/no requirements; metric = Decomposed Requirements Following Ratio (fraction of atomic requirements satisfied). Categories: content, linguistic, style, format, number. The decomposition-into-atoms idea is the scoring model the rubric wants (partial credit per latent rule).

Citation: Qin et al., "InFoBench: Evaluating Instruction Following Ability in Large Language Models," Findings of ACL 2024. arXiv:2401.03601 arxiv.org/abs/2401.03601 · aclanthology.org/2024.findings-acl.772

Adaptation: Adopt the DRFR decomposition as your per-rule scoring scheme, but replace GPT-4-annotated requirement checking with deterministic Python checks (requires restricting to programmatically-verifiable atoms — number/format/keyword, not "is it stylistically elegant"). Generate synthetic instances with known atom labels.

Rubric fit / weaknesses: DRFR = graded per-rule score is exactly criterion 10's partial-credit mechanism, and "expected/got" per atom is diagnostic (criterion 11). Weakness: as published, DRFR is scored by GPT-4 as annotator (fails criterion 2); the many linguistic/style atoms are not deterministically checkable and must be dropped.

7 · CFBench — comprehensive constraint taxonomy

What it is: 1,000 samples over 200+ scenarios / 50+ NLP tasks, with a systematic constraint taxonomy: 10 primary categories, 25+ subcategories, plus requirement-prioritization (primary vs secondary constraints). Useful mainly as a taxonomy source for enumerating independent latent rules.

Citation: Zhang, Tao et al., "CFBench: A Comprehensive Constraints-Following Benchmark for LLMs," ACL 2025. arXiv:2408.01122 arxiv.org/abs/2408.01122 · aclanthology.org/2025.acl-long.1581.pdf

Adaptation: Mine the taxonomy to define a balanced set of independent latent rules for your synthetic generator; the requirement-prioritization idea (primary vs secondary) is a natural authority/priority-ledger variant (Theme H tie-in). Regenerate instances; use only rule-checkable subcategories with deterministic scoring.

Rubric fit / weaknesses: Great as a rule-taxonomy blueprint (criteria 7, 10) and its priority mechanic supports the instruction-hierarchy angle. Weakness: evaluation uses multi-dimensional + LLM-assisted assessment (partial fail on criterion 2); scenarios are realistic/long, so you keep the taxonomy, not the data or scorer.

8 · ComplexBench — And / Chain / Selection / Nested composition

What it is: Complex instructions built by four composition operations — And, Chain, Selection, Nested — over 4 constraint types / 19 dimensions. Documents that performance degrades sharply as Chain/Selection depth grows. The composition operators are a principled way to make difficulty decompose and to tune the incremental curve.

Citation: Wen et al., "Benchmarking Complex Instruction-Following with Multiple Constraints Composition," NeurIPS 2024 D&B. arXiv:2407.03978 arxiv.org/abs/2407.03978 · openreview.net/forum?id=U2aVNDrZGx

Adaptation: Use the composition algebra (esp. Selection = conditional constraints: "if input has property P, obey rule A else rule B") to create latent rules a prompt must discover, not just copy — strongly serves criterion 4 (non-one-shot-guessable) and criterion 10 (independent rules). Instantiate with deterministic atoms + short outputs; regenerate synthetically.

Rubric fit / weaknesses: The Selection/Chain operators are the most compelling mechanism here for a graded, discovery-driven gap. Weakness: ComplexBench's own scoring augments rules with LLM evaluators and uses a dependency-structure aggregation (partial fail on criterion 2); adopt the composition structure with a pure-rule scorer only.

Authority / priority variant (Theme H, hot 2025-2026)

9 · IHEval — instruction hierarchy: system vs user vs tool

What it is: Evaluates whether models follow the instruction hierarchy — resolving conflicts across system prompts, user instructions, conversation history, and tool outputs (aligned vs conflicting settings). Directly supports the authority/priority-ledger variant of the seed domain.

Citation: Zhang et al., "IHEval: Evaluating Language Models on Following the Instruction Hierarchy," 2025. arXiv:2502.08745 arxiv.org/abs/2502.08745 · Framework origin: Wallace et al., "The Instruction Hierarchy," 2024, openreview.net/forum?id=vf5M8YaGPY · Related: "Control Illusion" arXiv:2502.15851.

Adaptation: Build a synthetic task where a system rule and an injected user rule conflict, and the correct short output is dictated by the higher-priority rule. Ground truth is deterministic (which rule wins → which token). The "latent rules" become the priority policy itself: capturing "system overrides user" adds score. Bound output to a single token / few words.

Rubric fit / weaknesses: Very clean deterministic scoring (criterion 2), naturally short outputs, strong 2025-2026 research interest (tie-breaker), and diagnostic failures ("model obeyed the wrong authority"). Weakness: the default-prompt-scores-low criterion (4) is only satisfied if base models genuinely mis-prioritize — literature ("Control Illusion") says they do, which is favorable, but you must verify a meaningful, incrementally closable gap rather than a binary the model either gets or doesn't. Fewer independent rules than the pure-constraint tasks (may need combining with format atoms for enough latent dimensions).

Low-floor and single-axis sources

10 · LIFEBench — length instruction following

What it is: Benchmark isolating length constraints (exact/min/max word or character budgets). Documents that even strong models hit only ~0.53 accuracy on length, and sub-4B models collapse to 3-21% compliance at tight budgets — a built-in low floor with a known ceiling (exact target length).

Citation: Wu et al., "LIFEBench: Evaluating Length Instruction Following in Large Language Models," 2025. arXiv:2505.16234 arxiv.org/abs/2505.16234 · Related: Yuan et al., "Following Length Constraints in Instructions," arXiv:2406.17744.

Adaptation: Length is trivially deterministic to check and outputs are short by construction — pair a small "answer in exactly N words" budget with a keyword/format atom so it's not a single guessable rule. Regenerate targets synthetically (contamination-proof — it's just a number).

Rubric fit / weaknesses: Excellent on criteria 2, 3, 8, 9 (ceiling = exact length, floor = documented small-model failure), and the low floor is empirically established, de-risking criterion 4. Weakness: length alone is a single latent rule — insufficient for criterion 10's "several independent rules." Use it as one axis in a multi-atom generator, not the whole task.

11 · RuleArena — rule-guided reasoning (latent-rule application)

What it is: LLMs given task instruction + reference rules + a user instance must apply the rules to compute an answer (airline baggage fees, NBA transactions, tax). Answers are numeric/deterministic. The "apply a set of explicit rules to an input to get an exact answer" shape decomposes cleanly into independent rules.

Citation: Zhou et al., "RuleArena: A Benchmark for Rule-Guided Reasoning with LLMs in Real-World Scenarios," ACL 2025. arXiv:2412.08972 arxiv.org/abs/2412.08972 · code github.com/skyriver-2000/RuleArena · Related 2026: DeonticBench arXiv:2604.04443.

Adaptation: Replace the real-world (contaminated) rule sets with a synthetic rule engine: generate K parameterized rules ("+3 if field A>5, forbidden if B=='x'"), sample an input, compute the exact answer programmatically. The correct short answer (a number or a label) is exact-match scored. Each rule the prompt surfaces adds accuracy → incremental (criteria 4, 10). Prompt optimization here means eliciting correct multi-rule application.

Rubric fit / weaknesses: Strongest on criterion 10 (rules are the task), deterministic exact-match scoring (criterion 2), short numeric/label output (criterion 3), fully synthetic & contamination-proof if you build the rule engine (criteria 7, 8). Diagnostic failures ("applied rule 2 but missed rule 4"). Weakness: closer to rule-guided reasoning than pure "constrained generation" — drifts toward multi-step arithmetic, which can add unwanted variance; keep rule counts small and arithmetic trivial so temp-0 determinism holds (criterion 5). This is a synthetic-family candidate more than an off-the-shelf dataset.

Verifier components to mine

12 · RewardBench 2 "Precise Instruction Following" / IF-RewardBench

What it is: RewardBench 2 adds a Precise Instruction Following category scored by verifier functions drawn from the IFBench taxonomy (e.g., "answer without the letter u"). IF-RewardBench (ACL 2026) benchmarks judge models for IF evaluation. Primarily a source of clean, atomic, programmatically-verifiable single constraints and negatives.

Citation: Malik et al., "RewardBench 2: Advancing Reward Model Evaluation," 2025. arXiv:2506.01937 arxiv.org/abs/2506.01937 · IF-RewardBench: arXiv:2603.04738, github.com/thu-coai/IF-RewardBench

Adaptation: Lift the verifier-function constraints (forbidden-letter, exact-count, casing) as atoms for the synthetic generator; these are short-output-checkable and non-obvious ("write without the letter e") → good for a non-guessable incremental gap. Ignore the reward-model framing; you only want the constraint checkers and negative examples.

Rubric fit / weaknesses: Provides ready deterministic checkers (criterion 2) that produce short outputs (criterion 3) and clean negatives for the failure-path machinery (criterion 13). The forbidden-letter style constraints are hard for small models but capturable by better prompts (good for criteria 4, 12). Weakness: it's a reward-model benchmark, not a generation task — you're mining components, not adopting it wholesale; single-constraint atoms need to be composed to satisfy criterion 10.

Cross-cutting synthesis for the designer

What to adopt

  • Best off-the-shelf generators (synthetic, exact, compositional): COLLIE (#2), VFF (#3), and the IFEval/IFBench checker libraries (#1, #4). These already give deterministic Python scoring and can be regenerated to dodge contamination. Recommended quick-test task: a synthetic multi-atom constrained micro-generation — base task "produce a short answer/description," plus 3-4 stacked verifiable atoms drawn from these libraries, scored as fraction-of-atoms-satisfied.
  • Best design mechanics to borrow: FollowBench's incremental level-stacking (#5) and ComplexBench's And/Chain/Selection operators (#8) give the graded, non-one-shot-guessable gap criterion 4 demands (Selection = conditional rules the prompt must discover). InFoBench's DRFR (#6) is the per-rule partial-credit scoring model.
  • Best for the authority/priority (Theme H) variant: IHEval (#9) + CFBench's requirement-prioritization (#7) — deterministic, short-output, hot 2025-2026 research area (tie-breaker).
  • Best low-floor guarantees: LIFEBench (#10) and VFF (#3) empirically document small-model failure, de-risking criterion 4; RuleArena-style synthetic rule engines (#11) give the cleanest independent-latent-rules decomposition.

Universal caveats

  • Most published scorers use LLM judges for soft constraints — restrict to the code-verifiable subset to honor criterion 2.
  • Most target long outputs — you must bound to short outputs (criterion 3), which also keeps temp-0 determinism tight (criterion 5).
  • Published prompts are contaminated — regenerate (criterion 8).
  • Verify empirically that stacked constraints give an incremental, not binary, gap on a small model before committing.
