Rebuild targets: the staging API and dr_code.metrics

dr-platform · build on the staging contract

The platform layer targets PR #47's surface (branch rebuild/09-kernel-deletion, tip of the unmerged stack #39–#47): PipelineDefinition/StageDefinition registered via PipelineRegistry, submit() + WorkInput → SubmissionReceipt, wrap_pipeline_workflows and register_scheduled_dispatcher, staging inspection/operations, and schema via upgrade_platform_schema. dr-platform no longer provides a failure taxonomy or throttle ledger — whetstone owns both, and the failure taxonomy here is already positioned for that role. The platform database starts fresh; the staging schema has no bridge from earlier schemas. Pin to the rebuild-stack head until it merges to main, then re-pin.

dr-code · build on dr_code.metrics

Metrics contracts live in the dr_code.metrics package on current main. Sampling, scoring profiles, SubmissionOutcome, and dataset parsing are available unchanged for the scoring layer.

Recommended sequencing: design upward from the deps, grow tests with the code

1 · Advance the dr-code and dr-graph pins when their consumption starts

Both are safe to bump at any time; do it as each library enters the new design so the rebuild is written against current APIs from the first line.

2 · Design the new platform integration against the staging API

Pipelines/stages, submission, and wrapped workflows come from dr-platform; failure classification and any throttling are whetstone-side responsibilities built on the existing eval_failures contract. Recreate the platform database against the staging schema.

3 · Grow tests with the new code, then run experiments

The suite currently covers everything that exists. Keep that property as layers are added: each new boundary lands with its direct tests, and experiments run only once the rebuilt pipeline is the intended design end-to-end.

The sequencing section is a recommendation, not applied work.
