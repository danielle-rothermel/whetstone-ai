Whetstone is now a minimal failure-and-LM core; the legacy layers are gone and the rebuild targets are known

whetstone-ai · post-deletion state and dependency alignment · 2026-07-19
The legacy platform/, db/, and records/ layers — ~6,900 lines built on the superseded dr-platform kernel contract — are deleted, along with their tests, the Alembic configuration, the old-design documentation, and the last old-design vocabulary (the encdec validators, the graph-node naming, and the node-op constant). Because nothing imports dr_code, dr_graph, or dr_platform anymore, the dependency drift that previously blocked updates is no longer breaking: the rebuild will target the current APIs — dr-platform's staging contract (PR #47 rebuild stack) and dr_code.metrics — directly, with no migration debt.

What this branch deleted, and why

The deleted layers hardcoded the superseded design at every level: platform/ (4,623 lines) was built on dr-platform's deleted kernel vocabulary (ExecutionTarget, registration, PlatformOperationCut, throttle ledger); db/ (1,432 lines) persisted the v6 schema whose tables have already been dropped; records/ (837 lines) carried the retired dr_code metrics payloads, a ServiceClass property from a deleted dr-platform concept, and kernel-shaped record types. Deleting rather than migrating removes both the dead contract and the anchoring bias the old code shape would exert on the rebuild.

  • Source: src/whetstone/platform/, src/whetstone/db/ (including migrations), src/whetstone/records/, alembic.ini.
  • Tests: the five platform/connection/records suites, plus the records-dependent cases and support fixtures of the recording tests.
  • Trimmed, not deleted: eval_failures/recording.py lost recordable_jsonb and the deleted-payload builder (persistence-design servants) but kept ensure_recordable, recordable_text, and the diagnostics extractor that policy.summarize_exception depends on.
  • Docs and config: the old-design scope audit (repository-scope-audit.html), the README's description of the old setup (rewritten to an honest minimal state), and the now-unused alembic/sqlalchemy direct dependencies. dbos and psycopg remain only because the retained failure-policy tests verify classification of their error types.
  • Old-design vocabulary (follow-up removal after the initial deletion): node_ops.py and its LLM_CALL_OP graph-op constant; validate_encdec_generation and validate_direct_generation — the direct/encdec pair was the old experiment shape (the neutral require_generation_text empty-output guard stays); and the graph-node naming in lm/boundary.py, where llm_request_for_node became llm_request_from_parameters.

No old-design vocabulary survives

The three residue items flagged after the initial deletion were resolved by removal: node_ops.py is gone, the encdec/direct validators are gone, and the LM boundary speaks in caller parameters rather than graph nodes. A vocabulary grep for the deleted layers, "node", and "encdec" across tracked source and tests returns nothing.

Rebuild targets: the staging API and dr_code.metrics, not the retired contracts

dr-platform · build on the staging contract

The rebuild consumes PR #47's surface (branch rebuild/09-kernel-deletion, tip of the unmerged stack #39–#47): PipelineDefinition/StageDefinition registered via PipelineRegistry, submit() + WorkInput → SubmissionReceipt, wrap_pipeline_workflows and register_scheduled_dispatcher, staging inspection/operations, and schema via upgrade_platform_schema only. The old kernel vocabulary (targets, registration, operation cuts, throttle ledger, FailureClass, ServiceClass) no longer exists and must not be reintroduced. Pin to the rebuild-stack head until it merges to main, then re-pin.

dr-code · build on dr_code.metrics

The retired dr_code.humaneval payload family (MetricsPayload, build_metrics_payload, metric_models) is gone from main; new metrics contracts live in the dr_code.metrics package. Sampling, scoring profiles, SubmissionOutcome, and dataset parsing survive unchanged and remain available to the new design.

Recommended sequencing: settle the residue, then design upward from the deps

1 · Advance the dr-code and dr-graph pins when their consumption starts

Both are safe to bump at any time now; do it as each library enters the new design so the rebuild is written against current APIs from the first line.

2 · Design the new platform integration against the staging API

Whetstone-side ownership of the failure taxonomy and any throttling is now a design requirement, not a migration artifact — dr-platform no longer provides either. The platform database starts fresh (old tables already dropped; the staging schema has no bridge).

3 · Grow tests with the new code, then run experiments

The remaining suite covers everything that exists. Keep that property as layers return: each new boundary lands with its direct tests, and experiments run only once the rebuilt pipeline is the intended design end-to-end.
