The dr-* stack covers every stage of the compression eval; the experiment loop and optimizer are the gap

whetstone-ai · foundation review before planning · 2026-07-19
Every stage of the target flow — ground-truth code, encoder/decoder LLM calls, lossless compression, sandboxed test scoring — already has a home in one of the five dr-* libraries, and whetstone's surviving core supplies the failure taxonomy and provider boundary that connect them. What does not exist anywhere yet is the layer this project is: the eval program that wires the stages together, the metric records that become correctness-vs-compression pareto plots, and the optimizer loop that proposes and selects encoder prompts. dspy's COPRO, MIPROv2, and GEPA are reviewed below as reference algorithms only — each has clear weaknesses worth improving in a from-scratch implementation.

Every stage of the fixed flow maps onto an existing entry point

The flow is ground truth → encoder → description → compress → decompress → decoder → reconstructed code → run ground-truth tests, scored as test pass rate vs compressed-description bytes over ground-truth bytes.

Whetstone today: a 576-line neutral core with no legacy design

eval_failures/ — classify, retry, record

  • One exception base carrying underlying + metadata; five classes mapping 1:1 onto dr-providers FailureClass, plus RecordingFailureError and EmptyGenerationError.
  • policy.py: chain-walking classification, should_retry_step, FailureSummary payloads.
  • recording.py: ensure_recordable converts any value to JSONB-safe payloads via dr-serialize, failing loudly as a recording failure.

lm/ — thin boundary over dr-providers

  • PlainPromptAdapter: system + user messages, no hidden framework formatting — the right substrate for controlled prompt experiments.
  • ProviderResult: text, usage, cost, finish reason, diagnostics; empty generations rejected by require_generation_text.
  • translate_provider_failure carries kernel failure records into the whetstone taxonomy 1:1.

All 47 unit tests pass; lint and typecheck are clean. Only dr-providers and dr-serialize are imported so far — dr-graph, dr-code, and dr-platform are declared dependencies awaiting the rebuild.

The five libraries: strengths and the seams they leave to us

dr-serialize — identity never depends on serialization policy

Two deliberate lanes: Serializer.to_jsonable (pluggable handlers, explicit depth/byte limits, lossy fallbacks) for storage, and canonical_json/sha256_json_digest (policy-free, "same input, same bytes, forever") for identity. Compose them for content-addressed artifacts. Golden fixtures pin digest stability.

Sync-only, no I/O. Errors carry .path and persist-safe .diagnostics().

dr-providers — provider differences are data, failures are records

Frozen LlmRequest/LlmResponse models; provider variation lives in ProviderConfig presets, not subclasses. Provider is a Protocol — HttpProvider (httpx, opt-in bounded retries, injectable sleep/rng) and ScriptedProvider (records every request; no network) both conform. ProviderFailure records classify; the caller owns retry policy. Post-hoc conformance warnings catch reasoning-not-observed, token-limit-exceeded, and model substitution.

Gaps we own: no caching or memoization (dr-serialize digests are the natural key), sync-only, and provider_metadata carries the raw response body — strip it before hashing if artifact IDs should be stable and privacy-safe.

dr-graph — whatever is searched over is data; whatever searches is code

Graph specs are valid by construction (acyclic, refs checked, deterministic topological order) and hashable: graph_digest covers the full spec including open parameters, making each pipeline variant a stable treatment identity. The interpreter is pure and strictly sequential; it never dispatches on op — one injected run_node callback does all the work. Failed nodes yield outcomes, dependents become BLOCKED, independent branches continue; completed gives cheap resume.

The README's own example is an encoder→decoder chain binding task.prompt and encoder.description — this library was shaped for this project.

dr-code — the entire non-LLM half of the pipeline, hardened

Beyond the stage coverage above: a metrics engine (extract_metrics) that plans content-addressed sandbox executions, dedups them through an injectable ExecutionCache, and emits MetricRecords carrying full producer/definition identity — operator failures become records, not exceptions. A resumable concurrent corpus runner (SQLite lease scheduling, Parquet output, execution fingerprinting) is a working template for batch evaluation. The synthetic package generates deterministic corruption datasets to stress the reconstruction leg before spending LLM calls.

Caveats: two scoring paths coexist ("parity twins", old humaneval.scoring vs new CodeTest, kept equal by tests, retirement planned) — prefer the metrics path. Sandbox blocks network/host/credentials but a candidate shares the runner's stdout, so adversarial-code score integrity isn't guaranteed. Requires Python ≥3.13 (others ≥3.12).

dr-platform — durable sweeps, permanently payload-agnostic

Campaign → run → work-item identity over a linear staged pipeline on DBOS + Postgres: randomized bounded admission, per-label selector capacity (throttle one model without stopping the sweep), pause/resume, retry, cancellation, sweep of abandoned work. Submission is idempotent by design — the top-up protocol (derive desired keys, check bulk_work_statuses, submit only absent keys as a new run) is built for iterate-and-resubmit optimization loops. Stage bodies are at-least-once and must return immutable output references; results live in the application's own store.

Only dr-serialize is imported (deterministic workflow IDs). It knows nothing of prompts, metrics, or pareto fronts — that read side is ours. Alpha status; CANCELLED work keys are permanently dead within a campaign, so an optimizer that cancels in-flight variants must mint fresh keys.

The three reference optimizers trade eval cost against proposal richness

COPRO — hill-climb the instruction

  • The seed meta-prompt proposes breadth−1 rewrites and evaluates the original last; each later depth proposes breadth rewrites from score-ranked unique history.
  • Greedy coordinate ascent per predictor; fixed iterations, no convergence check.
  • Consumes score only. Defaults: breadth 10, depth 3, temperature 1.4.
  • Weakness: every candidate costs a full trainset eval — no minibatching, no budget cap; no backtracking; deepcopy-per-candidate memory.

MIPROv2 — search a fixed candidate pool

  • Three stages: bootstrap few-shot demo sets from successful traces; propose grounded instructions (dataset summary + program code + demos + prompting tip); then Optuna TPE Bayesian search over instruction×demo combos.
  • Trials score on minibatches (size 35); every 5 trials the best-average combo is promoted to a full eval.
  • Optimizes instructions and demos jointly; search consumes score only.
  • Weakness: the pool is fixed up front — it cannot invent new instructions mid-run in response to failures; hard Optuna dependency; noisy minibatch scores patched by heuristics.

GEPA — evolve with textual feedback

  • Archive of candidates with per-task val scores; parent sampled from the per-task pareto frontier; a reflection LM rewrites one component's instruction from a minibatch of traces + textual feedback.
  • Child accepted only on strict minibatch improvement, then promoted to full eval; optional merge of ancestor candidates; budget in metric calls.
  • Richest signals: score, per-predictor traces, textual feedback, format failures.
  • Weakness: engine lives in a separate gepa package behind an adapter; depends on dspy trace capture and global LM context; predictor-level scores effectively unused.
GEPA's pareto archive is over validation tasks; our objective is a pareto front over correctness and compression. A from-scratch implementation can make the two-objective front the native selection structure rather than a per-task proxy — a real improvement opportunity, not just a port. The other obvious upgrades: budgeted minibatch evaluation everywhere (COPRO's absence), and proposal that can react mid-run (MIPROv2's absence).

A reimplementation needs three primitives, not the dspy stack

An editable program object

A container of named components, each with a mutable instruction string (and optionally demos). In our setting: the encoder prompt (and later the decoder prompt) as fields of a dr-graph spec — candidate identity falls out of graph_digest for free, replacing dspy's deepcopy-heavy program snapshots and id()-keyed bookkeeping.

A metric runner with minibatch support

Score a candidate over a task set, cheaply on minibatches and authoritatively on full sets, returning per-task records (pass/fail + compression ratio), not just aggregates. dr-code's metrics engine plus dr-platform's ledger are most of this.

A proposal LM channel

Meta-prompt calls that turn attempt history (COPRO), grounded context (MIPROv2), or feedback-annotated traces (GEPA) into new instruction text — just dr-providers calls with dedicated prompts, no global LM context switching needed.

The local dspy copy also carries optimizer_guides/ — a 19-page HTML site covering these three plus OPRO, EvoPrompt, text-gradient methods, and others — useful when we widen the algorithm search later.

What must be built: the whetstone layer itself

  • Stage implementations — the run_node bodies wiring encoder call → compression → decoder call → preprocessing → sandbox scoring, with whetstone failure classification at each step.
  • Metric and result records — per (treatment, task) rows carrying correctness, compressed bytes, ground-truth bytes, cost, and provenance, in a store the pareto/plot read side can query. dr-platform holds the completion ledger; the metric values need a home.
  • Treatment definition — where prompts, models, and codec parameters live inside the graph spec so graph_digest is the experiment key.
  • The evaluation harness — sweep a set of treatments over a task sample (locally first, dr-platform at scale) and emit the correctness-vs-compression pareto front.
  • The optimizer loop — proposal, budgeted evaluation, and selection over encoder prompts, informed by the three references but native to the two-objective front.
  • LLM call caching — content-addressed on request digests; nothing in the stack provides it.
One open design tension to settle during planning: the flow treats compression as a pipeline stage (the decoder sees decompressed text), while dr-code models compressed length as a metric operator. Since lossless round-trip means the decoder input equals the encoder output, measuring compression as a metric may suffice — worth deciding explicitly.
